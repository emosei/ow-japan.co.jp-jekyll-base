## 問題の現れ方

SPAやマルチページアプリケーションでWebpackなどのバンドラを使ってコード分割を行うと、予期しないタイミングでグローバルスコープの初期化が遅れることがあります。具体的には、アプリケーション起動時に実行されるべき初期化処理（設定の読み込み、APIクライアントのセットアップ、ログシステムの初期化など）が、メインチャンクの読み込み直後ではなく、その後に動的に読み込まれるチャンクの処理に左右されるケースです。

開発環境では気づかず、本番環境で特定のネットワーク遅延下でのみ顕在化することが多いため、判断が難しい問題です。

## チャンク分割時の実行順序の仕組み

Webpackがコードを複数のチャンクに分割する際、各チャンクは独立したファイルとして出力されます。しかし、グローバルスコープで定義されたモジュール初期化処理は、そのモジュールを含むチャンクが読み込まれるまで実行されません。

例えば、以下のような構成を考えてみます。

```javascript
// utils/config.js（メインチャンクに含まれる想定）
export const AppConfig = {
  apiBaseUrl: null,
  isInitialized: false
};

// 初期化処理（メインチャンクのエントリーポイント）
import { AppConfig } from './utils/config.js';
AppConfig.apiBaseUrl = 'https://api.example.com';
AppConfig.isInitialized = true;

// feature/dashboard.js（動的インポートで別チャンクになる想定）
import { AppConfig } from './utils/config.js';
export function initDashboard() {
  if (!AppConfig.isInitialized) {
    console.warn('Config not initialized yet');
  }
  // ダッシュボード初期化処理
}
```

メインチャンクとダッシュボード機能が別チャンクに分かれた場合、以下の実行順序が起こり得ます。

1. メインチャンクが読み込まれ、`AppConfig.isInitialized = true` が実行される
2. ダッシュボード機能が必要になり、別チャンクが読み込まれる
3. その別チャンク内の`initDashboard()`が実行される

一見すると問題がなさそうですが、実際には**共有モジュールの初期化タイミング**が複雑になります。

## 共有モジュール参照と遅延初期化

バンドラが複数チャンクから参照されるモジュール（例えば`config.js`）を検出すると、それを共有チャンク（vendor chunkやcommon chunkなど）に移動することがあります。この時点で、グローバル初期化の順序が曖昧になります。

具体的な検証環境を構築してみます。

```javascript
// webpack.config.js の簡略版
module.exports = {
  entry: './src/index.js',
  output: {
    filename: '[name].[contenthash].js',
    chunkFilename: '[name].[contenthash].js'
  },
  optimization: {
    splitChunks: {
      chunks: 'all',
      cacheGroups: {
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendors',
          priority: 10
        },
        common: {
          minChunks: 2,
          priority: 5,
          reuseExistingChunk: true
        }
      }
    }
  }
};
```

このような設定で、複数のエントリーポイントやダイナミックインポートがある場合、`minChunks: 2`の条件により、2つ以上のチャンクから参照されるモジュールは自動的に`common`チャンクに抽出されます。

問題は、**その抽出されたモジュール内のグローバル初期化コードが、どのチャンクより先に実行されるかが保証されない**という点です。HTMLで複数のスクリプトタグを記述する順序に依存します。

```html
<script src="vendors.js"></script>
<script src="common.js"></script>
<script src="main.js"></script>
<script src="dashboard.chunk.js"></script>
```

理想的には上記の順序で読み込まれますが、ネットワーク遅延やブラウザのキャッシュ状態によって、`common.js`が`main.js`より遅れて実行されるケースが発生します。

## 検証の進め方

実際に問題を再現するには、以下の手順が有効です。

**1. チャンク分割の実態を確認する**

Webpackのビルド統計を確認します。

```bash
webpack --profile --json > stats.json
```

`stats.json`を[webpack-bundle-analyzer](https://github.com/webpack-contrib/webpack-bundle-analyzer)で可視化すると、どのモジュールがどのチャンクに含まれているかが明確になります。

**2. グローバル初期化の依存関係を明示的に記録する**

初期化順序を追跡するため、簡単なログシステムを挿入します。

```javascript
// debug/initLog.js
export const initLog = [];

export function logInit(moduleName, stage) {
  initLog.push({
    module: moduleName,
    stage,
    timestamp: performance.now()
  });
  console.log(`[INIT] ${moduleName} - ${stage}`);
}

// config.js
import { logInit } from './debug/initLog.js';

logInit('AppConfig', 'start');
export const AppConfig = { /* ... */ };
logInit('AppConfig', 'end');
```

本番環境でも軽量なログを残すことで、実際のユーザー環境での初期化順序を把握できます。

**3. ネットワーク遅延をシミュレートする**

Chrome DevToolsのネットワークスロットリング機能を使い、3G相当の遅延下で挙動を確認します。特に、メインチャンクと共有チャンクの読み込み順序がズレやすくなります。

## 実装上の対策

### 1. 初期化処理を明示的に分離する

グローバル初期化と遅延初期化を明確に分けます。

```javascript
// bootstrap.js（メインチャンクに必ず含める）
export async function initializeApp() {
  // メインスレッドで即座に実行すべき処理
  setupGlobalErrorHandler();
  initializeLogging();
  loadCriticalConfig();
}

// main.js
import { initializeApp } from './bootstrap.js';

await initializeApp();
// ここまで完了してから、他の機能を読み込む
const { initDashboard } = await import('./feature/dashboard.js');
```

この方法では、`initializeApp()`の完了を待ってから次の処理に進むため、順序が保証されます。

### 2. チャンク分割戦略を見直す

`minChunks`の値を調整し、共有チャンクへの過度な抽出を防ぎます。

```javascript
cacheGroups: {
  common: {
    minChunks: 3,  // 2から3に引き上げ、より多くの参照で初めて抽出
    priority: 5,
    reuseExistingChunk: true,
    enforce: true  // 他の条件を無視して適用
  }
}
```

ただし、この方法はバンドルサイズが増加する可能性があるため、実装の複雑さと天秤にかける必要があります。

### 3. 初期化順序を制御するマニフェストを用意する

複雑なアプリケーションでは、初期化順序を明示的に定義するマニフェストを用意します。

```javascript
// init-manifest.js
export const initSequence = [
  { name: 'logging', fn: () => import('./init/logging.js') },
  { name: 'config', fn: () => import('./init/config.js') },
  { name: 'api-client', fn: () => import('./init/api-client.js') }
];

export async function executeInitSequence() {
  for (const step of initSequence) {
    const { default: init } = await step.fn();
    await init();
  }
}
```

## 実務投入時の確認ポイント

- **本番ビルドでの動作確認**：開発環境ではバンドラの動作が簡潔ですが、本番ビルド（minify、最適化適用）で挙動が変わることがあります
- **複数ブラウザでの検証**：ブラウザのJavaScript実行エンジンやメモリ管理により、チャンク読み込みのタイミングが微妙に異なります
- **キャッシュ戦略との整合性**：Service Workerやブラウザキャッシュがある場合、チャンク読み込み順序がさらに複雑になります
- **エラーハンドリング**：初期化失敗時の復旧戦略を事前に用意しておくことが重要です

## どんな案件で注意が必要か

グローバル状態に依存する大規模なSPA、複数のダイナミックインポートを持つアプリケーション、複数のチームが並行開発している場合に特に注意が必要です。一方、初期化処理がシンプルで、グローバル状態への依存が少ないアプリケーションであれば、この問題の影響は限定的です。

チャンク分割は確実にパフォーマンスを改善しますが、その恩恵を受けるには、初期化順序という目に見えない部分への丁寧な設計が不可欠です。
