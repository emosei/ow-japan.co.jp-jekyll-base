## ローカルで「速い」が本番で「遅い」の根本原因

開発環境やテスト環境でクエリが高速に動いているのに、本番環境に上げた途端に遅くなる。こうした経験は多くのエンジニアが持っています。原因を調べると、実行計画（Query Plan）が開発と本番で全く異なっていることがあります。

その背景にあるのが、**テストDBに施されたシード処理が本番スキーマと異なるインデックス構成を持つ**という状況です。DBMSのクエリオプティマイザーは、テーブルのサイズ、インデックスの有無、カラムの統計情報などを見て、最適だと判断した実行計画を選択します。テストDBにそもそもインデックスがなかったり、複合インデックスの順序が異なったりすると、オプティマイザーは別の経路を選んでしまうのです。

その結果、開発では数ミリ秒で返ってくるクエリが、本番では数秒かかる。あるいは逆に、開発では想定外のフルテーブルスキャンが走っていても、本番のインデックスのおかげで目立たない。こうしたズレは、デプロイ後に初めて顕在化することがほとんどです。

## なぜシード処理でインデックス構成が異なるのか

テストDBの初期化やシード処理は、開発効率を重視して設計されることが多いです。以下のような判断が現場では自然に起きます。

**開発初期段階での判断**
- 「とりあえず動く状態を作りたい」という優先度で、本番スキーマの全インデックスを複製しない
- テスト用の小さなデータセットなら、インデックスがなくても十分に速い
- 初期スキーマ定義ファイルと本番スキーマが分かれており、同期が手動になっている
- 新しいインデックスが本番で追加されても、テスト側への反映が遅れる

**マイグレーション管理の不整合**
- スキーママイグレーションツール（Flyway、Liquibase、Rails Migration等）で本番スキーマは更新されているが、テストDB初期化スクリプトは古いまま
- インデックス追加のマイグレーションが、テスト側では実行されていない

**データ量の違いによる判断ミス**
- テストDBは数百件のレコードで十分と考えられているが、本番は数百万件
- 小規模データではインデックスなしでもスキャンが速いため、インデックスの必要性が見えない

こうした状況では、開発環境でのテストはすべてパスするのに、本番で性能問題が露呈します。

## オプティマイザーの判断がどう変わるか

具体的にどのような実行計画の変化が起きるのか、例を示します。

ユーザーの注文履歴を取得するクエリを考えてみましょう。

```sql
SELECT o.id, o.user_id, o.created_at, p.name
FROM orders o
JOIN products p ON o.product_id = p.id
WHERE o.user_id = ? AND o.created_at > ?
ORDER BY o.created_at DESC
LIMIT 20;
```

**テストDB（インデックスなし、レコード数 1,000件）**
- orders テーブル：1,000件
- products テーブル：100件
- インデックス：PRIMARY KEY のみ

オプティマイザーの判断：
- user_id で絞り込んでも 100件程度残る → フルテーブルスキャンで十分
- products テーブルも小さいので、ネストループジョインで問題ない
- 実行時間：5ms

**本番DB（インデックスあり、レコード数 500万件）**
- orders テーブル：500万件
- products テーブル：50万件
- インデックス：`(user_id, created_at)` の複合インデックス

オプティマイザーの判断：
- user_id で複合インデックスを使い、created_at でフィルタ → 数十件に絞り込める
- products テーブルは product_id インデックスでジョイン
- ハッシュジョインまたはソートマージジョイン選択
- 実行時間：2ms

一見すると本番の方が速いのですが、問題は逆パターンでも起きます。テストDBに**過度に多くのインデックス**が存在すると、オプティマイザーが予期しないインデックスを選んでしまい、実は非効率な経路を走ることもあります。

## 現場で取るべき対策

### 1. テストDB初期化スクリプトを本番スキーマと同期させる

最も確実な方法は、テストDB の初期化を本番スキーマの定義から自動生成することです。

```bash
# 本番スキーマをダンプして、テスト用に流用
mysqldump --no-data production_db > test_schema.sql
# または
pg_dump --schema-only production_db > test_schema.sql
```

マイグレーションツールを使っている場合は、テスト環境の初期化時に本番と同じマイグレーションを実行する仕組みを入れます。

### 2. テストDBに本番相当のデータ量を用意する

インデックスの効果を検証するには、本番に近いデータ量が必要です。ただし全量をコピーするのは非現実的なので、以下の選択肢があります。

- **本番データのサンプリング**：ランダムに 1% 程度抽出し、統計情報を再計算
- **シンセティックデータ生成**：faker ライブラリなどで、本番と同じカーディナリティを持つダミーデータを生成
- **段階的テスト**：開発環境では小規模、ステージング環境では本番相当のデータを用意

### 3. 実行計画を定期的に確認する

デプロイ前に、本番と同じスキーマでクエリの実行計画を確認する習慣をつけます。

```sql
-- MySQL
EXPLAIN FORMAT=JSON SELECT ...;

-- PostgreSQL
EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
```

CI/CD パイプラインに組み込み、実行計画が大きく変わったら警告を出す仕組みも有効です。

### 4. インデックス定義をコードで管理する

スキーママイグレーションツールで、インデックスの追加・削除も含めて管理します。これにより、本番と開発のズレが自動的に検出されます。

```ruby
# Rails Migration の例
class AddIndexToOrders < ActiveRecord::Migration[6.0]
  def change
    add_index :orders, [:user_id, :created_at]
  end
end
```

## 中小規模チームが現実的に取れるアクション

リソースに限りがある場合は、優先順位をつけることが重要です。

| 対策 | 効果 | 手間 | 推奨度 |
|------|------|------|--------|
| テストDB初期化スクリプトの本番スキーマ化 | 高 | 中 | ★★★ |
| 本番相当データ量でのテスト | 高 | 高 | ★★★ |
| CI での実行計画チェック | 中 | 中 | ★★ |
| 定期的な性能プロファイリング | 中 | 中 | ★★ |

まずは「テストDB初期化スクリプトを本番スキーマと同期させる」ことから始めるのが現実的です。これだけでも、多くのインデックス関連の問題は防げます。

また、本番環境で性能問題が発生した場合は、まず実行計画を確認することを習慣づけてください。「開発では速かったのに」という報告があったら、それは高い確率でスキーマのズレが原因です。
