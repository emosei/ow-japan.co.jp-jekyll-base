## 現場で起こる追跡不可能な状況

Rubyプロジェクトで本番環境に不具合が出たとき、原因がモンキーパッチにあるかもしれないと疑うのに、どのgemがどのクラスやメソッドを拡張しているか一覧できる手段がないまま、ログを追い続けるという経験をしている人は少なくないでしょう。

数十個のgemが入っている環境では、Stringクラスの挙動が変わったとしても、それが標準ライブラリなのか、あるgemの初期化処理なのか、複数のgemの組み合わせ結果なのか、静的に把握するのは困難です。開発環境では動いていたのに本番で同じ症状が出ないこともあり、gemのロード順序やバージョン差が絡むと、再現性の確保も難しくなります。

この記事では、モンキーパッチの影響を本番環境で追跡・検証するための実務的な手法を、実装観点と運用観点から整理します。

## モンキーパッチが見えなくなる理由

Rubyのモンキーパッチは、既存のクラスやモジュールを実行時に拡張する機能です。便利さの反面、以下の理由で本番追跡が難しくなります。

**1. ロード時点が不確定**

gemの初期化順序はBundlerの依存関係解決に依存し、同じGemfile.lockでも環境やバージョン差で微妙に変わることがあります。開発環境ではgemAが先にロードされ、本番ではgemBが先にロードされる場合、モンキーパッチの上書き順序が逆になり、挙動が異なります。

**2. 静的解析ツールが追跡できない**

RuboCopなどの静的解析ツールは、ソースコード上の明示的な定義を見ます。しかし `String.class_eval` や `Module.prepend` で行われた拡張は、実行時にしか確定しません。

**3. エラースタックトレースに痕跡が残らない**

モンキーパッチされたメソッドが呼ばれても、スタックトレースには元のgemファイルパスが出ません。Rubyのメソッド呼び出しは動的なため、どのgemが最終的に実行されたメソッドを定義しているのか、ログだけからは判定できません。

## 検証の前提と観点

本番環境での追跡を実現するには、以下の観点で検証を進めます。

- **どのgemがどのクラスを拡張しているか、起動時に列挙できるか**
- **複数のgemが同じクラスを拡張した場合、ロード順序を制御できるか**
- **本番環境で拡張内容を確認・ログ出力できるか**
- **開発環境と本番環境でgemバージョンが異なる場合、差分を検出できるか**

## 実装による可視化戦略

### パターン1: 起動時にモンキーパッチを記録する

Rubyアプリケーション起動時に、各クラスのメソッド一覧をスナップショットとして保存し、どのgemがどのメソッドを追加・上書きしたかを記録します。

```ruby
# config/initializers/patch_tracker.rb

class PatchTracker
  def self.snapshot_methods
    tracked_classes = [String, Array, Hash, Integer, Time, ActiveRecord::Base]
    snapshot = {}

    tracked_classes.each do |klass|
      snapshot[klass.name] = {
        instance_methods: klass.instance_methods(false).map(&:to_s),
        singleton_methods: klass.singleton_methods(false).map(&:to_s),
        timestamp: Time.current
      }
    end

    snapshot
  end

  def self.compare_snapshots(before, after)
    diff = {}
    before.each_key do |class_name|
      next unless after[class_name]

      added_instance = after[class_name][:instance_methods] - before[class_name][:instance_methods]
      added_singleton = after[class_name][:singleton_methods] - before[class_name][:singleton_methods]

      diff[class_name] = {
        added_instance_methods: added_instance,
        added_singleton_methods: added_singleton
      } if added_instance.any? || added_singleton.any?
    end

    diff
  end
end

# 起動時に記録
if defined?(Rails)
  Rails.configuration.after_initialize do
    patch_log = PatchTracker.snapshot_methods
    Rails.logger.info "Applied patches: #{patch_log.to_json}"
  end
end
```

このアプローチでは、起動ログから拡張されたメソッドを確認できます。ただし、どのgemが追加したかは別途追跡が必要です。

### パターン2: gem側で拡張を宣言させる

自社gemや管理下のgemであれば、モンキーパッチを明示的に登録するインターフェースを用意します。

```ruby
# lib/patch_registry.rb

class PatchRegistry
  @@patches = []

  def self.register(gem_name, target_class, methods)
    @@patches << {
      gem: gem_name,
      class: target_class.name,
      methods: Array(methods),
      loaded_at: Time.current
    }
  end

  def self.all
    @@patches
  end

  def self.report
    @@patches.group_by { |p| p[:class] }.each do |klass, patches|
      puts "#{klass}:"
      patches.each do |patch|
        puts "  - #{patch[:methods].join(', ')} (from #{patch[:gem]})"
      end
    end
  end
end

# 各gemの初期化ファイルで
module MyGemExtensions
  def self.apply
    String.class_eval do
      def my_custom_method
        # ...
      end
    end
    PatchRegistry.register('my_gem', String, :my_custom_method)
  end
end
```

本番環境で `PatchRegistry.report` を呼ぶことで、拡張内容をいつでも確認できます。

### パターン3: Method Sourceで定義位置を追跡

`method_source` gemを使い、実行時にメソッドがどこで定義されたかを確認します。

```ruby
require 'method_source'

# 本番環境で疑わしいメソッドの定義位置を確認
method_obj = String.instance_method(:downcase)
puts method_obj.source_location
# => ["/path/to/gem/lib/string_ext.rb", 5]
```

この手法は、既存のコードに依存せず後付けで追跡できるメリットがあります。ただし、すべてのgemが `method_source` に対応しているわけではなく、C拡張で実装されたメソッドは位置情報が得られません。

## 本番環境で確認すべきポイント

実務投入する際は、以下を事前に検証します。

**1. gemロード順序の再現性確認**

```bash
# Bundlerのロード順序をダンプ
bundle exec ruby -e "Bundler.require(:default); puts $LOADED_FEATURES.grep(/gems/)"
```

開発環境と本番環境で同じ順序か確認し、差異があれば `Gemfile` で明示的に `require` 順序を制御します。

**2. バージョン差による挙動検証**

Gemfile.lockが同期されていても、本番デプロイ直前に新しいgemがリリースされ、意図せず新バージョンがインストールされるケースがあります。本番環境で `bundle show` を定期実行し、開発環境との差分を監視します。

**3. モンキーパッチの競合テスト**

複数のgemが同じクラスを拡張する場合、テスト環境で意図的にロード順序を変え、結果が変わらないか確認します。

```ruby
# spec/integration/patch_order_spec.rb

RSpec.describe 'Patch loading order' do
  it 'produces consistent behavior regardless of gem load order' do
    # gemAとgemBが両方ともStringを拡張する場合
    result_ab = load_gems_in_order(['gem_a', 'gem_b'])
    result_ba = load_gems_in_order(['gem_b', 'gem_a'])
    
    expect(result_ab).to eq(result_ba)
  end
end
```

## どんな案件で採用しやすいか

この検証・対策は、以下のような環境で特に有効です。

- **gemの数が多い（30個以上）** 
  - 依存関係が複雑になり、モンキーパッチの影響範囲が大きくなる

- **本番環境で不具合が再現しづらい**
  - 環境差が疑われるが、原因特定に時間がかかっている

- **複数チームで開発を進めている**
  - 各チームが独立してgemを追加し、相互影響を把握できていない

- **長期運用されている大型プロジェクト**
  - 初期段階で導入されたgemと新しく追加されたgemの相性問題が顕在化しやすい

一方、小規模なプロジェクトや、gemの数が限定的な場合は、ここまでの対策は過度になる可能性があります。その場合は、モンキーパッチを使わない設計（ラッパークラスやCompositionパターン）を検討する方が現実的です。

## 運用の心構え

モンキーパッチは便利ですが、本番環境での追跡コストは確実に上がります。採用する際は、メリットと追跡コストを天秤にかけ、以下を意識してください。

- **新しいgemを追加する前に、モンキーパッチの有無を確認する習慣**
- **本番デプロイ前に、拡張内容を記録・ログ出力する仕組み**
- **定期的に拡張内容をレビューし、不要な拡張を削除する**

これらを地道に積み重ねることで、本番環境での予期しない挙動を減らせます。
