## キャッシュと永続化ストアの「ズレ」が許容できなくなる時代

システムの性能要件が厳しくなるにつれ、メモリ上のキャッシュと永続化ストア（データベースなど）を分離するのは当たり前になってきました。応答速度を求めるなら、頻繁に読み込むデータをメモリに置くのは理に適っています。

ただ、この分離が進むほど、開発チームの中で「どちらを信頼するか」の判断が揺らぎやすくなります。

現場ではこういう場面が増えています：

- キャッシュに値がある場合、それを即座に返すか、ストアを確認してから返すか
- ストアに書き込んだ直後、キャッシュをどのタイミングで更新するか
- キャッシュとストアが矛盾していることに気づいたとき、どちらを正とするか
- 複数のキャッシュレイヤーがある場合、どの層を優先するか

これらの判断が「チームメンバーごとに異なる基準で行われる」というのが実務の課題です。設計時には明文化されていても、実装時の細部や障害対応時には「とりあえずこっちを優先しよう」という暗黙の判断が積み重なります。

## 「強い一貫性」と「最終的一貫性」の境界線が曖昧になる

多くのチームは、キャッシュ戦略を導入するときに「強い一貫性が必要な部分」と「最終的一貫性で良い部分」を分けようとします。理論としては正しいのですが、実装段階でこの線引きが曖昧になりやすいのです。

たとえば、ユーザーの権限情報をキャッシュするとします。

- 権限が変わったとき、キャッシュをすぐに無効化するべきか？
- それとも、次のアクセス時に確認すればいいのか？
- 有効期限を設けるなら、その時間は業務要件で決まるのか、性能で決まるのか？

業務要件では「権限削除は即座に反映すべき」と言われても、性能テストでは「キャッシュヒット率が90%以上必要」という制約が出てきます。この両立を求めると、設計判断の拠り所が不明確になります。

結果として、チーム内で「キャッシュを信頼する派」と「ストアを信頼する派」に分かれることになります。

## キャッシュレイヤーが複数になるとさらに複雑化する

近年のシステムは、単一のキャッシュレイヤーでは済みません：

- アプリケーションメモリ内のオンメモリキャッシュ
- Redis や Memcached などの分散キャッシュ
- CDN キャッシュ
- ブラウザキャッシュ

これらが同時に存在するとき、「どの層を信頼するか」の判断は複雑になります。

実装例として考えてみましょう。ユーザー設定情報を取得する処理：

```
1. ブラウザキャッシュを確認
2. なければ、アプリケーションのオンメモリキャッシュを確認
3. なければ、分散キャッシュを確認
4. なければ、データベースから取得して各層に書き込む
```

一見、階層的で合理的に見えます。しかし、実際には：

- ブラウザキャッシュは有効期限が不明確
- オンメモリキャッシュはサーバー再起動で消える
- 分散キャッシュは予期しない削除が起きる
- データベースの値が最新とは限らない（レプリケーション遅延）

各層が「ベストエフォート」で動いているのに、呼び出し元は「どこかから正しい値が返ってくる」と期待してしまいます。

## 現場で起きやすい判断ミス

設計時には「キャッシュが古かったら、ストアから取り直す」と決めていても、実装段階では以下のような判断が起きます：

**パターン1：性能優先で判断を先延ばしにする**

キャッシュがあれば返す。ストアの確認は「定期的なバッチ処理で検証する」という設計に変わります。すると、バッチ処理が止まったり遅延したりすると、不整合が放置されます。

**パターン2：安全を優先して過度に確認する**

毎回ストアを確認するようになり、キャッシュのメリットが消えます。性能要件を満たせなくなり、結局キャッシュを削除することになります。

**パター3：「ある程度の不整合は許容」という暗黙の仕様**

チームで「5分程度のズレなら大丈夫」と口頭で共有されても、実装では「10分」「30分」とズレていきます。監視も曖昧になり、障害時に「どちらが正しいのか」を判断できなくなります。

## 設計判断を明確にするための実務的なアプローチ

完全な一貫性を求めるのは非現実的です。代わりに、以下を明文化することが重要です：

**1. 各キャッシュレイヤーの「責務」を定義する**

- どのデータを保持するのか
- 有効期限はいくつか（理由も含める）
- 不整合が起きたときの検出方法は何か

**2. 「信頼の優先順位」を明示する**

曖昧に「必要に応じて確認」ではなく、具体的に：

- 権限情報：ストアを信頼、キャッシュは補助
- ユーザープロフィール：キャッシュを信頼、1時間ごとに確認
- アクセスログ：キャッシュを信頼、リアルタイム性は不要

**3. 監視・検証の仕組みを組み込む**

不整合を「後で気づく」のではなく、定期的に検証します。

```
定期タスク：
- キャッシュの値とストアの値を比較
- ズレを検出したら、ログに記録し、どちらを正とするか自動判定
- 判定ルールが不明な場合はアラート
```

**4. 障害時の対応を先に決めておく**

「キャッシュとストアが矛盾していた場合、どちらをリセットするか」を事前に決めておくと、対応が迅速になります。

## 中小規模チームが現実的に取れる選択肢

大規模な分散キャッシュの仕組みを全て導入するのは、保守負荷が高いです。

- **キャッシュは1層に絞る**（分散キャッシュか、オンメモリか、どちらか一方）
- **有効期限を短めに設定**（秒単位なら不整合の影響は限定的）
- **監視ツールで定期的に確認**（自動検証で人手を減らす）
- **業務要件で「許容できるズレ」を明確に定義**（曖昧な議論を避ける）

複数のキャッシュレイヤーが本当に必要か、一度問い直すのも有効です。

## 終わりに

キャッシュと永続化ストアの関係は、一度設計すれば終わりではなく、システムの成長とともに何度も見直す対象です。その過程で「どちらを信頼するか」の判断が揺らぎやすいのは、現場の実務的な制約があるからです。

理論的な完璧さを求めるのではなく、チームで「このシステムにおける信頼の基準は何か」を繰り返し確認し、実装に反映させることが、長期的には保守性と信頼性を両立させます。
