## 見落とされやすい、でも確実に起こる問題

モバイルアプリを運用していると、「ユーザーから『アプリが重くなった』という報告が増えるが、新機能も追加していないし、サーバー側も変わっていない」という相談を受けることがあります。原因を調べると、デバイスの内部ストレージが圧迫されていて、キャッシュやログファイルが蓄積されているケースが大半です。

一見すると『ストレージ容量の問題』に見えますが、実際には単なる容量不足ではなく、**ファイルシステムのパフォーマンス低下**が性能を蝕んでいます。ストレージが満杯に近づくと、OSの書き込み最適化が機能しなくなり、データベースクエリが遅くなり、ファイルI/Oが待機状況に陥ります。結果として、アプリ全体がもたつくようになるのです。

この問題は、開発段階ではほぼ見つかりません。テスト環境では十分な空き容量がありますし、数日のテスト期間では蓄積が目立たないからです。運用に入ってから、数ヶ月経ったユーザーデバイスで初めて顕在化します。

## キャッシュ戦略が曖昧だと起こること

モバイルアプリでキャッシュを使わないわけにはいきません。オフライン対応、通信量削減、レスポンス高速化のどれもが、ローカルキャッシュに依存しています。しかし、キャッシュを「一度書き込んだら放置」にしていると、以下のような連鎖が起きます。

- 画像キャッシュが月単位で蓄積される
- ログファイルが削除ルールなく書き込まれ続ける
- 一時ファイルが古いバージョンのまま残される
- SQLiteのデータベースファイルが肥大化し、VACUUMが追いつかない

その結果、ユーザーのデバイスでは「アプリは使い続けているのに、なぜか遅くなる」という体験になります。アプリ再起動で一時的に改善しても、数日で元に戻ります。

## 現場で機能する削除戦略の設計

実装上で重要なのは、**削除戦略を複数層で用意する**ことです。単一の判定基準では、予期しない状況に対応できません。

### 1. 容量ベースの上限管理

キャッシュディレクトリ全体に対して、最大サイズを設定します。超過した場合は、古いファイルから削除していく方式です。

```kotlin
fun pruneCache(cacheDir: File, maxSizeBytes: Long) {
    val files = cacheDir.listFiles()?.sortedBy { it.lastModified() } ?: return
    var totalSize = files.sumOf { it.length() }
    
    for (file in files) {
        if (totalSize <= maxSizeBytes) break
        totalSize -= file.length()
        file.delete()
    }
}
```

目安としては、アプリの用途にもよりますが、**100MB～500MB程度**が実用的です。ただし、これはデバイスの容量や想定ユーザー層によって判断が変わります。予約系アプリなら100MB、画像を多用するアプリなら500MB、という具合です。

### 2. 時間ベースの有効期限設定

キャッシュの種類によって、有効期限を変えます。たとえば：

- 画像キャッシュ：30日以上古いファイルは削除
- APIレスポンスキャッシュ：7日以上古いファイルは削除
- ログファイル：3日以上古いファイルは削除

```kotlin
fun deleteExpiredCache(cacheDir: File, ageInDays: Int) {
    val cutoffTime = System.currentTimeMillis() - (ageInDays * 24 * 60 * 60 * 1000)
    cacheDir.listFiles()?.forEach { file ->
        if (file.lastModified() < cutoffTime) {
            file.delete()
        }
    }
}
```

ここで注意すべきは、**有効期限と実際のデータ陳腐化の速度がズレないようにする**ことです。現場では「30日が標準だから」と機械的に設定されることが多いのですが、実際にはAPIの更新頻度や、ユーザーの行動パターンに基づいて判断すべきです。

### 3. アプリ起動時とバックグラウンドでの定期実行

削除処理は、アプリ起動時に軽量版を実行し、バックグラウンドで定期的に詳細版を実行するのが効果的です。

```kotlin
// アプリ起動時：軽量版（容量チェックのみ）
fun onAppStart() {
    val cacheSize = File(cacheDir).walk().sumOf { it.length() }
    if (cacheSize > CACHE_SOFT_LIMIT) {
        pruneCache(cacheDir, CACHE_HARD_LIMIT)
    }
}

// バックグラウンド：定期実行版（時間ベース削除も含む）
fun schedulePeriodicCacheMaintenance() {
    // WorkManager や JobScheduler を使用
    PeriodicWorkRequestBuilder<CacheMaintenanceWorker>(
        7, TimeUnit.DAYS
    ).build()
}
```

起動時に重い処理を走らせると、アプリの起動時間が悪化するため、バランスが重要です。

## SQLiteデータベースの肥大化への対応

キャッシュだけでなく、ローカルデータベースも蓄積の対象です。特にSQLiteは、削除したレコードの領域を自動的には回収しないため、定期的なVACUUMが必要です。

```kotlin
fun vacuumDatabase(db: SQLiteDatabase) {
    db.execSQL("VACUUM")
}

// アプリ起動時、または週1回程度の定期実行
fun onAppStart() {
    val db = openDatabase()
    val dbSize = File(db.path).length()
    
    if (dbSize > DATABASE_SIZE_THRESHOLD) {
        // 古いレコードを削除してからVACUUM
        db.delete("logs", "timestamp < ?", 
            arrayOf((System.currentTimeMillis() - 30*24*60*60*1000).toString()))
        db.execSQL("VACUUM")
    }
    db.close()
}
```

ただし、VACUUMは時間がかかるため、**バックグラウンドで実行するか、ユーザーが明示的に「キャッシュクリア」を実行した時点で走らせる**のが現実的です。起動時に毎回実行すると、ユーザーが起動遅延を感じるようになります。

## 運用段階での監視と判断

実装が完了しても、運用に入ると新たな課題が浮かぶものです。以下の観点で監視することをお勧めします。

**キャッシュサイズの可視化**：アプリの設定画面に「キャッシュサイズ」を表示し、ユーザーが手動削除できるようにします。同時に、その情報をアナリティクスに送信し、実際のユーザーデバイスでどの程度蓄積しているかを把握します。

**パフォーマンスメトリクス**：ストレージ使用量とアプリの応答時間の相関を監視します。「ストレージ使用量が500MBを超えたユーザーのアプリ起動時間が平均で200ms遅い」といった傾向が見えれば、上限値の調整が必要です。

**ユーザーフィードバック**：「アプリが重い」という報告が増えた時点で、該当ユーザーのストレージ使用量を確認します。デバイス容量が少ないユーザーほど、この問題が顕在化しやすいため、セグメント分析が有効です。

## 小規模チームでの現実的な導入ステップ

最後に、実装リソースが限られている場合の優先順位をお示しします。

1. **最初は容量ベースの上限管理だけ実装する**：複雑な時間ベース削除より、シンプルな「容量超過時のLRU削除」から始めます。

2. **ログファイルの削除ルールを明確にする**：キャッシュより、ログが蓄積しやすいケースが多いです。ここを整備するだけで効果が出ることもあります。

3. **バックグラウンド処理は後付けでOK**：最初はアプリ起動時の簡易版だけで運用し、ユーザー規模が増えてから定期実行を追加します。

4. **ユーザーが手動削除できる仕組みを用意する**：自動化より、ユーザーが「今すぐ削除」できるボタンの方が、信頼感につながります。

完璧な自動化を目指さず、**運用で観察しながら調整する**という姿勢が、実務では最も堅牢です。
