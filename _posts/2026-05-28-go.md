## 本番障害時に「どこから来たエラーか追えない」という現実

Goで複数のレイヤーを持つシステムを運用していると、こういう場面に遭遇します。

本番環境でエラーが発生し、ログを見ると：

```
error: database connection failed: context deadline exceeded
```

このメッセージだけでは、どのクエリが遅延したのか、どの関数から呼ばれたのか、そもそもタイムアウトが何秒設定だったのか、まったく判断できない。エラーの発生地点は分かっても、**なぜそこに至ったのか、どの処理フローで起きたのか**を追うのに30分以上かかることもあります。

これは Goのエラーラップ機能（`fmt.Errorf("%w", err)` や `errors.Is()`）が便利だからこそ陥りやすい落とし穴です。ラップを重ねるたびに「コンテキスト情報は増えるが、スタックトレース情報は失われていく」という逆説的な状況が生まれます。

## エラーラップの層が増えるほど、情報が散逸する理由

Goのエラーラップを簡潔に書くと、こうなります：

```go
// レイヤー1: DB接続
func queryUser(ctx context.Context, id string) (*User, error) {
    user := &User{}
    if err := db.QueryRowContext(ctx, "SELECT * FROM users WHERE id = ?", id).Scan(&user); err != nil {
        return nil, fmt.Errorf("query failed: %w", err)
    }
    return user, nil
}

// レイヤー2: ビジネスロジック
func getUserProfile(ctx context.Context, id string) (*Profile, error) {
    user, err := queryUser(ctx, id)
    if err != nil {
        return nil, fmt.Errorf("failed to get user: %w", err)
    }
    return &Profile{Name: user.Name}, nil
}

// レイヤー3: HTTPハンドラ
func handleGetProfile(w http.ResponseWriter, r *http.Request) {
    id := r.URL.Query().Get("id")
    profile, err := getUserProfile(r.Context(), id)
    if err != nil {
        log.Printf("error: %v", err)
        http.Error(w, "Internal Server Error", 500)
    }
}
```

ここまでなら読みやすいのですが、実際の本番コードでは：

- HTTPミドルウェアで再度ラップされる
- リトライロジックでラップされる
- トレーシング用にメタデータが付加される
- エラー変換ユーティリティを通される

こうして5層、6層と重なると、**ログに出力される時点では、元のエラーがどこで発生したのかを特定するために、ラップの層を逆向きに辿る必要**が出てきます。

さらに厄介なのは、Goに標準的なスタックトレース機能がないということです。JavaやPythonのようにフレームごとのファイル名・行番号がログに自動で含まれません。`errors.Is()` や `errors.As()` で型チェックはできますが、**発生地点の行番号情報は保持されていない**のです。

## 検証：ラップの層数とデバッグ効率の関係

実際にシミュレーションしてみました。同じエラーを異なる層数でラップして、ログ復元にかかる時間と確度を測ります。

### 検証条件

- Goバージョン: 1.21以上
- 3つのシナリオを比較
  - **シナリオA**: ラップなし（エラーをそのまま返す）
  - **シナリオB**: 2層ラップ（一般的な設計）
  - **シナリオC**: 5層ラップ（実際の複雑なシステム）
- 各シナリオで、同じ根本原因のエラーを発生させ、ログから原因特定までの情報量を記録

### シナリオAのコード例

```go
func simulateScenarioA() error {
    // 根本原因: SQL構文エラー
    err := errors.New("syntax error near 'SELEC'")
    return err
}
```

出力：
```
syntax error near 'SELEC'
```

問題：どの関数で起きたかは全く分からない。

### シナリオBのコード例

```go
func queryDB() error {
    err := errors.New("syntax error near 'SELEC'")
    return fmt.Errorf("database query failed: %w", err)
}

func fetchUserData() error {
    err := queryDB()
    return fmt.Errorf("user fetch failed: %w", err)
}
```

出力：
```
user fetch failed: database query failed: syntax error near 'SELEC'
```

改善：どのレイヤーで失敗したかは推測できる。ただし呼び出し元の関数名や行番号はない。

### シナリオCのコード例

```go
func queryDB() error {
    err := errors.New("syntax error near 'SELEC'")
    return fmt.Errorf("database query failed: %w", err)
}

func fetchUserData() error {
    err := queryDB()
    return fmt.Errorf("user fetch failed: %w", err)
}

func getUserProfile() error {
    err := fetchUserData()
    return fmt.Errorf("profile retrieval failed: %w", err)
}

func handleRequest() error {
    err := getUserProfile()
    return fmt.Errorf("request processing failed: %w", err)
}

func middleware(next func() error) error {
    err := next()
    return fmt.Errorf("middleware intercepted: %w", err)
}
```

出力：
```
middleware intercepted: request processing failed: profile retrieval failed: user fetch failed: database query failed: syntax error near 'SELEC'
```

問題：メッセージは増えたが、**「どの行番号で queryDB() が呼ばれたのか」「fetchUserData() は何秒で失敗したのか」といった実行時情報が全くない**。

## 本番デバッグ性を確保するための3つの設計判断

### 1. エラーラップにコンテキスト情報を埋め込む

メッセージだけでなく、関数名や入力値を含める：

```go
func queryUser(ctx context.Context, userID string) (*User, error) {
    user := &User{}
    if err := db.QueryRowContext(ctx, "SELECT * FROM users WHERE id = ?", userID).Scan(&user); err != nil {
        return nil, fmt.Errorf(
            "queryUser failed [userID=%s, timeout=%v]: %w",
            userID,
            ctx.Deadline(),
            err,
        )
    }
    return user, nil
}
```

これにより、ログに「userID=12345, timeout=2024-01-15 10:30:45」といった情報が残ります。

### 2. クリティカルパスのみスタックトレースを記録する

全エラーでスタックトレースを取得するのは重い処理です。代わりに、重要なレイヤー（DB接続、外部API呼び出し、認証）のみに限定する：

```go
import "runtime/debug"

func queryDatabase(query string) error {
    err := db.Exec(query)
    if err != nil {
        // 本番環境ではスタックトレースを記録
        if isProduction {
            stack := debug.Stack()
            log.Printf("critical error in queryDatabase: %s\nStack: %s", err, stack)
        }
        return fmt.Errorf("queryDatabase failed: %w", err)
    }
    return nil
}
```

### 3. エラーラップの層数に上限を設ける

設計段階で「このシステムのエラーラップは最大3層まで」と決めておくと、ログの可読性が保たれます。4層目以上は、ラップではなく**エラーハンドラで処理を終わらせる**ようにします：

```go
// NG: 何層もラップを重ねる
func layer4(err error) error {
    return fmt.Errorf("layer4: %w", err)
}

// OK: ここで処理を終わらせる
func errorHandler(err error) {
    if err == nil {
        return
    }
    
    // エラーの型と内容をログに記録
    log.Printf("[ERROR] type=%T, message=%v", err, err)
    
    // 必要に応じてアラート送信
    alerting.Notify(err)
}
```

## 実装時の注意点と運用への影響

### ログ集約ツールとの連携を前提に設計する

本番環境では、単純な `log.Printf()` では不十分です。構造化ログ（JSON形式）で出力し、ELKやDatadog等で検索・集計できるようにしておきます：

```go
import "encoding/json"

type ErrorLog struct {
    Timestamp   time.Time `json:"timestamp"`
    Level       string    `json:"level"`
    Message     string    `json:"message"`
    ErrorType   string    `json:"error_type"`
    UserID      string    `json:"user_id,omitempty"`
    RequestID   string    `json:"request_id,omitempty"`
}

func logError(err error, userID, requestID string) {
    el := ErrorLog{
        Timestamp: time.Now(),
        Level:     "ERROR",
        Message:   err.Error(),
        ErrorType: fmt.Sprintf("%T", err),
        UserID:    userID,
        RequestID: requestID,
    }
    data, _ := json.Marshal(el)
    log.Println(string(data))
}
```

### エラーの「種類」と「原因」を分離して記録する

ラップを重ねるとき、「どんなエラーが起きたのか」と「なぜそれが起きたのか」を混在させないようにします：

```go
// NG: 原因と処理が混在
return fmt.Errorf("user fetch failed: database query failed: syntax error: %w", err)

// OK: 種類と原因を分離
return fmt.Errorf("user fetch failed (reason: database.query.syntax_error): %w", err)
```

### テスト環境でのデバッグ効率と本番でのログ量のバランス

テスト環境では詳細なスタックトレースを出力し、本番環境では最小限に抑えるという設定が必要です。環境変数で制御するのが一般的です：

```go
var debugMode = os.Getenv("DEBUG_MODE") == "true"

func logWithTrace(err error) {
    if debugMode {
        log.Printf("Full trace: %+v", err)
    } else {
        log.Printf("Error: %v", err)
    }
}
```
