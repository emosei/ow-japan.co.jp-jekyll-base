## 現場で起きる「わかりやすい」インターフェース設計の落とし穴

Goでシステムを構築していると、インターフェース設計の判断で分かれ道に立つ場面がよくあります。特に複数の実装を扱う場面で「とりあえず広いインターフェースを定義しておこう」という判断が、本番環境で型アサーション（type assertion）の連鎖を生み出し、保守性を大きく損なう事例を何度も見てきました。

問題は単なる「設計が甘い」では済みません。Goの言語仕様とインターフェースの柔軟性が、一見「わかりやすく」見える設計を許容してしまうため、チーム全体がその危険性に気づきにくいのです。本記事では、実際の構成で検証しながら、型アサーション地獄に陥る具体的な流れと、それを避けるための設計判断を整理します。

## 検証の前提：なぜインターフェース設計で型アサーションが増えるのか

Goのインターフェースは、実装側が明示的に「このインターフェースを実装する」と宣言する必要がありません。メソッドシグネチャが一致すれば自動的に満たされます。この仕様は柔軟性をもたらす反面、設計者に大きな責任を課します。

```go
// 広すぎるインターフェース定義の例
type DataHandler interface {
    Read(ctx context.Context, id string) (interface{}, error)
    Write(ctx context.Context, data interface{}) error
    Delete(ctx context.Context, id string) error
    Validate(data interface{}) error
    Transform(data interface{}) interface{}
}
```

このインターフェースが複数の実装に使われると、呼び出し側では戻り値の `interface{}` に対して型アサーションを繰り返すことになります。そしてアサーション失敗時の処理が増え、テストケースも爆発的に増加します。

## 検証構成：段階的な型アサーション増加を追跡

実際の本番シナリオに近い構成で検証しました。以下の3段階を用意します。

**段階1：初期実装（単一の具体的な型）**
```go
type UserRepository interface {
    GetUser(ctx context.Context, id string) (*User, error)
    SaveUser(ctx context.Context, user *User) error
}
```

**段階2：拡張時（interface{}を導入）**
```go
type GenericRepository interface {
    Get(ctx context.Context, id string) (interface{}, error)
    Save(ctx context.Context, data interface{}) error
}
```

**段階3：複数の実装が混在**
```go
// UserRepository実装
func (r *userRepo) Get(ctx context.Context, id string) (interface{}, error) {
    user := &User{}
    // ... 取得ロジック
    return user, nil
}

// ProductRepository実装
func (r *productRepo) Get(ctx context.Context, id string) (interface{}, error) {
    product := &Product{}
    // ... 取得ロジック
    return product, nil
}

// 呼び出し側で型アサーションが必須に
repo := getRepository(entityType) // GenericRepository を返す
result, err := repo.Get(ctx, id)
if err != nil {
    return err
}

// ここから型アサーション地獄
user, ok := result.(*User)
if !ok {
    product, ok := result.(*Product)
    if !ok {
        order, ok := result.(*Order)
        if !ok {
            return fmt.Errorf("unknown type")
        }
        // Orderの処理
    }
    // Productの処理
}
// Userの処理
```

このパターンは一見「柔軟だ」と感じますが、実装が増えるたびに型アサーションのネストが深くなります。

## 本番で顕在化する問題

### 1. 型アサーション失敗時のバグ潜在化

型アサーションに失敗すると `ok` フラグが `false` になりますが、その処理が不十分だと、本番環境で予期しない型が流入したときに対応できません。

```go
// 危険な実装例
result, _ := repo.Get(ctx, id)
user := result.(*User) // panic の可能性
```

### 2. テストケースの組み合わせ爆発

複数の実装が増えると、各組み合わせに対するテストが必要になり、テストの保守コストが急増します。

### 3. IDE補完とリファクタリング支援の低下

`interface{}` を使うと、IDEが型推論できず、メソッド補完やリネーム時の自動置換が機能しなくなります。

## 設計改善の検証：ジェネリクスと型制約の活用

Go 1.18以降のジェネリクスを活用すると、型安全性を保ちながら汎用性を確保できます。

```go
// ジェネリック版の設計
type Repository[T any] interface {
    Get(ctx context.Context, id string) (*T, error)
    Save(ctx context.Context, data *T) error
    Delete(ctx context.Context, id string) error
}

// 実装側
type userRepository struct {
    db *sql.DB
}

func (r *userRepository) Get(ctx context.Context, id string) (*User, error) {
    // User型で直接返す
    user := &User{}
    // ... 取得ロジック
    return user, nil
}

// 呼び出し側
func getUserService(repo Repository[User]) (*User, error) {
    // 型アサーション不要、型安全
    return repo.Get(context.Background(), "123")
}
```

この方法では、型アサーションが完全に不要になり、コンパイル時に型チェックが完了します。

### 制約条件と実務上の注意

- **Go 1.18以降が必須**：レガシー環境では使用できません
- **複雑な型制約は可読性を損なう**：チーム全体が理解できるレベルに留めることが重要です
- **パフォーマンス**：ジェネリクスは現在、型ごとにコードが生成（monomorphization）されるため、バイナリサイズが増加します

## 現実的な折衷案：インターフェース分離とアダプターパターン

ジェネリクスが使えない環境では、インターフェースを細かく分割し、必要に応じてアダプターを挟む方法が有効です。

```go
// 共通のメソッドのみを含む最小限のインターフェース
type Reader interface {
    Read(ctx context.Context, id string) (interface{}, error)
}

// 具体的な型を返すインターフェース
type UserReader interface {
    ReadUser(ctx context.Context, id string) (*User, error)
}

// アダプター：汎用インターフェースを具体的な型に変換
type userReaderAdapter struct {
    reader Reader
}

func (a *userReaderAdapter) ReadUser(ctx context.Context, id string) (*User, error) {
    result, err := a.reader.Read(ctx, id)
    if err != nil {
        return nil, err
    }
    user, ok := result.(*User)
    if !ok {
        return nil, fmt.Errorf("expected *User, got %T", result)
    }
    return user, nil
}
```

この方法では、型アサーションをアダプター層に限定でき、ビジネスロジック側は型安全に保てます。

## 実務投入時の確認ポイント

1. **Go バージョンと組織の制約を確認**：ジェネリクスが使える環境か、レガシー環境か
2. **チームの習熟度**：ジェネリクスやアダプターパターンをチーム全体が理解できるか
3. **型アサーション箇所の明示**：やむを得ず型アサーションを使う場合は、エラーハンドリングを厳密に
4. **テストカバレッジ**：型アサーション失敗時のシナリオを必ずテストに含める
5. **ドキュメント**：インターフェース設計の意図と、どの層で型安全性が保証されるかを明記

## どんな案件で採用しやすいか

- **新規プロジェクト**：ジェネリクスを前提に設計できます
- **チーム規模が大きい場合**：型安全性の重要性が高まり、設計の統一が必須
- **長期保守が必要なシステム**：型アサーション地獄は保守性を著しく低下させるため、初期段階での改善投資が回収しやすい
- **複数のエンティティを扱う基幹系**：予約システムや管理画面など、データ型が多岐にわたるシステムでは特に効果的

逆に、単一の型のみを扱う小規模なツールでは、シンプルな具体的インターフェースで十分です。
