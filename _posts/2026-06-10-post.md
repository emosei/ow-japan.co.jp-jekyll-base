## 自動スキャンで「安全」と判定されたのに、本番環境で脆弱性が見つかる

セキュリティスキャンツールを導入して脆弱性チェックを自動化した。ローカルでもCI/CDパイプラインでも問題なく通過した。それなのに、本番環境で外部からの指摘によって初めて脆弱性が発見される──こういった経験は、中小規模の開発チームでも珍しくありません。

その原因の多くは、スキャンツール自体の検出範囲の限界にあります。特に「推移的な依存関係」に含まれる脆弱性は、見落とされやすい傾向にあります。直接指定したライブラリは検出されるが、そのライブラリが依存する先の先のライブラリ内の脆弱性は、ツールの設定や実装次第では見逃される。こうした現実を理解しておくことは、単なる知識ではなく、本番環境を守るための実務的な判断につながります。

## 推移的依存関係が検出漏れする仕組み

### ツールの検出方式による違い

脆弱性スキャンツールは大きく2つのアプローチで動作します。

**静的マニフェスト解析**：`package.json`や`Gemfile`、`pom.xml`といったマニフェストファイルを読み込み、宣言されたライブラリバージョンを脆弱性データベースと照合する方式です。実装が単純で高速ですが、実際にビルドされるまでは「どのバージョンが最終的に選ばれるのか」が確定しません。

**ビルド成果物解析**：実際にビルド・依存関係解決を行った後の成果物（`package-lock.json`、`Gemfile.lock`、`pom.lock`など）を解析する方式です。より正確な依存関係グラフが得られますが、ビルド環境の整備が必要になります。

現場では、前者の静的解析のみで運用しているケースが多く見られます。理由は簡単で、セットアップが楽だからです。しかし、この選択が検出漏れを招きます。

### バージョン解決ルールの複雑性

Node.jsの`npm`やRubyの`Bundler`では、依存関係のバージョン指定に柔軟性があります。`^1.2.3`のようなセマンティックバージョニング範囲指定を使う場合、実際にどのバージョンがインストールされるかは、マニフェストファイルだけからは判断できません。

さらに複雑なのは、複数のライブラリが同じ依存先を指定している場合です。A というライブラリが「lodash ^4.17.0」を要求し、B というライブラリが「lodash ^4.16.0」を要求している場合、どちらを満たす最新バージョンが選ばれるか。こうした解決ロジックは、ツール側で完全に再現されていないことがあります。

## 現場で起きやすい判断の誤り

### 「スキャン済み = 安全」の思い込み

開発チームがセキュリティスキャンツールを導入すると、無意識のうちに「ツールが通ったから脆弱性はない」という心理が働きます。これは人間の自然な反応ですが、危険です。

実際には、ツールが見落とした脆弱性はまだ存在しています。スキャンツールは「既知の脆弱性データベースに登録されているものを検出する補助ツール」に過ぎず、完全な保証ではありません。

### 推移的依存関係の可視化不足

`npm list`や`bundle show`で依存関係を確認することはできますが、その全体像を把握している開発者は少ないものです。「自分たちが明示的に指定したライブラリ」と「それが引き込む先の依存関係」の区別が曖昧になりやすく、脆弱性通知が来たときに「それは我々のコードじゃない」と他人事になってしまいます。

しかし本番環境では、推移的依存関係のコードも実行されます。脆弱性があれば、それは攻撃対象になります。

## 実務的な対応策

### 1. ビルド成果物を含めたスキャン

最も確実な方法は、ロックファイル（`package-lock.json`、`Gemfile.lock`など）を生成してからスキャンすることです。

```bash
# Node.js の場合
npm ci  # 正確な依存関係を解決
npm audit --production

# Ruby の場合
bundle install
bundle audit
```

CI/CDパイプラインに組み込む際は、この「ロックファイルを基準とした解析」を標準化することが重要です。マニフェストファイルのスキャンだけでなく、実際にビルドされた成果物を検査する段階を必ず含めてください。

### 2. 複数のツールを組み合わせる

単一のツールに依存しないことも有効です。例えば、NPM監査に加えて`Snyk`や`OWASP Dependency-Check`を併用すれば、検出漏れのリスクが低減します。ツールごとに脆弱性データベースやスキャンロジックが異なるため、相補的に機能します。

ただし、複数ツールの運用コストは増します。チームの規模と脆弱性検出の重要度のバランスを取って判断してください。

### 3. 推移的依存関係の定期的な可視化

3ヶ月ごと、あるいは重要なアップデートの前に、依存関係グラフを出力して目視確認することをお勧めします。

```bash
# Node.js の場合
npm ls --depth=3 > dependencies.txt

# Ruby の場合
bundle show --paths
```

この作業は地味ですが、「自分たちが何に依存しているのか」という認識を保つ上で有効です。現場では、この認識の有無が、脆弱性通知への対応スピードに大きく影響します。

### 4. 脆弱性通知への対応フロー

脆弱性が報告されたときに「それは推移的依存関係だから対応不要」という判断をしてはいけません。むしろ、以下のフローを決めておくことが大切です。

- 脆弱性の詳細を確認（CVE、影響範囲、攻撃可能性）
- 本番環境で実際に使用されているかを確認
- 対応方法を検討（依存先のアップデート、バージョン指定の変更、代替ライブラリへの移行）
- テスト環境で検証後、本番環境に適用

推移的依存関係であっても、対応の優先度付けは同じロジックで行うべきです。

## 中小規模チームが現実的に取れるアクション

完璧なセキュリティスキャンを目指す必要はありません。以下の3点を押さえれば、大多数の脆弱性リスクは低減できます。

1. **ロックファイルベースのスキャンを標準化する**──マニフェストファイルだけでなく、ビルド成果物を検査の基準にする。

2. **脆弱性通知への対応フローを文書化する**──推移的依存関係だからといって後回しにしない。優先度付けと対応方法を事前に決めておく。

3. **定期的に依存関係を見直す**──ツールに完全に任せず、人間の目で確認する機会を作る。

セキュリティツールは「補助」です。それを使いこなすのは、結局のところ開発チームの判断と運用規律です。
