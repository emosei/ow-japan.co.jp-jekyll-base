## なぜ複数エンジンの統合は「動く」のに「壊れる」のか

マイクロサービスの流行とは別に、モノリシックなRailsアプリケーション内で複数の独立した機能セットを「エンジン」として組み込む設計は、今でも現場では選ばれます。理由は単純です。デプロイが一度で済む、データベース接続が統一できる、認証情報の共有が容易だからです。

ところが、この設計を採用すると、開発中盤から運用初期にかけて、ルーティングの優先度が予測不可能になる問題が浮上します。統合テストでは成功するのに、本番環境の実際のアクセスパターンで特定の機能だけが動作しなくなる。あるいは、エンジンAのエンドポイントのつもりで、エンジンBの同名パスが先に捕捉されてしまう。こうした問題に直面したプロジェクトは少なくありません。

この記事では、複数エンジン統合時に何が起きているのか、そして統合テストだけでは防げない落とし穴が何なのかを、実装と運用の視点から整理します。

## Rails のルーティング優先度は「登録順」だが、その順序は不可視

Railsのルーターは、`routes.rb`に記述されたマッピングを上から順に評価し、最初にマッチしたものを採用します。これは明確なルールです。しかし複数のエンジンが同一プロセスで動く場合、その登録順序は開発者の意図とズレることがあります。

```ruby
# config/routes.rb の典型的な例
Rails.application.routes.draw do
  mount AdminEngine::Engine => '/admin'
  mount ApiEngine::Engine => '/api'
  
  resources :articles
end
```

一見すると、`/admin`と`/api`のパスは明確に分離されているように見えます。しかし問題は、エンジン内部のルーティングにあります。

```ruby
# engines/admin_engine/config/routes.rb
AdminEngine::Engine.routes.draw do
  get '/articles', to: 'articles#index'
  post '/articles', to: 'articles#create'
end

# engines/api_engine/config/routes.rb
ApiEngine::Engine.routes.draw do
  get '/articles/:id', to: 'articles#show'
end
```

この場合、`GET /admin/articles/1`にアクセスすると、AdminEngineの`/articles`ルートが先にマッチします。ここまでは問題ありません。

ところが、エンジンの初期化順序、gemの依存関係、あるいはRailsのバージョンアップによって、ルーティングテーブルの構築順序が変わることがあります。特に、エンジンが別のgemとして分離されている場合、その読み込み順序はRailsの内部的な依存解決に委ねられます。

```ruby
# Gemfile
gem 'admin_engine', path: './engines/admin_engine'
gem 'api_engine', path: './engines/api_engine'
```

この場合、どちらのエンジンが先に`routes.rb`に統合されるかは、gemの読み込み順序に依存します。開発環境と本番環境で読み込み順序が異なれば、ルーティングの優先度も変わります。

## 統合テストの限界──「期待するパスが通る」では検証不足

統合テストは通常、特定のシナリオに沿ったリクエストをシミュレートします。

```ruby
# test/integration/routing_test.rb
class RoutingTest < ActionDispatch::IntegrationTest
  test 'admin articles index' do
    get '/admin/articles'
    assert_response :success
  end
  
  test 'api article show' do
    get '/api/articles/1'
    assert_response :success
  end
end
```

これらのテストが緑になったからといって、ルーティングの優先度が正しいことは保証されません。なぜなら、統合テストは「このパスへのリクエストが期待する応答を返すか」しか検証していないからです。

実際の問題は、複数のエンジンが「同じパターンに近いルート」を持つ場合に発生します。

```ruby
# AdminEngine
get '/reports/:id', to: 'reports#show'

# ApiEngine  
get '/reports', to: 'reports#list'
```

`GET /admin/reports/summary`というリクエストが、意図せずApiEngineの`/reports`ルートにマッチしてしまう可能性があります。統合テストでは`/admin/reports/1`と`/api/reports`を個別に検証しているため、この衝突は見つかりません。

## 優先度の衝突を検出する──ルーティングテーブルの明示的な検証

回避策の第一歩は、ルーティングテーブルそのものを検証することです。Railsは`Rails.application.routes`経由でルーティング情報にアクセスできます。

```ruby
# test/unit/routing_priority_test.rb
class RoutingPriorityTest < ActiveSupport::TestCase
  test 'no route collision between engines' do
    routes = Rails.application.routes.routes
    
    # 各エンジンのパスプレフィックスを明示的に記録
    admin_routes = routes.select { |r| r.path.spec.to_s.start_with?('/admin') }
    api_routes = routes.select { |r| r.path.spec.to_s.start_with?('/api') }
    
    # 衝突の可能性をチェック
    admin_patterns = admin_routes.map { |r| r.path.spec.to_s }
    api_patterns = api_routes.map { |r| r.path.spec.to_s }
    
    # 同じパターンが複数エンジンに存在しないか確認
    collision = admin_patterns & api_patterns
    assert collision.empty?, "Route collision detected: #{collision.inspect}"
  end
end
```

ただし、このテストは静的なパターンマッチングに過ぎません。より確実な方法は、ルーティングの動的な解決をシミュレートすることです。

```ruby
# test/unit/route_resolution_test.rb
class RouteResolutionTest < ActiveSupport::TestCase
  test 'verify route resolution order' do
    test_paths = [
      '/admin/articles',
      '/admin/articles/1',
      '/api/articles',
      '/api/articles/1',
      '/articles',
      '/articles/1'
    ]
    
    test_paths.each do |path|
      # 実際にルーティングを解決
      route, _params = Rails.application.routes.recognize_path(path, method: :get)
      
      # 期待するエンジンにマッチしているか確認
      expected_engine = path.split('/')[1]
      actual_engine = route.engine.class.module_parent.name.downcase if route.engine
      
      assert actual_engine.include?(expected_engine),
        "Path #{path} resolved to #{actual_engine}, expected #{expected_engine}"
    end
  end
end
```

## 設計の段階で「名前空間」を明確に分離する

根本的な解決は、設計の時点で各エンジンのパス領域を完全に分離することです。

```ruby
# config/routes.rb
Rails.application.routes.draw do
  namespace :admin_v1 do
    mount AdminEngine::Engine => '/'
  end
  
  namespace :api_v1 do
    mount ApiEngine::Engine => '/'
  end
  
  resources :articles  # メインアプリケーション
end
```

この方法では、エンジン内部のルーティングが同じパターンでも、外側の名前空間によって必ず分離されます。

```ruby
# AdminEngine のルート定義
AdminEngine::Engine.routes.draw do
  resources :articles  # /admin_v1/articles になる
end

# ApiEngine のルート定義
ApiEngine::Engine.routes.draw do
  resources :articles  # /api_v1/articles になる
end
```

この設計には、バージョニングの明示という副次的な利点もあります。エンジンのAPIを変更する際、既存クライアントへの影響を最小限に抑えられます。

## 運用段階での監視と再検証

本番環境に移行した後も、ルーティングの優先度は定期的に再検証する必要があります。特に以下のタイミングでリスクが高まります。

- Railsのマイナーバージョンアップ
- エンジンの追加やgemの依存関係の変更
- デプロイパイプラインの変更（読み込み順序に影響する可能性）

簡単な監視スクリプトを用意しておくと、問題の早期発見につながります。

```ruby
# lib/tasks/check_routes.rake
namespace :routes do
  desc 'Check route resolution consistency'
  task check_consistency: :environment do
    critical_paths = [
      '/admin/dashboard',
      '/api/v1/users',
      '/articles'
    ]
    
    critical_paths.each do |path|
      begin
        route, _params = Rails.application.routes.recognize_path(path, method: :get)
        puts "✓ #{path} -> #{route.controller}##{route.action}"
      rescue ActionController::RoutingError => e
        puts "✗ #{path} -> #{e.message}"
      end
    end
  end
end
```

このタスクをCI/CDパイプラインに組み込み、デプロイ前に実行することで、ルーティングの意図しない変更を検出できます。

## 小規模チームが採用すべき現実的なアプローチ

複数エンジンの統合は、機能の独立性を高める一方で、ルーティングの複雑さを増します。小規模なチームであれば、以下の優先順位で対応することをお勧めします。

1. **名前空間による強制分離** – 最初の設計段階で必ず実施
2. **ルーティング解決テスト** – 統合テストとは別に、ルート認識の明示的な検証を追加
3. **ドキュメント化** – 各エンジンのパスプレフィックス、バージョン、責任範囲を明記
4. **定期的な監視** – デプロイ前後にルーティング整合性をチェック

複数エンジンの統合は、適切に設計・テスト・監視すれば、保守性と拡張性のバランスが取れた構成になります。ただし、その前提条件を見落とすと、本番環境での予測不可能な動作に直面することになります。
