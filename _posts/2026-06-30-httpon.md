## 現場で起きている、説明しづらい問題

基幹系のWebシステムを運用していると、こんな相談を受けることがあります。

「HTTPキープアライブを有効にしたら、スモークテストでは問題なかったのに、本番運用で数時間後からメモリ使用量が徐々に増え始めて、最終的にOOMで落ちてしまう。キープアライブを無効にすると安定するんです」

性能改善のためにキープアライブを導入したのに、かえって不安定になる。こうした状況では、単純に「キープアライブのせい」と判断してしまいがちです。しかし実際には、Webサーバーとアプリケーションランタイム間の接続管理の相互作用が、特定の業務パターンでメモリ圧力を加速させているケースが多いのです。

今回は、その仕組みと検証方法を整理してみました。

## キープアライブが「安定」と「リスク」を同時にもたらす理由

HTTPキープアライブ（HTTP persistent connection）は、TCP接続を再利用することで、新規接続のコストを削減します。これ自体は性能向上の正当な最適化です。

ただし、接続を長く保つということは、その接続に紐付く状態や資源も長く保持し続けるということです。多くの言語ランタイム（Java、Python、Node.js、PHPなど）は、HTTP接続ごとに内部的な接続オブジェクト、バッファ、キャッシュ領域を確保します。

**キープアライブなし**の場合、接続は1リクエスト後に切断され、これらのメモリは解放されます。

**キープアライブあり**の場合、接続が開いたままなので、これらのメモリはサーバー側で保持され続けます。

ここまでなら問題ありません。接続数が一定以下なら、メモリ使用量も安定します。

しかし、実際の業務パターンでは「接続数が一定」ではないのです。

## 業務パターンとメモリリークの関係

現場で見かけるケースは、こういった流れです：

1. **朝の営業開始時刻**に、営業担当者が一斉にクライアント（PCやモバイルアプリ）を立ち上げる
2. 各クライアントが複数の並列接続を確立し、Webサーバーに保持される接続数が急増
3. **業務中盤**で使用頻度が低下し、クライアント側では接続を閉じないまま放置される
4. Webサーバー側では「アイドル状態の接続」が蓄積され続ける
5. メモリ圧力が高まり、ガベージコレクション（GC）の頻度が上がり、最終的にOOMに至る

問題は、言語ランタイムのGCが、アイドル接続に紐付いたメモリを「生きた接続」と判定し続けることです。接続オブジェクトが参照を保持しているため、GCの対象にならないのです。

つまり、これは「メモリリーク」というより「接続の蓄積によるメモリ枯渇」なのですが、現象としてはメモリリークと区別しづらいのです。

## 検証の構成と手順

実際に検証してみましょう。ここでは、典型的なPython + Gunicornの構成を例に挙げます。

**環境：**
- Python 3.9以上
- Gunicorn（WSGIアプリケーションサーバー）
- Nginx（リバースプロキシ、キープアライブ設定あり）
- メモリ監視ツール（`psutil`、`memory_profiler`）

**テストアプリケーション（簡易版）：**

```python
from flask import Flask
import time
import threading

app = Flask(__name__)
connection_cache = {}

@app.route('/api/data', methods=['GET'])
def get_data():
    # リクエストごとに接続識別子を記録
    client_id = request.remote_addr
    if client_id not in connection_cache:
        connection_cache[client_id] = {
            'buffers': bytearray(1024 * 1024),  # 1MB確保
            'timestamp': time.time()
        }
    return {'status': 'ok', 'connections': len(connection_cache)}

@app.route('/api/idle', methods=['GET'])
def idle_endpoint():
    # 業務中盤の低頻度アクセスを模擬
    time.sleep(2)
    return {'status': 'idle'}

if __name__ == '__main__':
    app.run()
```

**Nginx設定（キープアライブON）：**

```nginx
upstream gunicorn {
    server 127.0.0.1:8000;
}

server {
    listen 80;
    
    location / {
        proxy_pass http://gunicorn;
        proxy_http_version 1.1;
        proxy_set_header Connection "";  # キープアライブを有効化
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }
}
```

**負荷テストスクリプト：**

```python
import concurrent.futures
import requests
import time
import psutil
import os

def simulate_user(user_id, duration=300):
    session = requests.Session()
    start = time.time()
    
    while time.time() - start < duration:
        try:
            # 最初は頻繁にアクセス
            if time.time() - start < 60:
                session.get('http://localhost/api/data')
                time.sleep(0.5)
            # その後、低頻度に
            else:
                session.get('http://localhost/api/idle')
                time.sleep(5)
        except Exception as e:
            print(f"Error: {e}")

# メモリ監視
process = psutil.Process(os.getpid())

with concurrent.futures.ThreadPoolExecutor(max_workers=50) as executor:
    futures = [executor.submit(simulate_user, i) for i in range(50)]
    
    # テスト実行中のメモリ使用量を記録
    while not all(f.done() for f in futures):
        mem = process.memory_info().rss / (1024 * 1024)
        print(f"Memory: {mem:.2f} MB")
        time.sleep(10)
    
    concurrent.futures.wait(futures)
```

**検証結果の見方：**

キープアライブONの場合、以下の傾向が観測されます：

- **初期フェーズ（0～60秒）**：メモリ増加は緩やか。アクティブな接続数に応じた増加
- **中盤フェーズ（60～180秒）**：アイドル接続が蓄積。メモリ増加が加速
- **後期フェーズ（180秒～）**：GC圧力が高まり、CPUスパイク。メモリは高止まり

キープアライブOFFの場合は、中盤以降もメモリが安定します。

## 実務投入時に確認すべきポイント

単に「キープアライブONが悪い」ではなく、以下の観点で判断してください。

**1. 接続タイムアウト設定の見直し**

Webサーバー側で、アイドル接続の切断時間を明示的に設定します。

```nginx
proxy_http_version 1.1;
proxy_set_header Connection "";
keepalive_timeout 30s;  # 30秒以上アイドルなら切断
keepalive_requests 100;  # 100リクエスト後は切断
```

アプリケーション側でも、接続ごとのメモリ確保に上限を設けることが重要です。

**2. 接続プールサイズの制限**

言語ランタイムが管理する接続プールに、明示的な上限を設定します。

```python
# Gunicornの場合
# --max-requests 1000 で、1000リクエスト後にワーカープロセスを再起動
# これにより、蓄積したメモリもリセットされる
```

**3. 業務パターンの把握**

「朝に一斉接続、日中は低頻度」といった業務パターンを事前に把握し、そのパターンで検証することが不可欠です。スモークテストでは、こうした時間軸の変化を再現できないのです。

**4. メモリ監視とアラート設定**

本番環境では、メモリ使用率が閾値を超えた際の自動リスタート機構を用意しておきます。Kubernetesなら`memoryLimit`、従来型ならsupervisordやsystemdのリスタート設定を活用します。

## 設計判断：キープアライブの採用基準

最後に、どんな案件でキープアライブを採用すべきか、判断の目安を整理します。

**キープアライブの採用が適している：**
- 短命な接続が大量に発生する（API呼び出しが頻繁）
- クライアント数が限定的（数十～数百程度）
- 接続の生存期間が予測可能（業務システムの営業時間など）

**採用を慎重にすべき：**
- 不特定多数のクライアントからのアクセス（Webサイト）
- クライアントの接続パターンが不規則
- ランタイムのメモリ管理が不透明な言語・フレームワーク

基幹系のシステムは、前者に該当することが多いため、キープアライブは有効な最適化です。ただし「設定したら終わり」ではなく、実際の業務パターンでの負荷検証と、本番環境での継続的な監視が必須なのです。
