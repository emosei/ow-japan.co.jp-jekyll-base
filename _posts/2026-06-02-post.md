## スプリント中盤の「急な仕様変更」はなぜ予想以上にコストがかかるのか

スプリントが始まって3日目。営業から「この機能、ユーザーからの要望で少し仕様を変えたい」という連絡が入ります。変更内容は「商品検索の並び順を、価格順だけでなく『ユーザーの過去購買履歴に基づく関連度順』も選べるようにしたい」というもの。

一見、既存の検索機能に新しいオプションを足すだけに見えます。でも実装を始めると、事態はそう単純ではありません。

現場ではこういう判断が起きやすいです。スプリント開始時に、検索結果の並び順は「DBクエリの ORDER BY 句で完結する」という前提で設計されていました。ソート条件がシンプルで、フロントエンドのロジックも軽い。だから「価格順」「新着順」といった静的な条件だけで十分だと考えられていたわけです。

ところが「ユーザーの過去購買履歴に基づく関連度順」となると、話が変わります。この判定にはユーザーのセッション情報や購買履歴DBへのクエリが必要になる。並び順の計算がメモリ上で行われるようになり、フロントエンドのロジックも複雑化します。場合によっては、キャッシュ戦略も見直す必要が出てきます。

つまり、スプリント中盤での優先度変更リクエストは、単に「新しい機能を足す」のではなく、「既に実装済みの設計判断を根本から揺さぶる」ことになるのです。

## なぜ「小さな変更」が大きなコストになるのか

設計判断の後戻りコストを理解するには、実装の層を意識する必要があります。

**データベース層での判断の固定化**

検索機能を実装する段階で、「並び順はDBのクエリレベルで完結させる」という判断がされていたとします。このとき、テーブル設計やインデックス戦略も、その前提で最適化されています。

- `products` テーブルに `price` と `created_at` のインデックスを張っている
- クエリプランナーの統計情報が、その前提で学習されている
- キャッシュレイヤー（Redisなど）も、ソート済みの結果セットを保存する想定で設計されている

ここに「ユーザー個別の関連度順」という要件が入ると、これらすべてが「使えない」または「不十分」になります。なぜなら、ユーザー個別の計算結果はキャッシュできないか、キャッシュの粒度が一気に細かくなるからです。

**アプリケーション層での分岐の増加**

次に、フロントエンドやAPI層を見ると、ソート条件の分岐が増えます。

```javascript
// 初期実装：シンプルな分岐
function sortResults(results, sortType) {
  if (sortType === 'price') {
    return results.sort((a, b) => a.price - b.price);
  } else if (sortType === 'newest') {
    return results.sort((a, b) => new Date(b.createdAt) - new Date(a.createdAt));
  }
}

// 優先度変更後：複雑な分岐
function sortResults(results, sortType, userId) {
  if (sortType === 'price') {
    return results.sort((a, b) => a.price - b.price);
  } else if (sortType === 'newest') {
    return results.sort((a, b) => new Date(b.createdAt) - new Date(a.createdAt));
  } else if (sortType === 'relevance') {
    // ユーザーの購買履歴を取得して、関連度スコアを計算
    const userHistory = await fetchUserHistory(userId);
    const relevanceScores = calculateRelevance(results, userHistory);
    return results.sort((a, b) => relevanceScores[b.id] - relevanceScores[a.id]);
  }
}
```

一見、`else if` を足しただけに見えます。しかし実際には：

- 非同期処理が混在するようになる（`fetchUserHistory`）
- エラーハンドリングが複雑化する（ユーザー履歴の取得に失敗したら？）
- テストケースが指数関数的に増える
- パフォーマンス特性が変わる（同期処理から非同期処理へ）

## 実装の「負債化」はいつ決定されるのか

重要な気づきは、この負債は「優先度変更リクエストが来た瞬間」ではなく、「スプリント開始時の設計判断」で既に決定されているということです。

スプリント計画フェーズで、検索機能の仕様が「価格順と新着順の2パターン」と明記されていたなら、その時点で実装方針が決まります。開発者は当然、シンプルなDBクエリベースの設計を選びます。これは正しい判断です。制約条件に対して、最適な設計を選んでいるのです。

ところが、スプリント中盤で新しい要件が入ると、その「最適な設計」が一気に「不最適な設計」に変わってしまいます。

現場では、この時点で以下のような判断が迫られます：

1. **既存実装を捨てて、新しい設計で作り直す** → スプリント内に完了しない
2. **既存実装の上に新しいロジックを無理に乗せる** → 技術的負債が溜まる
3. **優先度変更を次のスプリントに延期する** → ビジネス側の納期要求と衝突する

多くの現場では、結局2番目の判断に落ち着きます。なぜなら、1番目は時間がなく、3番目は営業から承認を得られないからです。その結果、コードは複雑化し、運用保守のコストが上昇していきます。

## 優先度変更に対応する現実的なアプローチ

では、どう対応するか。完全に防ぐことはできませんが、後戻りコストを最小化する方法があります。

### 1. スプリント開始時に「変更の受け入れ範囲」を明確にする

スプリント計画フェーズで、営業やプロダクトマネージャーと一緒に「このスプリント中に、どの機能なら変更要件を受け入れられるか」を事前に決めておきます。

例えば：
- 「検索機能の並び順は、スプリント開始後の変更は受け付けない」
- 「ただし、検索結果の表示件数やページネーション周りなら、柔軟に対応できる」

この「線引き」があるだけで、急な変更リクエストへの対応が劇的に効率化します。

### 2. 設計判断の「仮定」を文書化する

実装を始める前に、設計判断の背景を簡潔に記録しておきます。

```
【検索機能の設計判断】
- 並び順：DBクエリレベルで完結させる（パフォーマンス優先）
- 対応パターン：price, newest, rating（3パターンに限定）
- キャッシュ戦略：検索条件ごとにRedisに結果を保存

【変更が必要になる場合の判断基準】
- ユーザー個別の計算が必要な並び順が追加される場合 → 設計の見直し必須
- 新しいパターンが3パターン以上になる場合 → リファクタリングのタイミング
```

こうしておくと、変更リクエストが来た時に「これは現在の設計では対応できないレベルの変更」かどうかを、客観的に判断できます。

### 3. 段階的な対応を提案する

優先度変更が避けられない場合は、段階的な対応を提案します。

**第1段階（今スプリント）**
- 「ユーザーの過去購買履歴に基づく関連度順」の簡易版を実装
- 計算ロジックはフロントエンド側で行う（DBへの負荷を避ける）
- キャッシュは使わず、毎回計算する（一時的な妥協）

**第2段階（次スプリント）**
- バックエンド側に関連度計算ロジックを移動
- キャッシュ戦略を見直す
- パフォーマンステストを実施

この方法なら、ビジネス要求に応えつつ、技術的な後戻りコストも最小化できます。

## 運用フェーズでの影響

スプリント中の優先度変更の影響は、本番リリース後にも続きます。

複雑になった検索ロジックは、バグの温床になりやすいです。特に「ユーザー個別の計算結果」が絡むと、テストの網羅性が落ちやすい。すべてのユーザーパターンを事前テストすることは現実的でないからです。

その結果、本番環境で「特定のユーザーには関連度順が機能しない」「検索結果が時々空になる」といったバグが報告されるようになります。これらは、スプリント中盤の設計判断の後戻りが根本原因です。

運用フェーズでの対応を見越して、スプリント計画の段階で「変更受け入れ範囲」を厳しめに設定することの価値が、ここで初めて理解できます。

## 小規模チームでも実践できる判断ポイント

- **スプリント開始時に、機能ごとに「変更受け入れ可能度」を3段階評価する**
  - 緑：変更があっても対応できる（表示形式、テキストなど）
  - 黄：変更があると設計見直しが必要（データ取得ロジック）
  - 赤：変更は次スプリント以降（アーキテクチャレベルの判断）

- **設計判断の「仮定」を簡潔に記録する**
  - 「なぜこの設計にしたのか」を後から確認できるようにしておく
