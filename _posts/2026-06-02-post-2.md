## 現場で起こる判断の迷い

ユーザーが「バックアップコードを失くしてしまった」と連絡してきたとき、あなたのシステムは何をすべきか判断できていますか。

多要素認証（MFA）を導入したシステムでよく見かける光景です。スマートフォンの紛失や機種変更のときに、バックアップコード（回復用コード）が生命線になる。ところが、その再発行のタイミング、失効ルール、ユーザー側での保管方法まで含めた設計が、実装時点では後回しにされることが多い。

認証フロー自体は比較的シンプルに実装できます。でも、バックアップコード管理は「何度まで再発行できるのか」「古いコードはいつ無効にするのか」「再発行時に前のコードを即座に失効させるべきか」といった判断が、要件定義の段階で曖昧なまま進むケースが目立ちます。

その結果、運用が始まると「ユーザーサポートが対応方針を決めあぐねている」「管理画面の機能が不足している」といった問題が浮上するのです。

## バックアップコード設計で避けられない3つの判断点

バックアップコード管理で押さえるべき判断点は、実は3つに整理できます。

**1. 再発行の上限**

バックアップコードを何度でも再発行できるシステムと、回数制限を設けるシステムでは、セキュリティリスクと利便性のバランスが変わります。

無制限再発行は利便性が高いですが、ユーザーが古いコードを複数保有することになり、流出リスクが増します。また、攻撃者が管理画面へのアクセスを得た場合、何度も再発行させて新しいコードを収集することも理論的には可能です。

一方、再発行を「1回のみ」や「年1回まで」と制限すれば、古いコードの蓄積は防げます。ただし、ユーザーが再発行上限に達した後に本当に必要になったときは、管理者による手動対応が必須になります。

**2. 再発行時の前コード失効タイミング**

新しいコードを発行した瞬間に、古いコードをすべて失効させるか、一定期間の並行運用を許すかで、ユーザー体験が大きく変わります。

即座に失効させれば、古いコードの流出時の被害は限定的です。しかし、ユーザーが新コードの受け取りに失敗した場合、アクセス不可になるリスクがあります。

並行運用期間を設ければ（例：新コード発行後7日間は旧コードも有効）、ユーザーへの親切度は上がります。ただし、その期間中は旧コードが有効なままなので、紛失時のリスク期間が延びます。

**3. コード使用済み時の扱い**

バックアップコードは通常、1回限りの使用です。使用済みコードを記録して、二重使用を防ぐ必要があります。

ここで問題になるのが、「部分的な使用」です。例えば、10個のコードを発行した場合、ユーザーが3つ使用した後に「すべてのコードを再発行してほしい」と言ってきたとき、使用済みの3つはどう扱いますか。再発行時に履歴をリセットするのか、新しいセットの中には含めないのか。この判断が曖昧だと、監査ログや復旧手順が複雑になります。

## 実装検証：バックアップコード管理の設計パターン

実際に、複数のバックアップコード管理パターンを検証してみました。

### パターンA：無制限再発行＋即座失効

```
// ユーザーがバックアップコード再発行を要求
POST /api/user/backup-codes/regenerate

// サーバー側の処理フロー
1. 新しいコード10個を生成
2. 古いコード（未使用分）をすべて失効マーク
3. 新コードを返却
4. 使用済みコードはログに残す
```

**利点：**
- セキュリティ的にはシンプル。古いコードは即座に無効
- 監査ログが明確。再発行のたびに世代管理できる

**制約：**
- ユーザーがコードを受け取れなかった場合、アクセス手段が失われる
- 再発行履歴が増えると、ユーザーが「どのコードセットが有効か」を混同しやすい
- 管理画面で「現在有効なコード一覧」を常に表示する必要がある

### パターンB：回数制限＋並行運用期間

```
// バックアップコード再発行時の設定例
{
  "max_regenerations": 3,
  "parallel_validity_days": 7,
  "codes_per_set": 10
}

// データベーススキーマ例（概念図）
backup_codes:
  id
  user_id
  code_hash
  set_number (第1セット、第2セットなど)
  generated_at
  used_at (使用済みの場合)
  expires_at (失効日時)
  status (active / expired / used)
```

**利点：**
- ユーザーが新コード受け取り失敗時も、旧コードで対応可能
- 再発行回数制限により、流出リスクを統計的に抑制できる

**制約：**
- 並行運用期間中は複数セットが有効なため、ログ分析が複雑
- 有効期限の管理が増え、バッチ処理で期限切れコードの整理が必要
- ユーザーが「古いコード」と「新しいコード」を混在させて保管すると、混乱の原因に

## 運用で後手に回る理由

バックアップコード管理が運用で後手に回る本質的な理由は、「異常系の判断基準が実装時に決まっていない」ことです。

例えば：

- ユーザーが「コードを全部使ってしまった」と言ってきた → 再発行上限に達していたら？
- 攻撃者が再発行APIを何度も叩いた → ログから検知できる仕組みがあるか？
- ユーザーが古いコードと新しいコードを混ぜて使用 → どちらが有効か判定できるか？

こうした状況では、実装時に「仕様書に書いてない」という理由で対応が遅れます。あるいは、その場しのぎの対応をして、次回同じ問題が起きたときに判断がぶれます。

さらに、多くのシステムではバックアップコード管理の管理画面が簡素です。ユーザーのコード使用状況、再発行履歴、現在有効なコード一覧を一目で確認できない場合、サポート対応が手作業になります。

## 実装時に確認すべきチェックリスト

バックアップコード機能を実装する際は、以下を事前に決めておくことをお勧めします。

- **再発行上限の設定**：何回までか。上限に達したときの対応フロー
- **失効タイミング**：再発行時に旧コードを即座に失効させるか、並行運用期間を設けるか
- **使用済みコードの扱い**：再発行時にリセットするのか、履歴として保持するのか
- **コード形式と検証**：10個一括発行か、個別発行か。有効期限はあるか
- **管理画面の機能**：現在有効なコード一覧、使用済みコード一覧、再発行履歴を表示できるか
- **監査ログ**：再発行、使用、失効のすべてを記録できるか
- **レート制限**：再発行APIへの連続リクエストを防ぐ仕組みがあるか
- **ユーザーへの通知**：再発行時にメール通知するか、管理画面に警告を出すか

## 設計判断の指針

バックアップコード管理で「どのパターンを選ぶか」は、システムのセキュリティポリシーとユーザーサポート体制に左右されます。

**セキュリティを優先する場合**
- 再発行は回数制限
- 即座失効（並行運用期間なし）
- コード使用済みは履歴として永続保持

**利便性を優先する場合**
- 無制限再発行
- 並行運用期間を設定（7～14日）
- ユーザーが再発行のたびに古いコードを破棄する前提

実務では、「セキュリティと利便性のバランス」よりも、「サポート体制で対応できるか」という現実的な判断が重要です。再発行上限を設けると、その上限に達したユーザーの対応が発生します。その対応を管理者が手作業で行うのか、自動化するのか、あるいは本人確認を強化するのか。こうした運用フローが実装と同期していないと、問題が顕在化したときに対応が遅れます。

小規模なチームなら「管理画面で確認できれば対応できる」という設計で十分な場合もあります。ただし、その場合は管理画面の検索機能やフィルタリング機能に投資する必要があります。

## まとめ

バックアップコード管理は、「あると便利」な機能として軽く扱われやすい部分です。でも、実際に運用が始まると、再発行タイミング、失効ルール、ユーザー対応の判断が次々と現れます。

実装時に「異常系をどう扱うか」を明確にしておくことで、運用の後手感は大きく減ります。セキュリティと利便性のバランスは、チームの状況に応じて柔軟に判断してください。ただし、その判断を実装に反映させ、管理画面で確認できる仕組みまで含めて初めて、運用可能な設計になります。
