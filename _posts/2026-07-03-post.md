## 「動いている」チームと「動いていない」チームの正体

複数のチームが同じ共有ライブラリを使っているプロダクトで、ある機能が一部のユーザーには正常に動き、別のユーザーには失敗するという報告を受けたことがあります。調査してみると、原因はライブラリのバージョンが統一されていなかったというケースです。

チームAは半年前にライブラリのメジャーバージョンをアップデートしていて、チームBはセキュリティパッチのみを当てていた。同じ関数を呼び出しているのに、内部の処理ロジックが異なっていたのです。テストは各チーム内で通っていたため、本番環境でのみ問題が浮き出ました。

このような状況は、一見すると「バージョン管理がルーズ」という単純な問題に見えます。しかし実際には、組織の成長段階、マイクロサービスの分散化、CI/CDの段階的導入など、複数の要因が絡み合っています。

## バージョン統一を「強制しない」と何が起きるか

### 発生する具体的な問題

バージョン統一を厳密に管理しない環境では、以下のような現象が連鎖的に起こります。

**1. 同じ関数が異なる戻り値を返す**

ライブラリのマイナーバージョン更新で、エラーハンドリングの仕様が変わったとします。古いバージョンではnullを返し、新しいバージョンではempty配列を返すという変更です。呼び出し側のコードが「nullチェック」に依存していれば、新バージョンでは条件分岐が常に失敗します。

**2. 追跡不可能なバグになる理由**

本番環境ではロードバランサーを通じて複数のインスタンスに振り分けられます。インスタンスごとに異なるバージョンのライブラリが載っていると、同じAPIリクエストでも返ってくる結果が変わります。ログを見ると「ランダムに失敗している」ように見え、再現性がありません。

開発環境では特定のインスタンスで検証するため、問題が起きません。ステージング環境でも同じバージョンに統一されていれば気づきません。本番環境だけで、特定の条件下でのみ顕在化するのです。

**3. デプロイのタイミングによる時間差バグ**

チームAが新バージョンにアップデートしてデプロイし、その翌日にチームBがアップデートする。その間の数時間から数日間、システム全体が不整合な状態になります。この間に起きたトランザクションやデータ処理は、予測不可能な結果を生む可能性があります。

### なぜ統一が後回しになるのか

実務では、バージョン統一を強制しない判断が生まれる背景があります。

- **チーム間の独立性を重視する**：各チームが自分たちのペースで依存関係を管理したいという要望
- **アップデートのリスク回避**：新バージョンで予期しない動作が起きるのを恐れ、古いバージョンを使い続ける
- **マイクロサービス化の過程**：モノリスから分割する際に、各サービスが独立したライブラリバージョンを持つようになる
- **組織の成長速度**：小規模なときは暗黙的に統一されていたが、チーム数が増えると管理が緩くなる

これらは「怠慢」ではなく、現場の制約と判断の結果です。

## 設計と運用で取るべき対策

### バージョン統一を強制する仕組み

最も確実な方法は、ビルド時にバージョンを固定することです。

```
# 例：package.json での依存関係指定
{
  "dependencies": {
    "shared-lib": "2.5.3"  // 完全一致指定（^や~を使わない）
  }
}
```

セマンティックバージョニングの`^`（互換範囲）や`~`（パッチ範囲）を使うと、インストール時期によってバージョンが異なります。共有ライブラリについては、完全一致指定（ピニング）が必須です。

### CI/CDパイプラインでの検証

デプロイ前に、全チームが使用するライブラリのバージョンをスキャンし、不一致があれば失敗させるステップを入れます。

```
# 疑似コード：デプロイ前チェック
services = get_deployed_services()
lib_versions = {}

for service in services:
    version = get_dependency_version(service, "shared-lib")
    if "shared-lib" in lib_versions:
        if lib_versions["shared-lib"] != version:
            fail("Version mismatch detected")
    lib_versions["shared-lib"] = version
```

### 段階的アップデートの計画

全チームが同時にアップデートするのは難しい場合、以下のプロセスを取ります。

1. **新バージョンの互換性テスト**：本来の仕様変更と、実装上の予期しない動作を分離
2. **段階的ロールアウト**：1チーム→複数チーム→全体という順序で進める
3. **ロールバック計画**：問題が起きたら即座に戻せる状態を保つ
4. **アップデート期限の明示**：「3ヶ月以内に全チームが新バージョンに移行」という明確な期限

## 中小規模チームが現実的に取るアクション

### すぐに実行できる対策

**1. 依存関係の可視化**

各サービス・アプリケーションが何のバージョンを使っているか、一覧にして共有します。スプレッドシートでもいいので、定期的に更新する習慣をつけます。

**2. 共有ライブラリの定義**

「これは全チームで統一すべき」というライブラリを明確にします。ユーティリティ系や基盤系は必須ですが、UI部品など選択的なものは除外できます。

**3. アップデート通知の仕組み**

新バージョンがリリースされたら、Slack通知やメールで各チームに知らせ、対応期限を設定します。

### 組織が大きくなった場合

複数チームが存在する場合は、プラットフォームチームやインフラチームが「依存関係の管理」を責務として持つ必要があります。これは単なる「ガイドライン作成」ではなく、CI/CDツールの設定やポリシー実装まで含みます。

## 向くケースと向かないケース

**バージョン統一を厳密に管理すべき場合**
- 複数チームが同じライブラリを使う基幹系システム
- 金銭取引や重要な状態管理を扱うドメイン
- 本番環境での予測不可能な動作が許容できないサービス

**比較的柔軟に対応できる場合**
- 単一チームが管理する独立したマイクロサービス
- ライブラリの互換性が高く、バージョン差による動作変化が少ない
- ユーザーへの影響が限定的なバックエンド処理

実務では「完全な統一」と「完全な自由」の間で、現実的なバランスを取ることになります。その判断基準は、システムの複雑さ、チーム数、リスク許容度によって変わります。
