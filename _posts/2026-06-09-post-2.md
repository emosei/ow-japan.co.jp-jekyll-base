## 現場で起きやすい認証状態の不整合

Webアプリケーションの認証処理をミドルウェアで実装する際、多くの場合「リクエスト到達時に認証情報を検証し、レスポンス返却時にセッション情報を更新する」という二段階の流れになります。この設計自体は自然ですが、エラーが発生したパスでは想定外の状態不整合が生じやすいポイントです。

実務では、こういった不整合が本番環境で顕在化するのは、アクセス権限の喪失、セッション期限切れ、外部認証サービスの一時的な応答遅延といった「例外的だが完全には無視できない」シナリオが重なったときです。その結果、一部のユーザーだけが「認証状態は有効だと思っているが、実際は無効」という状態に陥ることがあります。

本記事では、このパターンの問題がなぜ起きるのか、どう検証するのか、実装上どこに気をつけるべきかを、実際の試行を通じて整理します。

## ミドルウェアの状態管理が複雑になる仕組み

認証ミドルウェアが「リクエスト前処理」と「レスポンス後処理」で異なる状態を管理する場合、以下のような流れになります。

**通常のリクエストフロー:**
1. クライアントからリクエスト到達
2. ミドルウェアがセッショントークンを検証（前処理）
3. 検証結果をコンテキストに格納
4. ハンドラー実行
5. ミドルウェアがセッション有効期限を延長（後処理）
6. レスポンス返却

ここで問題は、ハンドラー実行中に例外が発生したときの動作です。例えば、データベース接続エラーや外部API呼び出しのタイムアウトが起きた場合、ハンドラーは例外をスローするか、エラーレスポンスを返します。

その際、ミドルウェアの「後処理」が実行されるかどうかは、フレームワークの例外ハンドリング仕様に依存します。もし後処理がスキップされれば、セッション有効期限の延長が行われず、次のリクエストで「前回のリクエストは失敗したのにセッションの有効期限が延長されなかった」という矛盾が生じます。

## 検証環境と試行内容

この問題の実態を確認するため、Node.js + Express を用いて簡単な検証環境を構築しました。

**環境構成:**
- Express 4.18
- セッションストレージ: メモリ内（テスト用）
- 認証ミドルウェア: カスタム実装
- エラーハンドリング: 複数パターン

**検証シナリオ:**
- 正常なリクエスト → 認証成功 → ハンドラー成功 → レスポンス返却
- 認証成功 → ハンドラーでエラー発生 → エラーレスポンス返却
- 認証成功 → ハンドラーで例外スロー → フレームワークが例外キャッチ

以下は、ミドルウェアが状態を持つ実装例です。

```javascript
const authMiddleware = (req, res, next) => {
  // 前処理：認証検証
  const token = req.headers.authorization;
  const session = sessionStore.get(token);
  
  if (!session || session.expiresAt < Date.now()) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  
  // セッション情報をリクエストコンテキストに保存
  req.user = session.user;
  req.sessionToken = token;
  
  // 後処理用フラグをセット
  let sessionUpdated = false;
  
  // ハンドラー実行後に呼ばれるコールバック
  const originalSend = res.send;
  res.send = function(data) {
    // 後処理：セッション有効期限を延長
    if (!sessionUpdated) {
      const session = sessionStore.get(req.sessionToken);
      if (session) {
        session.expiresAt = Date.now() + 3600000; // 1時間延長
        sessionUpdated = true;
      }
    }
    return originalSend.call(this, data);
  };
  
  next();
};
```

この実装では、`res.send()` をオーバーライドして後処理を行っています。しかし、ハンドラーで例外がスローされた場合、`res.send()` が呼ばれず、セッション有効期限の延長が行われません。

## 検証結果：エラーパスでの不整合

実際に複数のエラーシナリオをテストした結果、以下の不整合が確認されました。

**シナリオ1: ハンドラー内でエラーレスポンスを返す場合**
```javascript
app.get('/api/data', authMiddleware, (req, res) => {
  const data = fetchData(); // データベースエラー発生
  if (!data) {
    return res.status(500).json({ error: 'Internal Server Error' });
  }
  res.send(data);
});
```
この場合、`res.send()` が呼ばれるため、セッション有効期限は延長されます。**不整合なし。**

**シナリオ2: ハンドラー内で例外がスローされる場合**
```javascript
app.get('/api/data', authMiddleware, (req, res) => {
  throw new Error('Unexpected error'); // 例外スロー
});
```
この場合、Express の例外ハンドラーが介入し、`res.send()` がスキップされることがあります。結果として、セッション有効期限が延長されず、次のリクエストで「認証状態の矛盾」が生じます。**不整合あり。**

**シナリオ3: 非同期処理内で例外が発生する場合**
```javascript
app.get('/api/data', authMiddleware, async (req, res) => {
  const result = await fetchDataAsync(); // Promise reject
  res.send(result);
});
```
この場合、Express 4.x では未処理の Promise rejection が例外として扱われ、同じく `res.send()` がスキップされる可能性があります。**不整合あり。**

## 不整合が起きる根本的な理由

問題の核心は、**ミドルウェアが「ハンドラーの成功を前提とした」状態管理をしている**ことです。

- 前処理では「このリクエストは認証済み」という状態を確立
- 後処理では「セッションを更新して次のリクエストに備える」という状態を進める
- しかし、その間にハンドラーが失敗すると、ミドルウェアの想定が崩れる

さらに、フレームワークの例外ハンドリングの仕様によって、後処理が「条件付きで実行される」という非決定性が生まれます。これが、環境やフレームワークバージョンによって挙動が異なる原因になります。

## 実装上の改善方針

この問題に対応するには、以下のような設計変更が有効です。

**方針1: 後処理を明示的に実行する**

Express では `res.on('finish')` イベントを使うことで、ハンドラーの成否に関わらず、レスポンス送信直前に処理を実行できます。

```javascript
const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization;
  const session = sessionStore.get(token);
  
  if (!session || session.expiresAt < Date.now()) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  
  req.user = session.user;
  req.sessionToken = token;
  
  // finish イベントで必ず実行される
  res.on('finish', () => {
    const session = sessionStore.get(req.sessionToken);
    if (session) {
      session.expiresAt = Date.now() + 3600000;
    }
  });
  
  next();
};
```

**方針2: 前処理と後処理を分離する**

認証検証と状態更新を異なるレイヤーで管理し、状態更新は必ず成功パスでのみ実行するよう明示的に制御します。

```javascript
const validateAuth = (req, res, next) => {
  // 認証検証のみ
  const token = req.headers.authorization;
  const session = sessionStore.get(token);
  
  if (!session || session.expiresAt < Date.now()) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  
  req.user = session.user;
  req.sessionToken = token;
  next();
};

const updateSession = (req, res, next) => {
  // 成功パスでのみ実行される専用ミドルウェア
  const session = sessionStore.get(req.sessionToken);
  if (session) {
    session.expiresAt = Date.now() + 3600000;
  }
  next();
};

// 使用例
app.get('/api/data', validateAuth, handler, updateSession);
```

**方針3: セッション更新を非同期で後処理する**

ハンドラー完了後、セッション更新を非同期タスクとして実行し、クライアントへのレスポンス速度には影響させません。ただし、更新漏れのリスクがあるため、定期的な同期メカニズムが必要です。

## 実務投入時の確認ポイント

この問題に対応する際、以下の点を確認しておくと、後々の運用トラブルを減らせます。

- **フレームワークの例外ハンドリング仕様**: 使用しているフレームワークで、ハンドラー内の例外がどう扱われるか、ドキュメントを確認する
- **エラーレスポンスの返却パターン**: エラー時に `res.send()` が必ず呼ばれるか、それとも例外ハンドラーが介入するか
- **非同期処理の扱い**: async/await を使う場合、Promise rejection が例外として扱われるかどうか
- **セッション更新の必須性**: 本当に「すべてのリクエスト後」にセッションを更新する必要があるか、設計段階で検討する
