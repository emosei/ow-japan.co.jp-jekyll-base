## はじめに──「後から気づく」ログセキュリティの実態

開発段階では気づかず、本番運用が数ヶ月経ってから発覚する。そんな場面が少なくありません。

Webアプリのミドルウェアでリクエスト・レスポンスをログに記録する際、認証情報（トークン、セッションID、APIキーなど）がそのままログに出力されていることに、運用チームが事後的に気づくケースです。

開発時は機能テストや動作確認が優先され、ログ内容の精査は後回しになりやすい。さらに、ミドルウェアは複数の層で処理が重なるため、「どこでどの情報が記録されるか」が設計段階では見落とされやすいのです。

本記事では、ロギングミドルウェアの構成パターンを検証し、認証情報の漏洩リスクを段階的に低減するマスキング戦略を実装レベルで整理します。

## ミドルウェアのロギング位置が問題になる理由

### ロギング層が複数存在する現実

多くのWebアプリケーションでは、以下のような層でロギングが行われます。

- HTTPミドルウェア（リクエスト全体の記録）
- 認証ミドルウェア（認証結果の記録）
- ビジネスロジック層（処理内容の記録）
- エラーハンドリング（例外とスタックトレース）

各層が独立して動作するため、開発者が「どの層でどの情報が出力されるか」を把握しきれないまま本番に進むことがあります。特に、HTTPミドルウェアで「リクエストボディ全体」を記録する実装は、一見デバッグに便利に見えるため、セキュリティレビューを通過してしまうことも多いのです。

### 認証情報がログに含まれやすい箇所

実装の現場では、以下のパターンが頻繁に見られます。

```javascript
// 典型的な「危険なログ出力」の例
app.use((req, res, next) => {
  console.log('Request:', {
    method: req.method,
    path: req.path,
    headers: req.headers,  // ← Authorization ヘッダが含まれる
    body: req.body,        // ← パスワード、APIキーが含まれる
  });
  next();
});
```

このコードは開発環境では問題ありませんが、本番環境では以下のリスクが生じます。

- ログを閲覧できる権限を持つ人員が増えると、認証情報が第三者に見られる可能性
- ログファイルがバックアップされ、長期保存される場合、漏洩の窓口が広がる
- ログ集約サービス（CloudWatch、Splunk など）を使う場合、アクセス権限の管理が複雑化

## 検証：段階的マスキング戦略の実装と効果

### 検証の前提条件

- Express.js を使用したNode.js アプリケーション
- 本番環境を想定し、ログは標準出力に出力
- 認証方式は Bearer Token（JWT）とBasic認証の両方を対象
- マスキング戦略は3段階に分けて実装・比較

### 第1段階：ホワイトリスト方式（許可する情報のみ記録）

最も安全なアプローチは、**記録してよい情報を明示的に指定する**ことです。

```javascript
const allowedHeaders = ['content-type', 'user-agent', 'accept-language'];
const sensitiveHeaderPatterns = ['authorization', 'cookie', 'x-api-key'];

function filterHeaders(headers) {
  const filtered = {};
  
  Object.keys(headers).forEach(key => {
    const isBlocked = sensitiveHeaderPatterns.some(pattern => 
      key.toLowerCase().includes(pattern)
    );
    
    if (!isBlocked && allowedHeaders.includes(key.toLowerCase())) {
      filtered[key] = headers[key];
    }
  });
  
  return filtered;
}

app.use((req, res, next) => {
  const safeLog = {
    method: req.method,
    path: req.path,
    headers: filterHeaders(req.headers),
    // body は記録しない
  };
  
  console.log('Request:', safeLog);
  next();
});
```

**利点**：
- 記録する情報が明確で、監査しやすい
- デフォルト安全（ブロックリストの漏れが少ない）

**制約**：
- デバッグ時にボディ情報が必要な場合、別途ログ出力を用意する必要がある
- ビジネスロジックに応じて許可リストをカスタマイズする手間

### 第2段階：マスキング方式（情報を部分的に隠す）

完全に除外できない情報については、**マスキング（一部を隠す）**という選択肢があります。

```javascript
function maskToken(token) {
  if (!token || token.length < 10) return '***';
  return token.substring(0, 3) + '***' + token.substring(token.length - 3);
}

function maskSensitiveData(body) {
  const masked = { ...body };
  
  const sensitiveFields = ['password', 'apiKey', 'token', 'secret', 'creditCard'];
  
  sensitiveFields.forEach(field => {
    if (masked[field]) {
      masked[field] = maskToken(masked[field]);
    }
  });
  
  return masked;
}

app.use((req, res, next) => {
  const safeLog = {
    method: req.method,
    path: req.path,
    body: maskSensitiveData(req.body),
    authHeader: req.headers.authorization 
      ? maskToken(req.headers.authorization.replace('Bearer ', ''))
      : undefined,
  };
  
  console.log('Request:', safeLog);
  next();
});
```

**利点**：
- デバッグ時に値の一部を確認できる（本当にトークンが送られているか、など）
- 完全除外よりも情報損失が少ない

**制約**：
- マスキング対象フィールドの定義が漏れると効果なし
- ネストされたオブジェクトに対応する場合、実装が複雑化

### 第3段階：環境別制御（本番では除外、開発ではマスキング）

現実的には、開発環境と本番環境で異なるログレベルを適用することが有効です。

```javascript
function createRequestLogger() {
  const isProduction = process.env.NODE_ENV === 'production';
  
  return (req, res, next) => {
    let logData = {
      method: req.method,
      path: req.path,
      timestamp: new Date().toISOString(),
    };
    
    if (isProduction) {
      // 本番：認証情報は一切記録しない
      logData.headers = filterHeaders(req.headers);
      // body は記録しない
    } else {
      // 開発：マスキングして記録
      logData.headers = req.headers;
      logData.body = maskSensitiveData(req.body);
    }
    
    console.log('Request:', JSON.stringify(logData));
    next();
  };
}

app.use(createRequestLogger());
```

**利点**：
- 本番環境では最小限のリスク
- 開発環境では十分なデバッグ情報を確保

**制約**：
- 環境変数の設定を確実にする必要がある
- ステージング環境でのテストが重要

## 実務投入時の追加確認項目

### 1. レスポンスログも同じルールを適用する

リクエストだけでなく、レスポンスにも認証情報が含まれないか確認が必要です。特に、認証API（ログイン）のレスポンスには新しいトークンが含まれます。

```javascript
app.use((req, res, next) => {
  const originalJson = res.json;
  
  res.json = function(data) {
    const safeData = maskSensitiveData(data);
    console.log('Response:', safeData);
    return originalJson.call(this, data);  // 実際のレスポンスは変更しない
  };
  
  next();
});
```

### 2. スタックトレースに機密情報が含まれないか

エラーハンドリングでスタックトレースを出力する際、ローカル変数に認証情報が含まれていないか確認してください。

```javascript
app.use((err, req, res, next) => {
  // スタックトレースは本番では出力しない
  if (process.env.NODE_ENV === 'production') {
    console.error('Error:', { message: err.message, code: err.code });
  } else {
    console.error('Error:', err);
  }
  
  res.status(500).json({ error: 'Internal Server Error' });
});
```

### 3. ログ集約サービスへの送信ルール

CloudWatch、Datadog、Splunk などのサービスを使う場合、送信前にマスキングを行うか、サービス側の設定でフィルタリングするか、事前に決めておきます。

```javascript
// Bunyan を使った例
const bunyan = require('bunyan');

const log = bunyan.createLogger({
  name: 'app',
  serializers: {
    req: (req) => ({
      method: req.method,
      path: req.path,
      headers: filterHeaders(req.headers),
    }),
  },
});

app.use((req, res, next) => {
  log.info({ req }, 'Incoming request');
  next();
});
```

## 推奨される採用パターン

### 小規模チーム・スタートアップ向け

ホワイトリスト方式 + 環境別制御の組み合わせがシンプルです。実装の負担が少なく、セキュリティレビューも簡潔に進みます。

### 大規模システム・複数チーム向け

マスキング方式 + ログ集約サービスの設定で、一元管理を推奨します。ログの保持期間、アクセス権限、監査ログなどの運用負荷を考慮してください。

### 既存システムへの後付け

段階的に対応する場合、まずは本番環境でのリクエストボディ記録を止め、次にマスキングを追加、最終的にホワイトリスト方式に移行するステップが現実的です。

## おわりに

ロギングミドルウェアのセキュリティ問題は、開発時には見えにくく、運用段階で初めて顕在化することが多いです。
