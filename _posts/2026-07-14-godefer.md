## 現場で起こる「想定外」

データベース接続、ファイルハンドル、ロック解放といったリソース管理で`defer`を使うのは、Go開発では一般的な手法です。しかし複数のリソースを扱う関数で`defer`を複数登録すると、実際の解放順序が直感と異なるために、思わぬバグや不安定性につながることがあります。

「コードを上から読むと、リソースAを取得してから、リソースBを取得する。だから解放もAが先だろう」という判断が、実行時には逆になってしまう。これは言語仕様としては正しい動作ですが、実装者の意図と乖離しやすい落とし穴です。

本記事では、この動作の仕組みと、現場で判断を誤らないための確認ポイントを検証します。

## `defer`の逆順実行が起こる仕組み

Goの`defer`は、それが登録された関数を抜ける際に、**登録順の逆順**で実行されます。これはスタック構造で管理されているためです。

```go
func processData() error {
    // リソースA取得
    connA, err := openConnection("A")
    if err != nil {
        return err
    }
    defer connA.Close()
    
    // リソースB取得
    connB, err := openConnection("B")
    if err != nil {
        return err
    }
    defer connB.Close()
    
    // 処理
    return nil
}
```

このコードでは、`connA.Close()`が先に登録されても、関数を抜ける際には**`connB.Close()`が先に実行され、その後`connA.Close()`が実行されます**。

一見すると自然に思えるかもしれません。しかし、以下のような場面では問題になります。

### 依存関係がある場合

リソースBがリソースAに依存している場合、Aを先に閉じるとBの操作が失敗する可能性があります。

```go
func transferData() error {
    // ソース接続
    source, err := db.Connect("source")
    if err != nil {
        return err
    }
    defer source.Close()
    
    // ソースから取得したハンドルを使って、
    // 宛先接続を確立
    dest, err := source.GetRemoteConnection()
    if err != nil {
        return err
    }
    defer dest.Close()
    
    // データ転送
    return source.Transfer(dest)
}
```

この場合、`dest.Close()`が先に実行されるため、正常に閉じられます。しかし逆のパターンでは危険です。

## 検証: 実際の実行順序を確認する

仕組みを理解するために、簡単な検証コードで動作を確認しましょう。

```go
package main

import (
    "fmt"
)

func testDeferOrder() {
    fmt.Println("=== defer実行順序の検証 ===")
    
    fmt.Println("Step 1: リソースA取得")
    defer fmt.Println("Step 1-cleanup: リソースA解放")
    
    fmt.Println("Step 2: リソースB取得")
    defer fmt.Println("Step 2-cleanup: リソースB解放")
    
    fmt.Println("Step 3: リソースC取得")
    defer fmt.Println("Step 3-cleanup: リソースC解放")
    
    fmt.Println("処理実行中...")
}

func main() {
    testDeferOrder()
}
```

実行結果は以下の通りです。

```
=== defer実行順序の検証 ===
Step 1: リソースA取得
Step 2: リソースB取得
Step 3: リソースC取得
処理実行中...
Step 3-cleanup: リソースC解放
Step 2-cleanup: リソースB解放
Step 1-cleanup: リソースA解放
```

登録順は1→2→3ですが、実行順は3→2→1になります。これは意図的な設計であり、通常のスタック処理では自然な動作ですが、複数のリソースを扱う際には**設計時に明示的に意識する必要があります**。

## 現場での判断ポイント

### 1. 依存関係を図示する

複数のリソースを扱う関数では、まず依存関係を整理してください。

- リソースAが独立している
- リソースBはAに依存している
- リソースCはAとBの両方に依存している

このような構造が明確になれば、`defer`の登録順をどう組むべきかが見えやすくなります。

### 2. 逆順実行を前提に設計する

`defer`の逆順実行は**変わらない仕様**です。これを前提に、依存関係を持つリソースはあとに取得して、あとに登録するようにします。

```go
func processWithDependency() error {
    // 独立したリソース（先に取得）
    baseConn, err := getBase()
    if err != nil {
        return err
    }
    defer baseConn.Close()
    
    // 依存したリソース（あとに取得）
    childConn, err := baseConn.CreateChild()
    if err != nil {
        baseConn.Close()
        return err
    }
    defer childConn.Close()
    
    return nil
}
```

この場合、`childConn.Close()`→`baseConn.Close()`の順で実行され、依存関係が正しく保たれます。

### 3. 複雑な場合は明示的に処理する

3つ以上のリソースがあり、依存関係が複雑な場合、`defer`に頼らず明示的に処理する方が安全です。

```go
func complexProcessing() error {
    resA, err := acquireA()
    if err != nil {
        return err
    }
    
    resB, err := acquireB()
    if err != nil {
        resA.Close()
        return err
    }
    
    resC, err := acquireC()
    if err != nil {
        resB.Close()
        resA.Close()
        return err
    }
    
    defer func() {
        resC.Close()
        resB.Close()
        resA.Close()
    }()
    
    // 処理
    return nil
}
```

無名関数で`defer`を一つ登録し、その中で明示的に順序を制御する手法です。可読性も向上します。

### 4. エラーハンドリング時の取りこぼし

エラーが発生した場合、`defer`は実行されますが、部分的に取得したリソースの処理を忘れやすくなります。

```go
func problematicCode() error {
    resA, err := acquireA()
    if err != nil {
        return err  // deferは実行されない（関数を抜けるため実行される）
    }
    defer resA.Close()
    
    resB, err := acquireB()
    if err != nil {
        return err  // resA.Close()は実行される
    }
    defer resB.Close()
    
    return nil
}
```

実は上記は問題ありません。Goでは`defer`は関数を抜ける際に必ず実行されます。しかし、複数の`defer`がある場合、エラーパスで意図しない順序になることがあります。

## 実務投入時の確認項目

- **リソースの依存関係図を引く**：3つ以上のリソースがあれば、紙に書いて依存関係を確認
- **エラーパスをテストする**：各リソース取得時にエラーを注入し、リソースリークがないか確認
- **逆順実行を前提にした設計になっているか**：コードレビューで「なぜこの順序なのか」を問い直す
- **複雑な場合は無名関数で明示化**：3つ以上のリソースがあれば、`defer`を1つにまとめることを検討

## どんな案件で注意が必要か

- データベース接続と外部API呼び出しを組み合わせるシステム
- ファイル操作とロック管理が絡むバッチ処理
- マイクロサービス間の複数接続を扱うミドルウェア

小規模な関数なら問題になりにくいですが、複数のチームが関わる保守性が求められるコードベースでは、この落とし穴が実装時の判断を鈍らせやすいです。
