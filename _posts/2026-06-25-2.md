## 短いサイクルと長い検証のズレ

アジャイル開発で2週間スプリントへの移行を決めた直後、よく聞く悩みがあります。「デプロイ頻度は上がったのに、本番環境での問題が増えた」というものです。

とくに基幹系やデータを扱うシステムでは、この矛盾が顕著になります。アプリケーションのコードは2週間ごとにリリースできるようになったのに、データベーススキーマの変更、既存データの移行、ロールバック時の復旧方法は、事前にそこまで詰められていない。結果として、本番でデプロイ失敗が起きたときの対応が泥沼化するケースが増えるのです。

問題は「アジャイルが悪い」ではなく、短いリリースサイクルに合わせて、データ移行戦略まで事前に決めておく作業が抜け落ちやすいという点にあります。

## なぜ事前検証が後回しになるのか

スプリント計画の段階では、機能開発のタスクが優先されます。データ移行やスキーマ変更は「付随する作業」と見なされ、実装者に任されることが多い。結果として以下のような状況が生まれます。

**実装フェーズでの判断遅延**
- スプリント中盤になって「このテーブルのカラムを追加する必要がある」という判断が出る
- 既存データとの互換性を確認する時間がない
- 本番環境でのデータ量が開発環境と大きく異なることに気づくのが遅い

**移行スクリプトの検証不足**
- マイグレーションスクリプトは実装の最後に書かれる
- 開発環境では数千件のテストデータで動作確認、本番では数百万件のデータがある
- 実行時間、ロック時間、メモリ使用量の見積もりが甘くなる

**ロールバック計画の曖昧さ**
- 「何か問題が起きたら前のバージョンに戻す」という漠然とした想定
- 実際には、データ移行後のロールバックは単なるコード巻き戻しでは済まない
- 移行済みデータをどう扱うか、どのタイミングで何を復旧するかが決まっていない

## 本番での復旧コストが跳ね上がる理由

デプロイ失敗時の対応を具体的に考えてみましょう。

**シナリオ1: スキーマ変更後にアプリケーション起動失敗**

新しいカラムを追加するマイグレーションを実行した直後、アプリケーションが起動しない。理由は、新カラムのNOT NULL制約が予期しない形で既存レコードに影響を与えているケース。

この場合、単に「前のバージョンのコードに戻す」では解決しません。データベーススキーマはすでに変更されています。戻したコードが古いスキーマを参照しようとして再度エラーが出ます。

対応としては以下の選択肢が必要になります。
- スキーマを前の状態に戻すロールバックスクリプトを実行（本番で初めてテスト）
- または、新しいスキーマのまま、アプリケーション側の対応を急いで実装
- その間、サービスは停止したままになる

**シナリオ2: データ移行スクリプトが本番で予想外の時間がかかる**

開発環境では5分で終わるマイグレーションが、本番環境では30分かかる。その間、テーブルがロックされてユーザーリクエストが詰まる。

事前に本番データ量での実行時間を見積もっていなければ、対応策は限られます。
- スクリプトを途中で止めて、部分的なロールバックを試みる（データ整合性が破壊される可能性）
- サービスを停止して、完了を待つ（ユーザーへの影響が大きくなる）

**シナリオ3: ロールバック後のデータ不整合**

マイグレーション後に問題が見つかり、スキーマを前の状態に戻した。しかし、その間に新しいアプリケーションコードが書き込んだデータが存在する。戻す際にそのデータをどう扱うか、事前に決まっていなければ、手作業での復旧が必要になります。

この種の対応は本番環境で急いでやるものではなく、落ち着いて検証できる環境で事前に計画しておくべきものです。

## 事前に決めておくべき項目

デプロイ前に、以下を明確にしておきましょう。

**1. マイグレーション実行の前提条件**
```
- 対象テーブルのレコード数と容量
- 実行予定時間（本番データ量での実測値）
- 実行中のロック対象とその影響範囲
- サービス停止が必要か、ダウンタイムなしで実行できるか
```

**2. ロールバック戦略の明文化**
```
- ロールバックスクリプトの実装と事前テスト
- データベーススキーマを戻す手順
- 移行後に書き込まれたデータの扱い（削除か保持か）
- ロールバック後のアプリケーション検証方法
```

**3. 実行時間超過時の判断基準**
```
- N分経ってもマイグレーションが終わらない場合は中止する
- その場合の復旧手順
- 誰が判断を下すか
```

## 実装例：安全なマイグレーション設計

小規模チームでも実践できる考え方を示します。

まず、マイグレーションを「即座に反映する部分」と「段階的に反映する部分」に分けます。

```sql
-- Phase 1: 新カラムを追加（NOT NULL制約なし）
ALTER TABLE orders ADD COLUMN status_v2 VARCHAR(50);

-- Phase 2: 既存データを段階的に更新（バッチ処理）
UPDATE orders SET status_v2 = CASE WHEN status = 'pending' THEN 'pending_v2' ... END
WHERE id BETWEEN 1 AND 10000;
-- 以降、バッチで分割実行

-- Phase 3: アプリケーション側で新カラムを使用開始
-- Phase 4: 旧カラムへの参照を削除
-- Phase 5: 旧カラムを削除（複数デプロイサイクル後）
```

このアプローチにより、各フェーズでロールバック可能な状態を保ちながら進められます。

また、ロールバックスクリプトも同時に用意します。

```sql
-- ロールバック手順
-- 1. アプリケーションを前のバージョンに戻す
-- 2. status_v2カラムへの参照を削除したコードを確認
-- 3. 必要に応じてstatus_v2カラムを削除
ALTER TABLE orders DROP COLUMN status_v2;
```

## 運用上の落とし穴

**デプロイ前チェックリストの不完全さ**

「マイグレーション実行時間を本番データで測定したか」という項目が抜けやすいです。テストデータではなく、本番と同じ規模のデータで実行時間を確認する時間を、スプリント計画に組み込む必要があります。

**ステージング環境の軽視**

ステージング環境で「コードが動く」ことだけ確認して、データ移行までは本番で初めてテストするというケースが多いです。ステージング環境を本番と同じデータ量で定期的に更新し、マイグレーション検証に使う仕組みが重要です。

**ロールバック計画の口頭伝承**

「問題が起きたら○○さんに連絡」という依存は避けるべきです。ロールバック手順はドキュメント化し、複数の人が実行できるようにしておきましょう。

## 小規模チームでの現実的な導入ステップ

1. **スプリント計画の段階で、データ移行が必要かスクリーニング**
   - 新機能に伴うスキーマ変更があるかを早期に把握

2. **マイグレーション設計を実装の前に進める**
   - コード実装と並行ではなく、先行して設計・検証

3. **本番データ量での実行時間を測定**
   - ステージング環境で事前テスト

4. **ロールバック手順を明文化し、実装者以外が読める形に**
   - オンコール対応時に参照できるドキュメント

5. **デプロイ当日は、マイグレーション実行担当者と復旧担当者を分ける**
   - 問題発生時に判断を下す人と実行する人の役割分離

短いリリースサイクルは強みですが、データを扱うシステムでは「速さと安全性のバランス」を事前に設計する作業が省けません。その作業をスプリントの中に組み込むことが、本番での復旧コストを大きく下げる鍵になります。
