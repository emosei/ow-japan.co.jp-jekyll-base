## 現場で繰り返される「型か柔軟性か」の選択

システム開発の初期段階で、言語選択の議論が出ます。「堅牢性重視なら型がしっかりした言語」「素早く試行錯誤するなら動的言語」──こうした判断基準は確かに存在します。ただ、実際の案件では初期判断と後年の要件が大きくずれることが多いのです。

特に予約管理や在庫管理といった基幹系システムでは、立ち上げ時の「確定した要件」が、3年後には「え、こんなことできます？」という機能追加要望に変わっていきます。そこで型による制約が活躍するのか、それとも足かせになるのか──この検証は、言語選択の後悔を減らすために欠かせません。

## 検証の設定：予約システムの簡略モデル

ここでは、旅行予約の基本的な流れをシミュレートしてみました。初期要件は「ユーザーが日付と人数を指定して予約を確定する」という単純なものです。

**初期要件:**
- ユーザーID、予約日時、人数、宿泊施設IDを保持
- 予約状態は「仮予約」「確定」「キャンセル」の3状態
- 確定時に料金を計算して記録

**後年の追加要件（実際に起こりやすい）:**
- 複数の宿泊施設を同時予約できるように
- キャンセル料金ルールが施設ごと、時期ごとに異なるように
- 顧客グループ向けの特別料金体系を追加
- 予約確定後に日程変更を可能に（手数料あり）

この流れで、Haskellのような関数型言語と、Pythonのような動的言語がどう対応するのかを比較しました。

## 関数型言語（Haskell）での実装：型による保証と修正コスト

Haskellで初期要件を実装する場合、型定義から始まります。

```haskell
data ReservationStatus = Tentative | Confirmed | Cancelled
  deriving (Eq, Show)

data Reservation = Reservation
  { userId :: Int
  , facilityId :: Int
  , checkInDate :: Date
  , nights :: Int
  , status :: ReservationStatus
  , totalPrice :: Maybe Double
  } deriving (Eq, Show)

confirmReservation :: Reservation -> Double -> Either String Reservation
confirmReservation res price
  | status res == Tentative = Right res { status = Confirmed, totalPrice = Just price }
  | otherwise = Left "Invalid state transition"
```

このアプローチの強みは明確です。

- 無効な状態遷移がコンパイル時に検出される
- `totalPrice` が `Maybe Double` 型なので、料金なしで確定した予約が存在しえない
- 3年後のコードレビューでも、型シグネチャを見れば関数の役割が一目瞭然

ところが、「複数施設の同時予約」要件が入ると、話が変わります。

```haskell
-- 修正後：複数施設対応
data Reservation = Reservation
  { userId :: Int
  , facilities :: [FacilityBooking]  -- ここを変更
  , status :: ReservationStatus
  , totalPrice :: Maybe Double
  }

data FacilityBooking = FacilityBooking
  { facilityId :: Int
  , checkInDate :: Date
  , nights :: Int
  }
```

この変更は、`Reservation` を扱うすべての関数に波及します。パターンマッチ、リスト操作、料金計算ロジック──修正箇所が連鎖的に増えるのです。さらに「キャンセル料金が施設ごとに異なる」という要件が加わると、料金計算の型も複雑化します。

```haskell
-- キャンセル料金計算：施設ごとのルールを型で表現
data CancellationPolicy = CancellationPolicy
  { daysBeforeCheckIn :: Int
  , chargePercentage :: Double
  }

calculateCancellationFee :: Reservation -> CancellationPolicy -> Double
-- ここで各施設のポリシーをどう管理するか？
-- 型を増やすか、リストを増やすか、判断が分岐する
```

実務では、こうした修正が「コンパイルが通るまで修正作業が続く」という状況を生みます。型安全性の恩恵は受けられますが、**変更の敏捷性は低下する傾向**があります。

## 動的言語（Python）での実装：初期の柔軟性と後年の技術債

同じ要件をPythonで実装すると、初期段階は驚くほど簡潔です。

```python
class Reservation:
    def __init__(self, user_id, facility_id, check_in_date, nights):
        self.user_id = user_id
        self.facility_id = facility_id
        self.check_in_date = check_in_date
        self.nights = nights
        self.status = "tentative"
        self.total_price = None
    
    def confirm(self, price):
        if self.status == "tentative":
            self.status = "confirmed"
            self.total_price = price
        else:
            raise ValueError("Invalid state transition")
```

複数施設対応への変更も、属性を追加するだけで済みます。

```python
class Reservation:
    def __init__(self, user_id, facilities, check_in_date, nights):
        self.user_id = user_id
        self.facilities = facilities  # リストで複数施設対応
        self.status = "tentative"
        self.total_price = None
```

変更が容易なため、要件が頻繁に変わる初期段階では開発速度が勝ります。しかし、運用が進むと別の問題が浮上します。

- 2年目：「あれ、この予約オブジェクトに `cancellation_fee` 属性があるのは、どの機能が追加したんだっけ」
- 3年目：「キャンセル料金計算で、ある顧客だけ計算結果が違う。どこで特別処理してるんだ」
- 4年目：「予約確定後に日程変更を許可したが、その際に料金再計算がされていない場合がある」

こうした問題は、型がないために**実行時エラーか、サイレント・バグ**として現れます。テストで検出できますが、テストの網羅性に依存することになります。

## 実務投入時の追加確認ポイント

どちらを選ぶにしても、以下を事前に確認しておくと判断がしやすくなります。

**要件変更の頻度と規模の見積もり**
- 初期要件が確定的か、それとも探索的か
- 3年以内に「想定外の機能追加」が起こりそうか
- 変更時に「既存コードへの影響範囲」がどれくらい許容できるか

**チームの技術水準**
- 関数型言語の型システムを使いこなせるメンバーがいるか
- 型エラーの修正に時間をかけられる環境か
- 動的言語で大規模コードを保守した経験があるか

**運用・保守の期間**
- 開発終了後、何年保守が続くのか
- 保守フェーズで頻繁な小変更が入るのか
- ドキュメント整備にどれくらい力を入れられるか

## 現場での使い分けの実際

小規模チームの案件では、初期段階で「要件が本当に確定しているのか」を徹底的に確認することが、言語選択より重要です。

もし「初期要件は確定的で、拡張性より堅牢性重視」なら、関数型言語の型による保証は大きな武器になります。ただし、その場合は**変更要件に対応するための設計戦略**（例えば、型クラスやレコード型の工夫）を最初から組み込んでおく必要があります。

逆に「要件が流動的で、試行錯誤が避けられない」なら、動的言語で素早く試しながら進める方が現実的です。ただし、**3ヶ月ごとにコード品質を測定し、技術債を意識的に返済する**ペースが必須になります。

実際には、どちらか一方に決めるのではなく、「初期段階は動的言語で要件を固める→確定後は型言語で実装」という組み合わせもあります。プロトタイプとプロダクションで言語を分ける判断も、現場では起こります。

## まとめ：選択は「今」ではなく「3年後」を見据えて

型による制約と動的言語の柔軟性は、トレードオフです。どちらが「正解」かは、案件の性質と環境次第です。重要なのは、その判断を「導入時の気分」ではなく、要件変更の頻度、チームの技術水準、保守期間を踏まえて、意識的に選ぶことです。

現場では「とりあえずPythonで始めよう」という判断も、「型安全性を確保するためHaskellで」という判断も、どちらも起こります。大事なのは、その選択が何に基づいているのか、そして3年後に「あの時の選択は正しかった」と言えるかどうかです。
