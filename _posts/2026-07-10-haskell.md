## 導入──本番環境で初めて気付く不完全なパターン

Haskellでコードレビューを受けたとき、「このパターンマッチ、全ケースを網羅していませんね」と指摘されることがあります。しかし開発環境では何度実行してもエラーが出ない。それでも本番環境に上がると、特定の入力パターンでだけ `Non-exhaustive patterns in function` という実行時エラーが発生する──こうした経験は、遅延評価言語を扱う現場では珍しくありません。

なぜこのようなことが起きるのか。そして、どうすれば事前に検証できるのか。この記事では、遅延評価がもたらす検証の盲点と、実務的な対策について掘り下げます。

## パターンマッチの不完全性が「実行時エラー」になる仕組み

### コンパイル時の警告と実行時エラーの違い

Haskellのコンパイラは、パターンマッチが網羅的でない場合、通常は警告を出します。

```haskell
data Status = Active | Inactive | Pending

describeStatus :: Status -> String
describeStatus Active = "稼働中"
describeStatus Inactive = "停止中"
-- Pending のケースがない
```

この場合、`ghc -Wall` でコンパイルすると警告が表示されます。しかし、**警告はあくまで警告です**。コンパイルは通ります。そして、もし `Pending` が実際に関数に渡されなければ、プログラムは何事もなく動作し続けます。

遅延評価の世界では、計算は必要になるまで実行されません。つまり、不完全なパターンマッチのコードが存在していても、そこに到達しなければエラーは起きないのです。

### なぜ開発環境では起きないのか

開発やテスト中は、テストケースが限定的になりやすいものです。特に以下のような場合、網羅的でないパターンマッチに到達しないまま進みます。

- テストデータが特定の値に偏っている
- エッジケースや例外的な状態を想定していない
- 複数のデータフロー経路があるが、一部のパスだけをテストしている
- 遅延評価によって、その関数が呼ばれるまで問題が顕在化しない

現場では、こうした状況が頻繁に起こります。開発環境では安定して動いていたのに、本番環境で「ある特定の条件下でだけ」エラーが発生するという現象は、遅延評価言語特有の難しさです。

## 検証の手順──網羅的でないパターンマッチを事前に見つける

### 第一段階：コンパイラの警告を厳密に読む

最初のステップは、コンパイラオプションを厳しくすることです。

```bash
ghc -Wall -Werror Main.hs
```

`-Werror` を指定すると、警告をエラーとして扱い、コンパイルを止めます。これにより、不完全なパターンマッチがあれば、開発時点で気付きやすくなります。

ただし、この方法だけでは不十分です。なぜなら、コンパイラが「このパターンは到達不可能」と判断する場合もあれば、「理論上は到達する可能性がある」と判断する場合もあるからです。

### 第二段階：型と値の領域を明確にする

パターンマッチの不完全性を減らすには、型設計の段階で工夫が必要です。

```haskell
-- 悪い例：String型で状態を表現
processOrder :: String -> String
processOrder "confirmed" = "確認済み"
processOrder "shipped" = "発送済み"
-- その他の文字列が渡された場合、エラー

-- 良い例：代数的データ型で限定する
data OrderStatus = Confirmed | Shipped | Pending
  deriving (Show, Eq)

processOrder :: OrderStatus -> String
processOrder Confirmed = "確認済み"
processOrder Shipped = "発送済み"
processOrder Pending = "保留中"
-- すべてのケースを網羅
```

代数的データ型を使うことで、コンパイラが「すべてのケースが網羅されているか」を静的に検証できるようになります。これは遅延評価言語の大きな利点です。

### 第三段階：動的検証──プロパティベーステスト

静的な型チェックだけでは、複雑なロジックの網羅性を完全には保証できません。ここで活躍するのが、プロパティベーステストです。

```haskell
import Test.QuickCheck

-- テスト対象の関数
safeLookup :: Eq a => a -> [(a, b)] -> Maybe b
safeLookup _ [] = Nothing
safeLookup key ((k, v):rest)
  | key == k = Just v
  | otherwise = safeLookup key rest

-- プロパティ：存在するキーは必ず見つかる
prop_found_if_present :: Int -> Int -> [Int] -> Bool
prop_found_if_present key val rest =
  safeLookup key ((key, val):rest) == Just val

-- 実行
-- > quickCheck prop_found_if_present
```

QuickCheckなどのツールを使うことで、ランダムに生成されたテストケースを大量に実行し、予期しない入力パターンでエラーが起きないか検証できます。

## 実務投入時の注意点

### 警告を無視する誘惑に抗う

現場では、スケジュールが厳しいとき、警告を無視してコンパイルを進めたくなる誘惑に駆られます。しかし、遅延評価言語では、その判断が本番環境での問題に直結しやすいです。

```haskell
-- 警告が出ているコード
processPayment :: PaymentMethod -> String
processPayment CreditCard = "クレジットカード決済"
processPayment BankTransfer = "銀行振込"
-- Digital Wallet のケースがない
```

このコードが決済処理に関わる場合、デジタルウォレットでの支払いが発生するまで問題は顕在化しません。本番環境で初めて実行時エラーが起きる可能性があります。

### 遅延評価の「見えない実行フロー」を意識する

遅延評価では、関数が呼ばれても、その結果が必要になるまで実際の計算は遅延されます。

```haskell
-- この関数は、result が実際に使われるまで calculateExpensive は実行されない
getResult :: Bool -> Int
getResult True = calculateExpensive
getResult False = 0
-- calculateExpensive が不完全なパターンマッチを含んでいても、
-- False のパスだけを実行するテストではエラーが見つからない
```

テストケースを設計するときは、「この関数の結果は、どのタイミングで実際に評価されるのか」を意識する必要があります。

### 段階的な導入と監視

遅延評価言語を採用する際は、以下の段階を踏むことをお勧めします。

1. **開発環境での厳密なコンパイル設定**：`-Wall -Werror` を標準にする
2. **テストカバレッジの確保**：QuickCheckなどで、予期しない入力パターンも試す
3. **本番環境でのログ監視**：実行時エラーが発生した場合、即座に対応できる体制を整える
4. **段階的なリリース**：新機能は一部のユーザーに限定してリリースし、問題がないか観察する

## 誰に向いているか、どんな案件で採用しやすいか

Haskellのような遅延評価言語は、以下のような案件では特に有効です。

- **型安全性が重要な基幹システム**：金融、決済、予約管理など、エラーが直結するドメイン
- **複雑なビジネスロジック**：型システムで不正な状態遷移を防げる
- **長期メンテナンスが前提**：静的な型情報が、将来の保守を容易にする

一方、以下のような場合は導入に慎重であるべきです。

- **スケジュールが非常に厳しい案件**：学習コストと検証の手間が増える
- **チーム内にHaskellの経験者がいない**：運用時のトラブルシューティングが難しくなる
- **頻繁な仕様変更が予想される**：型の再設計が頻繁に必要になり、かえって効率が落ちる可能性

## まとめ

遅延評価言語でのパターンマッチの不完全性が「たまにしか起きない」のは、計算が必要になるまで遅延されるという言語の性質と、テストが限定的になりやすい開発の実態が重なるためです。

対策は、コンパイラの警告を厳密に扱い、型設計で値の領域を明確にし、プロパティベーステストで動的に検証することです。これらを組み合わせることで、実行時エラーを大幅に減らせます。

ただし、完全な防止は難しいため、本番環境での監視と段階的なリリースも重要です。遅延評価言語の利点を活かしつつ、その落とし穴を認識した上で運用することが、安定したシステムづくりにつながります。
