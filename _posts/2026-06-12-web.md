## 現場で起きるミドルウェア層での判断ミス

Webアプリケーションのミドルウェア層でエラーが発生した時、その後のリクエスト処理をどう扱うかは実装者の判断次第です。「エラーが出たら即座に終了する」と「エラーを記録して処理を続ける」の二つの流れがありますが、この選択を誤ると本番環境で静かに問題が積み重なっていきます。

実務では、認証ミドルウェア、ロギングミドルウェア、レート制限ミドルウェアなど複数のミドルウェアが連鎖的に実行されます。その過程で「このエラーは本当に致命的なのか、それとも処理を続けても大丈夫なのか」を正確に判断できていないチームは多いです。結果として、本来は防ぐべき不正アクセスが通ってしまったり、逆に正常なリクエストが予期せず遮断されたりします。

今回は、ミドルウェアのエラーハンドリングにおいて「正常系（リクエスト継続）」と「異常系（早期終了）」をどう分けるのか、実装と運用の両面から検証してみました。

## ミドルウェア層のエラーを分類する視点

エラーハンドリングの方針を決める前に、発生するエラーの性質を整理する必要があります。大きく三つの視点から分けられます。

**1. リクエストの信頼性に関わるエラー**

認証トークンが不正、署名が改ざんされている、必須ヘッダが欠落している──こうしたエラーは「このリクエスト自体が信頼できない」という判定です。ここでリクエスト処理を続けると、意図しない動作や セキュリティリスクが生まれます。早期終了が正解です。

**2. 処理の進行に直結するエラー**

データベース接続失敗、外部APIのタイムアウト、キャッシュサーバが応答しない──こうしたエラーは「今この瞬間、特定の機能が使えない」という状態です。ただしリクエスト自体は悪くありません。エラーを記録しつつ、フォールバック処理やデグラデーション対応で継続するか、それとも即座に終了するかは、ビジネス要件に左右されます。

**3. 監視・ロギング層のエラー**

ログ書き込み失敗、メトリクス送信エラー、分散トレース情報の記録失敗──こうしたエラーは「観測機能が働かない」という問題であり、リクエスト処理自体の正否とは別です。ここで処理を止めると、ユーザーに見える障害になってしまいます。ログに記録して継続するべきです。

## 実装パターンの比較検証

実際にどう実装するか、いくつかのパターンを組み立てて動作を確認してみました。

### パターン1: 全エラーで即座に終了

```javascript
app.use((req, res, next) => {
  try {
    // 認証処理
    const token = req.headers.authorization;
    if (!token) {
      throw new Error('Token missing');
    }
    validateToken(token); // ここで例外発生の可能性
    
    // ロギング処理
    logRequest(req); // ここで例外発生の可能性
    
    next();
  } catch (err) {
    res.status(500).json({ error: 'Internal Server Error' });
  }
});
```

このパターンの問題点は、エラーの原因を区別していないことです。トークン検証失敗（セキュリティ上の問題）とログ書き込み失敗（監視機能の問題）を同じ扱いにしてしまい、正当なリクエストまで遮断される可能性があります。

### パターン2: エラーの種類で分岐

```javascript
app.use((req, res, next) => {
  try {
    const token = req.headers.authorization;
    if (!token) {
      return res.status(401).json({ error: 'Unauthorized' });
    }
    validateToken(token);
    next();
  } catch (err) {
    return res.status(401).json({ error: 'Invalid token' });
  }
});

app.use((req, res, next) => {
  try {
    logRequest(req);
  } catch (err) {
    // ログ失敗は記録して継続
    console.error('Logging failed:', err.message);
  }
  next();
});
```

認証ミドルウェアは失敗時に処理を止め、ロギングミドルウェアは失敗時に記録して継続させています。エラーの性質に応じた判断が入っています。

### パターン3: エラーハンドリングの統一インターフェース

より複雑な本番環境では、エラーの分類を明示的に扱うアプローチが有効です。

```javascript
class MiddlewareError extends Error {
  constructor(message, severity = 'fatal') {
    super(message);
    this.severity = severity; // 'fatal' | 'degraded' | 'ignorable'
  }
}

app.use((req, res, next) => {
  try {
    const token = req.headers.authorization;
    if (!token) {
      throw new MiddlewareError('Token missing', 'fatal');
    }
    validateToken(token);
    next();
  } catch (err) {
    if (err instanceof MiddlewareError && err.severity === 'fatal') {
      return res.status(401).json({ error: err.message });
    }
    next();
  }
});

app.use((req, res, next) => {
  try {
    logRequest(req);
  } catch (err) {
    // ロギングエラーは ignorable として扱う
    const logErr = new MiddlewareError(err.message, 'ignorable');
    console.error('Logging error (non-blocking):', logErr.message);
  }
  next();
});
```

このアプローチなら、エラーの深刻度を明示的に管理でき、ミドルウェアチェーン全体で一貫した判断が可能になります。

## 運用視点での判断基準

実装だけでなく、運用側でも「どのエラーで止めるのか」を事前に整理しておく必要があります。

**止めるべきエラー（早期終了）**

- 認証・認可の失敗
- リクエスト署名の検証失敗
- 必須ヘッダやパラメータの欠落
- CSRFトークンの不一致
- IPホワイトリスト違反

これらはセキュリティに直結しており、「疑わしきは処理しない」が原則です。

**継続すべきエラー（ログして続行）**

- ログ送信の失敗
- メトリクス収集の失敗
- 分散トレース情報の記録失敗
- オプショナルなヘッダの処理失敗

これらは「観測機能の問題」であり、ユーザーへのサービス提供を優先すべきです。

**判断が分かれるエラー（設計で決める）**

- 外部APIへの依存機能が失敗した場合
- キャッシュが応答しない場合
- データベース接続プール枯渇

これらは「ビジネス要件とSLA」で決めます。キャッシュがなくても主要機能が動くなら継続、キャッシュなしでは性能が著しく低下するなら中止、といった具合です。

## 検証環境での実装確認

複数のミドルウェアが連鎖する環境で、実際の挙動を確認しました。

**検証構成**

- Node.js + Express
- 認証ミドルウェア、ロギングミドルウェア、レート制限ミドルウェアの3層
- 各層で意図的にエラーを注入
- 正常系と異常系で、どのミドルウェアまで到達するかを追跡

**確認結果**

パターン1（全エラーで終了）では、ロギング失敗だけで正常なリクエストが遮断されました。監視機能の問題が、ユーザーへの障害に変わってしまったわけです。

パターン2（種類で分岐）では、認証失敗は適切に遮断され、ロギング失敗は記録されつつリクエストが続行されました。ただし、複数のミドルウェアが同じ分岐ロジックを持つと、保守が煩雑になります。

パターン3（severity で管理）では、エラーの分類が一元化され、新しいミドルウェア追加時も判断基準が明確でした。ただし、エラークラスの設計に手間がかかります。

## 実務投入時の追加確認ポイント

ミドルウェアのエラーハンドリング方針を決めたら、本番導入前に以下を確認してください。

- **ミドルウェアの実行順序**：認証→ロギング→ビジネスロジック、という順序なら、認証失敗時にログ処理が走らないよう設計します
- **エラーレスポンスの一貫性**：複数のミドルウェアが異なるステータスコードを返さないよう、統一フォーマットを決めます
- **監視・アラート設定**：「ロギング失敗が多発している」という異常を検知できるようにしておきます
- **段階的なデプロイ**：新しいエラーハンドリング方針は、カナリアリリースで検証してから全体展開します

## まとめ

ミドルウェア層のエラーハンドリングは、「エラーが出たら終了」という単純な判断では足りません。エラーの性質（セキュリティ、機能、監視）を分類し、リクエスト継続か早期終了かを明確に決めることで、初めて堅牢で保守性の高い設計になります。

現場では「ログが書けないからリクエストを止める」という事態が意外と起きています。エラーの分類と判断基準を事前に整理し、ミドルウェアチェーン全体で一貫した方針を持つことが、本番環境での予期しない問題を防ぐ鍵になります。
