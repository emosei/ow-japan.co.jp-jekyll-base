## 現場で起きやすい「どちらで対応するか」の曖昧さ

CORS（Cross-Origin Resource Sharing）への対応を決めるとき、チームの中で「リバースプロキシ層で設定する」と「アプリケーション層のミドルウェアで対応する」という二つの選択肢が浮かびます。どちらも正当性があるため、要件定義や設計段階では「両方やっておけば安全」という判断が生まれやすいのです。

実際の案件では、インフラ担当者がNginxやApacheで`Access-Control-Allow-Origin`ヘッダを設定し、同時にアプリケーション開発チームが別途ミドルウェアで同じヘッダを追加するというケースに遭遇します。一見すると「多層防御」に見えますが、実は予期しない矛盾や運用負荷を招きやすいのです。

本記事では、実際の構成で検証した結果をもとに、二重定義がなぜ問題になるのか、どこで判断を誤りやすいのかを整理します。

## 検証の前提と構成

### 検証目的

- リバースプロキシとアプリケーション層の両方でCORS対応を行った場合、レスポンスヘッダにどのような影響が出るのか
- キャッシュ戦略やクッキー設定と組み合わせたとき、予期しない動作が起きるのか
- 運用段階で設定変更が必要になったとき、どちらを修正すべきか判断に迷うのか

### 検証環境の構成

```
クライアント（ブラウザ）
    ↓
Nginx（リバースプロキシ）
    ↓
Node.js + Express（アプリケーション層）
```

**リバースプロキシ側の設定例：**

```nginx
server {
    listen 80;
    server_name api.example.local;

    location / {
        add_header 'Access-Control-Allow-Origin' '*' always;
        add_header 'Access-Control-Allow-Methods' 'GET, POST, OPTIONS' always;
        add_header 'Access-Control-Allow-Headers' 'Content-Type, Authorization' always;
        add_header 'Access-Control-Max-Age' '86400' always;

        proxy_pass http://localhost:3000;
    }
}
```

**アプリケーション層の設定例（Express + cors ミドルウェア）：**

```javascript
const cors = require('cors');
const express = require('express');
const app = express();

app.use(cors({
    origin: '*',
    methods: ['GET', 'POST', 'OPTIONS'],
    allowedHeaders: ['Content-Type', 'Authorization'],
    maxAge: 86400
}));

app.get('/api/data', (req, res) => {
    res.json({ message: 'Hello' });
});
```

この構成でプリフライトリクエスト（OPTIONS）とGETリクエストを送信して、レスポンスヘッダを確認しました。

## 検証結果：二重定義がもたらす問題

### 1. ヘッダの重複と矛盾

プリフライトリクエストを送ると、Nginxとアプリケーション両方から`Access-Control-Allow-Origin`が返されます。ブラウザの開発者ツールでは、最終的には一つのヘッダ値に統合されますが、実装の詳細によっては予期しない値が優先される可能性があります。

特に問題になるのは、プロキシ側では`*`を許可し、アプリケーション側では特定のオリジンのみを許可するといった**矛盾する設定**を持つ場合です。この場合、どちらの設定が有効になるかはミドルウェアの順序や実装に依存し、本番環境で初めて気づくことがあります。

### 2. キャッシュ戦略との相互作用

`Access-Control-Max-Age`がプロキシとアプリケーションで異なる値を持つと、ブラウザのプリフライトキャッシュが不安定になります。

例えば、Nginxで`Max-Age: 86400`（1日）、アプリケーションで`Max-Age: 3600`（1時間）と設定した場合、ブラウザはどちらを信頼するかが曖昧です。実際には、レスポンスヘッダの最初に現れた値を優先することが多いですが、プロキシの設定順序やミドルウェアの実行順序が変わるたびに動作が変わる可能性があります。

### 3. クッキー送信時の認証情報漏洩リスク

CORS対応でクッキーを含める場合（`credentials: 'include'`）、`Access-Control-Allow-Credentials: true`が正しく返される必要があります。

二重定義では、プロキシ側で`Credentials: true`を返しつつ、アプリケーション側で返していない（あるいはその逆）というミスマッチが起きやすいのです。この場合、ブラウザのセキュリティモデルが混乱し、クッキーが送信されない、または予期しない状態で送信されるといった挙動につながります。

## 設計判断：どこで対応すべきか

### リバースプロキシで対応するべき場合

- **複数のバックエンドサービスを統一的に保護したい**  
  マイクロサービス構成で、各サービスが独立したCORS設定を持つと管理が煩雑になります。プロキシ層で一元管理すれば、サービス側の変更が不要です。

- **セキュリティポリシーをインフラレベルで強制したい**  
  許可するオリジンを運用チームが管理する場合、アプリケーションコードの変更を避けられます。

- **プリフライトリクエストの負荷軽減**  
  プロキシで`OPTIONS`リクエストを処理して、バックエンドに到達させない構成にできます。

### アプリケーション層で対応するべき場合

- **サービスごとに異なるCORS要件がある**  
  エンドポイントによって許可するオリジンが異なる場合、ミドルウェアで細かく制御できます。

- **動的にオリジンを判定したい**  
  データベースやキャッシュから許可リストを読み込んで、実行時に判定する必要がある場合です。

- **認証・認可と組み合わせる**  
  ユーザーのロールに応じてCORS要件を変えるなど、ビジネスロジックと密接に関連する場合。

### 推奨：役割を明確に分ける

**原則として、どちらか一方で対応し、もう一方では設定しないことです。**

現場では「両方設定しておく」という判断が生まれやすいですが、それは設定の責任を曖昧にし、後々の変更や検証を複雑にします。

推奨パターンは以下の通りです：

1. **シンプな構成（単一バックエンド）**  
   → アプリケーション層で対応。コードと設定が一緒に管理でき、デプロイ時の検証が容易です。

2. **複数サービス + 統一ポリシー**  
   → リバースプロキシで対応。インフラ層で一元管理し、サービス側は設定を持たない。

3. **複数サービス + サービス固有の要件**  
   → プロキシで基本的なポリシー（許可するメソッド）を定義し、各サービスで詳細を上書きする方式。ただし、この場合も「プロキシが基本、アプリが例外」という役割分担を明確にしておく必要があります。

## 実装時の注意点

### ミドルウェアの順序を意識する

Express等のフレームワークでは、ミドルウェアの実行順序が重要です。CORS対応ミドルウェアを最初に配置し、その後の処理で追加のヘッダを付与しないようにします。

```javascript
// 良い例：CORSを最初に、その後のミドルウェアで上書きしない
app.use(cors({ origin: '*' }));
app.use(express.json());
app.use(authMiddleware);

app.get('/api/data', (req, res) => {
    res.json({ data: 'example' });
});
```

### 開発環境と本番環境で設定を分ける

開発時はプロキシを使わず、アプリケーション層だけで動作確認し、本番ではプロキシ層で対応するといった場合、設定の不一致が起きやすいです。少なくとも開発環境でもプロキシを経由させて、本番と同じ構成で検証することをお勧めします。

### テストケースを具体的に

単体テストだけでなく、実際のブラウザでプリフライトリクエストが正しく処理されるか確認します。特にクッキーを含むリクエストの場合、ブラウザのセキュリティモデルが動作するため、テストサーバーでの検証が必須です。

## 本番投入前に確認すべきポイント

- **プロキシ側の設定とアプリケーション側の設定が重複していないか**  
  構成管理ツール（Ansible、Terraformなど）で一覧化し、責任者が確認します。

- **許可するオリジンの定義が一元化されているか**  
  複数の場所で管理されていると、セキュリティポリシーの検証が困難になります。

- **キャッシュ関連のヘッダ（Max-Age、ETag等）が矛盾していないか**  
  CDNやプロキシのキャッシュ戦略と組み合わせた場合、予期しない動作が起きることがあります。

- **エラーレスポンスでもCORSヘッダが正しく返されるか**  
  認証失敗や500エラーの場合、CORS対応が漏れることがあります。

## まとめ：判断基準を持つことの重要性

CORS対応は一見シンプルに見えますが、キャッシュ戦略、認証、マイクロサービス構成など、多くの要素と相互作用します。「どこで対応するか」という設計判断は、後の運用効率と保守性に大きく影響するのです。
