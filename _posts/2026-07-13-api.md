## 現場で起きている判断の難しさ

モバイルアプリの開発・運用に携わっていると、避けられない問題があります。それは「ユーザーがアプリを更新してくれない」という現実です。

Webサービスなら、サーバー側を更新すれば全ユーザーに即座に反映されます。しかしモバイルアプリは違います。古いバージョンのアプリを使い続けるユーザーが、特に個人向けアプリでは相当な割合で存在します。App StoreやGoogle Playで強制アップデートを設定することもできますが、それは最終手段です。ユーザー体験を損なうし、ストア側の審査基準も厳しくなっています。

そうなると、バックエンド側は「古いAPIエンドポイントをいつまで維持するのか」という判断を迫られます。保守コストは増える一方で、廃止のタイミングは曖昧になりやすい。この記事では、その判断をどう立てるか、そして安全に廃止に向かうかを、実務的に整理します。

## なぜ古いエンドポイントの廃止が難しいのか

理由は単純ですが、複合的です。

**ユーザー分布の把握が難しい**

アプリのバージョン別ユーザー数を正確に把握していますか？多くの現場では、アナリティクスツールで「大体の傾向」は見えても、「何バージョン以上のアプリが何パーセント」という粒度の数字がありません。特に小規模なサービスでは、計測基盤そのものが整備されていないこともあります。

**廃止による影響の予測が立てにくい**

古いバージョンのアプリがどの程度APIを呼び出しているか、本当に誰も使っていないのか、それとも静かに使われ続けているのか。実際に廃止してみないと分からないことが多いのです。

**廃止のコスト判断が曖昧**

「このエンドポイントの保守に月に何時間かかっているのか」を正確に把握している組織は少ないでしょう。曖昧な判断のまま「とりあえず残しておこう」となりがちです。

## 廃止判断の前に整備すべき計測

廃止を検討する前に、最低限の計測を整備する必要があります。

**APIエンドポイントごとのアクセスログ分析**

古いエンドポイントへのリクエストがどの程度あるのか、そのリクエストはどのアプリバージョンから来ているのかを記録します。

```
# ログフォーマット例
timestamp | endpoint | app_version | status_code | response_time | user_id
2025-01-15T10:23:45Z | /api/v1/search | 2.3.1 | 200 | 145ms | user_12345
2025-01-15T10:23:46Z | /api/v1/search | 2.5.0 | 200 | 132ms | user_67890
2025-01-15T10:23:47Z | /api/v2/search | 3.1.0 | 200 | 98ms | user_11111
```

このログから、以下を集計します：

- エンドポイント別の1日あたりのリクエスト数
- アプリバージョン別の利用ユーザー数
- 古いバージョンのアプリからのリクエスト割合
- 過去3ヶ月、6ヶ月のトレンド

**実装上の注意**

リクエストヘッダーやユーザーエージェントからアプリバージョンを取得する場合、クライアント側で正しく送信されているかの検証が必要です。バージョン情報を偽って送信するアプリがあるかもしれませんし、古いバージョンのアプリが正しく送信していないかもしれません。

## 廃止判断の具体的な基準

計測が揃ったら、以下の基準で廃止を判断します。

### 1. アクセス量の閾値

「月間アクセス数が1000未満」「全アクセスの0.5%以下」といった数字を決めます。ここは組織の方針次第ですが、現場では以下を参考にしてください：

- **月間アクセス100未満**：廃止を強く検討すべき
- **月間アクセス100〜1000**：廃止に向けた準備段階
- **月間アクセス1000以上**：廃止判断を保留し、さらに詳細な分析が必要

### 2. ユーザー数の観点

古いバージョンのアプリを使い続けているユーザー数も重要です。「月1回使うユーザーが100人」と「月100回使う1人」では意味が違います。

```
古いバージョンのアプリユーザー数 / 全アプリユーザー数 = 廃止判断の比率
```

この比率が1%以下なら、廃止に向けた準備を開始しても問題ない可能性が高いです。

### 3. トレンドの確認

過去6ヶ月のアクセス数が一貫して減少しているか、それとも横ばいか。減少トレンドであれば、廃止のタイミングが読みやすくなります。

## 段階的な廃止戦略

いきなりエンドポイントを削除してはいけません。段階的に進めます。

### ステップ1：非推奨の明示（1〜2ヶ月）

レスポンスヘッダーに非推奨フラグを追加します。

```
HTTP/1.1 200 OK
Deprecation: true
Sunset: Wed, 15 May 2025 23:59:59 GMT
Link: </api/v2/search>; rel="successor-version"
```

これにより、API監視ツールを持っているクライアント開発者は警告を受け取ります。同時に、ドキュメントやリリースノートで「このエンドポイントは廃止予定」と明示します。

### ステップ2：レート制限の導入（2〜4週間）

古いバージョンのアプリからのアクセスに対して、段階的にレート制限を厳しくします。

```
# 古いバージョン（2.x系）からのアクセスに対して
- 初週：レート制限なし（警告ヘッダーのみ）
- 第2週：1秒あたり5リクエスト制限
- 第3週：1秒あたり2リクエスト制限
- 第4週以降：エラーレスポンス
```

この段階で、古いアプリからのリクエストが本当に減少するか、あるいは急増するかで、廃止の是非を再判断できます。

### ステップ3：エラーレスポンスへの切り替え（1〜2週間）

古いエンドポイントへのアクセスに対して、410 Gone（永続的に廃止）または 429 Too Many Requests（レート制限超過）を返すようにします。

```json
{
  "error": "endpoint_deprecated",
  "message": "このAPIエンドポイントはサポート終了しました。新しいバージョンをご利用ください。",
  "sunset_date": "2025-05-15",
  "migration_guide": "https://docs.example.com/api/migration/v1-to-v2"
}
```

この段階で、本当に影響を受けるユーザーが現れます。エラーレスポンスを見たユーザーがアプリを更新するまで、ここで1〜2週間は様子を見ます。

### ステップ4：エンドポイントの削除

アクセスがほぼゼロになったら、エンドポイント自体を削除してもいいでしょう。ただし、実装上は以下のいずれかの判断をします：

- **完全削除**：ルーティング定義から削除。ただし、ロードバランサーやリバースプロキシで404を返すようにしておく
- **スタブ化**：エンドポイントは残すが、常に410 Goneを返すだけにする

スタブ化の方が安全です。万が一、予期しないクライアントがアクセスしてきた時に、サーバーエラーではなく「廃止されている」という明確なメッセージを返せます。

## 実装上のハマりやすい点

**ログレベルの設定**

古いエンドポイントへのアクセスを「エラー」として記録すると、ログが肥大化します。計測用のタグを付けて、集計時に別途処理するのが無難です。

**クライアント側の対応遅延**

サーバー側で段階的に廃止を進めても、クライアント側が新バージョンへの対応をしていないと、結局ユーザーに迷惑がかかります。フロントエンド開発チームとの連携が重要です。

**キャッシュの影響**

CDNやプロキシキャッシュが古いレスポンスを返し続ける可能性があります。廃止フェーズでは、Cache-Controlヘッダーを短めに設定するか、キャッシュを無効化する必要があります。

## 小規模チームでの現実的なアプローチ

ここまで述べた方法は理想的ですが、小規模チームでは全て実装できないこともあります。その場合は、以下の優先順位で進めます：

1. **アクセスログの記録と集計**（最優先）
2. **非推奨ヘッダーの追加**
3. **リリースノートでの告知**
4. **レート制限の導入**（リソースに余裕があれば）
5. **エンドポイント削除**

最低限、ステップ1と3だけでも実施すれば、廃止のリスクは大きく軽減できます。

## 廃止判断を保留すべき場合

以下の場合は、廃止を延期することをお勧めします：

- **ユーザー数が5%以上**：まだ無視できない規模です
- **アクセストレンドが上昇中**：何か理由があるかもしれません
- **廃止の影響範囲が不明**：ユーザーからの問い合わせ対応に備えられていない
- **新バージョンへの移行パス が不明確**：ユーザーが更新
