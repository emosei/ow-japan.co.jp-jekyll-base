## 本番環境で「あの時のログが残っていない」という悔しさ

本番環境でバグが発生した時、真っ先にログを確認します。ところが、いざ問題を追跡しようとすると「あの操作の詳細ログが記録されていない」という経験は、開発チーム規模を問わず起こります。

原因は単純です。本番環境では**パフォーマンスとディスク容量を考慮してログレベルを制限**しているためです。開発環境では `DEBUG` レベルまで出力するのに対し、本番環境では `INFO` や `WARN` だけに絞るという判断ですね。これ自体は正当な設計判断なのですが、問題が起きた時に「あの時 `DEBUG` ログがあれば原因が5分で分かったのに」という後悔が生まれます。

一度トラブルシューティングを経験すると、多くのチームは「本番環境でもログレベルを上げておこう」と考えます。けれど、それもまた別の課題を招きます。ログ量が増えてディスク圧迫が起き、結果的にシステム全体が不安定になるケースもあります。

この「粗密のジレンマ」は、ログレベルの設定を**静的に決めるアプローチ**から抜け出すことで、かなり緩和できます。

## なぜ「本番でもDEBUGを出力」が現実的でないのか

ログレベルを一律に上げることが難しい理由は、パフォーマンスとディスク容量だけではありません。

**ノイズの増加**が実は大きな問題です。本番環境では、毎秒数千〜数万のリクエストが流れることもあります。そこで `DEBUG` レベルのログを全て記録すると、実際に問題が起きた瞬間の関連ログが、膨大なノイズに埋もれてしまいます。結果的に「ログが多すぎて原因が見つけられない」という逆の問題が発生します。

また、ログ出力自体が I/O 処理なので、ログレベルを上げすぎるとアプリケーションのレスポンスタイムが低下します。特に同期的なログ出力を使っている場合、この影響は顕著です。

さらに、ログ管理インフラの容量にも制約があります。クラウド環境でログを集約している場合、月額コストに直結するため、ログ量の増加は経営判断の対象になります。

つまり、「本番でもDEBUGを出力する」という単純な選択肢は、実務的には成立しにくいのです。

## 段階的な有効化設計──トラブル時に必要な情報を後付けする

では、どうするか。答えは**ログレベルを静的に決めるのではなく、実行時に段階的に有効化する**という設計です。

基本的な考え方は以下の通りです。

- **通常時**: INFO レベル以上のみ記録。パフォーマンスと容量を最優先
- **アラート検知時**: 自動的に該当モジュールの DEBUG レベルを有効化
- **手動調査時**: 管理画面やコマンドラインから特定のコンポーネントのログレベルを動的に変更
- **解決後**: 自動的に元の設定に戻す

実装例として、よく使われるパターンを紹介します。

```go
// ログレベルを動的に管理する構造体
type DynamicLogger struct {
    baseLevel  zapcore.Level
    moduleOverrides map[string]zapcore.Level
    mu         sync.RWMutex
}

func (dl *DynamicLogger) GetLevel(moduleName string) zapcore.Level {
    dl.mu.RLock()
    defer dl.mu.RUnlock()
    
    if override, exists := dl.moduleOverrides[moduleName]; exists {
        return override
    }
    return dl.baseLevel
}

func (dl *DynamicLogger) SetModuleLevel(moduleName string, level zapcore.Level) {
    dl.mu.Lock()
    defer dl.mu.Unlock()
    
    dl.moduleOverrides[moduleName] = level
}

func (dl *DynamicLogger) ResetModuleLevel(moduleName string) {
    dl.mu.Lock()
    defer dl.mu.Unlock()
    
    delete(dl.moduleOverrides, moduleName)
}
```

このパターンでは、**モジュール単位でログレベルを上書き**できます。本番環境でトラブルが発生した時、問題が「決済モジュール」にありそうだと判断したら、そのモジュールだけ DEBUG レベルを有効化します。他のモジュールは INFO のままなので、ノイズは最小限に抑えられます。

## 実装時の注意点──自動リセットと権限管理

この設計を本番環境に導入する際は、以下の点に注意が必要です。

**1. 自動リセットの仕組み**

手動でログレベルを上げたまま、誰も気付かずに運用が続く事態を防ぐ必要があります。一般的には、以下のような対策をします。

- ログレベルの変更を加えた時刻を記録し、一定期間（例：1時間）経過したら自動的に元に戻す
- 管理画面に「現在のオーバーライド状態」を可視化し、運用チームが定期的に確認できるようにする
- 変更ログを監査ログとして記録し、誰がいつ何を変更したかを追跡可能にする

**2. 権限管理**

ログレベルの変更は、セキュリティ観点で制限すべき操作です。開発チーム全員が変更できると、本番環境での不用意な変更が増えます。一般的には以下のような階層を設けます。

- 運用チームのみが変更可能
- 変更時には理由を記入必須
- 変更は監査ログに記録

**3. パフォーマンス測定**

ログレベルを上げることで、どの程度のオーバーヘッドが生じるかを事前に測定しておくことが重要です。特に高トラフィック環境では、DEBUG レベルの有効化が想定以上のレスポンス低下を招くことがあります。

## 中小規模チームが現実的に取れるアクション

大規模なログ管理インフラを持たないチームの場合、以下のステップから始めるのが現実的です。

1. **まずは基本的な動的ログレベル機能を実装する** – 上記の `DynamicLogger` のようなシンプルな仕組みで十分です。複雑にする必要はありません。

2. **ホットスポットを把握する** – 過去のトラブルシューティング経験から「このモジュールのログがよく必要になる」というモジュールを特定し、そこを優先的に対応します。

3. **HTTP API または CLI で動的に変更できる仕組みを用意する** – 管理画面を作るのは後でも構いません。curl や CLI ツールで変更できれば、運用上は十分です。

4. **ログの自動圧縮・削除ポリシーを設定する** – 動的ログレベルと並行して、古いログは自動削除する仕組みを整えておくと、ディスク容量の心配が減ります。

## 導入が向くケース、向かないケース

この設計パターンが活躍するのは以下のようなシーンです。

- **本番環境で頻繁にトラブルシューティングが必要な場合** – 特に複雑なビジネスロジックを持つシステム
- **24時間対応の運用体制がある場合** – ログレベルの動的変更に対応できる体制が整っている
- **ログ容量に制約がある場合** – クラウド環境でログ出力がコストに直結する

一方、以下のようなケースでは、シンプルな固定ログレベル設定で十分なこともあります。

- **ログ出力量が少なく、ディスク容量に余裕がある** – 小規模なシステムやバッチ処理中心の場合
- **トラブルが稀で、予測可能な場合** – 既に十分な品質が達成されている場合

重要なのは、**現在の運用課題に基づいて判断する**ことです。「いつか役に立つかも」という予測で複雑さを増やすのは避けるべきです。

## まとめ

ログレベルの粗密設定は、本番環境での現実的な制約と、トラブルシューティングの必要性のバランスを取る課題です。「全てのログを記録する」も「最小限のログだけ」も、どちらかに振り切ると問題が生じます。

段階的な有効化設計は、この二律背反を緩和する実用的なアプローチです。実装は複雑ではなく、基本的な動的ログレベル管理と、運用上のルール整備があれば機能します。

本番環境でトラブルが起きた時に「あの時のログが残っていない」という悔しさを感じたことがあれば、この設計を検討する価値は十分にあります。
