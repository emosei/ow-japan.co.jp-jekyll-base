## 現場で見落とされやすい問題

EC サイト、実店舗、電話注文、モバイルアプリなど複数のチャネルから同時に商品を販売する場合、在庫管理は基幹システムの中でも厄介な領域です。特に「数量ずれ」は月単位で蓄積していくため、気づいたときには既に顧客対応や棚卸し調整が必要になっています。

実務では、こういう判断が起きやすいです。

- 各チャネルが独立した在庫引き当てロジックを持っている
- チャネル間の在庫同期は「定期的に実行する」程度の認識
- リアルタイムで完全に揃っていることを前提にしていない

しかし、ここに落とし穴があります。**チャネル間の同期タイミングがズレると、同じ在庫を複数チャネルが同時に引き当ててしまう**のです。結果として、物理在庫より多くの数量が引き当てられ、納期遅延や欠品対応が増えます。

## なぜ数量ずれが蓄積するのか

典型的な設計は次のようなものです。

```
EC サイト: 在庫DB A から引き当て
実店舗POS: 在庫DB B から引き当て
モバイルアプリ: 在庫DB C から引き当て
↓
定期的に（例：1時間ごと）各チャネルの引き当て履歴を基幹システムに集約
↓
基幹システムが物理在庫と照合して調整
```

この設計では、**各チャネルが独立した在庫数を持つ時間帯が必ず存在**します。その間に、複数チャネルから同じ商品が引き当てられることがあります。

具体例を挙げます。

- 10時00分：基幹システムが在庫を「50個」と把握
- 10時00分～10時59分：EC サイト、POS、アプリが各々独立して引き当て
  - EC サイト：30個引き当て（DB A に「20個」を記録）
  - POS：25個引き当て（DB B に「25個」を記録）
  - アプリ：10個引き当て（DB C に「40個」を記録）
- 11時00分：各チャネルの引き当て履歴を集約
  - 実際には計65個が引き当てられた（50個を超過）
- 基幹システムが調整するが、この時点で既に顧客への約束が成立している

月間で数百件の取引があれば、こうした「微細な超過」が毎日蓄積し、月末には数千個単位のずれになることもあります。

## 同期タイミングの設計判断

この問題の本質は「**どのタイミングを真実の源泉（Single Source of Truth）とするか**」という設計判断です。

### パターン1：基幹システムが真実（強い一貫性）

基幹システムが唯一の在庫引き当て権を持ち、各チャネルはそこにアクセスして初めて在庫を確保します。

**メリット**
- 数量ずれがない
- 監査や棚卸しが単純

**デメリット**
- 基幹システムへのアクセス集中でボトルネック化
- チャネルの応答性が低下
- 基幹システム障害時に全チャネルが停止

### パターン2：各チャネルが独立（高速応答性）

各チャネルが独立した在庫を持ち、定期的に基幹システムと同期します。

**メリット**
- 各チャネルが高速に応答
- 基幹システム障害の影響が限定的

**デメリット**
- 同期間隔中に数量ずれが発生
- 月単位で蓄積する可能性

### パターン3：ハイブリッド（実務的な折衷）

在庫の「確保」と「確定」を分ける設計です。

- **確保段階**：各チャネルが独立して一時的に在庫を引き当て
- **確定段階**：基幹システムが定期的に集約し、物理在庫との差分を検出

この場合、確保と確定の間に「キャンセル可能な期間」を設けることで、ずれを最小化します。

## 実装での具体的な工夫

### 1. 同期間隔の短縮とバッチ処理の最適化

定期同期の間隔を短くすれば、ずれの蓄積は減ります。ただし、同期処理自体が重い場合は逆効果です。

```
現状：1時間ごとの同期
改善案：
- 5分ごとの軽量な集約（差分のみ転送）
- 1時間ごとの完全な照合（フルチェック）
```

軽量な集約では、各チャネルの「新しく引き当てられた分」だけを基幹システムに通知し、基幹システムはそれを在庫から差し引きます。完全な照合では、物理在庫と引き当て合計が一致しているか確認し、ずれていれば原因を特定します。

### 2. 予約在庫と確定在庫の分離

在庫を3つの状態で管理します。

```
物理在庫 = 確定在庫 + 予約在庫 + 未同期在庫

確定在庫：基幹システムが把握している、確実に存在する在庫
予約在庫：各チャネルが確保した、未確定の在庫
未同期在庫：同期遅延により、どちらにも計上されていない在庫
```

新規引き当ては「予約在庫」から行い、同期時に「確定在庫」に昇格させます。同期失敗時は予約をロールバックします。

### 3. 在庫引き当ての順序を統一

複数チャネルが同じ商品を引き当てるとき、引き当ての順序がランダムだと、どのチャネルが優先されるかが不確定になります。

```
優先度の設定例：
1. 実店舗（即日引き渡し）
2. EC サイト（配送予定日が早い順）
3. 電話注文（受付時刻順）
```

基幹システムが定期的に「どのチャネルの引き当てが優先されるべきか」を判定し、超過分をキャンセルまたは遅延対応に回します。

### 4. 同期失敗時の検知と自動修正

同期に失敗したとき、ずれが検知されずに蓄積することが多いです。

```
監視項目：
- 各チャネルから報告された引き当て数の合計が、
  基幹システムの確定在庫を超えていないか
- 前回同期から新しい報告がないチャネルはないか
- 同期処理の実行時間が閾値を超えていないか
```

これらをアラート化し、運用担当者が即座に気づけるようにします。

## 運用での落とし穴

### 在庫調整時の判断ミス

月末の棚卸し時に「システム上の在庫 vs 物理在庫」のずれが見つかります。ここで多くの企業がやってしまう判断が「物理在庫に合わせてシステムを修正する」というものです。

これは短期的には問題を解決しますが、**原因を隠蔽してしまう**ため、翌月も同じずれが繰り返されます。

正しい対応は「なぜずれたのか」を特定し、設計または運用を改善することです。

### チャネル追加時の再検討漏れ

新しいチャネル（例：SNS での受注）を追加するとき、既存の在庫同期設計をそのまま流用する企業が多いです。しかし、チャネル数が増えれば増えるほど、同期のタイミングずれは複雑になります。

チャネル追加のたびに「現在の同期設計で問題が起きないか」を改めて検討する必要があります。

## 小規模チームでも始めやすい改善ステップ

1. **現状把握**：過去3ヶ月の在庫ずれを集計し、パターンを分析する
2. **軽量な改善**：同期間隔を短くする、アラート機能を追加する
3. **仕組み化**：予約在庫と確定在庫の分離を実装する
4. **継続監視**：改善後も月1回、ずれの有無を確認する

いきなり完璧な設計を目指さず、現在の問題を特定してから、段階的に対応することが現実的です。

## まとめ

複数チャネルからの在庫引き当ては、一見シンプルに見えて、同期タイミングという見えない敵を抱えています。完全なリアルタイム同期は理想ですが、実務ではパフォーマンスや障害耐性とのバランスを取る必要があります。

大切なのは「**ずれが起きることを前提に、それを検知し、修正する仕組みを組み込むこと**」です。そうすることで、運用の負担も減り、顧客対応も迅速になります。
