## スレッド安全性の錯覚が生まれる理由

Rubyで長年マルチスレッドプログラミングをしてきた開発者ほど、Ruby 3.0以降のGVL（Global VM Lock）廃止という話題で落とし穴に陥りやすいです。理由は単純で、GVLが存在する環境では「Thread.safe と判定したコードが、実際には非常に限定的な条件下でのみ安全だった」という事実が見えにくいからです。

現場では、共有リソースへのアクセスを `Mutex` で保護したり、`Queue` を使ったり、`require 'thread/pool'` で並行数を制限したりする。これらの対策は確かに有効です。しかし、それらが有効だった理由の多くは「GVLが全体を覆っていたから」であり、GVL廃止後も同じロジックで動作保証できるとは限りません。

## GVLが隠していた「隙間」

具体的な例を考えてみましょう。Rubyの配列操作は「原子的」に見えます。

```ruby
@cache = {}
@lock = Mutex.new

def get_or_compute(key)
  @lock.synchronize do
    return @cache[key] if @cache.key?(key)
    @cache[key] = expensive_computation(key)
  end
end
```

このコードは、GVL環境では安全に見えます。Mutexで保護されているからです。しかし、GVL廃止後に複数のネイティブスレッドが同時に走ると、次のような問題が顕在化します。

**問題1：計算中の割り込み**

`expensive_computation` の実行中、他のスレッドが別の処理で割り込む可能性が高まります。GVL環境では、Rubyレベルのスレッド切り替えはタイムスライスで管理されていました。しかし、GVL廃止後は、ネイティブスレッドが本当に並行実行されるため、計算結果の順序や整合性の問題が生じやすくなります。

**問題2：Mutexの粒度が粗すぎる**

上記コードでは、計算全体がロック内に入っています。GVL環境ではこれで十分でしたが、GVL廃止後は、この長時間のロック保持が他スレッドの待機時間を大幅に増やし、スループットを低下させます。

**問題3：例外発生時のリソース管理**

`expensive_computation` が例外を発生させた場合、ロックは確実に解放されます（Mutexはensureで保護されているため）。しかし、部分的な状態更新があった場合、その不完全な状態を他スレッドが観察する可能性があります。

## 現場で見落とされやすい設計上の欠陥

長年Rubyで並行処理を扱ってきた経験から言うと、以下の3パターンの設計欠陥がGVL廃止で顕在化しやすいです。

### 1. 「読み取りはロック不要」という思い込み

```ruby
def current_value
  @state  # ロックなしで読み取り
end
```

GVL環境では、単純な読み取りはロック不要と判断されることが多いです。しかし、GVL廃止後は、読み取り中に他スレッドが書き込みを行う競合が発生します。特に複数フィールドの参照が必要な場合、部分的に古い値と新しい値が混在する可能性があります。

### 2. 「スレッドローカル変数は安全」という過信

```ruby
Thread.current[:user_id] = user_id
```

スレッドローカル変数自体は安全です。しかし、スレッドプールを使う場合、スレッドが再利用されるため、前回の実行結果が残る可能性があります。GVL廃止とは直接関係ありませんが、マルチスレッド環境の拡大に伴い、この問題がより頻繁に顕在化します。

### 3. 「イミュータブルなオブジェクトは競合しない」という勘違い

```ruby
CONSTANT_DATA = { a: 1, b: 2 }.freeze
```

凍結されたオブジェクトは変更できませんが、参照の置き換え自体は原子的ではありません。また、凍結されたハッシュの反復処理中に、別スレッドが新しいハッシュに置き換えた場合、反復が中断される可能性があります。

## 実装で確認すべきポイント

GVL廃止に向けた既存コードの見直しで、最初に確認すべきことは以下です。

### ステップ1：共有リソースの洗い出し

```ruby
class OrderProcessor
  def initialize
    @orders = []        # 共有リソース
    @stats = {}         # 共有リソース
    @lock = Mutex.new
  end

  def add_order(order)
    @lock.synchronize do
      @orders << order
      @stats[:count] ||= 0
      @stats[:count] += 1
    end
  end
end
```

上記の例では、`@orders` と `@stats` が共有リソースです。現在のコードでは単一のMutexで保護していますが、GVL廃止後は、読み取り頻度が高い場合、ロック粒度を細かくする必要があります。

### ステップ2：ロック保持時間の最小化

```ruby
# 改善前
@lock.synchronize do
  result = expensive_computation()  # ロック内で計算
  @cache[key] = result
end

# 改善後
result = expensive_computation()    # ロック外で計算
@lock.synchronize do
  @cache[key] = result
end
```

計算をロック外に出すことで、他スレッドの待機時間を減らします。ただし、計算結果が複数回行われる可能性がある点に注意が必要です。

### ステップ3：読み取り/書き込みの分離

```ruby
require 'thread'

class SafeCache
  def initialize
    @data = {}
    @read_lock = Mutex.new
    @write_lock = Mutex.new
  end

  def get(key)
    @read_lock.synchronize do
      @data[key]
    end
  end

  def set(key, value)
    @write_lock.synchronize do
      @data[key] = value
    end
  end
end
```

読み取りと書き込みを分けることで、読み取り頻度が高い場合のパフォーマンスを向上させられます。ただし、複数フィールドの一貫性を保つ必要がある場合は、この方法は向きません。

## ハマりやすい落とし穴と対策

### 落とし穴1：既存テストが通ってしまう

GVL廃止後も、単体テストは通る可能性が高いです。なぜなら、テストは通常、シングルスレッドか限定的なマルチスレッド環境で実行されるからです。本番環境で初めて競合が顕在化することもあります。

**対策**：ストレステストを導入し、複数スレッドでの並行実行を定期的に検証します。

### 落とし穴2：Rubyのバージョンアップと同時に行わない

GVL廃止はRuby 3.3で実験的、Ruby 4.0で本格化する予定です。既存コードをRuby 3.0に上げるだけでは、GVL廃止の影響は顕在化しません。段階的なアプローチが重要です。

**対策**：Ruby 3.2までの段階で、既存のスレッド設計を見直し、GVL廃止への準備を進めます。

## 小規模チームでの現実的な進め方

全コードを一度に見直すのは現実的ではありません。優先順位をつけることが重要です。

1. **高頻度で実行されるスレッド処理から始める**：バッチ処理やバックグラウンドジョブなど、スレッド数が多い処理から対策します。

2. **共有リソースの影響範囲を明確にする**：グローバル変数やクラス変数ではなく、インスタンス変数の共有を中心に見直します。

3. **段階的なテスト導入**：すべてのコードにストレステストを入れるのではなく、リスクが高い部分から始めます。

GVL廃止は避けられない流れです。しかし、それに向けた準備は、今から始めても十分間に合います。既存の「Thread.safe」という判定が、実は非常に限定的な条件下でのみ成立していたことを認識することが、第一歩になります。
