## 遅延ロードが「銀の弾」に見える理由

ORMの遅延ロード（Lazy Loading）機能は、ここ数年のトレンドの中でも特に「性能改善の切り札」として扱われることが増えました。関連データを必要になるまで読み込まない──この考え方は理論的には正しく、実際に不要なクエリを削減できます。

しかし現場では、この機能を導入した直後から予測不可能な性能問題が浮上することが多いです。問題の本質は、遅延ロードが「クエリの実行タイミングを開発者の手から奪う」という点にあります。

実際のところ、遅延ロードは「性能改善」というより「性能予測の複雑化」であることがほとんどです。

## なぜ実行タイミングが制御不可能になるのか

遅延ロードの仕組みを理解するために、簡単な例を考えてみましょう。

```javascript
// 例：ORMで関連データを遅延ロード
const user = await User.findById(userId);
console.log(user.name); // ここまでは1回のクエリ

// 後で別の処理で関連データにアクセス
const orders = user.orders; // このタイミングで追加クエリが実行される
```

問題は、このコードを書いた開発者が「いつクエリが実行されるのか」を明示的に制御していない点です。

実務では、こうした遅延ロードが複数の場所で連鎖します。

- Aというエンドポイントで user を読み込む
- user.orders にアクセスして追加クエリが実行される
- orders[0].items にアクセスしてさらに追加クエリが実行される
- items[0].warehouse にアクセスしてまた追加クエリ...

結果として、「1回のリクエストで10回のクエリが実行される」という状況が生まれます。これを **N+1 問題** と呼びますが、遅延ロードはこの問題を「隠蔽」するだけで、解決しません。

さらに厄介なのは、この問題が開発環境では顕在化しにくいということです。開発DBは小規模なテストデータしか入っていないため、10回のクエリも数ミリ秒で完了します。本番環境に行った途端、同じコードが数秒かかるようになります。

## 開発環境と本番環境のギャップ

遅延ロードの問題が見えにくい理由は、データ量の差です。

| 環境 | データ量 | クエリ実行時間 | 問題の見え方 |
|------|---------|--------------|-----------|
| 開発 | 数百件 | 数ミリ秒 | 「性能問題なし」と判断される |
| ステージング | 数千～数万件 | 100ミリ秒～1秒 | 時々遅いが、再現が難しい |
| 本番 | 数十万～数百万件 | 数秒～数十秒 | 明らかに遅い、タイムアウト |

開発チームが「これで十分」と判断したコードが、本番で突然破綻します。

また、遅延ロードの実行タイミングはビジネスロジックの深い場所で起こることもあります。APIエンドポイントから数層下のサービス層で、誰かが `user.profile.settings.notifications` にアクセスしたら、その時点で複数のクエリが同時に実行される──こうなると、どこで何が起きているのか追跡すら困難になります。

## 遅延ロードが向くケース、向かないケース

遅延ロードをすべて否定する必要はありません。適切に使い分けることが大事です。

**遅延ロードが向くケース：**
- ユーザーのアクションに応じて、条件付きで関連データを読み込む場合
- APIクライアントが明示的に「詳細情報を取得」というリクエストを送ってくる場合
- 関連データへのアクセスが稀で、実装の簡潔性を優先したい場合

**遅延ロードが向かないケース：**
- リスト表示画面で複数のレコードを取得し、各レコードの関連データにアクセスする場合（N+1の温床）
- レスポンスタイムがシビアなAPI（例：リアルタイムダッシュボード、決済処理）
- 複数の関連テーブルを同時にアクセスする必要がある場合

現場では、後者のケースで遅延ロードが導入されることが多いため、問題が生じるわけです。

## 実務的な対策：クエリ計画の可視化

遅延ロードを導入する場合、必ず実施すべきことがあります。

**1. 本番相当のデータ量でテストする**

開発環境に本番規模のデータセットをコピーするか、ステージング環境で定期的に性能テストを実施してください。数百件のテストデータでは、遅延ロードの問題は絶対に見えません。

**2. クエリログを監視する**

ORMの設定で「実行されたクエリをすべてログに出力する」という開発モードを有効にし、1回のリクエストで何回のクエリが実行されているかを常に把握してください。

```
// 例：Node.js + TypeORM の場合
logging: ["query"]
```

**3. 事前読み込み（Eager Loading）を明示的に指定する**

必要な関連データは、最初から読み込むようにコードを書き直します。

```javascript
// 遅延ロードに頼らず、必要なデータを明示的に指定
const user = await User.findById(userId, {
  relations: ["orders", "orders.items", "profile"]
});
```

この方が冗長に見えますが、「このエンドポイントでは何のクエリが実行されるのか」が明確になり、後々の保守が楽になります。

**4. 設計段階で「クエリの実行計画」を決める**

新しいエンドポイントを実装する前に、「このエンドポイントでは最大何回のクエリを実行するのか」を設計段階で決めておくことをお勧めします。遅延ロードに任せるのではなく、開発者が意図的に制御する姿勢が重要です。

## 中小規模チームが取るべき判断

遅延ロードは「便利だから使う」のではなく、「この部分では本当に必要か」を問い直す必要があります。

小規模なチームほど、性能問題が起きた時の対応コストが高くなります。新機能開発を中断して、性能改善に何日も費やすことになるからです。

そのため、導入段階では「遅延ロードを最小限に留める」という戦略をお勧めします。必要なデータは明示的に読み込む。その結果、クエリが少し増えるかもしれませんが、実行タイミングが予測可能で、本番環境での問題が少ないコードになります。

性能が本当に問題になった時点で、具体的なボトルネックを特定してから、遅延ロードの導入を検討する──その方が、結果的に堅牢で保守しやすいシステムになります。
