## 現場で起きている問題

Webアプリケーションの認可ミドルウェアを導入すると、ユーザーのロール情報をメモリやRedisにキャッシュして、リクエストのたびにデータベースを叩くのを避ける設計が一般的です。パフォーマンス面では正しい判断なのですが、実務ではここに落とし穴があります。

たとえば、管理者がユーザーの権限を「編集権あり」から「閲覧のみ」に変更したとします。その変更がデータベースには即座に反映されても、ミドルウェアのキャッシュに古いロール情報が残っていれば、ユーザーはしばらくの間、変更前の権限で動き続けます。セキュリティ観点ではこれは問題です。

では、キャッシュの有効期限をどう決めるか。「念のため短めに」と1分にするとデータベースへのアクセスが増えてパフォーマンスが落ちます。「効率重視で」と30分にするとセキュリティリスクが高まります。この判断は、案件の特性によって大きく変わります。

## キャッシュ有効期限の判断軸

まず整理しておくべき点は、キャッシュ有効期限は「技術的に最適な値」ではなく「ビジネス要件と運用能力で決まる」ということです。

**判断軸1：権限変更の緊急度**

権限剥奪が即座に反映される必要があるか、数分の遅延が許容されるかで大きく変わります。金融系や医療系では即座の反映が求められることが多いですが、社内システムや一般的なWebサービスでは数分の遅延は実務上許容されることがほとんどです。

**判断軸2：権限変更の頻度**

1日に何度も権限を変更するシステムと、月1回程度の変更では、キャッシュ戦略が変わります。変更が頻繁なら短めのキャッシュを、稀なら長めにしても影響は小さいです。

**判断軸3：ユーザー数とアクティブ率**

同時接続ユーザーが100人と10,000人では、キャッシュ有効期限の1分の差が生むデータベース負荷が全く異なります。また、ユーザーが常時アクティブか、スパイク的かによっても判断が変わります。

## 検証：3つのキャッシュ戦略の比較

実際に3つの戦略を検証してみました。前提条件は以下です。

- Node.js + Express環境
- Redisをキャッシュ層として使用
- ユーザー数：500人（実装テスト環境）
- 同時接続数：50セッション
- ロール情報：ユーザーごとに3～5個のロール
- テスト期間：10分間の継続リクエスト、その間に権限変更を3回実施

**戦略A：キャッシュなし（毎回DB問い合わせ）**

```javascript
// 毎回データベースから取得
async function getAuthorizationInfo(userId) {
  const roles = await db.query(
    'SELECT role FROM user_roles WHERE user_id = ?',
    [userId]
  );
  return roles;
}
```

結果：
- 権限変更の反映時間：即座（0～100ms）
- DB負荷：10分間で約3,000クエリ
- API平均レスポンス時間：120ms

**戦略B：TTL 5分のRedisキャッシュ**

```javascript
const redis = require('redis');
const client = redis.createClient();

async function getAuthorizationInfo(userId) {
  const cacheKey = `auth:${userId}`;
  
  // キャッシュから取得を試みる
  let roles = await client.get(cacheKey);
  
  if (!roles) {
    // キャッシュミスの場合のみDB問い合わせ
    roles = await db.query(
      'SELECT role FROM user_roles WHERE user_id = ?',
      [userId]
    );
    // 5分間のTTLで保存
    await client.setex(cacheKey, 300, JSON.stringify(roles));
  } else {
    roles = JSON.parse(roles);
  }
  
  return roles;
}
```

結果：
- 権限変更の反映時間：最大5分（平均2.5分）
- DB負荷：10分間で約150クエリ
- API平均レスポンス時間：8ms
- キャッシュヒット率：約95%

**戦略C：TTL 30秒のRedisキャッシュ**

```javascript
// 戦略Bと同じ実装、TTLのみ変更
await client.setex(cacheKey, 30, JSON.stringify(roles));
```

結果：
- 権限変更の反映時間：最大30秒（平均15秒）
- DB負荷：10分間で約900クエリ
- API平均レスポンス時間：25ms
- キャッシュヒット率：約70%

## 実務で採用すべき基準

検証結果から見えるのは、キャッシュ有効期限は「セキュリティリスク」と「パフォーマンス」のトレードオフであり、単純な正解がないということです。

**金融・医療・セキュリティシステム向け**

権限変更の即座の反映が求められます。キャッシュなしか、TTL 10～30秒程度が目安です。ただし、ユーザー数が多い場合はデータベース負荷を監視し、必要に応じて読み取り専用レプリカを用意することを推奨します。

**社内システム・管理画面向け**

TTL 5分程度が現実的です。この範囲なら、権限変更後に「反映に数分かかる可能性がある」と運用側に周知すれば、ほぼ問題が生じません。

**一般向けWebサービス向け**

TTL 5～15分でも多くの場合問題ありません。権限変更が稀で、ユーザーが気づきにくいため、実務上の支障が少ないです。

## 運用段階で追加で確認すべきポイント

キャッシュ有効期限を決めた後も、本番運用では以下を確認してください。

**1. キャッシュ無効化の仕組み**

権限変更時に、該当ユーザーのキャッシュを即座に削除する機能を実装しておくと、セキュリティリスクを大幅に軽減できます。

```javascript
async function updateUserRole(userId, newRole) {
  await db.query('UPDATE user_roles SET role = ? WHERE user_id = ?', 
    [newRole, userId]);
  
  // キャッシュを即座に削除
  await client.del(`auth:${userId}`);
}
```

**2. キャッシュ層の監視**

Redisのメモリ使用量、ヒット率、キー数を定期的に監視し、期待値と乖離していないか確認します。

**3. 権限変更の監査ログ**

権限がいつ、誰に、どう変更されたかを記録し、後から「この時刻に変更が反映されていなかった」という問い合わせに対応できるようにします。

## 誰に向いているか、どんな案件で採用しやすいか

キャッシュ戦略の選択は、チームの運用能力にも左右されます。

- **キャッシュなし**：セキュリティ要件が厳しく、データベース負荷が許容できる案件。チーム規模が小さく、複雑な仕組みを避けたい場合。
- **TTL 5分程度**：バランス型。ほとんどの実務案件に適しています。
- **TTL 30秒以下**：セキュリティ要件が高いが、ユーザー数が少ない案件。または、キャッシュ無効化の仕組みが確実に実装・運用できるチーム。

現場では「念のため短めに」という判断が起きやすいのですが、その結果データベースが過負荷になり、かえってセキュリティが損なわれる（サービス停止による権限管理の不可視化）という事態も起こります。要件と制約を整理した上で、判断することが大切です。
