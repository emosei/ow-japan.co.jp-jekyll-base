## 「同じDockerfileなのに、ローカルでは速いのに本番では遅い」

開発環境では数秒で終わるDockerイメージのビルドが、CI/CDパイプラインを通すと毎回5分以上かかる。そんな経験をしている組織は少なくありません。特に基幹系や予約システムのような複雑な依存関係を持つアプリケーションでは、この差が顕著になります。

原因を追うと、Dockerのレイヤキャッシュが本番環境で期待通りに効いていないことが多いです。ただし、これは単なる「キャッシュの使い方が悪い」という話ではありません。開発環境と本番環境の環境差そのものが、キャッシュの有効性を根本的に変えてしまうのです。

## レイヤキャッシュが「効く」と「効かない」の分水嶺

Dockerのレイヤキャッシュの仕組みは単純です。各ステップ（RUN、COPY、ADDなど）の実行結果がレイヤとして保存され、次のビルド時に同じ命令であれば、キャッシュされたレイヤが再利用されます。

しかし「同じ命令」というのが曲者です。Dockerは命令テキストそのものと、入力ファイルのチェックサムを比較しており、どちらかが変わっていればキャッシュは無効になります。

```dockerfile
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y curl
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
RUN python setup.py build
```

一見すると単純ですが、現場では以下のような環境差が発生します：

- **ローカル開発環境**：前回のビルドから数時間しか経っていない。キャッシュが生きている
- **CI/CDパイプライン**：キャッシュストレージが環境ごと、あるいはビルドエージェントごとに分離されている
- **本番デプロイ**：キャッシュレジストリが別システムで管理されている、あるいは意図的にクリアされている

特に問題になるのは、**ビルドコンテキスト（COPY . . で転送されるファイル群）のタイムスタンプやメタデータの差**です。

## 現場で起こる再ビルドの連鎖

実際の案件では、以下のようなシナリオが繰り返されます。

**シナリオ1：GitHubアクションでの無意識なキャッシュ破棄**

```dockerfile
COPY . .
RUN pip install -r requirements.txt && python -m pytest
```

ビルドエージェントが毎回新しいワーカーで起動する場合、前のビルドのキャッシュはアクセスできません。`COPY . .` の時点で、ソースコードのタイムスタンプが「今この瞬間」として解釈され、以降のすべてのレイヤがキャッシュミスになります。

**シナリオ2：マルチステージビルドでの部分的キャッシュ無効化**

```dockerfile
FROM node:18 AS builder
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:18
COPY --from=builder /app/dist ./dist
RUN npm ci --production
```

`package-lock.json` が変わっていなくても、`COPY . .` で全ソースコードをコピーする段階で、開発者のローカルで編集したファイルのタイムスタンプが反映されます。本番環境では、GitHubからチェックアウトされたファイルのタイムスタンプが「ビルド実行時刻」に統一されるため、ローカルとの差が生じます。

**シナリオ3：依存関係キャッシュの層別管理の欠落**

言語パッケージマネージャー（npm、pip、gemなど）のインストール結果は、`package.json` や `requirements.txt` のハッシュに基づいてキャッシュされるべきです。しかし、これらのファイルが頻繁に更新される場合、キャッシュの効率は急速に落ちます。

```dockerfile
# 悪い例：ソースコード更新のたびに依存関係が再インストールされる
COPY . .
RUN npm install

# 良い例：依存関係宣言ファイルだけを先にコピー
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build
```

この違いは、ローカルでは数回のビルドで気づきやすいのですが、本番のCI/CDでは「毎回新しいエージェントで実行される」という環境差に隠れてしまいます。

## 環境差を前提にした設計判断

キャッシュ効率を本番環境で確保するには、以下の判断が必要です。

### 1. キャッシュストレージの一元化

ローカル開発とCI/CD、本番デプロイで同じキャッシュレジストリを参照するようにします。Docker Buildkitの `--cache-from` と `--cache-to` オプションを使い、明示的にキャッシュを管理します。

```bash
docker buildx build \
  --cache-from=type=registry,ref=registry.example.com/myapp:buildcache \
  --cache-to=type=registry,ref=registry.example.com/myapp:buildcache,mode=max \
  -t registry.example.com/myapp:latest .
```

ただし、この方法はキャッシュレジストリへのネットワーク通信が増えるため、ビルド時間の短縮と引き換えに、レジストリへの負荷が高まる可能性があります。小規模な組織では、キャッシュレジストリの管理コストが意外と大きくなります。

### 2. ビルドコンテキストの正規化

ソースコードのタイムスタンプに依存しない設計にします。`.dockerignore` で不要なファイルを除外し、本当に必要なファイルだけをコピーするようにしましょう。

```
.git
.gitignore
node_modules
dist
.pytest_cache
__pycache__
.env.local
```

さらに、`COPY` の順序を「変更頻度の低い順」に並べることで、キャッシュの有効期間を延ばせます。

### 3. 層別キャッシュ戦略の明文化

チーム内で「どのレイヤはキャッシュに頼り、どのレイヤは毎回実行する」かを決めておきます。

- **ベースイメージ（FROM）**：本番環境で固定版を指定
- **システムパッケージ（apt-get、yum）**：セキュリティ更新のため、定期的に再ビルド
- **言語パッケージ（npm、pip）**：ロックファイル変更時のみ再インストール
- **アプリケーションコード（COPY . .）**：毎回コピーするが、以降の処理はキャッシュ活用

## 現実的なアクション

中小規模の開発組織では、完全なキャッシュ最適化よりも、**「本番デプロイのたびに全レイヤが再ビルドされても耐える」という前提で設計する**方が現実的です。

その上で、段階的に改善していきます：

1. **まずは `.dockerignore` の整備**：不要なファイルを除外し、キャッシュミスの原因を減らす
2. **依存関係ファイルのコピーを分離**：言語パッケージの再インストール頻度を下げる
3. **ビルド時間を計測**：各ステップの実行時間を記録し、ボトルネックを特定
4. **必要に応じてキャッシュレジストリを導入**：ビルド時間が5分を超える場合の検討対象

重要なのは、キャッシュの効率化を「自動化の一環」と見なすことです。手作業でのビルドが多い組織では、キャッシュの効きが悪くても、そこまで影響は大きくありません。しかし、デプロイが1日に複数回、あるいはコミットのたびに走る環境では、キャッシュ戦略の有無が運用負荷を大きく左右します。

環境差を前提に、段階的に改善していく。それが現場での現実的なアプローチです。
