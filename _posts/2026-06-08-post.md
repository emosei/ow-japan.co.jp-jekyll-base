## 現場で起きている判断の分かれ目

環境変数の扱い方は、開発初期段階ではあまり目立たない選択肢に見えます。開発環境では、設定ファイルに値を直書きしたり、ビルド時に環境変数を読み込んで成果物に埋め込んだりする方法が「シンプルで動く」からです。ただし、本番運用が始まると、この判断が開発チームと運用チームの間に大きな摩擦を生み出します。

「APIキーを変更したいが、再度ビルドして本番環境に再デプロイしないといけない」「データベース接続先を切り替えたいのに、ビルド成果物を作り直すまで待たされる」──こうした事態が増えると、簡単な設定変更が『本番環境への重大な変更』として扱われ始めます。

この記事では、なぜビルド成果物に環境変数を埋め込むことが後々の足かせになるのか、そして実務的にどう判断すべきかを整理します。

## ビルド時埋め込みが招く『変更のコスト化』

ビルド時に環境変数を読み込んで成果物に固定化する方式は、一見すると合理的です。

```dockerfile
# ビルド時に環境変数を読み込む例
ARG API_KEY=${API_KEY}
ARG DB_HOST=${DB_HOST}
RUN echo "API_KEY=${API_KEY}" > /app/config.properties
```

このアプローチは、開発初期や小規模なチームでは問題になりません。ただし、以下の場面で制約が顕在化します。

**APIキーのローテーション**  
セキュリティ上の理由で定期的にAPIキーを更新する場合、毎回ビルドが必要になります。緊急対応が必要なセキュリティインシデント時には、この遅延が深刻です。

**ステージング環境と本番環境の差異**  
同じコンテナイメージを複数環境で使い回したいという要望は、システムが成長するにつれて避けられません。ビルド時埋め込みではこれが困難です。

**データベース接続先の切り替え**  
フェイルオーバーやメンテナンス時に接続先を変更する必要が生じます。再ビルドを待つ間、その環境は動作不可になります。

こうした変更が『ビルドパイプラインを通す重大な作業』として扱われると、運用チームの判断スピードが低下し、本来なら数分で対応できる問題に数時間かかるようになります。

## 実行時に環境変数を読み込む設計への転換

対比として、実行時に環境変数を読み込む方式を見てみます。

```python
# 実行時に環境変数を読み込む例
import os

API_KEY = os.getenv('API_KEY')
DB_HOST = os.getenv('DB_HOST', 'localhost')
DB_PORT = int(os.getenv('DB_PORT', '5432'))

if not API_KEY:
    raise ValueError("API_KEY is required")
```

この方式では、ビルド成果物は環境に依存しない状態になります。同じイメージやバイナリを、異なる環境変数を与えて複数環境で実行できます。

**メリット**

- 本番環境での設定変更がデプロイなしで完了する
- ステージング環境と本番環境で同じ成果物を使用できる
- CI/CDパイプラインの複雑性が低下する
- 緊急対応が迅速になる

**デメリット・注意点**

- 実行時に環境変数が不足していると、起動時にエラーになる
- 環境変数の値が多数になると、管理が複雑化する
- 起動時のバリデーション漏れがあると、本番環境で初めてエラーが顕在化する

## 中小規模チームが現実的に取るべき判断基準

完全に実行時読み込みに移行することが常に正解とは限りません。以下の判断軸が実務的です。

**実行時読み込みを優先すべきケース**

- 本番環境での設定変更が頻繁に発生する
- 複数環境（開発・ステージング・本番）を運用している
- 運用チームと開発チームが分かれている
- コンテナ化やマイクロサービス化を検討している

**ビルド時埋め込みでも許容できるケース**

- 単一の本番環境のみで、設定変更がほぼない
- 小規模なスタートアップで、チーム全体が開発と運用を兼務している
- 秘密情報を含まない公開可能な設定のみ

**実務的な折衷案**

多くの現場では、次のような使い分けが現実的です。

- **秘密情報**（APIキー、データベースパスワード）→ 実行時読み込み必須
- **環境依存設定**（接続先ホスト、ポート番号）→ 実行時読み込み推奨
- **アプリケーション定数**（タイムアウト値、ページサイズ）→ ビルド時埋め込みでも許容

## 設計判断を遅延させないために

この判断は『後から変更できる』と後回しにされやすいテーマです。しかし、ビルドパイプラインの構造や成果物の配布方式に関わるため、後付けの変更は思った以上にコストがかかります。

新規プロジェクト開始時に、次の点を確認しておくことをお勧めします。

- 本番環境での設定変更の頻度はどの程度か
- 複数環境を同じ成果物で運用する予定があるか
- 秘密情報の管理方針はどうするか
- 運用チームの構成と権限はどうなるか

これらが明確でなければ、最初から実行時読み込みで設計する方が、後々の修正コストを避けられます。小規模なチームであっても、こうした構造的な判断は早期に決定する価値があります。
