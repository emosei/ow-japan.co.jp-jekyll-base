## 起動時キャッシュ再構築と、スケールアウトのタイミングが合わさるとき

複数のアプリケーションサーバーを運用している現場では、デプロイやインスタンス追加のタイミングで、新しいサーバーが起動するたびにキャッシュを一から再構築する設計を選ぶことがあります。これ自体は珍しくない判断です。ステートレスな設計を保ちたい、あるいはキャッシュの鮮度を保証したいという動機は理解できます。

ところが、スケールアウトが発生したときに複数のサーバーが同時に起動し、全キーの再構築を並行して始めると、本番環境で予期しないロックアップが起きることがあります。負荷テストでは再現しなかった現象が、実際の運用の中で突然顕在化する──こうした経験は、キャッシュ戦略に携わる多くの組織が通る道です。

この記事では、なぜそのロックアップが起きるのか、その仕組みと、実務的な対策を整理します。

## 同時起動時に何が起こるのか

### キャッシュウォーミングの基本的な流れ

通常のキャッシュウォーミング処理は、こうした流れで動きます：

```
1. アプリケーション起動
2. キャッシュストア接続確認
3. 必要なキーの一覧を取得（DB クエリなど）
4. ループで全キーを構築・格納
5. 起動完了、リクエスト受け付け開始
```

1台のサーバーであれば、この処理は通常、数秒から数十秒で完了します。問題は、スケールアウト時に複数のインスタンスが同じ処理を同時に開始することです。

### スケールアウト時の同時実行が引き起こす競合

スケールアウトで新しいインスタンスが3台同時に起動した場合を考えてください。

```
サーバーA: キャッシュ再構築開始 → キーセット全体をDB から取得 → 1000キー分ループ開始
サーバーB: キャッシュ再構築開始 → キーセット全体をDB から取得 → 1000キー分ループ開始
サーバーC: キャッシュ再構築開始 → キーセット全体をDB から取得 → 1000キー分ループ開始
```

このとき、以下の問題が連鎖します。

**1. データベースへの同時負荷集中**

キーセットを取得するためのクエリが3台から同時に飛びます。特に、キー一覧を取得するクエリが重い（全テーブルスキャンなど）場合、DBの接続プールが枯渇したり、クエリロックが発生したりします。

**2. キャッシュストアへの書き込み競合**

Redis や Memcached であれば、単一のキーに対する書き込みは比較的安全ですが、大量の書き込みが並行すると、ネットワーク I/O の飽和、あるいはキャッシュストア自体のメモリ圧迫が起きます。さらに、キャッシュストアが単一インスタンスの場合、スループットの上限に達してタイムアウトが多発します。

**3. アプリケーション層での待機**

キャッシュウォーミングが完了するまで、アプリケーションはリクエストを受け付けない設計になっていることが多いです。つまり、3台のサーバーが全てウォーミング中なら、その間ロードバランサーに登録されたサーバーがない状態になり、既存のリクエストは他の古いインスタンスに集中します。

**4. 既存インスタンスへの負荷集中**

スケールアウトの目的は負荷分散ですが、新しいインスタンスが起動中に受け付けられないため、既存インスタンスにトラフィックが集中し、それらも過負荷に陥ります。結果として、全体的なレスポンス時間が悪化し、タイムアウトやエラーが増加します。

## 設計上の見落としと実装の甘さ

### 起動時の制御不備

多くの場合、キャッシュウォーミングの実装には、以下のような制御が不足しています。

- **起動順序の制御がない**：複数インスタンスが完全に同期して起動する
- **ウォーミング中のリクエスト受け付けがない**：キャッシュ未完成の状態でリクエストを受け付ける選択肢がない
- **スケールアウト時の特別な処理がない**：通常起動と同じロジックで動く
- **タイムアウト設定が甘い**：ウォーミング処理自体にタイムアウトがない、あるいは非常に長い

### 負荷テストで再現しない理由

開発環境や負荷テストでは、通常1～2台のサーバーで検証します。このため、同時起動の問題が表面化しません。本番で初めて3台以上が同時に起動するシナリオに遭遇し、問題が顕在化するのです。

## 実務的な対策

### 1. 起動の段階化（Staggered Startup）

複数インスタンスの起動タイミングをずらします。

```
- オートスケーリングルール：1分間に1台ずつ起動
- ロードバランサー：新インスタンスをヘルスチェック後に段階的に登録
- ウォーミング中：新インスタンスをトラフィックから隔離
```

これにより、DBやキャッシュストアへの負荷が平準化されます。

### 2. ウォーミング中のリクエスト受け付け

キャッシュが完全に構築される前に、部分的なリクエスト受け付けを開始する設計に切り替えます。

```
- キャッシュ未命中時は DB から直接取得（キャッシュなしのフォールバック）
- ウォーミングはバックグラウンドで継続
- リクエスト受け付けは即座に開始
```

この方法なら、新インスタンスがすぐにトラフィックを分散でき、既存インスタンスの過負荷を防げます。

### 3. キー再構築の範囲制限

全キーではなく、アクセス頻度の高い「ホットデータ」のみをウォーミングする設計に変更します。

```
- 起動時：頻度上位100キーだけ再構築
- 残り：初回アクセス時に遅延構築
- 定期的な全キー更新：夜間バッチなど低トラフィック時に実行
```

### 4. 分散ロックの導入

複数インスタンスのうち、1台だけがウォーミングを担当する設計です。

```
- 起動時に分散ロック（Redis SETNX など）を試行
- ロック取得に成功した1台だけがウォーミング実行
- 他のインスタンスはウォーミング完了を待機（ポーリングまたは通知）
```

この方法は、DB負荷を完全に集約できます。ただし、ウォーミング担当インスタンスの障害時に全体が止まるリスクがあるため、タイムアウトと再試行ロジックが必須です。

### 5. 非同期ウォーミング

起動とウォーミングを完全に分離します。

```
- 起動時：最小限の初期化のみ
- ウォーミング：別プロセスまたはキューで非同期実行
- リクエスト受け付け：即座に開始、キャッシュ未命中時は DB フォールバック
```

## 中小規模組織が取るべき現実的な選択

スタートアップや小規模チームであれば、複雑な制御を入れる前に、まず以下を検討してください。

**本当にスケールアウトが頻繁に起きるのか**

スケールアウトが月1回程度なら、手動で段階的に起動する運用でも問題ありません。自動スケーリングが常時動く環境でなければ、複雑な制御は過剰投資です。

**キャッシュ全体の再構築は本当に必要か**

多くの場合、初回アクセス時の遅延構築で十分です。「キャッシュの鮮度を完全に保証したい」という要件が本当にあるのか、改めて確認することをお勧めします。

**段階化ルールをインフラ側で実装する**

アプリケーション側で複雑な制御を入れるより、オートスケーリング設定やロードバランサーのルールで起動間隔を制御する方が、保守性が高いことが多いです。

## まとめ

キャッシュウォーミングは、シンプルな設計ほど運用しやすいものです。スケールアウト時の同時起動によるロックアップは、設計段階で想定されていない場合がほとんどです。

本番環境で初めて複数インスタンスが同時起動するシナリオに遭遇したとき、その時点で修正するのは難しくなります。設計の段階で「複数インスタンスが同時起動したらどうなるか」を一度シミュレーションしておくことが、後々の苦労を大きく減らします。
