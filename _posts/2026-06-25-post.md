## 関数型設計の約束が現場で破綻する瞬間

関数型パラダイムを採用するときは、多くの場合「純粋関数なら単体テストが簡単になる」という期待を抱きます。入力が同じなら出力も同じ、副作用がない、だからモックやスタブが不要──こうした理屈は教科書的には正しいです。

ですが現場では、この純粋性がテスト時に保たれないケースが頻繁に起きます。

特に顕著なのは、**関数型言語やパラダイムを部分的に導入した環境**（TypeScriptやPythonなど）で、副作用を「外側に隔離する」設計を試みたときです。依存性注入（DI）を駆使して副作用を遠ざけようとしたのに、結果としてテストコードが複雑化し、かえって保守性が落ちてしまう──こういう状況を何度も見てきました。

今回は、この落とし穴がなぜ生じるのか、実装レベルで検証してみます。

## 検証の設定：純粋関数の「約束」と現実のギャップ

### 検証の目的

関数型設計で副作用を隔離したコードが、テスト時にどのような問題を抱えるのか、そして単純な依存性注入だけでは解決しない理由を実測する。

### 想定シナリオ

決済処理を例に取ります。以下の要件を想定します：

- ユーザーが商品を購入する
- 在庫をチェックし、減らす
- 決済ゲートウェイに問い合わせる
- ログを記録する
- メール通知を送る

この一連の処理を、関数型パラダイムで「純粋な計算部分」と「副作用部分」に分離する設計が一般的です。

## 実装例：DIで副作用を隔離した場合

```typescript
// 純粋な計算ロジック
function calculatePrice(basePrice: number, taxRate: number): number {
  return basePrice * (1 + taxRate);
}

// 副作用を受け取る形で設計
interface PaymentDependencies {
  checkInventory: (itemId: string) => Promise<number>;
  chargePayment: (amount: number, cardToken: string) => Promise<string>;
  recordLog: (message: string) => Promise<void>;
  sendEmail: (to: string, subject: string, body: string) => Promise<void>;
}

async function processPurchase(
  itemId: string,
  quantity: number,
  cardToken: string,
  deps: PaymentDependencies
): Promise<{ success: boolean; transactionId?: string }> {
  // 在庫確認（副作用）
  const stock = await deps.checkInventory(itemId);
  if (stock < quantity) {
    await deps.recordLog(`Insufficient stock for ${itemId}`);
    return { success: false };
  }

  // 価格計算（純粋）
  const price = calculatePrice(100, 0.1); // 簡略化
  const totalAmount = price * quantity;

  // 決済（副作用）
  const transactionId = await deps.chargePayment(totalAmount, cardToken);

  // ログ記録（副作用）
  await deps.recordLog(`Purchase completed: ${transactionId}`);

  // メール送信（副作用）
  await deps.sendEmail(
    "user@example.com",
    "Purchase Confirmation",
    `Transaction ID: ${transactionId}`
  );

  return { success: true, transactionId };
}
```

一見すると、この設計は「きれい」です。テストでは `deps` に対してモック実装を渡せば、副作用を制御できるはずです。

## テスト時に現れる問題

### 問題1：モック実装の複雑性と保守負担

```typescript
describe("processPurchase", () => {
  it("should process purchase successfully", async () => {
    // モック実装を用意する
    const mockDeps: PaymentDependencies = {
      checkInventory: jest.fn().mockResolvedValue(10),
      chargePayment: jest.fn().mockResolvedValue("TXN-12345"),
      recordLog: jest.fn().mockResolvedValue(undefined),
      sendEmail: jest.fn().mockResolvedValue(undefined),
    };

    const result = await processPurchase("ITEM-1", 2, "token-xyz", mockDeps);

    expect(result.success).toBe(true);
    expect(mockDeps.checkInventory).toHaveBeenCalledWith("ITEM-1");
    expect(mockDeps.chargePayment).toHaveBeenCalledWith(220, "token-xyz");
    expect(mockDeps.recordLog).toHaveBeenCalled();
    expect(mockDeps.sendEmail).toHaveBeenCalled();
  });
});
```

ここまでは良好に見えます。しかし、実務では以下の問題が生じます：

- **モックの状態管理**：複数のテストケースで同じ `deps` オブジェクトを使い回すと、前のテストの呼び出し履歴が残る
- **隠れた依存関係**：`recordLog` の呼び出し回数や順序が変わると、テストが脆くなる
- **統合テストの必要性**：モックを使ったテストでは、実際の非同期処理の遅延やタイムアウトを検証できない

### 問題2：「純粋」な関数の定義が曖昧になる

実際のコードでは、`processPurchase` は純粋ではありません。同じ入力を与えても、`deps` の実装によって結果が変わります。

```typescript
// このテストは成功するが...
await processPurchase("ITEM-1", 2, "token-xyz", mockDeps1);

// 別の deps を渡すと失敗するかもしれない
await processPurchase("ITEM-1", 2, "token-xyz", mockDeps2);
```

つまり、**関数は入力 `(itemId, quantity, cardToken, deps)` に対して純粋ですが、実装者の意図は「アプリケーションロジックは純粋であるべき」という願いです**。この乖離が、テストの複雑性を招きます。

### 問題3：エラーハンドリングの検証が漏れやすい

副作用を DI で隔離すると、各副作用の失敗ケースをテストする責任が分散します。

```typescript
it("should handle payment failure gracefully", async () => {
  const mockDeps: PaymentDependencies = {
    checkInventory: jest.fn().mockResolvedValue(10),
    chargePayment: jest.fn().mockRejectedValue(new Error("Card declined")),
    recordLog: jest.fn().mockResolvedValue(undefined),
    sendEmail: jest.fn().mockResolvedValue(undefined),
  };

  // ですが processPurchase は chargePayment の失敗に対応していない
  // エラーが伝播するだけで、適切なロールバックがない
  await expect(processPurchase("ITEM-1", 2, "token-xyz", mockDeps))
    .rejects.toThrow();
});
```

実際には、決済失敗時に在庫を戻す、ログを記録するなどの処理が必要ですが、関数型設計では「副作用の順序」を制御しにくくなります。

## 実務レベルでの改善案

### 1. 副作用を「層」として明確に分離する

依存性注入は便利ですが、すべての副作用を関数パラメータで受け渡すのではなく、**アーキテクチャレベルで副作用の責任を定義する**ほうが現実的です。

```typescript
// ビジネスロジック層（純粋に近い）
function validatePurchase(
  stock: number,
  quantity: number
): { valid: boolean; reason?: string } {
  if (stock < quantity) {
    return { valid: false, reason: "Insufficient stock" };
  }
  return { valid: true };
}

// アプリケーション層（副作用を明示的に管理）
class PurchaseService {
  constructor(
    private inventoryRepository: InventoryRepository,
    private paymentGateway: PaymentGateway,
    private logger: Logger,
    private emailService: EmailService
  ) {}

  async processPurchase(
    itemId: string,
    quantity: number,
    cardToken: string
  ): Promise<{ success: boolean; transactionId?: string }> {
    const stock = await this.inventoryRepository.getStock(itemId);
    const validation = validatePurchase(stock, quantity);

    if (!validation.valid) {
      await this.logger.warn(`Purchase failed: ${validation.reason}`);
      return { success: false };
    }

    // 以降、トランザクション管理、エラーハンドリングを明示的に
    try {
      const transactionId = await this.paymentGateway.charge(
        100 * quantity,
        cardToken
      );
      await this.inventoryRepository.decreaseStock(itemId, quantity);
      await this.logger.info(`Purchase completed: ${transactionId}`);
      await this.emailService.sendConfirmation("user@example.com", transactionId);
      return { success: true, transactionId };
    } catch (error) {
      await this.logger.error(`Purchase failed: ${error.message}`);
      // ロールバック処理
      return { success: false };
    }
  }
}
```

この設計では：

- **ビジネスロジック** (`validatePurchase`) は純粋で、単体テストが簡単
- **アプリケーション層** (`PurchaseService`) は副作用を明示的に管理し、モックのテストも現実的
- **責任の分離**が明確で、保守性が向上
