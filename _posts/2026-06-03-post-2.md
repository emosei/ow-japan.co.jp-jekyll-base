## レトロスペクティブの改善案が消える理由

スプリント終了時のレトロスペクティブ（ふりかえり）では、チームから改善提案が次々と挙がります。「テストカバレッジを上げよう」「デプロイ手順を自動化しよう」「デザインレビューを毎週やろう」。その場では全員が頷いて、改善案はバックログに入ります。

ところが次のスプリントが始まると、その改善案はほぼ実行されていません。

これはチームの意志の弱さではなく、組織の制約と優先度の構造が、改善案を「実行可能な仕事」に変換していないからです。現場では、この問題が静かに繰り返されています。

## 改善案が『タスク化』されていない

レトロスペクティブで「テストを増やそう」と決まったとしましょう。これは改善の*方向性*であって、実行計画ではありません。

実行されるには、以下の要素が必要です。

- **誰が担当するか**（アサインメント）
- **どの機能・領域から始めるか**（スコープの限定）
- **何をもって完了とするか**（定義）
- **他のスプリント目標とのバランス**（優先度の明示）

多くのチームでは、改善案がバックログに入った後、これらが曖昧なまま次のスプリント計画に進みます。スプリント計画では顧客向け機能が優先され、改善案は「やれたらやる」という扱いになり、結局実行されません。

## 優先度が『相対的』に下がる構造

ここが重要なポイントです。改善案が実行されないのは、その改善案の優先度が低いからではなく、**顧客向け機能や緊急対応と相対比較された時点で、自動的に後回しになる構造**だからです。

たとえば：

- スプリント開始時点で、顧客から「この機能を来週までに」という要望が入る
- デプロイパイプラインのトラブルが発生して、エンジニアの時間が割かれる
- 前スプリントからの未完了タスクが残っている

こうした日常的な圧力の中では、「来月でもいい改善」は自動的に優先度が下がります。これは組織横断的な制約であり、個々のチームの努力では解決しません。

## 改善案を『スプリント目標の一部』にする

実行される改善案にするには、スプリント計画の段階で、改善案を顧客向け機能と同じレベルの**スプリント目標**に組み込む必要があります。

具体例を示します。

あるチームで「デプロイ手順の自動化」が改善案として挙がったとします。これをスプリント目標に含める場合：

```
スプリント目標：
- 新規決済機能の実装（ストーリーポイント 13）
- デプロイパイプラインの自動化第1段階（ストーリーポイント 5）

制約：
- 決済機能は顧客向けリリースの必須要件
- デプロイ自動化は運用負荷軽減のため、スプリント内に着手する
- 両立できない場合は、スプリント計画時点で PO と調整
```

重要なのは、改善案を「タスク」ではなく「スプリント目標の一部」として宣言することです。これにより：

- スプリント開始時点で優先度が固定される
- チーム全体が改善案の実行を前提に動く
- 途中で優先度変更がリクエストされても、スプリント目標との衝突が明示的になる

## 改善案の『実行可能性』を事前に判定する

レトロスペクティブの場では、改善案の実現性が十分に検討されないことが多いです。「テストを増やそう」という提案は理想的ですが、実際には：

- テスト環境の整備が必要か
- 他チームの協力が必要か
- ツールやライセンスの購入が必要か
- 学習期間が必要か

こうした**実行に必要な前提条件**が明確でないと、スプリント計画で「できない」という判断になり、改善案は消えます。

改善案を挙げた時点で、簡易的な実現性チェックを入れることが効果的です：

```
改善案：自動テスト実行環境の構築

実現性評価：
- 必要なリソース：エンジニア 1 名、3 日
- 前提条件：CI/CD ツールの既存契約確認
- 依存関係：インフラチームとの調整必要
- 推奨開始時期：来スプリント（今スプリントでは他の目標が優先）
```

このように評価することで、「実行できるが優先度が低い」と「実行の前提が整っていない」が区別でき、スプリント計画での判断が明確になります。

## 組織横断的な制約を可視化する

改善案が実行されない背景には、チーム内の問題だけでなく、組織全体の制約があることが多いです。

- インフラチームの対応が必要
- セキュリティレビューが必要
- 経営判断（予算）が必要
- 他プロダクトチームとの調整が必要

こうした制約が改善案の実行を阻む場合、それを「チーム内の優先度問題」として扱うと、改善案は永遠に実行されません。

代わりに、制約を**明示的に組織に報告**し、実行の可否を判断してもらう必要があります。

```
改善案：デプロイの自動化
制約：インフラチームの作業キャパシティが満杯
対応案：
  A) 来月のインフラチームの空き時間を待つ
  B) 外部ツール導入で自前対応（コスト発生）
  C) この改善案は延期
```

この判断は、チームではなく、プロダクトマネージャーや経営層がすべきものです。チームは「実行できるか」を答え、組織は「実行すべきか」を答える。その境界を引くことで、改善案の実行可能性が初めて現実的になります。

## 小規模チームでの現実的な運用

大規模組織の制約管理は難しいですが、小規模チームでも改善案を実行する仕組みは作れます。

- レトロスペクティブで改善案を挙げる際、「実行時期」を同時に決める
- 改善案が「今スプリント」「来スプリント」「将来」のいずれかを明示する
- 「今スプリント」に入れた改善案は、顧客向け機能と同じスプリント目標に含める
- 「将来」の改善案は、定期的に見直す（毎月など）

この運用なら、改善案が消えることはなく、実行時期が明確になります。

## まとめ

レトロスペクティブの改善案が実行されないのは、改善案そのものが悪いのではなく、**改善案が『実行可能なスプリント目標』に変換されていない**からです。

改善案を実行するには：

1. 改善案を具体的なタスク（誰が、何を、いつまでに）に落とし込む
2. スプリント計画で、顧客向け機能と同じレベルの優先度判定をする
3. 実行に必要な組織横断的な制約を明示し、実行可否を組織に判断してもらう
4. 実行時期を明確に決める

この流れを回すことで、レトロスペクティブは「ふりかえりの場」から「改善を実行する仕組み」に変わります。
