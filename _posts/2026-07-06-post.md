## 現場で起きている問題

データベース設計で「論理削除」を採用している案件は少なくありません。削除フラグを立てるだけで物理的には行を残す方式です。監査証跡が必要な基幹系システムではとくに重宝されます。

ところが、ここに複合ユニーク制約が絡むと、予想外の制約違反が発生します。

具体的には、こういう場面です。

ユーザーが登録した「メールアドレス + 組織ID」の組み合わせに複合ユニーク制約がある。ユーザーが退職して削除フラグを立てた。しばらく後、同じメールアドレスで別の人が同じ組織に入社し、再登録しようとする。

**ここで制約違反エラーが出ます。**

削除フラグが立った行は物理的に存在しているため、ユニーク制約の検査に引っかかるのです。新しい行を挿入できません。

この矛盾は、スキーマ設計と削除戦略が噛み合っていない状態です。現場では「なぜこんなことが起きるのか」と混乱し、応急対応で済ませてしまうことが多い。しかし根本的には、設計段階で判断を誤っています。

## なぜユニーク制約が問題になるのか

ユニーク制約は「このカラムの値は重複してはいけない」という約束です。論理削除の場合、削除フラグがあっても行は存在しています。

データベースエンジン側からすれば、削除済みの行も「存在する行」です。制約チェック時に削除フラグの状態を区別することは、標準的なユニーク制約の動作ではありません。

つまり、**論理削除と通常のユニーク制約は、本来的に相性が悪い組み合わせ**なのです。

これが問題になるのは、以下のようなケースです。

- 人事異動で同じメールアドレスが再利用される
- 商品コードが廃止後、同じコードで新商品が登録される
- 顧客が一度削除された後、再度登録される

こうした「同じ値の再利用」がビジネス上あり得る場合、論理削除とユニーク制約の両立は設計の綻びになります。

## 実装の選択肢と判断基準

この問題に対して、現場で取られる選択肢は大きく3つです。

### 1. 削除フラグを制約に含める

複合ユニーク制約に、削除フラグを含めます。

```sql
-- メールアドレス + 組織ID + 削除フラグの複合ユニーク制約
ALTER TABLE users 
ADD CONSTRAINT uq_email_org_status 
UNIQUE (email, organization_id, deleted_flag);
```

この方式なら、`(email, organization_id, deleted_flag=0)` と `(email, organization_id, deleted_flag=1)` は別の組み合わせとして扱われます。削除済みの行があっても、新規登録時の制約チェックをパスします。

**メリット：** 実装が簡単で、既存ロジックへの影響が少ない。

**デメリット：** 削除フラグを常に制約に含める必要があり、クエリ側でも `WHERE deleted_flag = 0` を忘れやすくなります。削除済みデータを誤って参照する隠れたバグが増える可能性があります。

### 2. 物理削除に切り替える

監査が本当に必要なのか、改めて問い直します。

多くの場合、「何か問題が起きたときのために」という曖昧な理由で論理削除が選ばれています。実際には、監査ログテーブルを別途用意し、変更履歴をそちらに記録する方が、スキーマをシンプルに保てます。

```sql
-- メインテーブルは物理削除
DELETE FROM users WHERE id = ?;

-- 履歴テーブルに削除イベントを記録
INSERT INTO user_audit_log (user_id, action, timestamp, ...)
VALUES (?, 'DELETED', NOW(), ...);
```

**メリット：** スキーマが単純で、ユニーク制約が素直に機能します。クエリも明確。

**デメリット：** 履歴管理の仕組みが別途必要。削除時の処理が増えます。

### 3. 論理削除専用の設計に変更する

ユニークさが必要な値を、別テーブルに分離します。

```sql
-- メインテーブル（削除可能）
CREATE TABLE users (
  id INT PRIMARY KEY,
  user_profile_id INT NOT NULL UNIQUE,
  deleted_flag TINYINT DEFAULT 0
);

-- 値の一意性を管理するテーブル（削除不可）
CREATE TABLE user_identifiers (
  id INT PRIMARY KEY,
  email VARCHAR(255) NOT NULL,
  organization_id INT NOT NULL,
  UNIQUE (email, organization_id)
);
```

メールアドレスと組織IDの一意性は `user_identifiers` で保証し、こちらは物理削除のみとします。

**メリット：** 論理削除と制約の矛盾を完全に解決できます。

**デメリット：** スキーマ設計が複雑になり、テーブル間の整合性維持が運用負荷になります。

## 中小規模の開発現場での現実的な判断

正直に言うと、この問題に「最適解」はありません。プロジェクトの制約と優先順位で判断するしかありません。

**監査が本当に必要か、一度確認してください。** 「いつか必要になるかもしれない」という理由だけで論理削除を選ぶと、後々こうした設計の矛盾が積み重なります。

新規プロジェクトであれば、**削除フラグを制約に含める方法（選択肢1）が最もシンプル**です。ただし、クエリを書く側が常に `deleted_flag = 0` を意識する必要があります。ORM を使っているなら、デフォルトスコープで自動的に削除済みを除外するよう設定しておくと、ヒューマンエラーを減らせます。

既存システムで問題が顕在化している場合は、その時点で**どのデータが再利用される可能性があるか、実際のビジネス要件を洗い直す**ことをお勧めします。意外と「再利用は起きない」という判断もあり得ます。

## 運用段階での注意点

どの方式を選んでも、**削除と再登録の流れをテストケースに明示的に含める**ことが重要です。

現場では単体テストで「削除できる」「復旧できる」だけを確認し、「削除後に同じ値で再登録できるか」というシナリオが抜けやすいものです。

また、ユニーク制約の設計変更は、既存データが大量にある場合、マイグレーションが慎重になります。ダウンタイムなしで制約を追加するのは難しい場合もあります。早めに気づいて対応することが、後々の手戻りを減らします。
