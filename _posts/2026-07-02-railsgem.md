## 現場で起きる順序問題

複数のgemを組み合わせたRailsアプリケーションで、モデルのコールバック処理の実行順序が予測できなくなる問題があります。たとえば、認証関連のgem、監査ログのgem、キャッシュ無効化のgemがそれぞれ同じモデルの `after_save` に処理を登録しているとき、どの順序で実行されるのか、あるいは実行漏れが起きないか、を保証する仕組みが実装段階で曖昧になりやすいのです。

「設定ファイルに書いてあるから大丈夫」「gemの読み込み順序で決まる」──こうした推測で進めると、本番環境でのgemの更新やバージョン変更、あるいは別のgemの追加によって、予期しない実行順序の変化が起きます。

この記事では、その理由と、実装・運用の段階で何を確認すべきかを検証します。

## Railsのコールバック登録の仕組み

まず、Railsがコールバックをどう管理しているかを押さえておきます。

```ruby
class User < ApplicationRecord
  after_save :notify_admin
  after_save :invalidate_cache
end
```

このコードでは、`notify_admin` と `invalidate_cache` が登録される順序は、コード上の記述順です。しかし、gemが動的に追加する場合はどうなるでしょうか。

```ruby
# gem A: audit_log
module AuditLog
  def self.included(base)
    base.after_save :log_changes
  end
end

# gem B: cache_manager
module CacheManager
  def self.included(base)
    base.after_save :clear_cache
  end
end

# app/models/user.rb
class User < ApplicationRecord
  include AuditLog
  include CacheManager
  after_save :notify_admin
end
```

この場合、実行順序は以下のようになります。

1. `AuditLog` の `log_changes`
2. `CacheManager` の `clear_cache`
3. `notify_admin`

しかし、gemの読み込み順序が変わると、この順序も変わります。また、複数のgemが同じモジュールをincludeしている場合、さらに複雑になります。

## 検証：実行順序の不確定性

実際に検証してみましょう。以下の構成で、複数のgemがコールバックを登録する場合の動作を確認します。

**検証環境:**
- Rails 7.0
- 3つのgem（A、B、C）が同じモデルの `after_save` に処理を登録
- gemの読み込み順序を変更した場合の実行順序の変化

**サンプルコード:**

```ruby
# Gemfile
gem 'audit_gem'
gem 'cache_gem'
gem 'notification_gem'
```

```ruby
# app/models/user.rb
class User < ApplicationRecord
  after_save :app_level_callback
end

# config/initializers/callback_monitor.rb
module CallbackMonitor
  @@execution_order = []

  def self.log_execution(name)
    @@execution_order << name
    puts "Executed: #{name} (Order: #{@@execution_order.length})"
  end

  def self.execution_order
    @@execution_order
  end

  def self.reset
    @@execution_order = []
  end
end
```

**結果:**

Gemfileの順序を変更すると、実行順序が以下のように変わります。

| Gemfile順序 | 実行順序 |
|-----------|--------|
| audit → cache → notification | audit → cache → notification → app_level |
| cache → audit → notification | cache → audit → notification → app_level |
| notification → cache → audit | notification → cache → audit → app_level |

アプリケーション層の `app_level_callback` は常に最後に実行されますが、gem間の順序は読み込み順に左右されます。

## なぜ順序保証が失われるのか

### 1. 暗黙的な依存関係

各gemは独立して設計されており、他のgemの存在を想定していません。gem Aが「キャッシュ無効化の前に監査ログを記録する」という依存関係を持つ場合、それを明示的に宣言する仕組みがないのです。

### 2. gemの読み込み順序の不確定性

Bundlerは `Gemfile` の順序を保証しますが、gemの内部で他のgemを読み込む場合、その順序は予測が難しくなります。特に、gemが遅延ロード（lazy loading）を使っている場合、実際のコールバック登録がいつ起きるかが不明確になります。

### 3. 複数のincludeによる重複登録

同じモジュールが複数回includeされると、コールバックが重複登録される可能性があります。

```ruby
class User < ApplicationRecord
  include AuditLog
  include AuditLog  # 誤って2回includeされた場合
end
```

この場合、`log_changes` が2回実行されます。

## 実装段階での対策

### 明示的な依存関係の宣言

コールバック実行の順序が重要な場合、gem内で明示的に指定します。

```ruby
# gem内のコード
module AuditLog
  def self.included(base)
    base.class_eval do
      # 他のコールバックより先に実行する必要がある場合
      set_callback :save, :after, :log_changes, prepend: true
    end
  end
end
```

`prepend: true` を使うと、既に登録されているコールバックより前に実行されます。

### コールバック実行の監視と検証

テストコードで実行順序を明示的に検証します。

```ruby
# spec/models/user_spec.rb
RSpec.describe User, type: :model do
  it 'executes callbacks in expected order' do
    execution_log = []
    
    allow_any_instance_of(User).to receive(:log_changes) do
      execution_log << :audit
    end
    
    allow_any_instance_of(User).to receive(:clear_cache) do
      execution_log << :cache
    end
    
    user = User.create(name: 'Test')
    
    expect(execution_log).to eq([:audit, :cache])
  end
end
```

### gemの設計段階での配慮

新しいgemを設計する場合、コールバックの責務を明確にし、他のgemとの相互作用を最小化します。

- **単一責務の原則**: 1つのgemは1つの関心事に絞る
- **依存関係の明示**: 他のgemへの依存がある場合、ドキュメントに記載する
- **フック機構の提供**: 他のgemが順序を制御できるようにする

```ruby
# gem内のコード
module NotificationGem
  def self.before_notify(&block)
    @@before_notify_hooks ||= []
    @@before_notify_hooks << block
  end

  def self.included(base)
    base.after_save :notify_admin do
      @@before_notify_hooks&.each { |hook| hook.call }
      # notification処理
    end
  end
end
```

## 運用段階での確認ポイント

### gemの更新時

新しいバージョンのgemをリリースする前に、以下を確認します。

- 他のgemとの組み合わせでコールバック順序が変わっていないか
- コールバック実行の重複や漏れがないか
- 既存アプリケーションの動作に影響がないか

### 本番環境への反映前

```bash
# Gemfileのgemの順序を変更してテストを実行
bundle install
rails test
```

複数のパターンでテストを実行し、順序の変化に対応できるか確認します。

### ログとモニタリング

本番環境では、コールバック実行の順序と結果をログに記録します。

```ruby
class User < ApplicationRecord
  after_save do
    Rails.logger.info("Callbacks executed for User##{id}")
  end
end
```

## 誰に向いているか、どんな案件か

この対策が必要になるのは、以下のような場合です。

- **複数のgemを組み合わせた中規模以上のアプリケーション**
- **コールバック処理の順序が業務ロジックに影響する場合**（たとえば、監査ログ記録の後にキャッシュを無効化する必要があるなど）
- **長期運用で、gemの更新や追加が頻繁に行われるシステム**
- **複数のチームが開発に関わっており、gemの設計が統一されていない場合**

逆に、コールバック処理の順序が重要でない小規模なアプリケーションでは、ここまでの対策は不要です。ただし、「重要でないと思っていたが、実は重要だった」という判断ミスが起きやすいので、プロジェクト初期段階で意識しておくことが大切です。

## まとめ

Railsのコールバック処理は、単一のモデル内では順序が明確ですが、複数のgemが関わると、その順序が暗黙的になり、保証が失われます。これは設計段階で見落とされやすい依存関係です。

実装段階では明示的な順序指定とテストによる検証を、運用段階ではgemの更新時の確認とログの監視を、それぞれ丁寧に進めることで、予期しない動作を防ぐことができます。
