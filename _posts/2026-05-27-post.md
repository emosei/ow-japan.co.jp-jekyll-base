## 「たまに遅くなる」という曖昧な苦情の正体

外部APIやSaaS連携を組み込んだシステムを運用していると、ユーザーから「たまに処理が遅くなる」「時間帯によって重くなる」といった曖昧な報告が上がることがあります。ログを見ても特段のエラーは出ていない。サーバーリソースも余裕がある。それなのに、なぜか応答時間が伸びる──。

こういう場面では、多くの場合、外部サービスの**レート制限（Rate Limiting）** が静かに効いていることが原因です。

レート制限は、外部APIの提供者が「1秒あたり10リクエストまで」「1分間に100リクエストまで」といった形で課している制約です。この制限に引っかかると、APIは単にエラーを返すのではなく、リクエストを一時的に遅延させたり、429ステータスコードを返したりします。

問題は、この遅延が**自社システムの同期処理に直接影響する** ということです。ユーザーのリクエストを受けた時点で、外部APIへの呼び出しが同期的に実行されていれば、レート制限による待機時間がそのままユーザーの待ち時間になってしまいます。

## 同期連携の落とし穴：見えにくい依存関係

外部サービス連携を最初に設計するとき、多くの開発者は「とりあえず同期で呼び出す」という選択をします。理由は単純です。実装が簡潔で、結果をすぐに返せるから。

```javascript
// 同期的な外部API呼び出しの例
app.post('/order', async (req, res) => {
  const order = await createOrder(req.body);
  const paymentResult = await externalPaymentAPI.charge(order.id); // ← ここで待機
  const notification = await notificationService.send(order.id);   // ← さらに待機
  res.json({ success: true, orderId: order.id });
});
```

初期段階では問題が顕在化しません。トラフィックが少ないうちは、レート制限に引っかかることがめったにないからです。

しかし、ユーザー数が増えたり、複数の外部サービスを連携させたり、バッチ処理で大量の呼び出しが発生するようになると、状況が変わります。ある時間帯に処理が集中すると、外部APIのレート制限に一気に引っかかり、その結果、自社システム全体の応答時間が悪化します。

さらに厄介なのは、この問題が**スケーラビリティの限界として現れる** ということです。サーバーを増やしても、データベースを最適化しても、外部APIのレート制限は変わりません。むしろ、サーバーを増やすことで、より多くのリクエストが外部APIに殺到し、問題が悪化することもあります。

## レート制限の影響を見極める3つのチェックポイント

設計段階で「同期か非同期か」を判断する際は、以下の点を確認することが実務的です。

**1. 外部サービスの制限値と自社のトラフィック予測**

APIの仕様書に「1秒あたり10リクエスト」と書かれていても、それが自社システムの要件に対して十分か確認する必要があります。ピーク時に100ユーザーが同時にアクセスするなら、その全員が外部APIを呼び出す場合、制限値は足りません。

ただし、現実はもっと複雑です。すべてのユーザーアクションが外部API呼び出しに繋がるわけではないし、キャッシュやローカル処理で吸収できる部分もあります。だからこそ、**実際のユーザーシナリオに基づいた呼び出し頻度の推定** が重要になります。

**2. 結果の即時性がビジネス要件か**

「ユーザーが操作した直後に結果が必要か」という問いは、同期・非同期の選択を大きく左右します。

決済処理なら、ユーザーに即座に成功・失敗を知らせる必要があります。一方、メール通知やデータ分析用のログ送信なら、数秒から数分の遅延は許容できることが多いです。この区別が曖昧なままだと、すべてを同期で実装してしまい、後で痛い目を見ます。

**3. 外部サービスの障害時の影響範囲**

同期連携では、外部APIが遅い、あるいは落ちていると、自社システムも応答不可になります。これが許容できるか、それとも自社システムは独立して動作すべきかで、設計が変わります。

## 非同期キューへの転換：実装の現実的な判断基準

「レート制限が問題になりそう」と判断したら、非同期キューの導入を検討します。ただし、非同期化は万能ではなく、導入コストと運用負荷が増えます。

**向くケース：**
- 外部API呼び出しが頻繁に発生する
- 結果の即時性が不要、あるいは非同期通知で対応できる
- 複数の外部サービスを組み合わせている
- バッチ処理で大量の呼び出しが発生する

**向かないケース：**
- ユーザーが即座に結果を必要とする（決済、在庫確認など）
- キューイングシステムの運用負荷を吸収できない
- 外部サービスのレート制限が十分に余裕がある

非同期化を進める場合、実装上の注意点として、以下のような判断が必要になります。

```javascript
// 非同期キューを使った実装例
app.post('/order', async (req, res) => {
  const order = await createOrder(req.body);
  
  // 即座にレスポンスを返す
  res.json({ success: true, orderId: order.id, status: 'processing' });
  
  // バックグラウンドでキューに追加
  await jobQueue.enqueue({
    type: 'process_payment',
    orderId: order.id,
    amount: order.total
  });
  
  // キューの処理結果はWebSocketやポーリングで通知
});
```

この場合、ユーザーは即座に「注文受け付けました」という応答を得られ、実際の決済処理は後で行われます。ただし、ユーザーが結果を知るための仕組み（WebSocketでの通知、ポーリング、メール通知など）が別途必要になります。

## 中小規模の開発組織が現実的に取れるアクション

非同期キューの導入は、RabbitMQやRedis、あるいはマネージドサービスなど、選択肢が多く、判断が難しいです。

現実的には、以下のような段階的なアプローチが有効です。

1. **まずは計測する**：外部API呼び出しの応答時間と頻度を記録し、実際にレート制限に引っかかっているか確認する。推測ではなく、ログに基づいた判断が重要です。

2. **キャッシュやリトライロジックで対応できないか検討する**：非同期化の前に、キャッシュやエクスポーネンシャルバックオフによるリトライで問題が緩和できることもあります。

3. **小さく始める**：非同期化が必要と判断したら、最初は優先度の低い処理（ログ送信、分析用データ送信など）から始めて、運用ノウハウを蓄積します。

4. **外部サービスの上位プランの検討**：レート制限を緩和できるプランがあれば、コスト面での判断も含めて検討する価値があります。

## 設計判断の落とし穴：「今は大丈夫」という思考停止

最後に、現場で見かけやすい失敗パターンを指摘しておきます。

「今のところ問題が出ていないから、同期でいい」という判断は、スケーリングの必要性が生じた時点で一気に崩れます。その時点で非同期化を進めようとしても、既存の同期処理に依存するコードが蓄積していて、リファクタリングコストが膨大になることが多いです。

逆に、「念のため非同期化しておこう」という過度な先制投資も、運用負荷を増やすだけで、実際の効果が出ないことがあります。

大切なのは、**外部サービス連携の性質と自社システムの成長予測に基づいた、時間軸を含めた設計判断** です。初期段階では同期でいいが、ユーザー数が増えたら非同期に移行する、という段階的な見直しも現実的なアプローチです。
