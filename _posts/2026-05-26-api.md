## 「まだ古いバージョンが使われている」という判断の落とし穴

モバイルアプリやWebクライアントの開発では、バージョンチェック機能がよく実装されます。ユーザーが古いバージョンを使い続けていないか監視し、必要に応じて更新を促すための仕組みです。

この機能自体は悪くありません。しかし現場では、このバージョンチェック結果が、サーバー側の廃止API対応を後送りにする根拠として使われることがあります。

「ユーザーの5%しか古いバージョンを使っていないから、廃止APIの置き換えはまだ急がなくていい」

こうした判断は一見合理的に見えますが、実は後々大きな負債を生み出す可能性があります。

## クライアント側の認識とサーバー側の現実のズレ

バージョンチェック機能は、クライアント側が「自分がどのバージョンか」を把握する仕組みです。しかし、これはあくまで**クライアント側の認識**に過ぎません。

実際のサーバー側では、以下のような現実が起きています。

- **古いバージョンからのリクエストが継続して来ている**：バージョンチェック機能があっても、ユーザーは更新を無視することができます。特に企業向けシステムやBtoB的なアプリでは、端末管理の制約から古いバージョンの使用が強制されることもあります。

- **廃止予定のAPIが実装コストの理由で削除されない**：「まだ5%のユーザーが使っている」という情報があると、廃止APIの削除判断が先延ばしになります。結果、バックエンド側は複数のAPI仕様を並行維持することになり、実装の複雑性が増します。

- **テスト対象が増え続ける**：新しいAPIと古いAPIの両方を検証する必要が生じ、リグレッションテストの範囲が膨らみます。

現場ではこうした状況を目にします。バージョンチェック機能が「対応を先送りにしても大丈夫」という誤った安心感を生み出しているのです。

## 「まだ使われている」と「対応が必要」は別の問題

ここで重要な区別があります。

**「クライアントが古いバージョンを使用している」という事実と、「サーバー側がそれに対応し続けるべき」という判断は、別の問題**です。

前者は観測可能な事実ですが、後者は経営判断や運用設計に関わる決定です。

たとえば：

- ユーザーベースが小さい場合、古いバージョンの使用率が高い傾向にあります。一方で、サーバー側の保守コストは使用率に比例しません。むしろ、複数バージョン対応の複雑性は指数関数的に増えます。

- セキュリティパッチが必要になった場合、古いバージョンへの対応を無視できるかは、事業リスク評価の問題です。単なる「使用率」では判断できません。

- 新機能の開発速度と、レガシー対応の負荷はトレードオフです。バージョンチェック機能の数字だけでは、このバランスを測れません。

## 後方互換性の負債が膨らむシナリオ

具体的には、こんな流れで負債が増えていきます。

1. **初期段階**：新しいAPIへの移行を開始するが、「古いバージョンがまだ使われている」を理由に、廃止APIの削除を延期。

2. **中期段階**：新しいAPIと古いAPIの両方を運用し続ける。テストケースが増え、バグ修正の際も両方を検証する必要が出てきます。

3. **後期段階**：さらに新しいAPI仕様が必要になるが、古い仕様の削除がまだできていない。コード上に3世代のAPI実装が共存するようになります。

4. **危機段階**：新しい要件に対応しようとしても、レガシーコードとの相互作用が複雑になり、変更が難しくなります。最終的には、古いバージョン対応を一気に削除するしかなくなり、その時点でのユーザー影響は大きくなります。

現場ではこのシナリオを何度も見てきます。バージョンチェック機能が存在することで、「対応を先送りしても大丈夫」という心理的な安心感が生まれ、実際の負債化を見落としやすくなるのです。

## 現実的な対応設計

では、どう考えるべきか。いくつかの実務的なアプローチがあります。

### 廃止スケジュールを事前に決める

バージョンチェック機能の数字に依存するのではなく、**廃止予定日を最初から決める**ことです。

例えば：

```
古いAPI (v1) → 廃止予定日: 6ヶ月後
新しいAPI (v2) → 本番運用開始

クライアント側の実装：
- v2への移行を段階的に進める
- v1は廃止予定日まで並行運用
- 廃止予定日を過ぎたら、v1へのリクエストは エラーレスポンスを返す
```

この方が、曖昧に「まだ使われている」と判断するより、開発チーム全体で対応計画を立てやすくなります。

### バージョンチェック機能の役割を限定する

バージョンチェック機能は、**ユーザーへの更新促進**という目的に限定し、サーバー側の廃止判断には使わないというルールを引く方法もあります。

- クライアント側：古いバージョンの使用率を監視し、ユーザーに更新を促す
- サーバー側：廃止スケジュールは独立して決定し、廃止予定日に従う

### 段階的な廃止戦略を設計する

すべてのユーザーを一度に新バージョンに移行させるのは難しい場合、段階的な廃止が現実的です。

例えば、新規ユーザーは新APIのみ対応、既存ユーザーは一定期間は古いAPIもサポート、というアプローチです。ただしこの場合も、「いつまでサポートするか」を明確に決めておく必要があります。

## 中小規模の開発組織が取るべきアクション

最後に、実務的なチェックリストを示します。

- [ ] 現在のバージョンチェック機能が、何のために存在するか明確にしたか
- [ ] 廃止予定のAPIについて、削除予定日を決めているか
- [ ] その予定日が、クライアント側の更新計画と整合しているか
- [ ] 古いバージョン対応による、サーバー側の実装コストを定量的に把握しているか
- [ ] セキュリティパッチが必要になった場合の対応方針を決めているか
- [ ] クライアント側とサーバー側で、廃止スケジュールについて共通認識を持っているか

バージョンチェック機能は、あくまで**監視ツール**です。それを根拠に対応を先送りすれば、後々より大きなコストがかかります。「使われている」という事実と、「対応が必要か」という判断は、しっかり分けて考えることが大事です。
