## 「スケールするから」で読み取り専用レプリカを追加した後の現実

読み取り専用レプリカ（Read-Only Replica）を増やすことは、スケーラビリティの基本戦略です。マスターデータベースへの読み取り負荷を分散し、応答時間を改善できます。ただ現場では、レプリカ数を増やした直後から「古いデータが返される」という報告が増え始めることがあります。

これは単なる「レプリケーション遅延」ではなく、アーキテクチャ全体でデータ鮮度の許容基準を見誤った結果です。レプリカが1台のときは「遅延は無視できる範囲」だったのに、3台、5台と増やすと、統計的に古いデータが返される確率が急速に高まります。そのメカニズムと、実務での対応策を整理します。

## なぜレプリカ数と遅延リスクが非線形に増加するのか

### 個別のレプリケーション遅延ばらつき

マスターから各レプリカへの同期は、ネットワーク状態、ディスク I/O、バイナリログ処理の速度によって異なります。理想的には全レプリカが同じタイミングで同期されますが、実際には：

- レプリカ A：平均 50ms 遅延（最大 200ms）
- レプリカ B：平均 80ms 遅延（最大 300ms）
- レプリカ C：平均 120ms 遅延（最大 500ms）

こうしたばらつきが出ます。レプリカ数が増えるほど、「最も遅いレプリカ」が遅延の支配要因になります。

### 読み取り分散時の「最悪ケース」の確率が増える

負荷分散ロードバランサーがラウンドロビンやランダムでレプリカを選択する場合、同じクライアントが常に同じレプリカに接続するわけではありません。データ更新直後に、偶然にも最も遅いレプリカに接続される確率は、レプリカ数に比例して高まります。

例えば：
- レプリカ 1 台：「古いデータが返される」確率 = 5%（仮定）
- レプリカ 3 台：確率 = 15〜20%（単純計算では 15%）
- レプリカ 5 台：確率 = 25%以上

実際には、バースト的な更新が集中したとき、複数のレプリカが同時に遅延し、この確率はさらに上昇します。

## アーキテクチャ設計で見落とされやすい判断ポイント

### 1. 「許容できる遅延時間」の定義が曖昧なまま進む

多くのチームは RTO（目標復旧時間）や RPO（目標復旧地点）は定義しても、読み取りレプリカの「許容遅延」を明示的に決めていません。結果として：

- 在庫管理システム：1秒以上の遅延は許されない（売上超過の原因）
- 分析ダッシュボード：数分の遅延は許容できる
- ユーザープロフィール表示：数秒なら実用上問題ない

こうした要件は業務ロジックによって全く異なるのに、「読み取り専用レプリカなら使える」と一括判断されやすいのです。

### 2. レプリカ選択の責任が曖昧になる

アプリケーションコードで「このクエリはマスターから読む」「このクエリはレプリカから読む」を明確に分けていないと、後から「なぜこのデータが古いのか」を追跡できなくなります。

ORM のレプリケーション自動振り分けは便利ですが、その裏で「どのクエリがどのレプリカに行くのか」が隠蔽されます。

## 実装上の具体的な対応策

### パターン 1：クリティカルな読み取りはマスターから

```python
# 在庫確認は常にマスターから
inventory = db.master.query(Inventory).filter_by(product_id=123).first()

# 過去の注文履歴はレプリカでも OK
history = db.replica.query(Order).filter_by(user_id=456).all()
```

マスターへの読み取り負荷が心配な場合は、キャッシュレイヤー（Redis など）を挟みます。

### パターン 2：遅延許容度を明示的にタグ付けする

```python
class QueryIntent(Enum):
    STRONG_CONSISTENCY = "master"      # 最新データが必須
    EVENTUAL_CONSISTENCY = "replica"   # 数秒の遅延は許容
    ANALYTICS = "replica_slow"         # 数分の遅延も許容

def get_user_balance(user_id, intent=QueryIntent.STRONG_CONSISTENCY):
    db_connection = get_db(intent)
    return db_connection.query(UserBalance).filter_by(id=user_id).first()
```

このように意図を明示すれば、後から「なぜこのクエリはレプリカを使ったのか」が追跡可能になります。

### パターン 3：レプリケーション遅延の監視と閾値設定

```sql
-- MySQL の場合
SHOW SLAVE STATUS\G

-- 確認すべき項目
-- Seconds_Behind_Master: マスターとの遅延秒数
-- Slave_IO_Running, Slave_SQL_Running: 同期プロセスの状態
```

監視ツール（Prometheus、Datadog など）で遅延を常時追跡し、閾値を超えたら：
- アラート発火
- 自動的にそのレプリカを読み取り対象から除外
- 管理者に通知

```yaml
# Prometheus アラート例（概念）
alert: ReplicationLagTooHigh
condition: seconds_behind_master > 5
action: exclude_replica_from_load_balancer
```

### パターン 4：読み取り後の検証

クリティカルな操作の直後は、同じマスターから読み直して検証する手法もあります。

```python
# 在庫を更新
db.master.execute("UPDATE inventory SET quantity = quantity - 1 WHERE id = ?", [item_id])

# 更新直後の確認は必ずマスターから
updated_qty = db.master.query(Inventory).filter_by(id=item_id).first().quantity

if updated_qty < 0:
    # ロールバック処理
    db.master.rollback()
```

## 運用時にハマりやすい落とし穴

### 落とし穴 1：「レプリカが落ちた」ときの自動フェイルオーバー

レプリカ 5 台のうち 1 台が落ちた場合、ロードバランサーがそれを検知するまでに数秒〜数十秒かかることがあります。その間、他の 4 台に負荷が集中し、遅延が悪化します。

**対応策**：レプリカの健全性チェック間隔を短縮（例：5秒ごと）し、落ちたレプリカを素早く除外する仕組みを構築します。

### 落とし穴 2：レプリケーション遅延と「クエリの遅さ」の混同

遅延が 100ms でも、レプリカ自体が遅いクエリ（全テーブルスキャンなど）を実行していれば、応答時間は 1 秒を超えます。「レプリカが遅い」という報告の背景に、遅延ではなく不適切なクエリが隠れていることは珍しくありません。

**対応策**：遅延と応答時間を分けて監視し、原因を正確に特定します。

### 落とし穴 3：レプリカの増設で「スケール感」を勘違いする

読み取り負荷が 10 倍になったからレプリカを 10 台追加する、という判断は危険です。レプリカが増えるほど、マスターからのバイナリログ送信負荷も増え、マスター自体がボトルネックになります。また、複数のレプリカ同期で遅延ばらつきも大きくなります。

**現実的な目安**：同じスペックのレプリカなら 3〜5 台程度が限度。それ以上の負荷分散が必要なら、読み取り専用キャッシュやデータウェアハウスの検討も必要です。

## 小規模チームでの導入ステップ

1. **現状把握**：マスターへの読み取りクエリの割合を計測。実際に負荷がかかっているか確認
2. **要件整理**：各機能ごとに「許容遅延」を定義。全てのクエリが強い一貫性を必要とするわけではない
3. **段階的導入**：レプリカ 1 台で始め、遅延と応答時間を 1 週間以上監視
4. **クエリ分類**：クリティカルな読み取り（マスター）と許容可能な読み取り（レプリカ）を明確に分ける
5. **監視設定**：遅延閾値を定め、超過時は自動で読み取り対象から除外する仕組みを構築

## 最後に

読み取り専用レプリカは強力なツールですが、「増やせば増やすほど良い」という単純な話ではありません。レプリケーション遅延は、レプリカ数の増加に対して非線形に悪化します。

アーキテクチャ設計の段階で、どのデータがどの程度の鮮度を必要とするのかを明確にし、その上でレプリカ数を決めることが、後の運用トラブルを大きく減らします。
