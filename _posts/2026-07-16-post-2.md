## 問題の背景

トークンベースの認証を導入するとき、セキュリティと利便性のバランスを取るために「リフレッシュトークン」を使う設計は一般的です。しかし現場では、リフレッシュトークンの有効期限ポリシーとローテーション戦略を独立して決めてしまい、その結果セッション期間の予測が失われるケースが少なくありません。

特にGoなどの軽量フレームワークでAPIサーバーを構築する際、ミドルウェアレベルでトークン検証を実装していると、この矛盾が潜在的な問題として残りやすくなります。

実装者の視点では「リフレッシュトークンがあれば、有効期限が切れても新しいトークンを取得できるから大丈夫」と考えがちです。しかし運用側や監査の視点では「ユーザーセッションが実際にいつまで続くのか」が不透明になり、ログ分析やセキュリティインシデント対応のときに判断が難しくなります。

## 何が起きるのか

具体的な場面を想像してみてください。

APIサーバーの設計として、以下のようなポリシーが定められているとします：

- **アクセストークン**: 有効期限15分
- **リフレッシュトークン**: 有効期限7日間
- **ローテーション戦略**: リフレッシュトークンは使用されるたびに新しいトークンを発行する

一見すると合理的に見えます。短命なアクセストークンでセキュリティを確保しつつ、リフレッシュトークンで利便性を保つ──という典型的なアプローチです。

しかし、ここに落とし穴があります。

ユーザーが最後にリフレッシュトークンを使用してから7日間、一度もアプリを開かなかった場合、古いリフレッシュトークンは有効期限切れになります。ユーザーが8日目にアプリを開いて、保存されていた古いリフレッシュトークンで新しいアクセストークンをリクエストすると、サーバーは「このトークンはもう有効ではない」と拒否します。

ここまでは設計通りです。問題は、その後の挙動が環境やミドルウェアの実装によって異なることです。

## 検証の目的と前提

この記事では、以下の点を検証しました：

- リフレッシュトークンの有効期限とローテーション戦略が矛盾した場合、アプリケーションのセッション期間がどのように不確定になるのか
- Goのミドルウェア実装で、この問題がどの層で顕在化するのか
- 実務的な対策として、どのような設計判断が有効か

**検証環境**:
- Go 1.21以降のHTTPサーバー
- JWT（JSON Web Token）ベースの実装
- SQLiteでトークンのローテーション履歴を記録
- クライアント側でトークンの自動リフレッシュを実装したシナリオ

## 実装パターンと失敗例

まず、不適切な設計例を示します。

```go
// リフレッシュトークンの検証ロジック（簡略版）
func validateRefreshToken(token string) (bool, error) {
    // トークンの署名を検証
    claims, err := jwt.ParseWithClaims(token, &CustomClaims{}, func(token *jwt.Token) (interface{}, error) {
        return secretKey, nil
    })
    
    if err != nil {
        return false, err
    }
    
    customClaims := claims.Claims.(*CustomClaims)
    
    // 有効期限チェック（JWTの exp クレーム）
    if customClaims.ExpiresAt < time.Now().Unix() {
        return false, errors.New("token expired")
    }
    
    // ここまで。データベースの確認がない。
    return true, nil
}
```

このコードの問題点は、JWTの有効期限チェックだけで、実際にそのトークンがローテーションされたかどうかを確認していないことです。

もし「ローテーション戦略として、一度使われたリフレッシュトークンは無効にする」という方針があるなら、データベースでトークンの使用状況を追跡する必要があります。しかし多くの場合、この追跡は忘れられるか、後付けされます。

```go
// より適切な実装例
func validateRefreshToken(token string) (bool, error) {
    claims, err := jwt.ParseWithClaims(token, &CustomClaims{}, func(token *jwt.Token) (interface{}, error) {
        return secretKey, nil
    })
    
    if err != nil {
        return false, err
    }
    
    customClaims := claims.Claims.(*CustomClaims)
    
    // 1. 有効期限チェック
    if customClaims.ExpiresAt < time.Now().Unix() {
        return false, errors.New("token expired")
    }
    
    // 2. データベースでトークンの状態を確認
    tokenRecord, err := db.GetTokenRecord(customClaims.TokenID)
    if err != nil {
        return false, err
    }
    
    // 3. すでに使用済みか、無効化されているか確認
    if tokenRecord.IsRevoked || tokenRecord.IsUsed {
        return false, errors.New("token has been revoked or used")
    }
    
    return true, nil
}
```

## セッション期間の予測が失われる理由

この問題の本質は、「リフレッシュトークンの有効期限」と「ローテーション時の新トークン発行タイミング」が独立した決定になっているところです。

例えば、以下のシナリオを考えてみます：

**シナリオ1**: ユーザーがアプリを毎日使用している場合
- リフレッシュトークンは毎回ローテーションされるため、常に新しい7日間の有効期限を得る
- 実質的なセッション期間は「ユーザーが使い続ける限り無制限」に近くなる

**シナリオ2**: ユーザーが3日間アプリを使わない場合
- その間、リフレッシュトークンはローテーションされない
- 4日目に再度アプリを開くと、3日前のリフレッシュトークンは有効だが、古いローテーション戦略に基づいて発行されたものかもしれない
- セッション期間の正確な終了時刻が不確定になる

この不確定性は、セキュリティ監査やアクセスログの分析のときに問題になります。「このユーザーのセッションはいつまで有効だったのか」という質問に、明確に答えられなくなるのです。

## 設計判断の分岐点

現場では、この問題に直面したとき、大きく3つの判断に分かれることが多いです。

**パターンA: 有効期限を絶対視する**
- リフレッシュトークンの有効期限を厳密に守る
- ローテーション戦略は簡略化し、新トークン発行時に有効期限を再計算する
- 実装は単純だが、ユーザーが長期間使わないと再ログインを強制される

**パターンB: ローテーション戦略を絶対視する**
- リフレッシュトークンを使用するたびに新しいトークンを発行し、常に新しい有効期限を得る
- セッション期間は実質的に無制限に近くなり、セキュリティリスクが増す
- ユーザー体験は良いが、セッション監査が難しくなる

**パターンC: 両者を調整する（推奨）**
- リフレッシュトークンの有効期限を固定する（例：7日間）
- ローテーション時に新しい有効期限を付与するのではなく、元のトークンの有効期限を継承する
- さらに「最後の使用日時」を記録し、一定期間未使用なら無効化する

## 実装上の注意点

パターンCを採用する場合、以下の点に注意が必要です。

**1. トークン発行時刻の管理**
```go
type RefreshToken struct {
    TokenID   string
    UserID    string
    IssuedAt  int64  // 最初に発行された時刻
    ExpiresAt int64  // 有効期限（IssuedAtから相対的に計算）
    LastUsedAt int64 // 最後に使用された時刻
    IsRevoked bool
}

// リフレッシュトークン生成時
func issueRefreshToken(userID string) string {
    now := time.Now().Unix()
    claims := &CustomClaims{
        TokenID: generateTokenID(),
        UserID: userID,
        IssuedAt: now,
        ExpiresAt: now + (7 * 24 * 60 * 60), // 7日間
        StandardClaims: jwt.StandardClaims{
            ExpiresAt: now + (7 * 24 * 60 * 60),
        },
    }
    
    token := jwt.NewWithClaims(jwt.SigningMethodHS256, claims)
    return token.SignedString(secretKey)
}
```

**2. ローテーション時の有効期限の扱い**

新しいトークンを発行するとき、元のトークンの有効期限を引き継ぐことが重要です。

```go
func rotateRefreshToken(oldToken string) (string, error) {
    // 古いトークンをパース
    oldClaims, err := parseToken(oldToken)
    if err != nil {
        return "", err
    }
    
    // 新しいトークンは、元のExpiresAtを保持する
    newClaims := &CustomClaims{
        TokenID: generateTokenID(),
        UserID: oldClaims.UserID,
        IssuedAt: time.Now().Unix(),
        ExpiresAt: oldClaims.ExpiresAt, // ← 元の有効期限を継承
        StandardClaims: jwt.StandardClaims{
            ExpiresAt: oldClaims.ExpiresAt,
        },
    }
    
    newToken := jwt.NewWithClaims(jwt.SigningMethodHS256, newClaims)
    return newToken.SignedString(secretKey)
}
```
