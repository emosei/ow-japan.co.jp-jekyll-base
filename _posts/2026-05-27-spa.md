## 現場で起きている問題

SPAで状態管理をローカルストレージに依存する実装をしていると、複数タブを開いた時点で予期しない動作が報告されることがあります。ユーザーが同じWebアプリケーションを複数のブラウザタブで開いて、それぞれ異なる操作を進めると、一方のタブで保存したデータが他方に反映されない、あるいは古いデータで上書きされてしまう──こうした競合が生じるのです。

「ローカルストレージは単一プロセスのメモリ管理と同じ感覚で使えばいい」という誤解が根底にあります。実際にはブラウザの複数タブは独立したJavaScriptコンテキストで動作し、ローカルストレージは共有リソースであるにもかかわらず、同期機構が明示的に実装されていないケースが大半です。この記事では、その理由を検証し、現場で採用できる対策を整理します。

## ローカルストレージが「共有」だと認識されない理由

ローカルストレージの仕様を改めて確認しましょう。

ローカルストレージは同一オリジン内のすべてのタブ・ウィンドウで共有されます。これは仕様上、明確に定義されています。しかし、JavaScriptコードを書く側では「自分のタブのメモリに値を保存している」という感覚で扱ってしまいやすいのです。

なぜか。理由は単純で、通常の開発では複数タブの同時操作を想定してテストしないからです。ローカル環境での開発、単一タブでの機能確認、ユニットテストの範囲では、この問題は顕在化しません。本番環境でユーザーが複数タブを開いて初めて気づく──という流れが繰り返されています。

## 検証の前提と構成

実際に動作を確認するため、シンプルなシナリオで検証しました。

**検証環境:**
- Chrome、Firefox の最新版
- 同一オリジン内での複数タブ動作
- 状態管理ライブラリなし（素のJavaScriptのみ）
- 同期メカニズムなし（ベースラインとして）

**テストシナリオ:**
1. タブAで状態を更新してローカルストレージに保存
2. タブBで同じデータを読み込んで表示
3. タブAで再度更新
4. タブBが古いデータのままになっていないか確認

## 同期なしの場合──何が起きるか

まず、同期機構なしで実装した場合のコード例です。

```javascript
// 状態管理（同期なし）
class SimpleStateManager {
  constructor() {
    this.state = this.loadFromStorage();
  }

  loadFromStorage() {
    const stored = localStorage.getItem('appState');
    return stored ? JSON.parse(stored) : { count: 0 };
  }

  updateState(newState) {
    this.state = { ...this.state, ...newState };
    localStorage.setItem('appState', JSON.stringify(this.state));
  }

  getState() {
    return this.state;
  }
}

const manager = new SimpleStateManager();

// ボタンクリック時
document.getElementById('incrementBtn').addEventListener('click', () => {
  const current = manager.getState();
  manager.updateState({ count: current.count + 1 });
  document.getElementById('display').textContent = manager.getState().count;
});
```

この実装で複数タブを開いて実験すると：

- **タブA** で「+」ボタンを3回押す → ローカルストレージに `count: 3` が保存される
- **タブB** は初期化時に `count: 0` をメモリにロードしたまま → 画面上は `0` のまま
- タブBで「+」ボタンを押す → メモリ上の `0` に `1` を加えて `count: 1` を保存 → タブAの `3` が上書きされる

結果として、タブAの更新が失われます。これが「競合」の実体です。

## 検証：storage イベントを使った同期

ブラウザは `storage` イベントで、別のタブからのローカルストレージ変更を通知します。これを活用すれば、複数タブ間での同期が可能になります。

```javascript
// 同期機構を持つ状態管理
class SyncedStateManager {
  constructor() {
    this.state = this.loadFromStorage();
    this.listeners = [];
    this.setupStorageListener();
  }

  loadFromStorage() {
    const stored = localStorage.getItem('appState');
    return stored ? JSON.parse(stored) : { count: 0 };
  }

  setupStorageListener() {
    window.addEventListener('storage', (event) => {
      if (event.key === 'appState' && event.newValue) {
        this.state = JSON.parse(event.newValue);
        this.notifyListeners();
      }
    });
  }

  updateState(newState) {
    this.state = { ...this.state, ...newState };
    localStorage.setItem('appState', JSON.stringify(this.state));
    this.notifyListeners();
  }

  subscribe(listener) {
    this.listeners.push(listener);
  }

  notifyListeners() {
    this.listeners.forEach(listener => listener(this.state));
  }

  getState() {
    return this.state;
  }
}

const manager = new SyncedStateManager();

// UIの更新を購読
manager.subscribe((state) => {
  document.getElementById('display').textContent = state.count;
});

document.getElementById('incrementBtn').addEventListener('click', () => {
  const current = manager.getState();
  manager.updateState({ count: current.count + 1 });
});
```

この実装で同じシナリオを実行すると：

- タブAで更新 → `storage` イベント発火 → タブBのメモリが更新される
- タブBで「+」ボタンを押す際は、常に最新の状態から計算される
- 結果として、どのタブで操作しても一貫性が保たれます

## 実装時の注意点と落とし穴

検証を通じて、いくつかの実務的な課題が浮かび上がります。

### 1. storage イベントは同一タブでは発火しない

重要な制限です。`storage` イベントは **別のタブ** で変更が起きた時だけ通知されます。同じタブ内での変更は検知されません。つまり、上記のコード例では、タブ自身の更新を `notifyListeners()` で手動で呼び出す必要があります。これを忘れると、同じタブ内での状態同期が失敗します。

### 2. ローカルストレージの容量制限

一般的に5〜10MBの制限があります。複雑な状態をそのまま保存していると、すぐに限界に達します。現場では「キャッシュとしてのローカルストレージ」と「永続化が必要なデータ」を明確に分けることが重要です。

### 3. 同期タイミングの遅延

`storage` イベントの通知は即座ではなく、ブラウザの内部スケジューリング次第です。高頻度での更新が続く場合、タブ間で数十ミリ秒の遅延が生じることがあります。UIの反応性が求められる場面では、この遅延が目立つことがあります。

### 4. 複数の状態管理ライブラリとの競合

ReduxやVuexなどの状態管理ライブラリを使っている場合、ローカルストレージとの同期戦略が複雑になりやすいです。ライブラリ側の更新ロジックとローカルストレージの同期が二重になり、バグの温床になります。

## 実務投入前に確認すべきポイント

### ユースケースの整理

ローカルストレージの同期が本当に必要なのか、まず問い直します。

- **不要な場合:** ユーザーが同時に複数タブを開かない、あるいは開いても独立した操作をしない業務アプリケーション
- **必要な場合:** ショッピングカート、フォーム下書き、ユーザープリファレンスなど、複数タブで一貫性が求められる機能

### サーバー側の状態管理との関係

ローカルストレージはあくまで「クライアント側のキャッシュ」です。真実の源泉はサーバーにあるべきです。複数タブで同期できても、サーバーとの不整合が生じれば意味がありません。特に、ユーザーが複数のデバイスやブラウザから同じアプリケーションにアクセスする場合は、サーバー側の設計が重要になります。

### 代替手段の検討

状態管理ライブラリ（Redux Persist、Vuex Persist など）を使うと、ローカルストレージとの同期が自動化されます。ただし、ライブラリの学習コストと依存関係が増えるトレードオフがあります。小規模なプロジェクトなら、上記の `storage` イベントの手動実装で十分です。

## まとめ：どんな案件で採用すべきか

複数タブ間でのローカルストレージ同期は、以下のような案件で実装価値があります。

- **社内システム・管理画面:** ユーザーが複数タブで同じ画面を開く可能性が高い
- **フォーム入力系:** 下書きを複数タブで共有する場合
- **キャッシュ戦略が複雑な場合:** 手動でキャッシュ無効化を制御する必要がある

一方、以下の場合は慎重に判断します。

- **リアルタイム性が高い業務:** storage イベントの遅延が許容できない
- **大規模な状態管理:** ローカルストレージの容量制限に引っかかりやすい
- **モバイルアプリのWebView:** 環境によって `storage` イベントの動作が異なることがある

現場では「ローカルストレージは単純な永続化機構」という認識を改め、複数タブという現実的なユースケースを設計段階から織り込むことが、後々のバグ報告や保守負荷を減らすコツです。
