## 現場で起こる「実は実行済み」問題

ドメインイベントを非同期で複数のリスナーに配信する仕組みを持つシステムで、こんな場面に遭遇したことはないでしょうか。

イベントが発火され、3つのリスナーが順に処理される予定だった。ところが2番目のリスナーでデータベース接続エラーが発生し、イベント配信が途中で失敗に終わった。その後、運用チームから「このイベント、再度配信してもいい？」と聞かれて、答えに窮してしまう──。

1番目のリスナーは既に実行済みだ。再配信すれば重複実行される。でも、3番目のリスナーはまだ実行されていない。スキップするわけにもいかない。

こうした状況は、イベント駆動のアーキテクチャを採用した現場では珍しくありません。特に複数の外部システムやマイクロサービスと連携するシステムでは、各リスナーの実行状態を正確に把握できないまま運用が進むことがあります。

## なぜこの設計が見落とされるのか

多くの場合、イベント駆動設計の初期段階では「イベントが発火される→リスナーが実行される」という単純な流れに目が向きます。フレームワークが提供する基本的なイベント機構を使い、リスナーを登録して終わり──という実装になりやすいのです。

しかし本番環境では、ネットワーク障害、一時的なリソース枯渇、外部API呼び出しのタイムアウト、データベースロックなど、様々な理由で途中失敗が発生します。その時点で「どのリスナーまで完了したのか」を知る手段がないと、以下のような判断を迫られます。

- **再実行時の重複リスク**：既に実行済みのリスナーをもう一度走らせてしまう
- **部分的な実行状態の放置**：一部のリスナーだけ未実行のまま、データの不整合が生じる
- **手作業による確認と修正**：ログを手で追ったり、データベースを直接確認したり、属人的な対応が増える

こうした状況は、システムが成長して連携先が増えるほど、深刻化します。

## 追跡設計に必要な3つの要素

途中失敗時の状態を把握し、安全に復旧するには、次の3つを組み込む必要があります。

### 1. イベント配信の実行履歴記録

イベントが発火された時点で、そのイベント自体に一意のIDを付与し、配信ログを記録します。

```
EventLog テーブル例
- event_id (UUID)
- event_type (string)
- aggregate_id (string)
- payload (JSON)
- created_at (timestamp)
- status (enum: pending, in_progress, completed, failed)
```

イベントが「pending」から「in_progress」に遷移した時点で、その配信が開始されたことが明確になります。

### 2. リスナー単位の実行記録

各リスナーの実行結果を個別に記録することが重要です。

```
ListenerExecutionLog テーブル例
- event_id (UUID)
- listener_name (string)
- status (enum: pending, success, failed)
- error_message (text, nullable)
- executed_at (timestamp, nullable)
- retry_count (integer)
```

このテーブルがあれば、あるイベントに対して「AというリスナーはOK、Bはエラー、Cはまだ実行されていない」という状態が一目でわかります。

### 3. べき等性の確保

リスナーの実装側では、同じイベントが複数回配信されても副作用が重複しないよう設計します。これはイベント駆動の基本原則ですが、実装時に後回しにされやすい部分です。

例えば、「注文確定イベント」を受けたリスナーが在庫を減らす場合、単純に `UPDATE inventory SET quantity = quantity - 1` では危険です。

```
-- 危険な実装（べき等性がない）
UPDATE inventory SET quantity = quantity - 1 WHERE product_id = ?;

-- 安全な実装（べき等性がある）
INSERT INTO inventory_deduction (event_id, product_id, quantity)
VALUES (?, ?, 1)
ON CONFLICT (event_id, product_id) DO NOTHING;

UPDATE inventory SET quantity = quantity - 1 
WHERE product_id = ? 
AND EXISTS (
  SELECT 1 FROM inventory_deduction 
  WHERE event_id = ? AND product_id = ?
);
```

イベントIDをキーに重複チェックを行い、同じイベントからの重複実行を防ぐわけです。

## 復旧フローの設計

実行履歴があれば、以下のような復旧フローが可能になります。

1. **失敗したイベントを特定**：ログから `status = 'failed'` のイベントを検索
2. **実行済みリスナーを確認**：そのイベントに対する `ListenerExecutionLog` で `status = 'success'` のものをチェック
3. **未実行リスナーのみ再実行**：`status = 'pending'` のリスナーだけを選別して再度配信
4. **べき等性により安全に再実行**：既に実行済みのリスナーが再度走っても、二重実行の影響はない

このフローなら、運用チームが手作業で判断する必要はなく、自動化された復旧ロジックで対応できます。

## 実装時の現実的な工夫

小規模なチームでも実装可能な工夫をいくつか紹介します。

**段階的な導入**：最初から全リスナーに履歴記録を付けるのではなく、重要度が高いリスナー（外部API呼び出し、金銭に関わる処理）から始める。

**既存フレームワークの活用**：多くのイベント駆動フレームワークは、ミドルウェアやインターセプタの機構を持っており、ここに履歴記録ロジックを挿入できます。各リスナーを修正する必要がありません。

**監視とアラート**：実行履歴ログから「同じイベントが複数回失敗している」「特定のリスナーが常に失敗している」といったパターンを検出し、早期に気付くことができます。

## 見落としやすい落とし穴

**タイムアウトと部分実行の境界**：リスナー実行中にタイムアウトが発生した場合、そのリスナーが「失敗」したのか「実行中」なのか判定が曖昧になりやすいです。実行開始時点で `status = 'in_progress'` に更新し、完了時に `status = 'success'` に変更することで、この曖昧性を減らせます。

**リスナーの実行順序依存**：リスナーAの結果に基づいてリスナーBが動作する場合、単純な非同期配信では順序が保証されません。この場合は、イベント駆動ではなくサガパターンやワークフローエンジンの導入を検討すべき信号です。

**ログ肥大化**：大量のイベントが発火されるシステムでは、実行履歴ログが急速に増え、クエリ性能が低下します。定期的なアーカイブ、インデックス戦略、ログ保持期間の設定が必要です。

## まとめ

ドメインイベントの非同期配信は、システムの疎結合性を高める強力な手段です。しかし、その利便性の裏側には「途中失敗時の状態把握」という運用上の課題が隠れています。

アーキテクチャとして「イベント駆動」を採用する判断をした時点で、その復旧設計も同時に考える必要があります。実行履歴の記録、リスナー単位での追跡、べき等性の確保──これらは事後的に付け足すと工数が跳ね上がります。初期設計の段階で組み込むことが、後々の運用負荷を大きく減らします。
