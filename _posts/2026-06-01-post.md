## 在庫確認と決済の間に「すき間」が生まれるのはなぜか

予約システムを構築するとき、誰もが最初に心配することがあります。それは「同じ席を2人以上に売ってしまう」という二重予約です。

現場では実装者も経営層も「これは絶対に起こしてはいけない」と考えます。でも正直に言うと、完全に防ぎ切ることは難しい。それは技術的な限界というより、**実務的な制約と理論的な理想のズレ**から生まれます。

旅行や宿泊の予約では、以下のような流れが起きます。

1. ユーザーが「この日程、この部屋を予約したい」と申し込む
2. システムが在庫（空き部屋）を確認する
3. 決済処理を実行する
4. 決済成功後、在庫を確定的に減らす

この流れの中で、確認と決済の間に**タイムラグ**が発生します。その間に別のユーザーの予約が入ると、理論上は二重予約が起こり得るのです。

## トランザクション分離レベルの落とし穴

多くのエンジニアが最初に考えるのは「トランザクション分離レベルを上げればいい」という解法です。

SQLiteやPostgreSQLでは以下のような分離レベルが用意されています。

- **READ UNCOMMITTED**: 最も低い。ダーティリード（確定していない変更が見える）が起こる
- **READ COMMITTED**: 確定済みのデータしか見えない。予約システムではこれが標準
- **REPEATABLE READ**: トランザクション中の読み込みが一貫性を保つ
- **SERIALIZABLE**: 最も厳密。すべてのトランザクションが順序付けられて実行される

理屈上は「SERIALIZABLEにすれば二重予約は起こらない」と思えます。でも実務では違います。

```sql
-- SERIALIZABLE で実装した場合
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
  SELECT available_rooms FROM inventory WHERE room_id = 101 FOR UPDATE;
  -- 在庫確認
  
  -- ここで決済APIを呼ぶ（外部サービス）
  CALL external_payment_api(user_id, amount);
  
  UPDATE inventory SET available_rooms = available_rooms - 1 WHERE room_id = 101;
COMMIT;
```

問題は「決済API呼び出し」です。外部のクレジットカード決済や銀行振込の確認は、トランザクション内で数秒から数分かかることがあります。その間、他のユーザーのトランザクションは**ロック待ちになり、システム全体が遅くなります**。

現場ではこれを「デッドロック地獄」と呼びます。予約が殺到する時間帯（年末年始、GWなど）に、SERIALIZABLEで運用すると、ユーザーは「予約ボタンを押したのに何十秒も反応がない」という経験をします。

## 実務では「楽観的ロック」と「確認メール」の組み合わせになる

そこで現場の予約システムは、理論的な完全性を諦めて、**実用的な妥協**を取ります。

典型的なアプローチは以下です。

```sql
-- テーブル設計
CREATE TABLE room_inventory (
  room_id INT,
  available_count INT,
  version INT,  -- 楽観的ロック用
  PRIMARY KEY (room_id)
);

-- 予約時の処理（疑似コード）
BEGIN TRANSACTION ISOLATION LEVEL READ COMMITTED;
  -- 1. 在庫確認（読み込みのみ）
  SELECT available_count, version FROM room_inventory 
    WHERE room_id = 101;
  
  -- 2. 決済処理（トランザクション外で実行）
  payment_result = call_external_payment_api(user_id, amount);
  
  -- 3. 決済成功後、楽観的ロックで在庫更新を試みる
  UPDATE room_inventory 
    SET available_count = available_count - 1, version = version + 1
    WHERE room_id = 101 AND version = <読み込み時のversion>;
  
  -- 4. 更新行数が0なら、別のユーザーが先に予約した
  IF affected_rows == 0 THEN
    -- 決済は既に成功しているので、キャンセル処理を実行
    call_cancel_payment_api(payment_id);
    RETURN "申し訳ありません。他のユーザーが先に予約されました。";
  END IF;
COMMIT;
```

このアプローチのポイント：

- **在庫確認は読み込みのみ**で、ロックを取らない
- **決済処理はトランザクション外**で実行し、システム全体の応答性を保つ
- **楽観的ロック（version番号）**で、更新時に競合を検出する
- 競合が起きたら、既に成功した決済をキャンセルする

この方法では「ごく稀に」二重予約が発生する可能性が残ります。でも実装が単純で、決済が遅い環境でも応答性を失いません。

## 現場で起こる「隠れた二重予約」

しかし、ここからが厄介です。楽観的ロックで競合を検出しても、**決済のキャンセルが失敗する**ケースが現実にあります。

- クレジットカード決済は成功したが、キャンセルAPIが通信エラーで失敗
- 銀行振込は既に着金しているが、システムは「キャンセル待ち」状態に
- 決済サービスのタイムアウトで、決済の成否が不明（Schrödinger's payment）

こうなると、ユーザーには「予約できませんでした」と表示されているのに、バックエンド側では決済が成功している。その後の手動対応で、実質的な二重予約状態が生まれます。

## 現実的な対策：確認メールと管理画面

完全性を求めるより、以下の実装が現場では有効です。

**1. 予約確定メールを「即座に」送らない**

```
予約申し込み → システム確認中 → 3分後に確認メール送付
```

この間に、バックエンドで決済とロック競合の最終確認を行い、問題があれば別メールで通知します。ユーザーは「確認メールが来た時点で、予約が本当に確定した」と判断できます。

**2. 管理画面で「決済済みだが予約未確定」の案件を検出**

決済成功したのに在庫更新に失敗した案件を自動抽出し、管理者が手動で対応する仕組みを作ります。

**3. 定期的な整合性チェック**

```sql
-- 日次バッチで実行
SELECT payment_id, user_id, room_id 
FROM payments 
WHERE status = 'success' 
  AND NOT EXISTS (
    SELECT 1 FROM reservations 
    WHERE reservations.payment_id = payments.payment_id
  );
```

決済は成功したのに予約レコードがない案件を検出し、自動で予約レコードを作成するか、管理者に通知します。

## 設計判断：どこまで「完全性」を求めるか

結局のところ、二重予約対策は**ビジネス要件と技術的コスト**のバランスです。

- **高級ホテルの限定スイート**: 二重予約は絶対に許されない → SERIALIZABLEでも遅延を受け入れる
- **一般的なホテル予約**: 稀な二重予約は起こるが、手動対応で対応 → 楽観的ロック + 確認メール
- **大量在庫（航空券など）**: 二重予約の確率は無視できる → READ COMMITTED で十分

小規模なチームであれば、最初は「楽観的ロック + 確認メール + 管理画面での検出」から始めるのが現実的です。その上で、実際に問題が起こったら、その時点で強化する。これが現場の判断です。
