## 本番環境で突然増える「予期しない型」の報告

Rubyで開発した機能が、ステージング環境では問題なく動いていたのに、本番環境に上がった途端に型関連のエラーが大量に発生する──こういう経験は、Ruby開発チームなら一度は経験しているのではないでしょうか。

エラーログを見ると `String expected (TypeError)` や `undefined method 'xxx' for nil:NilClass` といったメッセージが並び、「なぜステージングで検出されなかったのか」という問い合わせが現場から上がります。

この現象は、Rubyの動的型付けの特性と、開発環境と本番環境で流れるデータの質が大きく異なることが原因です。静的解析ツールだけに頼っていると、こうした問題は検出されません。本記事では、その仕組みと対策を検証します。

## 静的解析ツールが「見逃す」型の不一致

Rubyの開発現場では、RuboCop や Sorbet といった静的解析ツールが導入されていることが増えました。これらは非常に有用ですが、重要な制限があります。

**静的解析が検出できるのは、コード上に明示されている型情報の矛盾だけです。**

例えば、以下のようなコードを考えてみます。

```ruby
class OrderProcessor
  def calculate_total(items)
    items.map { |item| item[:price] * item[:quantity] }.sum
  end
end
```

このメソッドは `items` が配列で、各要素がハッシュで、`:price` と `:quantity` キーを持つことを前提としています。しかし、Rubyのコード上には型注釈がなければ、静的解析ツールはこの前提を理解できません。

問題は、**開発時には常に正しい形のデータが流れてくるが、本番では異なる形のデータが混在する可能性**があることです。

### データソースの違いが型不一致を招く

実務では、こういう状況が頻繁に発生します。

- APIから取得したJSONが、時には文字列で、時には数値で返される
- 外部サービスの仕様変更により、予期しない `null` が含まれるようになった
- レガシーシステムとの連携で、想定外のデータ形式が流入した
- ユーザー入力が、想定した形式以外で送信されてきた

開発環境では、テストデータが綺麗に整形されているため、こうした問題は発生しません。本番環境では、実際のユーザー行動や外部システムの予期しない変化により、データの質が変わります。

## 実行時検証が後付けで大量に必要になる理由

Rubyで型安全性を確保するには、**実行時の型チェックが不可欠**です。

以下は、前述の `calculate_total` メソッドを改善した例です。

```ruby
class OrderProcessor
  def calculate_total(items)
    raise TypeError, 'items must be an Array' unless items.is_a?(Array)
    
    items.map do |item|
      raise TypeError, 'item must be a Hash' unless item.is_a?(Hash)
      
      price = item[:price]
      quantity = item[:quantity]
      
      raise TypeError, 'price must be numeric' unless price.is_a?(Numeric)
      raise TypeError, 'quantity must be numeric' unless quantity.is_a?(Numeric)
      
      price * quantity
    end.sum
  end
end
```

このレベルの検証を最初から組み込むと、コード量が大幅に増えます。開発初期段階では「とりあえず動く」ことが優先されるため、こうした検証は後付けになりがちです。

しかし、本番環境で予期しないデータが流入すると、こうした検証が一気に必要になります。既存のコードベースに遡って検証ロジックを追加する作業は、想像以上に手間がかかります。

## 検証戦略：層別で実装時期を分ける

現場では、すべてのメソッドに完全な型チェックを入れるのは現実的ではありません。代わりに、**データが外部から流入する境界層を中心に検証を配置する**という戦略が有効です。

### 1. 入口層での検証（最優先）

APIやフォーム入力など、外部からのデータ受け取り箇所では、必ず型と形式を検証します。

```ruby
class OrderAPI
  def create(params)
    # 入口層での検証
    items = validate_items(params[:items])
    OrderProcessor.new.calculate_total(items)
  end
  
  private
  
  def validate_items(items)
    raise ArgumentError, 'items must be an Array' unless items.is_a?(Array)
    items.each do |item|
      raise ArgumentError, 'item must be a Hash' unless item.is_a?(Hash)
      raise ArgumentError, 'price is required' unless item.key?(:price)
      raise ArgumentError, 'quantity is required' unless item.key?(:quantity)
    end
    items
  end
end
```

### 2. 内部メソッドの検証（段階的）

入口層で検証済みのデータに対して、内部メソッドは最小限の検証に留めます。

```ruby
class OrderProcessor
  def calculate_total(items)
    # 入口層で検証済みと仮定
    items.map { |item| item[:price] * item[:quantity] }.sum
  end
end
```

### 3. 外部ライブラリ呼び出しの返却値検証

データベースやHTTPクライアントなど、外部ライブラリからの返却値は、その仕様が変わる可能性があるため検証します。

```ruby
def fetch_user(id)
  response = external_api.get_user(id)
  
  raise TypeError, 'response must be a Hash' unless response.is_a?(Hash)
  raise TypeError, 'user_id is missing' unless response[:user_id]
  
  response
end
```

## 実務投入時の確認ポイント

本番環境で型関連エラーを最小化するには、以下を確認します。

**1. データソースの仕様確認**
- APIドキュメントに記載されている型定義と、実際の返却値が一致しているか
- 外部サービスの仕様変更予定はないか
- レガシーシステムとの連携では、予期しない形式が流入する可能性はないか

**2. 本番データの事前確認**
- ステージング環境で、本番と同じデータボリューム・パターンを使用しているか
- 実際の本番データサンプルを取得して、開発時の想定と比較しているか

**3. ログレベルと監視の設定**
- 型エラーが発生した際、詳細なスタックトレースとコンテキストが記録されるか
- 型エラーの発生率を監視し、閾値超過時にアラートが出るか

**4. フェイルセーフの設計**
- 型エラーが発生した際、システム全体が停止するのか、その処理だけがスキップされるのか
- ユーザーに対して、どのようなエラーメッセージが返却されるのか

## 向き・不向きと案件での採用判断

**Rubyの動的型付けが向いている案件：**
- 要件が頻繁に変わり、型定義の変更コストが高い場合
- プロトタイピングや初期開発段階
- データ構造がシンプルで、外部との連携が少ない

**型検証に力を入れるべき案件：**
- 複数の外部APIやシステムと連携する基幹系
- 本番環境で大量のデータが流入する
- エラーが金銭的損失や信頼喪失につながる

現場では、「Rubyだから型安全性は諦める」のではなく、「Rubyだからこそ、入口層での検証に投資する」という判断が現実的です。
