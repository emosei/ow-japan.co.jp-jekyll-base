## 「あとから変更できない」という沼

マルチテナント対応のシステムを設計するとき、データベース層の構成方法として「テーブル分割」か「スキーマ分離」かの選択を迫られます。この判断は、一度実装が進むと変更が極めて難しくなります。

理由は単純です。テーブル分割で始めたシステムをスキーマ分離に変更するには、本番環境のデータを動かしながら、かつ顧客の利用を止めずに段階的に移行する必要があります。逆も然りです。その作業は、初期設計の数倍のコストがかかります。

実際、旅行予約システムの運用保守をしていた時代、最初はテーブル分割で対応していたのですが、顧客数が増えるにつれてクエリ性能が落ちていきました。その時点で「スキーマ分離に移行しよう」という判断が出ましたが、実現には3ヶ月以上を要しました。初期設計で正しい選択をしていれば、その時間と人員は別の機能開発に充てられたはずです。

## テーブル分割の現実的なメリットと限界

テーブル分割（1つのテーブルに全テナントのデータを格納し、tenant_idカラムで区別する）は、実装が単純です。

```sql
CREATE TABLE reservations (
  id BIGINT PRIMARY KEY,
  tenant_id INT NOT NULL,
  customer_id INT NOT NULL,
  booking_date TIMESTAMP,
  -- ...
  INDEX idx_tenant_customer (tenant_id, customer_id)
);
```

初期段階では、クエリもシンプルで、テナント追加時の開発作業も最小限です。「新しい顧客が来た」というときに、アプリケーション側でtenant_idを切り替えるだけで対応できます。

しかし、テナント数が増え、各テナントのデータ量が大きくなると問題が顕在化します。

- **インデックス戦略の複雑化**：すべてのクエリでtenant_idフィルタが必須になり、インデックスの組み方が限定される
- **バックアップと復旧**：特定テナントだけを復旧したいときに、全体をリストアして不要なデータを削除する手間が生じる
- **監査ログやデータ削除**：テナント退出時に、そのテナントのデータだけを確実に削除する処理が複雑になる
- **テナント間のクエリ分離漏れ**：where句でtenant_idを指定し忘れると、他テナントのデータが混在する重大なバグが起こる

最後の点は特に危険です。複数の開発者が関わるプロジェクトでは、うっかり条件を漏らすリスクが常に存在します。

## スキーマ分離の代償と現実

スキーマ分離（テナントごとに別のスキーマ、または別のデータベースを用意）は、テナント間のデータ隔離が完全です。

```sql
-- テナントA
CREATE SCHEMA tenant_a;
CREATE TABLE tenant_a.reservations (
  id BIGINT PRIMARY KEY,
  customer_id INT NOT NULL,
  booking_date TIMESTAMP,
  -- ...
);

-- テナントB
CREATE SCHEMA tenant_b;
CREATE TABLE tenant_b.reservations (
  id BIGINT PRIMARY KEY,
  customer_id INT NOT NULL,
  booking_date TIMESTAMP,
  -- ...
);
```

クエリはシンプルで、テナント間の誤混入がスキーマレベルで防がれます。バックアップも復旧も単位が明確です。

しかし、代償があります。

- **接続管理**：アプリケーションは、リクエストごとにどのスキーマに接続すべきかを判定し、接続プールを管理する必要がある
- **マイグレーション**：新しいカラムやテーブルを追加するとき、全スキーマに同じDDLを実行しなければならない。100テナントあれば100回実行する
- **リソース消費**：スキーマ数が増えると、メモリ使用量、コネクション数、バッファプール管理が複雑になる
- **監視と運用**：テナントごとの負荷監視、スロークエリログの分析が煩雑になる

小規模システムでスキーマ分離を採用すると、運用の手間だけが増えて、その恩恵は受けられません。

## 判断の軸足：テナント数と成長予測

この選択は、システムの成長予測に大きく左右されます。

**テーブル分割が現実的なケース：**
- テナント数が10〜20程度で、かつ今後も大きく増えない見込み
- 各テナントのデータ量が小〜中程度（数GB以下）
- クエリ複雑度が低く、tenant_idフィルタを入れやすい構造
- 監視や監査の要件が軽い

**スキーマ分離が必要なケース：**
- テナント数が50を超える、または今後超える見込みがある
- 大規模テナントが混在し、テナント間のリソース競合を避けたい
- 監査ログやコンプライアンス要件が厳しく、テナント間の完全隔離が求められる
- テナントごとにカスタマイズされたスキーマが必要になる可能性がある

実務的には、「今は20テナントだが、営業が100社を目指すと言っている」というケースが最も判断に困ります。その場合、初期段階でスキーマ分離の基盤を作っておき、実装はテーブル分割から始めるというハイブリッドアプローチもあります。

## ハイブリッド戦略：段階的な対応

実は、最初からスキーマ分離の「仕組み」を用意しておきながら、初期段階ではテーブル分割で運用するという方法があります。

```python
# 疑似コード：接続管理層
class TenantDB:
    def __init__(self, config):
        self.config = config
        self.pool = {}
    
    def get_connection(self, tenant_id):
        # 現在はすべてのテナントが同じDBを使う
        # 将来的には、tenant_idに応じて別のDBを返すように変更可能
        return self.pool.get('default')
    
    def execute_query(self, tenant_id, query):
        conn = self.get_connection(tenant_id)
        # テナント隔離ロジックはアプリケーション層で担当
        return conn.execute(query)
```

このアプローチなら、成長に応じて段階的に移行できます。ただし、アプリケーション層で常にtenant_idを意識する必要があり、その規律を保つのは簡単ではありません。

## 実装時のハマりどころと対策

**1. テナント情報の取得タイミング**

リクエストからtenant_idを抽出するのは、可能な限り早い段階で行うべきです。ミドルウェアレベルで実装し、以降のすべての処理で利用できるようにします。遅れると、あちこちでtenant_idの判定が散らばり、漏れが生じます。

**2. テスト環境での複数テナント検証**

開発環境で1テナントだけでテストしていると、本番で複数テナントが同じDBを使う状況をシミュレートできません。最低でも、ローカル開発環境で2〜3テナントを立ち上げて、データ分離がちゃんと機能しているか確認する癖をつけてください。

**3. バックアップとリストア手順の事前準備**

「テナントAのデータが破損したから復旧したい」というときに、本番環境で慌てるのは最悪です。テーブル分割なら部分復旧の手順を、スキーマ分離なら各スキーマの独立復旧を、事前にドキュメント化し、テストしておくべきです。

## 最後に

マルチテナント設計は、要件定義の段階で十分な議論が必要です。営業や経営層の「今後の成長予測」を聞くのは、エンジニアの責務です。その予測に基づいて、テーブル分割で十分なのか、スキーマ分離が必要なのか、あるいはハイブリッドで段階的に対応するのかを判断してください。

一度実装が進むと、設計の変更は極めて高くつきます。初期段階での判断が、その後の運用負荷を大きく左右します。
