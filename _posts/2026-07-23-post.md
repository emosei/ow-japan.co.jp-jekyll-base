## 現場で起きている問題

基幹システムの一部機能を新しいシステムに移行したり、モバイルアプリから在庫を直接更新できるようにしたり、あるいは外部サービスとの連携を増やしたりすると、必ずこういう場面に直面します。

「在庫数」「売上金額」「顧客の購入履歴」といった、ビジネスにとって唯一の真実（Single Source of Truth）であるべきデータが、複数のシステムから同時に更新されるようになる。そして、その瞬間から検証作業が指数関数的に増え始めます。

たとえば、基幹システムのWebインターフェースから在庫を更新できて、同時にモバイルアプリからも更新でき、さらに倉庫管理システムからの自動連携もある。こうなると、月次の決算時に「基幹システムの在庫数」と「実際に倉庫にある商品数」が合わなくなる。どのシステムのどの更新が原因なのか追跡するのに、数日かかることになります。

これは設計の甘さというより、複数の更新経路が増えることで、**検証すべき組み合わせの数が爆発的に増える**という根本的な構造的問題です。

## なぜ検証コストが増え続けるのか

複数のシステムから同じデータを更新する場合、検証すべき項目は単純な足し算では増えません。

**更新経路が増えるたびに、その経路同士の相互作用を検証する必要が出てきます。**

たとえば、更新経路が3つあるとします。

- 経路A（Webインターフェース）
- 経路B（モバイルアプリ）
- 経路C（自動同期API）

この場合、検証すべき組み合わせは以下のようになります。

- A単独で更新 → 正しく反映されるか
- B単独で更新 → 正しく反映されるか
- C単独で更新 → 正しく反映されるか
- Aで更新した直後にBで更新 → 競合は起きないか、どちらが優先されるか
- Aで更新した直後にCが自動同期 → データが上書きされないか
- BとCが同時に更新しようとしたら → どちらが優先されるか
- A→B→C→Aの順で連続更新 → 最終的に正しい状態に落ち着くか
- ネットワーク遅延でBの更新が遅延した状態でCが同期したら → 古いデータが復活しないか

経路が3つで、これだけの組み合わせです。4つになると、さらに倍以上の組み合わせが増えます。

さらに厄介なのは、これらの検証は**テスト環境では再現しにくい**ということです。本番環境では、複数のユーザーが同時に更新しますし、ネットワーク遅延のタイミングも不規則です。テスト環境では「同時更新は起きないだろう」と想定していても、本番では必ず起きます。

## 更新経路の統一か、読み取り専用の分散か

この問題を根本的に解決する方法は2つです。

### 方法1：更新経路を1つに統一する

最も確実な方法は、「在庫数」のような単一の真実は、**1つのシステムからのみ更新を許可する**ことです。

他のシステムが在庫を変更したい場合は、そのシステムから「在庫を100個減らしてほしい」というリクエストを、中央の基幹システムに送ります。基幹システムがそれを処理して、在庫を更新します。その結果は、すべてのシステムが読み取ります。

この場合、検証すべき項目は大幅に減ります。なぜなら、競合や相互作用が起きないからです。

```
モバイルアプリ → [リクエスト] → 基幹システム → [更新] → 在庫DB
             ← [読み取り] ←
             
倉庫管理システム → [リクエスト] → 基幹システム → [更新] → 在庫DB
                ← [読み取り] ←
```

ただし、この方法には**レイテンシーのコスト**があります。モバイルアプリから在庫を更新しようとする度に、基幹システムへのリクエストが必要になります。基幹システムが遅い場合や、ネットワークが不安定な場合、ユーザーは待つことになります。

### 方法2：読み取り専用のレプリカを分散させる

別のアプローチは、在庫の**更新は基幹システムだけが行い、他のシステムは読み取り専用のコピーを持つ**という設計です。

```
基幹システム（在庫DB） → [非同期レプリケーション] → モバイルアプリ用キャッシュ
                                              → 倉庫管理システム用キャッシュ
                                              → 検索エンジン用インデックス
```

この場合、各システムは最新の在庫情報を読めますが、更新はできません。在庫を変更したい場合は、基幹システムへのリクエストを送ります。

ただし、**レプリケーションには遅延が発生します**。基幹システムで在庫を100個減らした直後に、モバイルアプリから在庫を確認しても、まだ古い数字が表示されるかもしれません。この遅延をユーザーに説明し、許容できるレベルに保つ必要があります。

## 現場で起きやすい妥協と、その後のツケ

多くの場合、プロジェクトの初期段階では「複数システムから更新できるようにしよう」という決定が、十分な検討なく進みます。理由は、それが「便利」に見えるからです。

「基幹システムで在庫を更新することもあるし、モバイルアプリからも更新できた方が現場は楽だろう」

現場の声は確かに正当です。ただ、その便利さの代償として、後々、運用チームが毎月数日かけて整合性を検証することになります。

現場で見かけやすい妥協パターンは以下です。

- 複数経路からの更新を許可し、「最後の更新が優先」というルールにする
- 夜間バッチで整合性チェックを走らせ、差異があれば管理画面に通知する
- 月次決算前に、手作業で各システムのデータを照合する

これらは「動く」ようには見えますが、検証コストを先送りにしているだけです。

## 設計時に決めるべきポイント

新しいシステムを基幹システムと連携させるとき、最初に決めるべきことは以下です。

1. **どのデータが「単一の真実」か明確にする**
   - 在庫数、売上金額、顧客情報など、ビジネスにとって正確でなければならないデータを洗い出す

2. **その真実を更新できるシステムは1つか、複数か**
   - 複数の場合、競合時のルール（タイムスタンプ、優先度、マージロジック）を明確に決める
   - 決められない場合は、更新経路を1つに統一する方が、後々のコストが低い

3. **更新の頻度と遅延の許容度を把握する**
   - リアルタイムが必須か、数分の遅延なら許容できるか
   - 遅延が許容できるなら、レプリケーションベースの設計が現実的

4. **検証・監視の仕組みをどこに置くか**
   - 更新時に同期的に検証するか（レスポンス遅延のリスク）
   - 非同期バッチで検証するか（検出遅延のリスク）
   - その検証ロジックは、今後増える更新経路にも対応できるか

## 実装段階での注意点

もし複数経路からの更新を避けられない場合、最低限以下を実装しておくべきです。

```
更新リクエスト時のチェックリスト：
- 更新前後のデータを全て記録する（監査ログ）
- 更新を行ったシステムID、ユーザーID、タイムスタンプを記録
- 更新の順序が重要な場合は、トランザクションIDで追跡可能にする
- 定期的に全システムのデータを比較し、差異があれば自動通知
```

また、監視の仕組みは「検証パスが増えるたびに複雑になる」という前提で設計します。新しい更新経路が追加される度に、監視ロジックも増えることを想定し、ルールエンジンやメタデータベースの活用を検討する方が、後々の保守コストが低くなります。

## まとめ

複数のシステムから同じデータを更新できるようにすることは、一見便利に見えますが、検証コストは更新経路の数に対して指数関数的に増えます。

設計の段階で「更新経路は1つに統一するか、それが難しければ読み取り専用レプリカで分散させるか」を判断することが、後々の運用負荷を大きく左右します。便利さと検証コストのトレードオフを、プロジェクトの初期段階で明確に議論することをお勧めします。
