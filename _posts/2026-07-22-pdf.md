## 開発環境では完璧なのに、本番で突然崩れる理由

基幹システムの帳票をPDF化する案件は、一見すると単純に見えます。データベースから抽出したデータを、あらかじめ用意したテンプレートに流し込んで、PDFで出力する──それだけです。

ところが、開発環境では何の問題もなく動作していたのに、本番環境に上げた途端に「日本語が四角い記号になる」「行の途中で勝手に改ページされる」といった現象が起きることがあります。開発者が本番環境で再現を試みても、同じ条件では起きない。これは非常に厄介な問題です。

実際の現場では、このような環境差異による問題は「何度も繰り返される」ケースが多いです。一度解決しても、別のシステムで同じ落とし穴にはまる。その理由は、問題の本質が「フォント」と「ライブラリのバージョン」という、一見地味な要素にあるからです。

## なぜ開発環境では起きないのか

開発環境で帳票が正しく表示される理由は、ほとんどの場合「開発者のマシンに必要なフォントがインストールされている」からです。

基幹システムの帳票出力では、一般的にJavaやC#などの言語で、iTextやApache PDFBoxといったライブラリを使います。これらのライブラリは、指定されたフォント名を探す際、以下の順序で検索します：

1. PDF内に埋め込まれたフォント
2. システムにインストールされているフォント
3. ライブラリが提供するデフォルトフォント

開発環境では、開発者のマシンに「MS ゴシック」や「ヒラギノ角ゴ」といった日本語フォントが当たり前のようにインストールされています。そのため、たとえPDFに埋め込まれていなくても、システムフォントから自動的に補完されて、見た目上は問題が起きません。

しかし本番環境（特にLinuxサーバー）では、日本語フォントがインストールされていないことがほとんどです。その場合、ライブラリはデフォルトフォント（通常はラテン文字用）で代用しようとするため、日本語が正しく表示されません。

## 改ページが本番だけで起きるメカニズム

文字化けと同じくらい厄介なのが「改ページ位置のズレ」です。

開発環境では「1ページに収まっている」のに、本番環境では「2ページに分かれてしまう」というケースがあります。これは以下の理由で発生します：

**フォントサイズの計算差異**

PDFライブラリは、テキストの幅や高さを計算する際にフォント情報を使用します。埋め込まれたフォントと、システムフォントでは、同じサイズを指定しても実際のグリフ幅が異なる場合があります。特に日本語フォントは、ウェイト（太さ）やメトリクス情報がバージョンによって異なるため、行の折り返し位置や改ページタイミングが変わります。

また、ライブラリのバージョンアップによっても、フォント測定ロジックが改善されることがあり、それが改ページ位置を変える原因になります。

## 確実な解決策：フォント埋め込みの明示的な指定

根本的な解決には、**PDFにフォントを埋め込む**ことが必須です。埋め込まれたフォントは、環境に依存しません。

### iTextを使った実装例

```java
import com.itextpdf.io.font.PdfEncodings;
import com.itextpdf.kernel.font.PdfFont;
import com.itextpdf.kernel.font.PdfFontFactory;
import com.itextpdf.kernel.pdf.PdfDocument;
import com.itextpdf.kernel.pdf.PdfWriter;
import com.itextpdf.layout.Document;
import com.itextpdf.layout.element.Paragraph;

public class EmbeddedFontPdfGenerator {
    public static void createPdfWithEmbeddedFont(String outputPath, String fontPath) throws Exception {
        PdfWriter writer = new PdfWriter(outputPath);
        PdfDocument pdfDoc = new PdfDocument(writer);
        Document doc = new Document(pdfDoc);
        
        // フォントを埋め込み指定で読み込む
        PdfFont font = PdfFontFactory.createFont(
            fontPath,
            PdfEncodings.IDENTITY_H,  // 日本語対応の文字エンコーディング
            PdfFontFactory.EmbeddingStrategy.FORCE_EMBEDDED  // 強制埋め込み
        );
        
        Paragraph para = new Paragraph("これはテストです").setFont(font).setFontSize(12);
        doc.add(para);
        
        doc.close();
    }
}
```

重要なポイントは `EmbeddingStrategy.FORCE_EMBEDDED` を指定することです。これにより、フォントが確実にPDFに埋め込まれます。

### Apache PDFBoxを使った場合

```java
import org.apache.pdfbox.pdmodel.PDDocument;
import org.apache.pdfbox.pdmodel.PDPage;
import org.apache.pdfbox.pdmodel.PDPageContentStream;
import org.apache.pdfbox.pdmodel.font.PDFont;
import org.apache.pdfbox.pdmodel.font.PDType0Font;

public class PdfBoxEmbeddedFont {
    public static void createPdf(String outputPath, String fontPath) throws Exception {
        PDDocument document = new PDDocument();
        PDPage page = new PDPage();
        document.addPage(page);
        
        // フォントを埋め込み指定で読み込む
        PDFont font = PDType0Font.load(document, new File(fontPath));
        
        PDPageContentStream contentStream = new PDPageContentStream(document, page);
        contentStream.beginText();
        contentStream.setFont(font, 12);
        contentStream.newLineAtOffset(100, 700);
        contentStream.showText("これはテストです");
        contentStream.endText();
        contentStream.close();
        
        document.save(outputPath);
        document.close();
    }
}
```

## 本番環境での準備チェックリスト

フォント埋め込みを実装した後でも、本番環境でうっかり見落とされやすい点があります：

- **フォントファイルの配置**: サーバー上のどこにフォントファイルを置くか決めておく。相対パスは避け、絶対パスまたは環境変数で指定する
- **ファイルパーミッション**: アプリケーション実行ユーザーが、フォントファイルを読み取れることを確認する
- **ディスク容量**: 埋め込みフォントを使うとPDFサイズが増加します。特に大量出力の場合、ディスク容量の余裕を確保しておく
- **ライブラリバージョン**: 開発環境と本番環境で、PDFライブラリのバージョンを統一する。バージョン違いによる改ページ位置のズレはよく起きます

## 開発フェーズでの検証方法

本番環境を用意する前に、開発環境でも環境差異をシミュレートすることができます：

1. **開発マシンから日本語フォントを一時的に削除する**（または別の環境で試す）
2. **Linuxコンテナで開発環境を構築し、本番環境と同じフォント設定にする**
3. **PDFの内部構造を確認する**: `pdfinfo` や `pdftotext` コマンドで、フォントが実際に埋め込まれているか検証する

特に2番目の方法は、本番環境での問題を事前に検出する有効な手段です。Docker等でLinux環境を用意し、フォントをインストールせずに帳票出力テストを実行すれば、本番環境での動作をかなり正確に予測できます。

## 実装時の判断ポイント

フォント埋め込みにはメリットとデメリットがあります。実装を決める際の判断軸は以下の通りです：

| 観点 | フォント埋め込み | 非埋め込み |
|------|-----------------|----------|
| 環境依存性 | なし（推奨） | あり（リスク） |
| ファイルサイズ | 大きくなる | 小さい |
| 処理速度 | やや遅い | 速い |
| 運用の手間 | フォント管理が必要 | 不要 |

基幹システムの帳票は、出力頻度が高く、正確性が求められるため、**ファイルサイズや処理速度よりも環境依存性の排除を優先すべき**です。月間で数千枚以上の帳票を出力する場合でも、フォント埋め込みのコストは許容範囲内です。

## 本当の落とし穴は「後付け対応」

この問題の最大の落とし穴は、実装後期や運用開始後に「なぜか本番だけ文字化けする」という報告が上がることです。その時点では、すでに多くの帳票出力プログラムが本番環境で稼働しており、全てを修正するコストが膨大になります。

設計段階で「PDFにはフォントを埋め込む」という判断を明確にしておくことが、後々の手戻りを防ぐ最も確実な方法です。
