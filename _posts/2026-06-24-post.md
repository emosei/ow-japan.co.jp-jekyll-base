## 複数デバイスログインで起きる「あるはずの認証が無い」問題

モバイルアプリの運用が進むと、ユーザーが複数のデバイスから同じアカウントでログインするシーンが増えます。スマートフォン、タブレット、あるいは新しい端末への機種変更時です。

こうした場面で、実装がシンプル過ぎると予期しない不整合が生まれます。典型的な症状は「片方のデバイスでログインしたはずなのに、もう片方のデバイスでセッションが無効化される」「特定のデバイスだけ頻繁に再認証を求められる」といった現象です。

原因は多くの場合、トークンの管理層が単一化されているためです。具体的には、ユーザーを識別するトークン（ユーザートークン）とデバイスを識別するトークン（デバイストークン）が分離されていない、あるいは分離されていても管理ロジックが混在しているケースです。

## トークン分離が必要な理由──ユーザーとデバイスは独立した概念

セッション管理の実装では、認証後に何らかのトークンをクライアント側に保持させます。その際、「ユーザーが誰であるか」と「そのリクエストがどのデバイスから来たのか」は、本来は独立した概念として扱う必要があります。

分離しない設計では、以下のような問題が起きやすくなります。

**ユーザートークンの更新がデバイス全体に波及する**

ユーザートークンを単一管理していると、セキュリティ上の理由（定期的なローテーション、パスワード変更時の再発行など）でトークンを更新するたびに、すべてのデバイスに影響します。同期メカニズムがなければ、古いトークンを持ったデバイスは突然認証失敗に陥ります。

**デバイス固有の状態管理ができない**

デバイスAで「プッシュ通知の購読状態」「オフライン用キャッシュ」「ローカル設定」といった状態を持つとき、その状態はデバイスAに紐付いているべきです。しかしユーザートークンだけで管理していると、デバイスBから同じユーザーでアクセスした際に、デバイスA固有の状態が混在してしまいます。

**トークンの失効判定が曖昧になる**

「このトークンはまだ有効か」を判定するとき、ユーザーレベルの有効期限とデバイスレベルの有効期限が区別されていないと、片方のデバイスでログアウトした場合の処理が不透明になります。

## 設計の分離パターン

実装では、以下のように層を分けることが有効です。

**ユーザートークン（認証トークン）**
- ユーザーが本人であることを証明するトークン
- サーバー側で一元管理される有効期限を持つ
- パスワード変更やセキュリティイベント時に無効化される
- 複数デバイス間で共通

**デバイストークン（セッショントークン）**
- そのリクエストが特定のデバイスから来たことを証明するトークン
- デバイスごとに異なる値を持つ
- デバイスローカルストレージに保存される
- デバイス紛失時の個別無効化の対象となる

実装では、APIリクエスト時に両方を送信し、サーバー側で両方を検証するパターンが一般的です。

```
クライアント側の保持情報:
{
  "userId": "user_12345",
  "userToken": "eyJhbGc...", // ユーザートークン
  "deviceId": "device_abc789",
  "deviceToken": "eyJkZXY..." // デバイストークン
}

APIリクエストヘッダ:
Authorization: Bearer {userToken}
X-Device-Token: {deviceToken}
X-Device-ID: {deviceId}
```

サーバー側では、両方を独立して検証します。

```
ユーザートークン検証:
1. トークンの署名を確認
2. ユーザーIDを抽出
3. ユーザーレベルの無効化リストをチェック（パスワード変更など）
4. 有効期限を確認

デバイストークン検証:
1. トークンの署名を確認
2. デバイスIDを抽出
3. デバイスレベルの無効化リストをチェック（デバイス登録解除など）
4. デバイスごとの有効期限を確認
```

## 複数デバイスログイン時の具体的な流れ

実際の運用では、以下のようなシーケンスになります。

**デバイスAでログイン**
1. ユーザーが認証情報を入力
2. サーバーがユーザートークンを発行
3. デバイスAが初回アクセス時にデバイストークンをリクエスト
4. サーバーがデバイスAに紐付けたデバイストークンを発行・記録
5. デバイスAが両トークンをローカルストレージに保存

**デバイスBで同じアカウントでログイン**
1. ユーザーが再度認証情報を入力
2. サーバーが同じユーザートークンを発行（既存ユーザーの場合）
3. デバイスBが初回アクセス時にデバイストークンをリクエスト
4. サーバーがデバイスBに紐付けた別のデバイストークンを発行・記録
5. デバイスBが両トークンをローカルストレージに保存

この時点で、デバイスAとデバイスBは同じユーザートークンを持ちながら、異なるデバイストークンを持つ状態になります。

**ユーザーがパスワードを変更**
1. サーバーが新しいユーザートークンを発行
2. サーバーがユーザーレベルで古いユーザートークンを無効化リストに追加
3. デバイスAが次のAPIリクエストで古いユーザートークンを送信 → 失敗 → ローカルキャッシュから新しいトークンを取得するか、再認証を促す

デバイスBは別途、新しいユーザートークンの取得フローを実行します。

## 見落とされやすい運用上の注意点

**デバイストークンの無効化メカニズムが必須**

デバイスを紛失したり、不正アクセスの疑いが生じた場合、ユーザーがそのデバイスを個別に「登録解除」できる機能が必要です。ユーザートークンを無効化すると全デバイスに影響するため、デバイス単位での制御が不可欠です。

**トークンの有効期限を別々に管理する**

ユーザートークンは長めの有効期限（数日～数週間）、デバイストークンは短めの有効期限（数時間～1日）とすることが一般的です。これにより、デバイス紛失時のリスク最小化と、ユーザーの利便性のバランスが取れます。

**リフレッシュトークンの導入も検討**

デバイストークンの有効期限が短い場合、定期的にリフレッシュするメカニズムが必要です。その際、ユーザートークンの有効性を再確認してからリフレッシュするフローにします。

**オフライン状態での判定が曖昧になりやすい**

デバイストークンの有効期限切れに気づかずオフラインモードで動作していた場合、オンラインに戻った時点で突然認証失敗になります。ローカルキャッシュの有効期限とトークンの有効期限を別管理し、ユーザーに事前に通知するUI設計が必要です。

## 小規模チームでの導入ステップ

いきなり完璧な実装を目指さず、段階的に進めることをお勧めします。

1. **まずはデバイスIDの記録から始める**
   - 各デバイスに一意なIDを割り当て、ログイン時に記録する
   - サーバー側で「このユーザーはどのデバイスからログインしているか」を追跡可能にする

2. **次にデバイストークンを導入する**
   - ユーザートークンとは別に、デバイス固有のトークンを発行
   - APIリクエストで両方を検証するロジックを実装

3. **デバイス管理画面を用意する**
   - ユーザーが自分のログイン中のデバイス一覧を見られるようにする
   - デバイスを個別に「ログアウト」できる機能を追加

4. **トークン無効化ロジックを強化する**
   - パスワード変更時、セキュリティアラート時の動作を定義
   - ユーザートークン無効化時の全デバイスへの通知メカニズムを実装

## まとめ

複数デバイスからのログインが当たり前になった今、トークンの管理層は「ユーザーを認証する層」と「デバイスを識別する層」に分離しておくことが現実的な必須要件です。

最初は手間に感じるかもしれませんが、後々のセキュリティ対応やデバイス管理機能の追加が格段に容易になります。現場では「なぜか片方のデバイスだけセッションが消える」という不可解な問題に時間を費やすことになるより、初期段階で設計を整えておく方が、長期的には確実に効率的です。
