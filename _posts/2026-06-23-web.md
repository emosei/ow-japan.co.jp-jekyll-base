## 現場で起こる問題

リクエストハンドラーの前段でミドルウェアが実行され、リクエストIDやユーザー情報、トレーシングコンテキストをスレッドローカル変数に格納する。その後、同期的にレスポンスを返す分には問題ない。ところが、キューイングシステムやバックグラウンドジョブへ処理を委譲する非同期タスクが発生すると、そのコンテキスト情報が別スレッドで実行される際に消えてしまう。

結果として、ログに相関ID（correlation ID）が記録されず、分散トレーシングの追跡が途切れ、本番環境で問題が発生した際の原因特定が難しくなる。こういう状況は、スケーリングが進むにつれてバックグラウンドジョブの比率が高まる案件ほど顕著になります。

## 検証の前提と比較観点

ここでは、以下の構成を想定して検証を進めました。

- **フレームワーク**: Express.js（Node.js）、Spring Boot（Java）の2つの環境で確認
- **ミドルウェアの役割**: リクエストIDの生成、ユーザーコンテキストの格納、分散トレーシング用スパンの初期化
- **非同期処理**: Promise チェーン、async/await、スレッドプール経由のジョブ実行
- **比較軸**: スレッドローカル vs. コンテキスト伝播ライブラリ vs. 明示的な引き回し

検証の目的は「どの方式がコンテキスト喪失を防ぎ、かつ実装・保守コストが現実的か」を判断することです。

## スレッドローカルだけでは不十分な理由

```javascript
// Express.js での典型的なミドルウェア
const AsyncLocalStorage = require('async_hooks').AsyncLocalStorage;
const contextStorage = new AsyncLocalStorage();

app.use((req, res, next) => {
  const context = {
    requestId: generateId(),
    userId: extractUserId(req),
    timestamp: Date.now(),
  };
  contextStorage.run(context, () => next());
});

app.get('/api/data', async (req, res) => {
  const context = contextStorage.getStore();
  console.log('In handler:', context.requestId); // ✓ ここは取得できる

  // バックグラウンドジョブをキューに投入
  await queue.add({
    userId: context.userId,
    // コンテキストそのものは渡さない
  });

  res.json({ status: 'processing' });
});

// キューワーカー（別スレッドで実行）
worker.process(async (job) => {
  const context = contextStorage.getStore();
  console.log('In worker:', context); // ✗ undefined
});
```

Node.js の `AsyncLocalStorage` は非同期フローを追跡しますが、キューイングシステムを経由した場合、別プロセスやスレッドプールで実行されるジョブはコンテキストを失います。Java の `ThreadLocal` も同様で、スレッドプール内で別スレッドが割り当てられるとコンテキストは参照できません。

現場では、ここで「ログが繋がらない」という報告が上がり始めます。

## 設計パターンの比較検証

### パターン1: コンテキスト情報を明示的に引き回す

最もシンプルで確実な方法です。ジョブやメッセージにコンテキスト情報を含める。

```javascript
app.get('/api/data', async (req, res) => {
  const context = contextStorage.getStore();
  
  await queue.add({
    userId: context.userId,
    requestId: context.requestId,  // ← 明示的に含める
    timestamp: context.timestamp,
    payload: { /* 実際のデータ */ },
  });

  res.json({ status: 'processing' });
});

worker.process(async (job) => {
  const { requestId, userId, timestamp, payload } = job.data;
  
  // ジョブ開始時に新しいコンテキストを構築
  contextStorage.run(
    { requestId, userId, timestamp },
    async () => {
      await processData(payload);
    }
  );
});
```

**利点**:
- 依存関係が明確で、コンテキストの出所が追跡しやすい
- テストが書きやすい
- 言語やフレームワークを選ばない

**制約**:
- 全てのジョブ投入箇所でコンテキストを明示する必要がある
- 引き回す情報が増えると、ジョブペイロードが肥大化する
- 既存コードへの影響が大きい

### パターン2: コンテキスト伝播ライブラリ

Java の場合、Spring Cloud Sleuth や Micrometer を使うと、スレッドプール経由でもコンテキストが伝播します。

```java
// Spring Boot + Micrometer での例
@RestController
public class DataController {
  @Autowired private Tracer tracer;
  @Autowired private TaskExecutor taskExecutor;

  @GetMapping("/api/data")
  public ResponseEntity<?> getData() {
    Span currentSpan = tracer.currentSpan();
    
    // スレッドプール経由で実行しても、スパン情報が伝播
    taskExecutor.execute(() -> {
      try (Tracer.SpanInScope spanInScope = tracer.withSpan(currentSpan)) {
        processData();
      }
    });
    
    return ResponseEntity.ok("processing");
  }
}
```

**利点**:
- ミドルウェア層の設定で一括対応でき、ハンドラー側の変更が少ない
- スレッドプール、非同期処理、分散トレーシングが統合される
- 既存コードへの影響が相対的に小さい

**制約**:
- フレームワークやライブラリに依存する
- セットアップが複雑で、バージョン互換性の確認が必要
- メモリオーバーヘッドがある（スパン情報の保持）

### パターン3: カスタムデコレータ・インターセプタ

非同期タスク実行の入口に、コンテキスト復元ロジックを挟み込む方式です。

```javascript
// Node.js での例
function withContext(contextData) {
  return async (job) => {
    return new Promise((resolve, reject) => {
      contextStorage.run(contextData, async () => {
        try {
          resolve(await job());
        } catch (err) {
          reject(err);
        }
      });
    });
  };
}

// 使い方
worker.process(async (job) => {
  const { requestId, userId, payload } = job.data;
  
  await withContext({ requestId, userId }, async () => {
    await processData(payload);
  });
});
```

**利点**:
- ライブラリに依存しない、制御が手元にある
- 複数のコンテキスト伝播方式を混在させやすい
- テストやデバッグがしやすい

**制約**:
- 毎回、ジョブハンドラーで `withContext` を呼び出す必要がある
- ネストが深くなると可読性が落ちる

## 実装時の注意点

### コンテキスト情報の粒度を決める

全てのリクエスト属性を引き回すのではなく、「トレーシングに必須な情報」を限定しましょう。

必須:
- リクエストID / トレーシングID
- ユーザーID
- タイムスタンプ

不要なことが多い:
- リクエストボディ全体
- レスポンスヘッダー
- セッション全体

### キューイングシステムのデータ型制約

Redis、RabbitMQ、SQS など、キューシステムによってはシリアライズ形式が限定されます。複雑なオブジェクトは JSON にフラット化し、復元時に再構築する設計にしましょう。

### 非同期タスク失敗時のコンテキスト記録

リトライやデッドレターキューに移行する際も、コンテキスト情報を保持することで、後からのデバッグが格段に容易になります。

```javascript
worker.on('failed', (job, err) => {
  const { requestId, userId } = job.data;
  logger.error('Job failed', {
    requestId,
    userId,
    jobId: job.id,
    error: err.message,
  });
});
```

## 実務投入時の判断基準

| 判断軸 | パターン1（明示的引き回し） | パターン2（ライブラリ） | パターン3（カスタム） |
|--------|---------------------------|----------------------|----------------------|
| 導入コスト | 低 | 中〜高 | 中 |
| 保守性 | 高 | 中（バージョン追従が必要） | 中 |
| 既存コード影響 | 大 | 小 | 小〜中 |
| スケーラビリティ | 高 | 高 | 中 |
| マイクロサービス対応 | 要検討 | 良好 | 要実装 |

小規模で同期処理が主体なら **パターン1** で十分です。スケールしており、複数のジョブシステムを運用しているなら **パターン2** の検討が現実的です。既存ライブラリに依存したくない場合は **パターン3** で、ただし汎用化と再利用を意識した実装が必須です。

## 最後に確認すべきポイント

- 本番環境で実際のジョブスループット下でコンテキスト喪失がないか
- ログ集約ツール（Datadog、Splunk など）で相関IDが正しく記録されているか
- 非同期タスクの失敗時、コンテキスト情報がアラートに含まれているか
- 分散トレーシング（Jaeger、Zipkin など）を導入している場合、スパン情報が途切れていないか

ミドルウェアの設計は、同期フローだけを想定しがちですが、非同期処理が増えるほど、その効果が問われます。現場では「ログが繋がらない」という地味だが深刻な問題として現れるので、早期に検証して設計を固めることをお勧めします。
