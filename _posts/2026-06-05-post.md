## なぜ本番環境で急に「古いデータが返される」のか

キャッシュの無効化をイベント駆動で実装した案件で、本番環境に上がった直後から「ユーザーが古い情報を見ている」という報告が上がるケースが増えています。開発環境やステージング環境では問題なく動いていたのに、です。

理由は単純です。**イベント駆動のキャッシュ無効化は、イベントが確実に到達することを前提としている設計になりやすく、その前提が本番環境の負荷や障害によって簡単に崩れるから**です。

現場では「スケーラビリティのためにイベント駆動にしよう」という判断は理解しやすく、マイクロサービスやメッセージングの文脈でも自然に聞こえます。しかし、キャッシュ無効化という「必ず成功しなければならない処理」に対して、イベント駆動の「ベストエフォート性」を当てはめると、想像以上に複雑な問題が生じるのです。

## イベント駆動でキャッシュ無効化を実装するときの落とし穴

### パターン1: キューのドロップ

商品情報の更新があったとき、Redisのキャッシュを無効化するためにメッセージキューにイベントを流す設計を考えてみます。

```
商品更新API → イベント発行 → メッセージキュー → キャッシュ無効化ワーカー → Redis DEL
```

一見、きれいに見えます。でも本番環境で何が起きるか：

- キューが満杯になると、新しいイベントがドロップされる
- ネットワーク遅延で、ワーカーが消費する速度よりイベント発行の速度が速くなる
- ワーカーがクラッシュして再起動される間、イベントが処理されない

結果、「商品情報は更新されたのに、キャッシュには古いデータが残ったまま」という状態が発生します。ユーザーは古い情報を見続けます。

### パターン2: 非同期性による順序の逆転

複数の更新が短時間に連続したときに起こります。

```
時刻 10:00:01 → 商品Aを「1000円」に更新 → イベント①発行
時刻 10:00:02 → 商品Aを「900円」に更新 → イベント②発行

実際の処理順序：
イベント②がイベント①より先に到達
→ キャッシュが「900円」で無効化される
→ イベント①が到達
→ キャッシュが「1000円」で無効化される

ユーザーが見るキャッシュの値は「1000円」（古い方）
```

開発環境では同期的に実行されるため、この問題は見えません。本番環境でトラフィックが増えると、顕在化します。

### パターン3: 部分的な障害の検出漏れ

イベント駆動の実装では、「イベントが発行されたから、キャッシュ無効化も成功した」と考えがちです。しかし実際には：

- メッセージキューには到達したが、ワーカーが処理に失敗した
- Redisへの接続がタイムアウトした
- 無効化対象のキーが既に削除されていて、DELコマンドが実行されたが何もしなかった

こうした「サイレント失敗」は、ログを注視していないと見逃されます。

## スケーラビリティとデータ一貫性の選択を迫られる現実

イベント駆動のキャッシュ無効化が注目される背景には、確かに理由があります：

- **水平スケーリングが容易**：キャッシュ無効化を専用ワーカーに委譲できる
- **更新APIの応答性が上がる**：無効化処理をブロッキングしない
- **疎結合な設計**：更新処理とキャッシュ無効化が独立する

しかし、これらのメリットは「キャッシュ不整合を許容できるシステム」を前提としています。在庫管理、金銭取引、予約システムなど、データ一貫性が重要な領域では、このトレードオフが致命的になります。

## 現実的な設計判断

### 同期的な無効化が向くケース

- ユーザーが直後に確認する可能性が高い更新（商品情報、ユーザープロフィール）
- トランザクション内で更新とキャッシュ無効化を一体処理できる規模
- レイテンシーよりも一貫性を優先する必要がある場合

```
UPDATE product SET price = ? WHERE id = ?;
DEL cache:product:{id};
COMMIT;
```

この場合、キャッシュ無効化が失敗すれば、トランザクション全体がロールバックされます。確実性が担保されます。

### イベント駆動が向くケース

- 集計データやランキング、レコメンデーション（多少の遅延が許容される）
- キャッシュがあくまで「高速化のためのオプション」で、なくても動作する
- 無効化に失敗しても「次のアクセス時にDB問い合わせして再取得」できる設計

イベント駆動を採用するなら、**必ずフォールバック機構**を組み込みます：

```
キャッシュから取得 → なければDB問い合わせ → キャッシュに再設定
```

この「キャッシュミス時の自動リカバリ」があれば、イベント駆動の無効化が漏れても、ユーザーには影響しません。

## 中小規模の開発組織が取るべきアクション

1. **キャッシュ無効化の重要度を分類する**
   - 必ず成功しなければならない ← 同期的に
   - 遅延が許容される ← イベント駆動を検討
   - 失敗時のリカバリが可能 ← イベント駆動で許容

2. **イベント駆動を選ぶなら、監視を最初から組み込む**
   - キューの滞留監視
   - ワーカーの失敗ログ
   - キャッシュ無効化の遅延時間の計測

3. **本番環境での負荷試験に「キャッシュ無効化の遅延」を含める**
   - ステージング環境では見えない問題が出やすい
   - イベント駆動を選んだなら、必ず実施すべき

4. **疑わしいときは同期的な無効化から始める**
   - 後から非同期化することは容易
   - 最初から非同期化して、後から同期化するのは難しい

## 結論

イベント駆動は確かにスケーラブルですが、キャッシュという「データの正確性に直結する処理」に対しては、慎重に適用する必要があります。「トレンドだから」「他の企業が使っているから」という理由ではなく、システムの要件と制約の中で、同期と非同期のバランスを取る判断が求められます。

本番環境で「古いデータが返される」という報告を受けてからの修正は、想像以上に手間がかかります。設計段階での選択を、丁寧に検討することをお勧めします。
