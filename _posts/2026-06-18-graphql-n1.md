## 開発環境と本番環境で『見える問題』が異なる理由

GraphQL を導入するプロジェクトで、よく耳にする相談があります。「開発環境では動いているのに、本番でレスポンスが急に遅くなった」という報告です。その原因の多くが、GraphQL の N+1 問題です。

ただし注意深く聞いていると、チームが「N+1 問題は知っている」と答えることがほとんどです。知識はあるのに、なぜ本番環境で顕在化するのか。その答えは、データ量とクエリの複雑さが、開発環境と本番環境で桁違いに異なるからです。

REST API であれば、エンドポイント設計の段階で「どのデータを一度に返すか」が決まります。一方 GraphQL では、クライアント側が自由にフィールドを選択できるため、同じエンドポイントでも「どれだけ複雑なクエリが来るか」は実行時まで予測しにくいのです。開発環境では、テストケースが限定的で、単純なクエリばかり実行されます。だから問題が表面化しません。

本番環境では、様々なクライアント（Web フロントエンド、モバイルアプリ、管理画面など）が異なるクエリを投げ始めます。そこで初めて「このクエリ形状だと、1件のリクエストが 100 回のデータベースクエリを発行する」という現実に直面するのです。

## スケール閾値を意識した設計判断

N+1 問題自体は新しくありません。しかし GraphQL では、その『検出の難しさ』が増します。

理由は三つあります。

**一つ目は、クエリの複雑さが動的である点です。** REST API なら「このエンドポイントは何回のクエリを打つ」と固定的に測定できます。GraphQL では、同じエンドポイント（query）でも、クライアントが選択するフィールド数によって実行クエリ数が変わります。

**二つ目は、ネストされたフィールドの深さです。** 「ユーザー → 購入履歴 → 商品 → カテゴリ → 関連商品」というように、クライアントが深くネストしたクエリを書くと、各層で N+1 が発生します。開発環境では 2 層までしかテストしていなくても、本番では 5 層のクエリが来ることがあります。

**三つ目は、並列リクエストのパターンです。** 複数のクライアントが同時に複雑なクエリを投げると、データベースコネクションプールが枯渇し、接続待ちが発生します。開発環境での単一ユーザーテストでは、この競合状態は再現しません。

こうした閾値を超えると、レスポンスタイムは線形ではなく、指数関数的に悪化します。

## 開発環境での検出を『仕組み化』する

では、どうするか。重要なのは、開発環境で本番環境の「データ量パターン」をシミュレートすることです。

実装上の対策としては、以下のアプローチが現実的です。

**DataLoader パターンの導入**
バッチ処理でクエリをまとめ、N+1 を回避する手法です。多くの GraphQL ライブラリが対応しています。ただし、すべてのリゾルバに適用するのではなく、「アクセス頻度の高い関連データ」に限定するのが実務的です。

**クエリの複雑さ制限**
クエリの深さやフィールド数に上限を設ける仕組みです。例えば「ネストの深さは 3 層まで」「1 回のクエリで取得できるレコード数は最大 100 件」といったルールです。これにより、極端に複雑なクエリの実行を防げます。

**開発環境でのプロファイリング自動化**
テストコード内で、各クエリが発行する SQL 文の数をカウントし、閾値を超えたら失敗するテストを書くことです。例えば「このクエリは最大 5 個の SQL を発行してよい」という制約を CI に組み込みます。

```javascript
// 疑似コード：クエリの SQL カウントをテスト
test('User query should not exceed 5 SQL statements', async () => {
  const query = `
    query {
      user(id: 1) {
        name
        orders {
          id
          items {
            name
            category { name }
          }
        }
      }
    }
  `;
  const sqlCount = await countExecutedQueries(() => executeQuery(query));
  expect(sqlCount).toBeLessThanOrEqual(5);
});
```

## 監視の盲点：本番環境での『遅いクエリ』の可視化

開発環境での対策だけでは不十分です。本番環境で「どのクエリが遅いのか」を把握する仕組みも必須です。

多くのチームが見落としているのは、GraphQL クエリの実行時間を、クエリの形状ごとに分類して記録することです。REST API なら URL パスで分類できますが、GraphQL では「同じエンドポイントなのに、クライアントが選択したフィールドによって性能が全く異なる」という現実があります。

実務的には、以下の情報をログに記録することが有効です。

- クエリの操作名（operation name）
- 実行時間
- 発行された SQL クエリ数
- 取得されたレコード数
- クライアント ID（Web か Mobile か管理画面か）

これらを APM（Application Performance Monitoring）ツールやカスタムログで集約し、「このクエリ形状は本番で遅い」という傾向を見つけることが重要です。

## 中小規模チームが現実的に取れるアクション

「全部やるのは難しい」という声も聞きます。その場合、優先順位をつけることが大切です。

**段階 1：クエリ深さ制限の実装（即座）**
複雑さ制限ライブラリを導入し、ネスト深さと フィールド数に上限を設ける。これだけで、極端な N+1 の 8 割は防げます。

**段階 2：頻出パターンへの DataLoader 適用（1～2 週間）**
本番ログから「よく実行されるクエリパターン」を特定し、その部分に限定して DataLoader を適用します。全面的ではなく、ターゲットを絞ることが現実的です。

**段階 3：本番ログの自動分析（継続的）**
遅いクエリを自動検出し、チームに通知する仕組みを作ります。完全な自動化は不要で、週 1 回の手動確認でも構いません。

重要なのは、「GraphQL は性能管理が REST より複雑」という認識を、チーム全体で持つことです。その上で、段階的に対策を積み重ねていく。その過程で、プロジェクト固有の「スケール閾値」が見えてきます。
