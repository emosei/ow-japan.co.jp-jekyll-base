## 現場で起きる「所有権の壁」

Rustでマルチスレッド処理を書いていると、ある時点で必ず衝突します。複数のスレッドから同じデータにアクセスしたい、でも所有権システムが「それは許さない」と言ってくる場面です。

コンパイラのエラーメッセージは丁寧ですが、その丁寧さの向こうには「あなたの設計を変えてください」というメッセージが隠れています。小規模な案件なら割り切れますが、既存システムの一部をRustで置き換える、あるいはマイグレーション中の並行処理が必要な局面では、この制約が実装の自由度を大きく制限することがあります。

実務では「Rustの安全性は素晴らしい」と理論では分かっていても、納期と仕様の圧力の中では「とにかく動かせる方法」を探さざるを得ません。その時に知っておくべきは、どの「逃げ口」が本当に安全で、どれが後で運用負荷になるのかということです。

## 所有権制約が特に厳しくなるケース

マルチスレッド環境では、所有権システムが以下の場面で実装を難しくします。

**複数スレッドでの共有状態管理**  
スレッドA、スレッドBが同じデータ構造を更新したい。Rustの所有権は「所有者は1つ」という原則なので、素朴には書けません。

**長寿命な共有リソース**  
サーバープロセスが起動してから終了まで、複数のワーカースレッドが同じコネクションプール、キャッシュ、統計情報を参照・更新する場合、所有権の「誰が所有するのか」が不明確になりやすいです。

**動的なスレッド生成と同期**  
スレッド数が実行時に決まり、各スレッドがメインスレッドのデータを必要とする場合、ライフタイムと所有権の関係が複雑になります。

これらは「本来は避けるべき設計」かもしれませんが、現実のシステムではよく出現します。既存の非Rust言語で動いているシステムをRustに移行する際に特にそうです。

## 主な「逃げ口」と特性

Rustで所有権制約を回避する手段は複数ありますが、それぞれ異なる代償があります。

### Arc + Mutex

```rust
use std::sync::{Arc, Mutex};
use std::thread;

let shared_data = Arc::new(Mutex::new(vec![0; 100]));

for i in 0..4 {
    let data = Arc::clone(&shared_data);
    thread::spawn(move || {
        let mut locked = data.lock().unwrap();
        locked[i] = i as i32;
    });
}
```

**特性**  
- 複数スレッドでの所有権共有を実現する標準的な手段
- `Arc`（Atomic Reference Counting）で所有権を複製、`Mutex`でアクセス制御
- コンパイル時の安全性チェックが機能する

**代償**  
- ロック競合による性能低下
- デッドロックのリスク（Rustのコンパイラは防げない）
- ロック保持中の例外・パニックに対する脆弱性

**実務判断**  
競合が低い、または許容できるなら安定した選択肢です。ただしロック保持時間が長い、または頻繁にロックが争う場合は性能が劣化します。

### RwLock

```rust
use std::sync::{Arc, RwLock};

let data = Arc::new(RwLock::new(vec![0; 100]));

// 読み取り専用アクセス
let read_guard = data.read().unwrap();

// 書き込みアクセス
let mut write_guard = data.write().unwrap();
```

**特性**  
- 読み取りは複数スレッドで並行、書き込みは排他的
- 読み取り主体のワークロードで性能向上

**代償**  
- 読み取り/書き込みの判定ロジックが増える
- 書き込みが多いと逆に性能低下
- 同じく長いロック保持時間は危険

**実務判断**  
参照が圧倒的に多く、更新が稀な場合（キャッシュ参照、統計情報の読み出しなど）に有効です。ただし「読み取り主体のはず」という前提が実装段階で崩れることもあります。

### Channel

```rust
use std::sync::mpsc;
use std::thread;

let (tx, rx) = mpsc::channel();

thread::spawn(move || {
    tx.send(42).unwrap();
});

let received = rx.recv().unwrap();
```

**特性**  
- スレッド間でのデータ移動（所有権の移譲）
- 共有状態ではなく、メッセージパッシング
- デッドロックのリスクが低い

**代償**  
- 双方向通信は複雑
- バッファサイズ制限（`bounded channel`）でブロッキング
- 設計段階で通信パターンを明確にする必要

**実務判断**  
生産者・消費者パターンが明確な場合、最も安全です。ただし既存の共有状態設計をChannelに無理矢理変換すると、かえって複雑になります。

### unsafe ブロック

```rust
use std::sync::atomic::{AtomicUsize, Ordering};
use std::sync::Arc;

let counter = Arc::new(AtomicUsize::new(0));
counter.fetch_add(1, Ordering::SeqCst);
```

**特性**  
- 所有権チェックを明示的に回避
- 高速（ロック不要）
- Rustの安全性保証の外

**代償**  
- プログラマが安全性を保証する責任
- バグが起きた時の原因特定が困難
- チームメンバーの理解が必要

**実務判断**  
高性能が必須で、アクセスパターンが単純かつ検証済みの場合に限定すべきです。運用フェーズで問題が出た時の対応コストが高い。

## 「逃げ口」の選択判断フレームワーク

実装段階で判断する際のポイントをまとめます。

| 判断軸 | Arc + Mutex | RwLock | Channel | unsafe |
|--------|------------|--------|---------|--------|
| ロック競合の予想 | 中程度まで | 読み取り多い | なし | なし |
| 実装複雑度 | 低 | 低～中 | 中 | 高 |
| 性能要件 | 標準的 | 読み取り主体 | 高～超高 | 超高 |
| 運用時のデバッグ難易度 | 中 | 中 | 低 | 高 |
| チーム全体の理解度が必要 | 低 | 低 | 中 | 高 |

**選択の流れ**

1. まずChannelで実装できないか検討する（メッセージパッシングで解決できれば最も安全）
2. 共有状態が避けられなければArc + Mutexから始める
3. ロック競合が測定で問題になったらRwLockを検討
4. unsafeは、性能要件が明確で、チーム内に検証・保守できる人がいる場合のみ

## 実装時の落とし穴と運用上の注意

理論通りに選択しても、現場では予想外の問題が起きます。

**ロック保持中のI/O操作**  
ロックを取った状態でファイルやネットワークI/Oを実行すると、他のスレッドが待機し続けます。これは性能低下だけでなく、タイムアウトやデッドロック状態を招きます。

```rust
// 悪い例
let mut guard = data.lock().unwrap();
let result = expensive_io_operation().unwrap();  // ロック保持中に重い処理
guard.update(result);
```

**パニック時のロック状態**  
Rustのスレッドがパニックを起こしてもロックは解放されますが、`Mutex`の`poisoning`機構により、その後のロック取得は失敗します。これは堅牢性を上げる設計ですが、実装側が対応を忘れると全体が停止します。

**Channel の送信側が全て閉じた場合**  
受信側が `recv()` を呼ぶと即座にエラーになります。長時間待機するワーカースレッドがあると、予期しないシャットダウンが起きます。

## 「正解」ではなく「実装可能性と運用コストのバランス」

Rustの所有権システムは、単一スレッドのプログラムでは本当に素晴らしいです。メモリ安全性がコンパイル時に保証されます。ただしマルチスレッド環境では、その厳しさが「制約」に変わります。

大切なのは、理論的な「正しさ」よりも、チームの技術レベル、納期、保守性のバランスを取ることです。Arc + Mutexで十分に動くなら、無理にunsafeやChannelに複雑化させる必要はありません。逆に性能要件が明確なら、その段階で最適化を検討する。

現場ではこういう判断が繰り返されます。Rustの所有権は「逃げ口」を用意してくれていますが、どの逃げ口を選ぶかは、システムの要件と制約の中で決まるのです。
