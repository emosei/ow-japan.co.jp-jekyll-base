## 問題の発生場面

Goで複数の構造体を埋め込むコードを書いていると、意図しないメソッドが呼ばれるという経験をしたことはないでしょうか。特に、レイヤー化された設計で中間層の構造体が上位層の構造体を埋め込み、さらにそれが下位層を埋め込むような多段階の埋め込みがあると、どのレシーバのメソッドが実行されるのかが曖昧になりやすいです。

現場では、こうした構造は「既存コードに機能を追加する」「複数の責務を持つ型を段階的に設計する」という場面で自然に発生します。その時点では動作していても、後から別の開発者が修正を加えたり、新しい埋め込み層を追加したりすると、予期しない動作につながることがあります。

今回は、この「メソッドセット解決の順序」を検証し、実装時に注意すべき点をまとめました。

## 検証の前提と目的

**検証の目的**
- 複数レイヤーの構造体埋め込みにおいて、同じメソッド名が存在する場合、どのレシーバが実際に呼ばれるのかを確認する
- 埋め込みの深さ、埋め込みの順序、ポインタ型と値型の違いが解決順序に与える影響を把握する
- 実装時に曖昧さを回避するための設計判断を示す

**前提条件**
- Go 1.18以降（メソッドセット解決の仕様は安定している）
- 単一パッケージ内での埋め込み（パッケージ外のメソッド昇格は含めない）
- インターフェース実装による動的ディスパッチは対象外

## 実装パターンと検証結果

### パターン1: 2段階埋め込みでの同名メソッド

まず、基本的なケースから確認します。

```go
package main

import "fmt"

type Base struct {
	name string
}

func (b *Base) Process() {
	fmt.Println("Base.Process called")
}

type Middle struct {
	Base
}

func (m *Middle) Process() {
	fmt.Println("Middle.Process called")
}

type Top struct {
	Middle
}

func main() {
	t := &Top{
		Middle: Middle{
			Base: Base{name: "test"},
		},
	}
	t.Process()
}
```

**結果**: `Middle.Process called` が出力されます。

これは「メソッドセット解決の深さ優先探索」という原則に従っています。`Top` 型のメソッドセットを解決するとき、直接埋め込まれた `Middle` が優先され、その中で `Process` メソッドが見つかるため、さらに深い `Base` の `Process` は呼ばれません。

### パターン2: 同じレベルの複数埋め込みでの衝突

```go
package main

import "fmt"

type Logger struct{}

func (l *Logger) Log() {
	fmt.Println("Logger.Log called")
}

type Cache struct{}

func (c *Cache) Log() {
	fmt.Println("Cache.Log called")
}

type Service struct {
	Logger
	Cache
}

func main() {
	s := &Service{}
	s.Log()  // コンパイルエラー: ambiguous selector s.Log
}
```

**結果**: コンパイル時にエラーになります。

同じレベルの埋め込みで同じメソッド名があると、どちらを呼ぶのかが曖昧なため、明示的に指定する必要があります。

```go
s.Logger.Log()  // Logger.Log called
s.Cache.Log()   // Cache.Log called
```

### パターン3: ポインタ型と値型の混在

```go
package main

import "fmt"

type A struct{}

func (a *A) Do() {
	fmt.Println("A.Do (pointer receiver)")
}

type B struct {
	*A
}

func (b B) Do() {
	fmt.Println("B.Do (value receiver)")
}

type C struct {
	B
}

func main() {
	c := C{B: B{A: &A{}}}
	c.Do()
}
```

**結果**: `B.Do (value receiver)` が呼ばれます。

埋め込み構造体のレシーバ型（ポインタか値か）は、メソッドセット解決には影響しません。埋め込みの深さと構造のみが優先順位を決定します。

## 実務で気をつけるべき点

### 設計段階での曖昧さ排除

複数の構造体を埋め込む場合、その意図を明確にしておくことが重要です。現場では以下のような判断が起きやすいです：

- 「既存の型を流用したいから埋め込もう」という安易な判断
- 「後で拡張しやすいように多層化しておこう」という先制的な設計

ただし、埋め込みの層が増えるほど、メソッド呼び出しの追跡が難しくなります。特に、新しい開発者が参画したときに「このメソッドはどこで定義されているのか」を追うのに時間がかかります。

### 名前衝突の早期発見

同じレベルの埋め込みで同じメソッド名を持つ型を組み合わせる場合、コンパイラが即座に教えてくれます。しかし、層を深くした場合は、設計段階で気づきにくくなります。

新しいメソッドを追加するときは、既に埋め込まれている型に同じ名前のメソッドがないか、一度確認する習慣をつけておくと防げます。

### インターフェース経由でのメソッド呼び出し

埋め込みの複雑さを避けたい場合、インターフェースを経由してメソッドを呼び出す設計も検討できます。

```go
type Processor interface {
	Process() error
}

type Service struct {
	processor Processor
}

func (s *Service) Execute() error {
	return s.processor.Process()
}
```

この方法なら、埋め込みの深さに関わらず、どの実装が呼ばれるかが明確です。ただし、インターフェース定義のコストが増えるため、全てのケースに適用するわけではなく、責務の境界が明確な部分に限定するのが現実的です。

## 実装時の確認チェックリスト

埋め込みを含む設計をコードレビューするとき、以下の点を確認するといいでしょう：

- 埋め込みの層が3段階を超えていないか
- 同じメソッド名が複数の埋め込み型に存在していないか
- 新しいメソッドを追加するとき、埋め込み先の型に同じ名前がないか確認したか
- ドキュメントやコメントで「このメソッドはどの層から呼ばれるのか」が明記されているか

## まとめ

Goの埋め込みは便利な機能ですが、複数レイヤーで同じメソッド名が存在する場合、メソッドセット解決の順序を正確に理解していないと、予期しない動作につながります。

現場での実装では「埋め込みは2層まで」「同じメソッド名は避ける」といったシンプルなルールを引くことで、後からの保守性が大きく向上します。既存コードの複雑さを引き継ぐことになった場合は、段階的に埋め込み構造を整理し、責務を明確にしていく方が、長期的には効果的です。
