## 現場で起きている問題

オリジンサーバーの負荷軽減を目的にリバースプロキシを導入したのに、期待ほどキャッシュヒット率が上がらず、オリジンへのリクエストが減らないという相談を受けることがあります。ログを確認してみると、キャッシュ可能なはずのレスポンスが毎回オリジンから取得されている。設定を見直すと、キャッシュヘッダーの判定ルールが想定より厳密に機能していた、というケースです。

これは、リバースプロキシの設定値とHTTPキャッシュ仕様の間に、思わぬギャップが生まれるときに起きやすい問題です。設定者の意図は「安全にキャッシュする」だったのに、結果として「ほぼキャッシュしない」という逆効果になってしまう。こうした判断の落とし穴を、実装と運用の両面から整理してみます。

## キャッシュ判定ロジックの「厳密さ」が招く問題

リバースプロキシ（Nginx、Apache Traffic Server、Varnishなど）は、HTTPレスポンスのキャッシュ制御ヘッダーを読み込んで、キャッシュ対象かどうかを判断します。一般的には以下のようなヘッダーが基準になります。

- `Cache-Control` ヘッダーの `max-age`、`s-maxage`、`no-cache`、`no-store` など
- `Expires` ヘッダー
- `Pragma` ヘッダー
- `Set-Cookie` の有無

ここで問題が起きやすいのは、**プロキシが「キャッシュしてよい」と判断する条件を、実装側で想定より厳しく設定してしまう**というパターンです。

例えば、Nginxの場合、デフォルト設定では `Set-Cookie` ヘッダーが存在するレスポンスはキャッシュされません。これ自体は安全な判断です。しかし、オリジンが意図せず `Set-Cookie` を付与していたり、セッション管理とは無関係な追跡用クッキーを含めていたりすると、本来キャッシュ可能なコンテンツまでキャッシュ対象外になってしまいます。

さらに厄介なのは、こうした設定が「黙って」機能するという点です。キャッシュミスのログは出ますが、「なぜミスしたのか」という理由まで詳細に記録されないことが多いため、気付かないまま運用されることがあります。

## 検証環境での確認手順

実際にどう判定されているかを検証するには、以下のような環境構成が有効です。

**検証環境の構成例：**

```
クライアント → リバースプロキシ（Nginx） → オリジンサーバー（Node.js等）
```

オリジン側で複数パターンのレスポンスを返すエンドポイントを用意し、リバースプロキシのキャッシュ判定がどう動作するかを確認します。

```javascript
// Node.js + Express の例
app.get('/api/cached', (req, res) => {
  res.set('Cache-Control', 'public, max-age=3600');
  res.json({ data: 'キャッシュ可能', timestamp: Date.now() });
});

app.get('/api/cached-with-cookie', (req, res) => {
  res.set('Cache-Control', 'public, max-age=3600');
  res.set('Set-Cookie', 'tracking=abc123; Path=/');
  res.json({ data: 'クッキー付き', timestamp: Date.now() });
});

app.get('/api/no-cache', (req, res) => {
  res.set('Cache-Control', 'no-cache');
  res.json({ data: 'no-cache', timestamp: Date.now() });
});
```

Nginx側の設定例：

```nginx
upstream origin {
  server origin-server:3000;
}

proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m;

server {
  listen 80;

  location / {
    proxy_cache my_cache;
    proxy_cache_valid 200 1h;
    proxy_cache_key "$scheme$request_method$host$request_uri";
    proxy_pass http://origin;
    add_header X-Cache-Status $upstream_cache_status;
  }
}
```

ここで重要なのが `proxy_cache_valid 200 1h;` という設定です。これは「200ステータスのレスポンスを1時間キャッシュする」という指示ですが、**オリジンの `Cache-Control` ヘッダーと矛盾する場合、どちらが優先されるか**という問題が生まれます。

実際にリクエストを送ってみます：

```bash
# 1回目
curl -i http://localhost/api/cached

# 2回目（キャッシュ期待）
curl -i http://localhost/api/cached
# レスポンスヘッダの X-Cache-Status を確認
```

期待値は2回目で `X-Cache-Status: HIT` が出ることですが、実際には `MISS` が続くことがあります。その理由を掘り下げるには、Nginxのアクセスログを詳細に記録する必要があります。

## よくある設定の落とし穴

### 1. `proxy_cache_valid` が厳しすぎる場合

```nginx
proxy_cache_valid 200 10s;  # 10秒しかキャッシュしない
```

オリジンが `max-age=3600` を返していても、Nginxの設定が10秒であれば、10秒後に再検証が発生します。これ自体は設定の意図通りですが、「キャッシュを効かせたい」という目的と矛盾していることに気付かないまま運用されることがあります。

### 2. `Cache-Control: no-cache` の解釈違い

`no-cache` は「キャッシュするな」ではなく「キャッシュは可能だが、使う前に再検証しろ」という意味です。にもかかわらず、多くのプロキシ設定では `no-cache` が存在すると完全にキャッシュ対象外にされます。これは安全性重視の判断ですが、頻繁に更新されないコンテンツであれば、再検証でも十分な効果が得られます。

### 3. `Set-Cookie` の有無で判定する際の過度な厳密性

```nginx
# Nginxのデフォルト動作
# Set-Cookie があるレスポンスはキャッシュしない
```

これは安全ですが、セッション情報と無関係なクッキー（分析用、広告用など）が含まれている場合、本来キャッシュ可能なコンテンツまでキャッシュ対象外になります。

## 実装・運用での判断ポイント

### キャッシュ可能性の判定フロー

現場で判断するときは、以下の順序で確認することが有効です。

1. **オリジンが返すヘッダーを確認する**
   - `curl -i` でレスポンスヘッダを見る
   - 想定外のヘッダ（Set-Cookie、Pragma など）が付与されていないか

2. **プロキシの設定を確認する**
   - `proxy_cache_valid` の値がオリジンの `max-age` と矛盾していないか
   - `proxy_ignore_headers` で意図的に無視されているヘッダがないか

3. **実際のキャッシュ動作を検証する**
   - キャッシュステータスヘッダ（`X-Cache-Status` など）でHIT/MISS/REVALIDATEを確認
   - ログに記録されているレスポンスタイムの変化を見る

### 調整の方向性

キャッシュヒット率が低い場合、以下の調整を検討します。

- **オリジン側の改善**：不要な `Set-Cookie` を削除、キャッシュヘッダを明示的に追加
- **プロキシ側の緩和**：`proxy_cache_valid` を大きくする、特定ヘッダを無視する設定を追加
- **キャッシュキーの工夫**：クエリパラメータやクッキーをキャッシュキーに含めるかどうかの判断

```nginx
# 例：Set-Cookie を無視してキャッシュする
proxy_ignore_headers Set-Cookie;
proxy_cache_valid 200 1h;
```

ただし、この調整は「キャッシュを優先する」という判断なので、セッション管理やユーザー固有のデータが混在していないことを事前に確認する必要があります。

## 実務投入前の確認事項

リバースプロキシの設定を本番に反映する前に、以下を確認しておくと、後の問題を減らせます。

- オリジンアプリケーションが、キャッシュ可能なコンテンツに対して適切なヘッダを返しているか
- プロキシの設定値が、期待するキャッシュ期間と一致しているか
- キャッシュヒット率をモニタリングする仕組みが用意されているか
- キャッシュが効かない場合の調査方法が明確か（ログの見方、デバッグコマンドなど）

小規模な案件では、こうした確認をスキップしてしまうことがありますが、キャッシュの挙動は本番環境でしか完全には検証できません。ステージング環境での検証時点で、実際のアクセスパターンに近い負荷をかけ、キャッシュ統計を記録しておくことが重要です。

## まとめ
