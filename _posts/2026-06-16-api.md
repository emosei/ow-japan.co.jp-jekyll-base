## 外部APIとの接続で『制限に引っかかる』という経験

外部のAPIサービスと連携するシステムを運用していると、必ず直面する問題があります。それが「レート制限」です。

「1秒間に10リクエストまで」「1分間に1000リクエストまで」といった制約ですね。これ自体は昔からあるものですが、最近の実務では**制限の仕組みそのものを選択する段階で、後々の運用コストが大きく変わる**という現実に気づく機会が増えています。

特に、秒単位でリセットする方式と、スライディングウィンドウ方式のどちらを採用するかという選択が、スパイク対応の容易さと利用者間の公平性のバランスに直結しているのです。

## 『秒単位リセット』の見た目の簡潔さと隠れた落とし穴

秒単位リセット方式は、シンプルです。

```
時刻 09:00:00 → カウント 0/10
時刻 09:00:01 → カウント 0/10（前秒のカウントはリセット）
時刻 09:00:02 → カウント 0/10
```

毎秒、カウンターをゼロに戻す。実装も理解も簡単で、APIドキュメントに「秒間10リクエスト」と書かれていれば、多くの開発者はこのイメージで実装してしまいます。

ところが、現場ではこの仕組みが**瞬間的なスパイクに弱い**という問題が浮上します。

たとえば、複数のユーザーが同時にアクションを起こした場合、あるいは定期的なバッチ処理が秒の境界で重なった場合、その秒内で制限を超えるリクエストが発生することがあります。秒単位リセット方式では、その瞬間に「制限超過」と判定され、リクエストが弾かれます。

```
09:00:00.100 → リクエスト1（OK）
09:00:00.200 → リクエスト2（OK）
09:00:00.300 → リクエスト3（OK）
...
09:00:00.900 → リクエスト10（OK）
09:00:00.950 → リクエスト11（NG：制限超過）
```

この場合、リクエスト11は単純に失敗します。リトライロジックを実装していないと、エラーになります。実装していても、リトライが次の秒に入るまで待つ必要があり、その間のレイテンシが増加します。

## スライディングウィンドウ方式の『公平性』と複雑さ

一方、スライディングウィンドウ方式は、時間を『幅』として捉えます。

```
過去60秒間に送信したリクエスト数が1000以下か？
→ はい → リクエスト許可
→ いいえ → リクエスト拒否
```

常に「過去N秒間」というウィンドウを動かしながら、その中にいくつのリクエストが入っているかで判定します。

この方式の利点は、**時間帯による不公平が生じにくい**という点です。秒単位リセット方式では、秒の直前にリクエストを送った人と直後に送った人で、同じ制限内でも受け取れるリクエスト数が異なります。スライディングウィンドウでは、そうした時間的な『運不運』が減ります。

ただし、実装と運用が複雑になります。

- 過去N秒間のリクエスト履歴をすべて保持する必要がある
- タイムスタンプの精度が重要になる
- 分散システムでは、複数サーバー間でのカウント同期が必要
- メモリ使用量が秒単位リセット方式より増える可能性がある

## 現場で判断する基準

では、どちらを選ぶべきか。これは**APIの特性と、システムの許容度**で決まります。

### 秒単位リセットが向くケース

- **リアルタイム性が重要**で、スパイク時の瞬間的な失敗が許容できる場合
- 外部APIの仕様がすでに秒単位リセットで固定されている場合
- リトライロジックが実装しやすく、一時的なエラーが業務上の問題にならない場合
- 実装の簡潔さを優先したい小規模システム

### スライディングウィンドウが向くケース

- **複数ユーザーが同時利用**し、公平性が重要な場合
- 制限超過による失敗が業務に大きな影響を与える場合
- 長期的な安定運用を優先する場合
- インフラに余裕があり、メモリやCPUコストの増加が許容できる場合

## 実装時の落とし穴と運用上の注意

どちらを選んでも、実装段階で気をつけるべき点があります。

**秒単位リセットの場合**、リトライ戦略が重要です。単純な即座リトライではなく、指数バックオフ（待機時間を増やしながらリトライ）を組み込むことで、スパイク時の雪崩を防げます。

**スライディングウィンドウの場合**、クライアント側でカウンターを持つなら、サーバー側との時刻ズレに注意が必要です。クライアント時刻がサーバー時刻より遅れていると、実際には制限内でも『制限超過』と判定されることがあります。

また、どちらの方式であっても、**外部APIのレート制限に依存する設計は、長期的には脆弱**という点は変わりません。前回の記事でも触れましたが、キューイングや非同期処理で制限を吸収する設計を、並行して検討する価値は高いです。

## 最後に

レート制限の仕組みは、一見すると『APIサービス側の都合』に見えます。ですが、その仕組みをどう受け止めるかは、システムの信頼性と運用コストに直結する**設計判断**なのです。

秒単位かスライディングウィンドウか。その選択が、後々の障害対応やリトライロジックの複雑さを左右します。APIドキュメントを読むときは、単に「制限値」だけでなく、「リセット方式」まで確認する習慣をつけておくと、実装時の判断がずっと楽になります。
