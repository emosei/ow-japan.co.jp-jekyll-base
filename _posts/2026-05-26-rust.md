## 現場で起きる「モジュール分割の後悔」

Rustで中程度の規模のプロジェクトに取り組んでいると、ある時点で必ず直面する課題があります。それは「モジュール間の依存関係が複雑化して、循環依存を避けるための設計判断が、かえって実装を難しくしている」という状況です。

初期段階では気になりません。数個のモジュールを `mod.rs` で整理し、公開する型や関数を絞って `pub` キーワードで制御する。これだけで十分に見えます。しかし、ビジネスロジックが増え、複数の機能が相互作用し始めると、以下のような判断に迫られます。

- 型Aを定義したモジュールAが、型Bを定義したモジュールBに依存している
- しかし型Bもまた、型Aの情報が必要になった
- 循環依存を避けるために、共通の型を別モジュールに抽出するべきか
- それとも、設計自体を見直すべきか

この判断を誤ると、プロジェクトが進むにつれて「本来なら簡潔に書ける処理が、モジュール構造の制約で複雑になる」という事態が生じます。

## 循環依存がRustで「避けなければならない」理由

まず前提として、Rustのコンパイラは循環依存を許しません。言語レベルで禁止されています。これはC++やPythonと異なる点です。

```rust
// module_a.rs
use crate::module_b::TypeB;

pub struct TypeA {
    pub b: TypeB,
}

// module_b.rs
use crate::module_a::TypeA;  // コンパイルエラー

pub struct TypeB {
    pub a: TypeA,
}
```

このコードはコンパイルに失敗します。Rustは依存グラフが非環状（DAG: Directed Acyclic Graph）であることを強制するのです。

これ自体は良い制約です。循環依存は保守性を低下させ、変更の影響範囲を不透明にします。しかし、この強制力が「避けるためにどうするか」という判断を複雑にしているのが実情です。

## 設計判断の分岐点：共通モジュール抽出の罠

循環依存を避けるために最も一般的な手法は「共通の型を別モジュールに抽出する」というアプローチです。

```rust
// common/types.rs - 両方のモジュールに依存されるモジュール
pub struct SharedData {
    pub id: u64,
    pub name: String,
}

// module_a.rs
use crate::common::types::SharedData;

pub fn process_a(data: &SharedData) { /* ... */ }

// module_b.rs
use crate::common::types::SharedData;

pub fn process_b(data: &SharedData) { /* ... */ }
```

これで循環依存は解消されます。しかし、ここに落とし穴があります。

**プロジェクトが成長すると、「共通」だと思っていた型が、実は異なる責務を持つようになる** ということです。

たとえば、初期段階では `SharedData` は単なるデータコンテナでした。しかし、モジュールAの処理が進化すると、`SharedData` に対してA固有の検証ロジックが必要になります。一方、モジュールBでは異なる検証が必要です。結果、以下のようなことが起きます。

- `SharedData` に両方の検証ロジックが混在する
- または、検証ロジックをモジュールA、Bの側に持たせるが、その過程で型を変換する処理が増える
- 「共通」と思っていたモジュールが、実は特定の機能に依存し始める

現場では、この段階で「最初の設計判断が間違っていた」と気づくのです。

## 検証すべき3つのポイント

では、どう判断すればよいのか。実務的には、以下の3つを検証する必要があります。

### 1. 依存の方向性は本当に一方向か

まず、モジュール間の依存関係を図に描いてみてください。実際に矢印を引いて「AはBに依存しているか」「BはAに依存しているか」を可視化します。

```
例1: 明確な一方向
UserService → Repository → Database

例2: 曖昧な依存
OrderService ↔ PaymentService
（両方が相手の情報を必要としている）
```

例2のような場合、実装段階では「どちらが先か」を決めるために、共通モジュールを作りたくなります。しかし本質的には、この2つのサービスが同じレベルの責務を持っているということを示唆しています。

設計上、このような依存は「機能の粒度が大きすぎる」か「責務が混在している」かのどちらかです。

### 2. 共通モジュールに入れるべき型は、本当に「変わらない」か

共通モジュールに型を抽出する際、その型が「プロジェクトを通じて変わらないもの」かを問い直してください。

変わりやすい特徴：
- ビジネスルールに依存する検証ロジック
- 特定の機能でのみ使われるフィールド
- 将来、異なる表現が必要になる可能性がある構造

変わりにくい特徴：
- 基本的なデータ型（ID、タイムスタンプなど）
- 複数の機能で同じ意味で使われる概念
- インターフェース層での標準化が必要な型

前者を共通モジュールに入れると、後々の変更が困難になります。

### 3. 実装時の複雑さはどこで増えるか

コードを書き進めるとき、「この処理を書くために、なぜこんなに型変換が必要なのか」と感じる瞬間があります。これは設計判断の歪みを示す信号です。

例えば：
- 共通型から機能固有の型への変換処理が頻繁に発生する
- 共通型に「Aの場合はこのフィールド、Bの場合はこのフィールド」という分岐がある
- 共通型をラップして、機能固有のメソッドを追加する

こうした状況では、「共通化」の判断が早すぎた可能性が高いです。

## 実務的な構造判断のポイント

では、実際にプロジェクトに投入する際、どこで判断を検証すればよいか。

**初期段階（プロトタイプ～小規模機能）では、循環依存を避けることより「責務を明確にすること」を優先してください。** 

モジュール構造は、ビジネスロジックが安定してから整理しても遅くありません。むしろ、早期に共通モジュールを作ると、後々の変更で足かせになります。

具体的には：

1. **責務を言語化する** ── 各モジュールが「何をするのか」を1文で説明できるか確認する
2. **依存の理由を記録する** ── なぜAがBに依存するのか、コメントに残す。この理由が時間とともに薄れれば、設計の見直しが必要
3. **共通化は遅延させる** ── 3つ以上のモジュールが同じ型を必要とするまで、共通モジュール化を延期する

## Rustの型システムが「複雑さを露呈させる」側面

Rustの強い型安全性とモジュールシステムは、設計の甘さを容赦なく指摘します。これは利点でもあり、課題でもあります。

他の言語なら「とりあえず動く」で済むかもしれない設計が、Rustではコンパイルエラーで止まります。つまり、**Rustでのモジュール設計の複雑さは、実は「隠れていた設計課題が顕在化している」という側面もある** のです。

この視点を持つと、循環依存の回避は単なる「技術的な制約への対応」ではなく、「設計の質を高めるための検証機会」と捉え直すことができます。

## 判断基準をまとめる

大規模プロジェクトでRustのモジュール設計に取り組むなら、以下を確認してください。

- 循環依存が発生した理由は「責務の混在」か「粒度の問題」か
- 共通モジュール化で本当に解決するのか、設計の見直しが必要か
- 実装時に型変換や分岐が増えていないか

これらを丁寧に検証することで、プロジェクト後期での大きな後悔を避けることができます。
