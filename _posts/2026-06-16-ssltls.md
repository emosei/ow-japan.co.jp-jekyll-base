## 現場で起きている判断の迷い

Webサーバーとクライアント間の通信を最適化する際、SSL/TLSセッションの再開機構は有効な手段です。初回接続時の完全なハンドシェイクをスキップして、セッションIDやセッションチケットを再利用すれば、レイテンシを削減できます。ところが実装を進めると、こんな現象に直面することがあります。

「サーバー側ではセッション再開を正しく処理しているのに、クライアント側が新しいハンドシェイクを毎回要求してくる」

ブラウザ、モバイルアプリ、APIクライアントなど、クライアント実装の種類によって、セッション再開への対応度合いが大きく異なります。プロトコル仕様では対応しているはずなのに、実装レベルでは無視される。その結果、期待していたパフォーマンス改善が得られず、サーバー負荷が想定より高くなる──こうした齟齬は、設計段階では気づきにくいものです。

本記事では、セッション再開がクライアント側で無視される場面を実際に再現し、原因の切り分けと対応方針を検証します。

## 検証の目的と前提条件

### 何を調べるのか

以下の3点を中心に検証します。

1. **セッションIDとセッションチケット（TLS 1.3）の動作差異**
   - サーバーが提供する再開機構をクライアントがどう扱うか
   
2. **クライアント実装による対応バラつき**
   - ブラウザ、curl、Go標準ライブラリ、モバイルアプリなど、各実装がセッション再開をサポートしているか否か

3. **運用上の見落としやすいポイント**
   - キャッシュ設定、クッキー、プロキシの挙動がセッション再開に与える影響

### 前提環境

検証環境は以下の構成です。

- **サーバー側**: OpenSSLまたはGoの標準ライブラリでTLS 1.2/1.3対応
- **クライアント側**: curl、Go http.Client、ブラウザ（Chrome/Firefox）、モバイルアプリ（iOS/Android標準ライブラリ）
- **観測方法**: Wiresharkでのハンドシェイク追跡、ログ出力

## セッション再開の仕組みと落とし穴

### TLS 1.2とTLS 1.3の違い

TLS 1.2ではセッションIDまたはセッションチケット（RFC 5077）でセッション再開を実現します。サーバーが`ServerHello`でセッションIDを返し、クライアントが次の接続で同じIDを送信すれば、サーバーはセッション情報をメモリから復元して、簡略化されたハンドシェイクを行います。

TLS 1.3では仕様が変わり、セッションチケット（Pre-Shared Key, PSK）が唯一の再開メカニズムになります。サーバーは`NewSessionTicket`メッセージでチケットを送信し、クライアントはそれを保存して次回接続時に`pre_shared_key`拡張で提示します。

理論上は両方とも再開をサポートするはずですが、実装側の判断が分かれます。

### クライアント側が再開を無視する主な理由

**1. セッションストレージの非実装**

curl の初期状態では、セッション情報をメモリに保持しません。`-b`オプションでクッキージャーを指定しても、セッションIDは保存されません。Go の `http.Client` も同様で、デフォルトではセッション再開機能を持たないトランスポートを使用します。

**2. TLSハンドシェイクキャッシュの無効化**

ブラウザはセッション情報をディスクキャッシュに保存することが多いですが、プライベートモード（シークレットウィンドウ）ではこれが無効化されます。また、クライアント側で明示的にキャッシュをクリアすれば、セッション再開は利用できなくなります。

**3. プロキシやロードバランサーの影響**

クライアントとサーバーの間にプロキシが存在すると、接続がプロキシで一度終了し、プロキシからサーバーへの新しい接続が開かれます。クライアント側のセッション情報は、プロキシ先のサーバーとは異なる場合があり、再開が機能しません。

**4. クライアント実装の制限**

モバイルアプリで使用されるOkHttp（Android）やURLSession（iOS）は、セッション再開をサポートしていますが、設定によっては無効化されます。特にセキュリティポリシーで「すべてのハンドシェイクを再検証する」という厳格な要件がある場合、再開機構は意図的に使わないことがあります。

## 実装側の検証手順

### サーバー側の準備

Go標準ライブラリでセッション再開に対応したサーバーを実装します。

```go
package main

import (
	"crypto/tls"
	"fmt"
	"log"
	"net/http"
)

func main() {
	// TLS設定：セッション再開を有効化
	tlsConfig := &tls.Config{
		Certificates: []tls.Certificate{
			// 証明書を読み込む
		},
		SessionTicketsDisabled: false, // TLS 1.3のセッションチケット有効
	}

	server := &http.Server{
		Addr:      ":8443",
		TLSConfig: tlsConfig,
		Handler: http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
			fmt.Fprintf(w, "OK")
		}),
	}

	log.Fatal(server.ListenAndServeTLS("cert.pem", "key.pem"))
}
```

この設定では、TLS 1.3のセッションチケットが自動的に発行されます。

### クライアント側の動作確認

**curl での確認（セッション保存なし）**

```bash
curl -k https://localhost:8443 --tlsv1.3
curl -k https://localhost:8443 --tlsv1.3
```

この場合、2回目の接続でも新しいハンドシェイクが実行されます。Wiresharkで確認すると、毎回`ClientHello`から始まります。

**Go http.Client での検証**

```go
package main

import (
	"crypto/tls"
	"fmt"
	"io"
	"net/http"
)

func main() {
	client := &http.Client{
		Transport: &http.Transport{
			TLSClientConfig: &tls.Config{
				InsecureSkipVerify: true,
			},
		},
	}

	for i := 0; i < 3; i++ {
		resp, _ := client.Get("https://localhost:8443")
		io.ReadAll(resp.Body)
		resp.Body.Close()
		fmt.Printf("Request %d completed\n", i+1)
	}
}
```

Go の `http.Client` では、同一の `Transport` インスタンスを再利用する限り、接続プールが機能し、セッション再開が活用されます。ただし、新しい `Transport` を毎回作成すれば、セッション再開は利用されません。

## 現場で見落としやすい設定項目

### 1. キャッシュ制御ヘッダーとセッション再開の混同

セッション再開とHTTPキャッシュは別の層です。`Cache-Control: no-store`でHTTPレスポンスをキャッシュしなくても、TLSセッションは再開できます。逆に、HTTPキャッシュが有効でも、TLSセッションが再開されないこともあります。

### 2. ロードバランサーのセッションアフィニティ

複数のバックエンドサーバーがある場合、セッション再開の成功はクライアントが同じサーバーに接続できるかに依存します。ロードバランサーが接続を異なるサーバーに振り分けると、セッション再開は失敗します。この場合、セッション情報を共有ストレージ（Redis等）に保存する必要があります。

### 3. モバイルアプリでのセッション永続化

iOS の `URLSession` や Android の `OkHttp` は、デフォルトではセッション情報をメモリに保持し、アプリ終了時に破棄します。セッション再開を活用するには、明示的にセッション設定を保存する必要があります。

## 運用投入前に確認すべきポイント

1. **クライアント多様性の把握**
   - 利用されるクライアント（ブラウザ、SDK、自社アプリ）がセッション再開をサポートしているか確認する
   - サポート状況に応じて、期待できるパフォーマンス改善を現実的に見積もる

2. **接続パターンの測定**
   - 実際のトラフィックで、同一クライアントからの再接続がどの程度あるか測定する
   - 一時的なアクセスが多い場合、セッション再開の効果は限定的です

3. **プロキシ環境の考慮**
   - 企業ネットワーク、CDN、WAFなどを経由する場合、セッション再開が機能しない可能性を検討する
   - その場合の代替案（HTTP/2 Keep-Alive、接続プーリング）を用意する

4. **デバッグ手段の整備**
   - Wireshark、OpenSSL s_client、ブラウザ開発者ツールでハンドシェイク状況を可視化できるようにしておく

## まとめ

セッション再開はプロトコル仕様では標準機能ですが、クライアント実装の多様性と運用環境の複雑さから、期待通りに機能しないことが少なくありません。サーバー側で対応を整えても、クライアント側の制約や設定で無視されることは珍しくないのです。

重要なのは、「セッション再開が自動的に効く」と仮定せず、実装段階で各クライアントの対応状況を調査し、本番環境での動作を測定することです。その上で、効果が期待できる部分に集中投資し、そうでない部分は別のアプローチ（HTTP/2、接続プーリング）を組み合わせる判断が
