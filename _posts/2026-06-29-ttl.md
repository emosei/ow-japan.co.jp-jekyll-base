## キャッシュ層の「引き継ぎ問題」が本番で顕在化する瞬間

機能仕様を変更した直後、データベースには新しい値が入っているのに、ユーザーが見ている画面には古い値のまま──こういう不整合が数時間から数日続く経験をされたことはないでしょうか。

現場では、キャッシュの有効期限（TTL）を設定して「その時間が経つまで待つ」という判断をしてしまいがちです。特に小規模チームほど、キャッシュ戦略を「TTLを決めたら完結」と捉えてしまう傾向があります。ところが実際には、機能変更や業務ルール更新といった「データの意味が変わるタイミング」では、TTL待ちでは対応できない状況が繰り返し起こります。

本記事では、TTL単一戦略がなぜ本番で問題になるのか、そしてどのような多層戦略を組み合わせるべきかを、実装と運用の観点から整理します。

## TTLだけ頼みで失敗する理由

### キャッシュの「意図しない長生き」

TTLを設定する際、多くの場合は「パフォーマンスと鮮度のバランス」を考えます。例えば、ユーザー情報キャッシュを5分、商品マスタを30分、というように。しかし問題は、その有効期限が「データの物理的な更新」と「ビジネスロジックの変更」を区別していないという点です。

仮に、旅行プランの割引ルールが変わったとします。データベースの割引テーブルは即座に更新されますが、計算結果がキャッシュされていれば、TTLが切れるまでユーザーは古い価格を見続けます。30分のTTLなら、最大30分間の不整合が生じます。ただし、実務ではこれが「30分で済む」ことはまれです。

### アプリケーション再起動時の「ウォームアップ問題」

本番環境で障害対応やデプロイを行う際、アプリケーションを再起動することがあります。その時点で、メモリキャッシュはすべてクリアされます。しかし外部キャッシュストア（RedisやMemcached）に保存されているデータは残ります。再起動直後、古いキャッシュを読み込んだアプリケーションが起動することになり、コードロジックが変わっているのにキャッシュの内容は古いまま──という状況が生じます。

この状態は「TTLが切れるまで」続きます。業務時間中のデプロイなら、その間ずっとユーザーに古いデータが返される可能性があります。

### スケーラビリティの罠

複数のアプリケーションインスタンスが同じキャッシュストアを参照する場合、一つのインスタンスでキャッシュを無効化しても、他のインスタンスは古いキャッシュを読み続けます。無効化コマンドがすべてのインスタンスに確実に伝わるまでの間、システム全体で不整合が発生します。

## 多層戦略の設計判断

実務では、以下の三層を組み合わせることが現実的です。

### 第1層：イベント駆動の即座無効化

データの更新が確定した直後に、キャッシュ無効化を明示的に実行する方法です。例えば、割引ルール更新APIが呼ばれた時点で、関連するキャッシュキーをすぐに削除します。

```python
def update_discount_rule(rule_id, new_percentage):
    # DBに書き込み
    db.update_discount_rule(rule_id, new_percentage)
    
    # キャッシュ無効化を明示的に実行
    cache.delete(f"discount_rule:{rule_id}")
    cache.delete("all_discount_rules")  # 全体キャッシュも削除
    
    # 複数インスタンスへの通知（イベント発行）
    publish_cache_invalidation_event("discount_rule", rule_id)
```

ここで重要なのは、**無効化を同期的に実行する**という判断です。非同期でイベントを発行するだけでは、インスタンス間の無効化タイミングにズレが生じます。

### 第2層：バージョン管理によるキャッシュ隔離

キャッシュキーに「バージョン番号」を含める方法です。ビジネスロジックが変わった時点で、キャッシュキーのバージョンをインクリメントします。

```python
DISCOUNT_RULE_CACHE_VERSION = 2  # ロジック変更時にここを +1

def get_discount_rule(rule_id):
    cache_key = f"discount_rule:v{DISCOUNT_RULE_CACHE_VERSION}:{rule_id}"
    cached = cache.get(cache_key)
    if cached:
        return cached
    
    # キャッシュミス時はDBから取得
    rule = db.get_discount_rule(rule_id)
    cache.set(cache_key, rule, ttl=1800)  # 30分
    return rule
```

この方法の利点は、**古いキャッシュを強制削除する必要がない**という点です。バージョンが上がれば、古いキャッシュは自動的に使われなくなり、TTLで自然に消滅するのを待つだけです。デプロイ時にも、バージョン番号を上げるだけで対応できます。

### 第3層：TTLによる最後の防線

第1層と第2層で対応できない、予期しない不整合を検出するための防線です。ここでは短めのTTLを設定します。例えば第2層で1時間のTTLを設定していても、TTLは5分にしておくといった具合です。

```python
# キャッシュ設定の例
cache.set(
    key=cache_key,
    value=rule,
    ttl=300  # 5分。第1層・第2層で対応できない不整合を限定
)
```

## 実装上の注意点

### 無効化の漏れを防ぐ

イベント駆動の無効化を採用する場合、更新処理のあらゆる経路でキャッシュ削除を実行する必要があります。APIエンドポイント、バッチ処理、管理画面、外部連携──すべての更新元でです。

現場では「APIからの更新は無効化しているが、バッチ処理からの更新は無効化し忘れていた」という落とし穴が起きやすいです。対策として、**データベース層やORM層で無効化を一元管理する**のが有効です。

```python
class DiscountRule(BaseModel):
    def save(self):
        super().save()
        # モデル層で無効化を一元管理
        cache.delete(f"discount_rule:{self.id}")
        cache.delete("all_discount_rules")
```

### 複数インスタンス環境での同期

キャッシュストア自体はすべてのインスタンスで共有されますが、メモリ上の設定値（バージョン番号など）は共有されません。バージョン番号をインクリメントしても、他のインスタンスはそれを知りません。

対策として、バージョン番号自体をキャッシュストアに保存する方法があります。

```python
def get_cache_version(key):
    version = cache.get(f"version:{key}")
    return version or 1

def increment_cache_version(key):
    current = get_cache_version(key)
    cache.set(f"version:{key}", current + 1, ttl=None)  # 無期限
```

## 運用の観点から

### デプロイ時の手順

コードロジックが変わるデプロイを行う場合、以下の手順が安全です。

1. **デプロイ前に、バージョン番号をインクリメント**する設定変更をマージ
2. **コードをデプロイ**
3. 古いインスタンスが完全に停止するまで待つ
4. 必要に応じて、手動でキャッシュキーを削除

これにより、新旧インスタンスが混在する期間中も、バージョンの違いで自動的に不整合が回避されます。

### 監視とアラート

キャッシュの有効性を監視することは、TTL単一戦略よりも重要になります。例えば、キャッシュヒット率が突然低下した場合、何らかの無効化漏れが起きている可能性があります。

```
キャッシュヒット率の低下 → 無効化漏れまたは過度な無効化
キャッシュサイズの急増 → 無効化されないべきデータが蓄積
```

こうした指標を監視することで、ロジックの問題を早期に検出できます。

## 小規模チームでの導入ステップ

すべてを一度に導入する必要はありません。段階的に進めることをお勧めします。

1. **第1段階**：重要な更新処理で明示的なキャッシュ削除を実装
2. **第2段階**：バージョン管理をコード化（設定ファイルで管理可能にする）
3. **第3段階**：キャッシュヒット率などの監視を追加

最初は「TTL + 明示的な削除」だけで十分です。問題が繰り返されたら、バージョン管理を導入する、というアプローチで構いません。

## 最後に

キャッシュ戦略は、システムのスケールと複雑さに応じて進化させるものです。小規模なら単純で良いですが、機能変更が頻繁になったり、インスタンスが増えたりすると、多層戦略が必須になります。

「TTLで十分」という判断が、本番で数日間の不整合を招くことは珍しくありません。むしろ、その可能性を想定した上で、最初からいくつかの手段を組み合わせておく方が、実務的には安心です。
