## 非同期タスクの失敗とリトライが常態化する理由

非同期タスクキューを導入するシステムでは、ネットワーク遅延やタイムアウト、一時的なリソース不足といった理由でジョブが失敗することは日常茶飯事です。だからこそ、失敗したタスクを自動で再実行する仕組み（リトライロジック）が必須になります。

ところが現場で起きやすいのは、このリトライロジックが「失敗を検知して再実行する」という単純な条件だけで組まれてしまい、同じ処理が複数回実行されても問題ないかどうかが検証されないまま本番に入るケースです。結果として、重複実行が蓄積し、数日後に「なぜか同じ注文が二重に計上されている」「ユーザーポイントが重複加算されている」といった問題が顕在化します。

この落とし穴は、冪等性（べきとうせい）という概念の理解不足というより、**冪等性を実装する際の難易度と現場の判断基準のズレ**にあります。

## 冪等性は「理想」ではなく「実装の手間」

冪等性とは、同じ操作を何度実行しても、1回実行した場合と同じ結果になることを指します。数学的には美しい概念ですが、実装観点では大きく異なります。

たとえば、以下のようなケースを考えてみてください。

**ケース1: ユーザーの残高に1000ポイント加算する**

```python
def add_points(user_id, points):
    user = db.get_user(user_id)
    user.balance += points
    db.save(user)
```

このコードは冪等ではありません。同じ処理を3回実行すれば、3000ポイント加算されます。リトライで2回実行されれば、2000ポイント加算されてしまいます。

**ケース2: ユーザーの残高を1000ポイントに設定する**

```python
def set_points(user_id, points):
    user = db.get_user(user_id)
    user.balance = points
    db.save(user)
```

このコードは冪等です。何度実行しても、残高は1000ポイントのままです。

しかし現場では、ケース1のような「加算」「減算」「ステータス遷移」といった相対的な変更が大半です。これらを冪等にするには、以下のような手間が発生します。

- **トランザクションIDの記録**: 同じジョブが重複実行されていないか確認する仕組み
- **べき等チェック**: 既に実行済みの操作なら、その結果を返す
- **ロック機構**: 同時実行を防ぎながら状態を更新する

これらは追加の実装工数であり、納期やリソースの制約がある現場では「とりあえず動かす」という判断が優先されやすいのです。

## リトライロジックが単純だと、問題は時間差で顕在化する

重要なのは、重複実行の問題が**すぐには見つからない**という点です。

テスト環境では、ジョブが失敗するシナリオ自体が少ないため、リトライが何度も発動する状況を再現できません。本番環境でネットワーク遅延やデータベース接続エラーが頻発すると、リトライが繰り返され、その度に同じ処理が実行されます。

一方、データベースの更新ログやアプリケーションログを見ると「ジョブが再実行されている」ことは分かるのですが、その結果として**何が重複しているのか**は、時間が経ってから初めて気付きます。

- 在庫が実際の出荷数と合わない
- ユーザーの獲得ポイント履歴に同じ記録が複数ある
- 請求額が月次で数万円ずれている

こうした問題が発見されるのは、通常は翌日以降の集計タイミングや、ユーザーからの問い合わせ時です。

## 設計段階で「冪等性が必須か」を判断する

すべてのジョブを冪等にする必要はありません。むしろ、冪等性の実装コストと、重複実行時のリスクを天秤にかけて判断すべきです。

**冪等性が必須のケース:**

- 金銭や在庫など、重複が直接的な損失につながる処理
- ユーザーへのポイント付与、クレジット加算
- 外部APIへの送信（決済、配送手配など）
- ステータス遷移が複数段階に渡る場合

**冪等性が不要（リトライ回数を制限すれば足りる）ケース:**

- 読み取り専用の集計・レポート生成
- 通知メール送信（重複は許容できるなら）
- ログ記録（ログ重複は許容できるなら）

判断の際は、以下を確認してください。

1. **重複実行時の影響**: システムとユーザーの両面で、何が起きるのか
2. **検出難易度**: 重複がどのくらい経ってから気付くのか
3. **修復コスト**: 重複データを手動で修正する手間

影響が大きく、検出が遅れやすく、修復コストが高いほど、冪等性の実装に投資する価値があります。

## 実装観点での現実的なアプローチ

完全な冪等性を目指さずとも、現場で実行可能な対策があります。

**1. リトライ回数と間隔の制限**

無限リトライではなく、最大3回、指数バックオフで再実行するといった制限を設ける。ネットワーク遅延なら数秒で回復するはずなので、過度なリトライは不要です。

**2. デッドレターキューの活用**

リトライが規定回数を超えたジョブは、別のキューに移し、人間が確認してから対応する。自動リトライの無限ループを防げます。

**3. ジョブIDベースの重複チェック**

少なくとも、同じジョブIDが短時間に複数回実行されていないか、簡易的にチェックする。完全な冪等性ではなく、明らかな重複だけは防ぐという現実的な折衷案です。

```python
def execute_job(job_id, task_data):
    # 最近1時間以内に同じジョブIDが実行されたか確認
    recent = cache.get(f"job:{job_id}")
    if recent:
        return recent  # 既に実行済みならその結果を返す
    
    result = perform_task(task_data)
    cache.set(f"job:{job_id}", result, ttl=3600)
    return result
```

このアプローチは完全ではありませんが、大半の重複実行を防ぎ、実装コストも低く抑えられます。

## リトライ戦略の監視と運用

設計段階の判断と同じくらい重要なのが、本番環境での監視です。

- **リトライ発生率**: 1日あたり何件のジョブが再実行されているか
- **最終失敗率**: リトライを全て消費しても失敗するジョブの割合
- **処理時間の分布**: 再実行されたジョブの処理時間が異常に長くないか

こうした指標を定期的に確認することで、「リトライが多発しているなら、キューの設定やタイムアウト値を見直すべき」という判断につながります。

重複実行の問題は、一度起きると修復が難しく、その間のビジネスへの影響も大きいものです。だからこそ、設計段階で「何が冪等である必要があるのか」を明確にし、実装段階で現実的な対策を講じ、運用段階で継続的に監視する──この三層の対応が必要になるのです。
