## 機能抽出は見た目より複雑になる

モノリシックなRailsアプリケーションを段階的にモジュール化する過程で、機能ごとにgemとして抽出することは自然な判断です。関心の分離、テストの独立性、再利用性の向上——これらは理想的な目標です。

しかし実際には、抽出作業を進める中で**循環依存という目に見えない落とし穴**に引っかかることが多くあります。単体のテストは通り、個別のgemは動作しているのに、本体アプリケーションとの統合時にだけ問題が顕在化する。そういった経験は、恐らく多くのエンジニアが持っているのではないでしょうか。

循環依存は、gemの抽出段階では気づきにくく、後から修正するほど手間がかかります。なぜそうなるのか、そしてどう検出・回避するのかを、実装の観点から整理してみます。

## 「独立している」という錯覚が生まれる理由

モノリシックなRailsアプリから機能を抽出する際、多くの場合は以下のような流れになります。

1. `app/models/feature_a` と `app/controllers/feature_a` をまとめて `gems/feature_a` へ移動
2. `app/models/feature_b` と `app/controllers/feature_b` を `gems/feature_b` へ移動
3. 各gemが独立してテストできることを確認
4. 本体アプリケーションで両方のgemを読み込む

ここまでの段階では、各gemが「独立している」ように見えます。しかし実際には、以下のような依存が隠れていることが多いのです。

**共通モデルへの暗黙的な依存**

例えば、ユーザー認証を扱う `AuthGem` と、ユーザーのプロフィール管理を扱う `ProfileGem` があるとします。一見すると独立していますが、実際には両方が `User` モデルに依存しています。

```ruby
# gems/auth_gem/lib/auth_gem/user.rb
class User < ApplicationRecord
  has_many :sessions
end

# gems/profile_gem/lib/profile_gem/user.rb
class User < ApplicationRecord
  has_many :profiles
end
```

本体アプリケーション内では、これら両方のgemが同じ `User` テーブルを参照しようとします。結果として、どちらのgemが `User` の定義を「所有」するのか不明確になり、循環依存の温床になります。

**コールバックと暗黙的な処理フロー**

Railsのコールバック機構も落とし穴です。`after_create` や `after_update` で他のgemの処理をトリガーするような設計になると、依存関係が見えにくくなります。

```ruby
# gems/feature_a/lib/feature_a/models/item.rb
class Item < ApplicationRecord
  after_create :notify_feature_b
  
  private
  
  def notify_feature_b
    FeatureB::Notifier.call(self)
  end
end

# gems/feature_b/lib/feature_b/models/notification.rb
class Notification < ApplicationRecord
  after_create :update_item_status
  
  private
  
  def update_item_status
    FeatureA::Item.find(item_id).update(status: 'notified')
  end
end
```

こうなると、`FeatureA` は `FeatureB` に依存し、`FeatureB` は `FeatureA` に依存する循環が完成します。

## 循環依存が見つかりにくい理由

循環依存は、**単体テストの段階では露見しない**ことがほとんどです。

各gemのテストスイートは、本体アプリケーションをロードせずに実行できます。つまり、テスト時には循環依存の両方の側が同時にロードされないため、エラーが起きないのです。

しかし本体アプリケーションを起動した途端に、以下のようなエラーが発生します。

```
NameError: uninitialized constant FeatureA::Item
```

または

```
Gem::LoadError: circular dependency detected
```

問題は、これらのエラーが「どの依存関係が循環しているのか」を明確に示さないことです。スタックトレースは深く、複数のファイルにまたがります。

## 循環依存を検出する実装的アプローチ

### 1. 依存グラフの可視化

最も直接的な方法は、gemの依存関係を明示的に列挙することです。各gemの `gemspec` ファイルで、依存する他のgemを明確に記述します。

```ruby
# gems/feature_a/feature_a.gemspec
Gem::Specification.new do |spec|
  spec.name = 'feature_a'
  spec.version = '0.1.0'
  spec.summary = 'Feature A'
  
  spec.add_dependency 'rails', '>= 6.0'
  # 他のgemへの依存は明示的に書く
  # spec.add_dependency 'feature_b' は書かない
end
```

その後、以下のようなRakeタスクで依存グラフをチェックします。

```ruby
# lib/tasks/gem_dependencies.rake
namespace :gems do
  desc 'Check for circular dependencies'
  task :check_circular do
    require 'tsort'
    
    gems = Dir.glob('gems/*/').map { |p| File.basename(p) }
    dependencies = {}
    
    gems.each do |gem_name|
      gemspec_path = "gems/#{gem_name}/#{gem_name}.gemspec"
      spec = Gem::Specification.load(gemspec_path)
      dependencies[gem_name] = spec.dependencies
        .map(&:name)
        .select { |dep| gems.include?(dep) }
    end
    
    begin
      TSort.tsort(dependencies)
      puts '✓ No circular dependencies detected'
    rescue TSort::Cyclic => e
      puts "✗ Circular dependency detected: #{e.message}"
      exit 1
    end
  end
end
```

このタスクをCI/CDパイプラインに組み込むことで、循環依存の導入を防げます。

### 2. イベント駆動設計への転換

コールバックの連鎖を避けるために、イベント駆動のアーキテクチャに転換することも有効です。

```ruby
# gems/feature_a/lib/feature_a/models/item.rb
class Item < ApplicationRecord
  after_create do
    Rails.application.config.event_bus.publish('item.created', item: self)
  end
end

# 本体アプリケーション側で購読
Rails.application.config.event_bus.subscribe('item.created') do |event|
  FeatureB::Notifier.call(event.item)
end
```

この方式では、`FeatureA` は `FeatureB` の存在を知りません。本体アプリケーションが購読関係を管理するため、循環依存は構造的に起きにくくなります。

### 3. 共有モデルの分離

複数のgemが同じモデルに依存する場合、その共有部分を独立した小さなgemに分離することが有効です。

```ruby
# gems/shared_models/lib/shared_models/user.rb
module SharedModels
  class User < ApplicationRecord
    self.table_name = 'users'
  end
end

# gems/feature_a/feature_a.gemspec
spec.add_dependency 'shared_models'

# gems/feature_b/feature_b.gemspec
spec.add_dependency 'shared_models'
```

このように共有部分を明示的に分離することで、依存関係が一方向になり、循環を避けられます。

## 実装時の現実的な判断

小規模チームでは、すべてのgemを完全に独立させることは難しいかもしれません。その場合は、以下の優先順位で進めることをお勧めします。

1. **必ず分離する**: ビジネスロジックが全く異なる機能（例：決済と在庫管理）
2. **段階的に分離する**: 共通部分が多い機能は、まず共有gem化してから個別化
3. **無理に分離しない**: 密結合が避けられない場合は、本体アプリケーションの一部として残す

循環依存の検出を自動化することで、後から問題が大きくなることを防げます。本体アプリケーションのスタートアップ時に依存グラフをチェックするテストを追加することも、シンプルで有効な対策です。

抽出は一度きりではなく、段階的に進めるものです。焦らず、検出可能な状態を保ちながら進めることが、後々の保守性につながります。
