## はじめに：制約の強さと拡張性のジレンマ

型安全性が強い言語を採用すると、初期設計で「この型はこの振る舞いを必ず満たす」という約束を型システムに組み込めます。これは本来、システムの堅牢性を高める強みです。ところが現場では、この約束が強いほど、後から「実はこの既存の型に新しい振る舞いを足したい」という要望に応えられなくなる局面に直面します。

型クラスの仕組みを持つ言語（代表的にはHaskellなど）で案件を進めると、この矛盾がはっきり見えます。今回は、なぜこういう状況が起きるのか、そして現場でどう判断すべきかを検証していきます。

## 型クラス制約が「閉じた世界」を作る仕組み

型クラスの基本的な役割を整理しましょう。型クラスは「ある型がある振る舞いを満たすことを保証する」ための仕組みです。

たとえば、「数値のように足し算ができる」という性質を `Addable` という型クラスで定義したとします。すると、その型クラスのインスタンスになった型は、必ず足し算の実装を持つことが保証されます。

```haskell
class Addable a where
  add :: a -> a -> a

instance Addable Int where
  add x y = x + y

instance Addable String where
  add x y = x ++ y
```

ここまでは柔軟に見えます。新しい型を作って、そこに `Addable` インスタンスを足せば済みます。

ところが、実務では別の問題が起きます。

## 「既存の型」に「新しい型クラス」を後付けしたいケース

現場で起きやすいのは、こういう場面です。

開発の初期段階で、ユーザー情報を表す `User` という型を定義しました。当初は「ユーザーの基本情報を保持する」という役割だけでした。

```haskell
data User = User
  { userId :: Int
  , userName :: String
  , email :: String
  }
```

数ヶ月後、営業から「ユーザーの権限レベルに応じた表示制御が必要」という要件が入ります。そこで、権限を判定する型クラス `Authorizable` を新たに設計します。

```haskell
class Authorizable a where
  canAccess :: a -> String -> Bool
```

ここで問題が生じます。既に本番で動いている `User` 型に対して、新しく `Authorizable` インスタンスを追加したいのですが、**既存の型定義と実装が「閉じた」状態になっている場合、後付けが難しい** という状況に陥ります。

これは言語の仕様というより、設計の段階で「この型が将来どんな振る舞いを持つか」を決め切ってしまったことが原因です。

## 制約が強いほど「拡張ポイント」が限定される理由

型クラスの制約が厳しいシステムほど、この問題が顕著になります。理由は3つあります。

### 1. 型シグネチャが「その時点での全責任」を宣言している

関数が受け取る型に厳しい制約をかけると、その関数は「この型クラスのインスタンスなら何でもいい」という前提で動きます。

```haskell
processUser :: (Addable a, Authorizable a) => a -> String
```

こうなると、後から「実は `Authorizable` だけでいい」「いや、別の型クラス `Auditable` も必要」という変更は、この関数のシグネチャ全体に波及します。

### 2. 既存インスタンスの「実装の固さ」

一度 `User` に対して `Authorizable` インスタンスを実装してしまうと、その実装は「その時点での理解」に基づいています。

後から「権限判定のロジックを変えたい」「複数の権限体系に対応させたい」という要件が入ると、既存の実装を修正するか、別の型を作るかの二者択一になります。

### 3. 型クラスの「単一責任」が曖昧になりやすい

初期段階では明確だった「型クラスが何を保証するか」が、後付け要件で曖昧になります。

たとえば、`Authorizable` が「権限判定」を意味していたのに、後から「監査ログ記録も必要」という要件が入ると、同じ型クラスに責任が増えていきます。すると、既存の実装との矛盾が生じやすくなります。

## 実務での検証：段階的な拡張を想定した設計

実際の案件では、こういう状況を避けるために、いくつかの設計パターンを試します。

### パターン1：「最小限の型クラス」に絞る

最初から「あらゆる振る舞いを型クラスで保証する」のではなく、本当に必須な部分だけを型クラスにします。

```haskell
class HasId a where
  getId :: a -> Int

data User = User
  { userId :: Int
  , userName :: String
  , email :: String
  }

instance HasId User where
  getId = userId

-- 権限判定は型クラスではなく、単純な関数として定義
canUserAccess :: User -> String -> Bool
canUserAccess user resource = userId user > 0  -- 簡略例
```

このやり方なら、後から権限判定ロジックを変えても、`User` 型や `HasId` インスタンスには影響しません。

### パターン2：「振る舞いを型に持たせる」設計

型クラスではなく、振る舞いそのものをデータとして持たせる方法です。

```haskell
data User = User
  { userId :: Int
  , userName :: String
  , email :: String
  , permissions :: [String]  -- 権限を直接保持
  }

canAccess :: User -> String -> Bool
canAccess user resource = resource `elem` permissions user
```

この方法なら、権限の追加・変更は `User` の構築時に柔軟に対応できます。

### パターン3：「新しい型でラップする」

既存の `User` 型には手を入れず、権限情報を持つ新しい型を作ります。

```haskell
data AuthorizedUser = AuthorizedUser
  { baseUser :: User
  , role :: String
  , permissions :: [String]
  }

class Authorizable a where
  canAccess :: a -> String -> Bool

instance Authorizable AuthorizedUser where
  canAccess au resource = resource `elem` permissions au
```

既存の `User` 型への影響がゼロなので、後付けが容易です。ただし、データ変換のコストが増えます。

## 現場での判断ポイント

型クラス設計で後付け拡張を見据えるなら、以下を確認しましょう。

**初期設計段階で問う**
- この型クラスは、本当に「型安全性が必須」な部分か、それとも「実装の共通化」の都合か
- 6ヶ月後、新しい振る舞いが追加される可能性は何%か
- その時に「既存の型に足す」と「新しい型を作る」のどちらが現実的か

**実装段階で確認する**
- 型クラスのインスタンスは「その型の本質的な性質」か、「その時点での要件」か
- 複数の型に同じインスタンスを実装するなら、その実装は本当に「共通の振る舞い」か

**運用段階で対応する**
- 後付け要件が入ったとき、既存インスタンスの修正が他の部分に波及していないか
- 新しい型クラスを足す場合、既存の型との組み合わせで矛盾が生じていないか

## まとめ：制約と柔軟性のバランス

型クラスの制約が強いほど、初期段階では「安全で明確な設計」が得られます。しかし、その代わりに「後付けの拡張性」を失いやすくなります。

現場では、この矛盾に直面することが多いです。重要なのは、**設計時に「制約の強さ」と「将来の変更可能性」のバランスを明確に判断する** ことです。

型クラスを使うなら、「最小限の制約」「段階的な拡張を見据えた構成」「新しい型でのラップ戦略」など、複数の選択肢を現場の状況に応じて使い分けることが、実務では重要になります。
