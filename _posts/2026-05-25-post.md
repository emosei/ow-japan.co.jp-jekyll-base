## はじめに──認証基盤の更新は「一度に切り替える」では終わらない

セキュリティ対策として、パスワードハッシュアルゴリズムを新しいものに切り替える判断は珍しくありません。MD5やSHA-1から bcrypt や Argon2 へ移行する、あるいは既存の bcrypt から Argon2 へ段階的に進める。これは正当な対応です。

ただ、実際の本番環境では「新しいアルゴリズムへの切り替え」を発表した後でも、古いハッシュが大量に残り続ける事態が起きます。なぜか。それは、ユーザーがパスワードを変更するまで、古いアルゴリズムで作られたハッシュが有効なままだからです。

現場では、この現実を見落とすと「もう古いアルゴリズムは使っていない」という認識と、実際に保存されているハッシュの状態がズレたまま運用が続きます。結果として、セキュリティ監査やコンプライアンス対応で「なぜまだ脆弱なハッシュが残っているのか」という指摘を受けることになります。

## 検証の前提──本番で何が起きているのか

まず、状況を整理しましょう。

**シナリオ:**
- 既存システムでは MD5 でパスワードをハッシュ化している
- セキュリティ改善として、新規登録と変更時は bcrypt に切り替える決定をした
- ログイン時には「MD5 または bcrypt」の両方に対応する検証ロジックを実装した

**ここで起きる問題:**

```
ユーザーA: 過去に登録 → MD5ハッシュのまま保存
ユーザーB: 過去に登録 → MD5ハッシュのまま保存
ユーザーC: 新規登録 → bcryptで保存
ユーザーD: パスワード変更 → bcryptで保存

→ ユーザーA、Bは次にパスワード変更するまで MD5ハッシュが残る
```

3ヶ月後、アクティブユーザーの 60% が MD5ハッシュのままという状況が生まれます。

## 段階的な再ハッシュがないと何が困るのか

### 1. セキュリティ監査で説明がつかない

外部監査やペネトレーションテストで「なぜ脆弱なハッシュアルゴリズムが本番に存在するのか」と指摘されます。

「新しいアルゴリズムに切り替えた」という説明は、技術者にとっては「新規登録と変更時は」という但し書きを前提としていますが、監査側には「全体的に古いハッシュが大量に存在する」という事実が見えます。ここのギャップが埋まりません。

### 2. セキュリティインシデント時のリスク評価が甘くなる

パスワードデータベースが漏洩した場合、MD5ハッシュは現代の GPU クラッキングで数時間で復号される可能性があります。

「古いハッシュは 20% だけ」という見積もりが実は「60% 残っていた」では、インシデント対応の初期判断が大きく変わります。

### 3. 運用コストが増える

ログイン検証ロジックが「MD5 または bcrypt」の両方に対応したままになります。

- 複数のハッシュアルゴリズムを同時にサポート
- レガシーコード（MD5検証部分）が削除できない
- テストケースも増える

数年経つと「なぜこんなコードが残ってるの？」という状態になり、新しいメンバーが混乱します。

## 検証：段階的な再ハッシュの仕組み

実際に、ログイン時に古いハッシュを自動的に新しいアルゴリズムで再ハッシュ化する方法を検証します。

### 検証の構成

- **旧ハッシュ:** MD5（脆弱、検証用）
- **新ハッシュ:** bcrypt
- **ハッシュ形式の識別:** プレフィックスで判定（`$2a$` → bcrypt、それ以外 → MD5）
- **再ハッシュのタイミング:** ログイン成功時

### 実装例

```python
import hashlib
import bcrypt

# ユーザーテーブルの password_hash カラムに格納されているハッシュ

def verify_and_rehash(plain_password, stored_hash):
    """
    ログイン時の検証と段階的な再ハッシュ
    返り値: (認証成功フラグ, 新ハッシュが必要かフラグ, 新ハッシュ)
    """
    
    # ハッシュ形式を判定
    if stored_hash.startswith('$2a$') or stored_hash.startswith('$2b$'):
        # bcrypt
        is_valid = bcrypt.checkpw(
            plain_password.encode('utf-8'),
            stored_hash.encode('utf-8')
        )
        return (is_valid, False, None)
    else:
        # MD5（レガシー）
        md5_hash = hashlib.md5(plain_password.encode('utf-8')).hexdigest()
        is_valid = (md5_hash == stored_hash)
        
        if is_valid:
            # 認証成功 → 新しいハッシュを生成
            new_hash = bcrypt.hashpw(
                plain_password.encode('utf-8'),
                bcrypt.gensalt(rounds=12)
            ).decode('utf-8')
            return (True, True, new_hash)
        else:
            return (False, False, None)

# ログイン処理での使用例
def login_user(username, plain_password):
    user = db.query(User).filter_by(username=username).first()
    
    if not user:
        return {"success": False, "message": "ユーザーが見つかりません"}
    
    is_valid, needs_rehash, new_hash = verify_and_rehash(
        plain_password,
        user.password_hash
    )
    
    if not is_valid:
        return {"success": False, "message": "パスワードが違います"}
    
    # ここから認証成功時の処理
    if needs_rehash:
        # バックグラウンドタスクで更新する、または同期更新する
        user.password_hash = new_hash
        db.commit()
        # ログに記録（監視用）
        logger.info(f"Password rehashed for user: {username}")
    
    return {"success": True, "user_id": user.id}
```

### 検証結果

**利点:**

1. **段階的な移行が実現する**
   - ユーザーがログインするたびに古いハッシュが新しいものに置き換わる
   - 強制的な一括変更ではなく、自然な形で進む

2. **ユーザーへの負担がない**
   - パスワード変更を強要しない
   - ログイン体験は変わらない

3. **セキュリティ監査対応が可能**
   - 「90日以内に 95% が新ハッシュに移行」という進捗を示せる
   - 具体的な移行スケジュールを提示できる

**制約と注意点:**

1. **全ユーザーがすぐに移行するわけではない**
   - 非アクティブユーザーは古いハッシュのままになる可能性
   - 対策：定期的に古いハッシュの残存数を監視

2. **再ハッシュ時のパフォーマンス**
   - bcrypt は意図的に遅い（CPU 負荷を高くする）
   - ログイン時に bcrypt 生成を同期実行すると、ログイン遅延が発生する可能性
   - 対策：バックグラウンドキューで非同期処理、または最初のレスポンスの後に実行

3. **エラーハンドリング**
   - 再ハッシュに失敗した場合、ユーザーはログインできるが古いハッシュのままになる
   - 対策：再ハッシュ失敗をログに記録し、管理画面で監視

## 実務投入時の追加確認ポイント

### 1. 移行期間中のハッシュ形式の混在対応

複数のハッシュアルゴリズムをサポートする期間は、テストが複雑になります。

- MD5 のみでログイン成功するケース
- bcrypt のみでログイン成功するケース
- 両方で検証するケース（移行中のみ）
- 古いハッシュで失敗、新しいハッシュで成功するケース

これらをテストカバレッジに含めることが重要です。

### 2. 古いハッシュの残存状況を可視化する

```sql
-- 定期的に実行するクエリ例
SELECT 
  CASE 
    WHEN password_hash LIKE '$2a$%' OR password_hash LIKE '$2b$%' THEN 'bcrypt'
    ELSE 'md5'
  END as hash_type,
  COUNT(*) as count,
  ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM users), 2) as percentage
FROM users
GROUP BY hash_type;
```

このクエリを定期実行し、古いハッシュの割合を監視します。

### 3. 完全移行後のクリーンアップ

古いハッシュが十分に少なくなったら（例：1% 以下）、レガシーコードを削除します。

- MD5 検証ロジックの削除
- テストケースの削除
- ドキュメントの更新

この判断を曖昧にしておくと、「いつ削除してもいいのか」という疑問が残ります。事前に「古いハッシュが 1% 以下になったら、3ヶ月後に削除」といった基準を決めておくことをお勧めします。

## どんな案件で採用しやすいか

この段階的な再ハッシュアプローチは、以下の条件で特に有効です：

- **既存ユーザーベースが大きい場合**
  - 数万人以上のアクティブユーザーがいる
  - 全員にパスワード再設定を強要できない

- **認証基盤が複数のシステムで共有されている場合**
  - 一度の変更が複数のサービスに影響する
  - 段階的な
